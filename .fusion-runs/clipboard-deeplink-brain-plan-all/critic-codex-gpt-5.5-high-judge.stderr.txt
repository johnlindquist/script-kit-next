Raw leaks can still happen through non-obvious labels: `render_builtins/clipboard.rs`, `ai/context_selector/mod.rs`, Agent Chat portal labels, Day Page context serialization, and pinned clipboard brain indexing.

Links can decay if `brain_kept` rows are pruned, cleared as “unpinned,” manually deleted, or blob files are cleaned up.

Search quality will drop if brain FTS no longer indexes clipboard content. That is the privacy tradeoff. The replacement should be clipboard-history search as a separate runtime provider, not duplicated raw brain docs.

Legacy fragments are a major risk: old clipboard-origin fragments contain raw bodies. Do not create new ones; handle old ones as compatibility only.

codex
Several consensus claims survive, but only with narrower wording.

The canonical URI claim is well supported: `kit://clipboard-history?id=<id>` already parses and resolves, while `scriptkit://clipboard/<id>` is provenance-shaped. The dangerous overreach is treating that as enough for durability. Current source has more than one deletion path: `prune_old_entries`, db-worker `prune_impl`, `clear_unpinned_history`, `clear_history`/`clear_impl`, explicit `remove_entry`, and oversized text trimming. A plan that only changes prune SQL will still ship dead links.

The “stable IDs” claim also needs qualification. `add_entry` reuses an ID only while the row still exists via content hash lookup; after deletion, identical content gets a new UUID. So retention must protect the referenced row identity, not just trust dedupe.

The strict “no raw values” requirement is not satisfied by forward-only markdown changes unless the final plan explicitly scopes that to new writes. Existing day pages/fragments and `brain_docs` can still contain raw clipboard content. The brain indexer currently promotes pinned clipboard content into `brain_docs`, so synthesis must force a product decision: either “no raw values in brain markdown only” or also stop raw clipboard duplication in brain search storage.

Day Page behavior is under-specified. A new `ClipboardRef` line needs undo/reject handling, and `format_source_label` needs a `kit://clipboard-history` branch or it can render the literal URI as a provenance label. Image entries also need an explicit exclusion or image-aware resolver; the existing single-entry MCP resolver returns `text/plain` content, which is not enough for blob-backed images.
