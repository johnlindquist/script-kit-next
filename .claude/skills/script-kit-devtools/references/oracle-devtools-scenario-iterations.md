On my machine it sometimes overlaps the parent item, so I click the wrong action.” why this scenario is hard in a customizable scripted app: The menu anchor is not a fixed component. A user script can create actions with different labels, icons, keycaps, disabled rows, separators, and nested routes. The submenu position depends on runtime row height, current popup scale, monitor bounds, and whether the parent menu resized after async action metadata arrived. user-path reproduction idea: Open a script popup that exposes an Actions button. Open the actions dialog, hover or keyboard-focus a parent action with children, then open the nested popup route. Repeat after the script dynamically updates one sibling label to a longer value, causing the parent menu width to change. required DevTools API surfaces and fields: devtools.inspect needs target identity for the popup host, active actions dialog id, active popup menu route stack, parent action id, child popup route id, anchor element id, anchor rect, parent menu rect, child menu rect, viewport rect, device scale factor, layout generation id, route generation id, focused item id, hovered item id, item role, item enabled state, item label, item keycap, and warnings for stale anchor or changed layout generation. what devtools.inspect should prove first: It should prove that the child popup is anchored to the intended parent action row, not merely to the parent menu container or last hovered item.
