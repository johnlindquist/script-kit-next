# Markdown Files As Memory Substrate

The memory layer's canonical storage is markdown files on disk — the QMD model (github.com/tobi/qmd): one day page per day, fragment files for long captures (with provenance frontmatter, referenced from the day page by excerpt + link), and notes as files. SQLite (FTS, embeddings, attention signals, backlink graph) is a derived, rebuildable index over those files, never the source of truth. The brain finds; the filesystem owns.

We chose this over the current inversion — `notes.sqlite` rows and `todos.jsonl` as canonical with markdown only for links/snippets — because the product's ownership promise ("inspect, edit, version, and migrate the artifacts you run") must extend to memory: day pages should be readable in any editor, greppable, syncable, and survivable if the app is deleted. We chose excerpt-plus-fragment over inlining long captures because a day page that embeds a 3,000-word paste stops being a human-readable diary. We chose day-keyed files over per-note files as the primary structure because humans recall in time blocks ("what happened yesterday"), and time-keyed files make that question a file read.

Constraints: the existing brain retrieval recipe (`src/brain/search.rs`, hybrid BM25 + vectors + RRF — already the qmd recipe natively) is unchanged; only the substrate under it moves. The app has not shipped, so there is no migration path or backwards compatibility: existing `notes.sqlite` content is abandoned and `todos.jsonl` is removed outright — `;todo` appends a task-list line to today's day page instead. Capture gestures (`;note`, `;todo`, `;link`, `;snippet`) keep their distinct two-second semantics; only where the bytes land unifies. The cutover plan lives in `docs/specs/qmd-migration.md`.
