# Automation

Automation relies on stdin JSON control, deterministic targeting, and proof-bearing capture.

The repo splits that work between protocol-level transactions and Bun-based runtime helpers instead of fixed sleeps or CLI launch shortcuts.

## Key Facts

These points capture the stable automation behavior that runtime tooling depends on.

- The autonomous harness uses stdin JSON messages to drive the app. Command-line script args are explicitly not the contract.
- Protocol-level `waitFor` and `batch` commands are backed by a transaction executor that records receipts, observations, and actionable failure suggestions.
- The repo's `scripts/agentic/` tools choose between protocol-level batch input, GPUI event simulation, and native input instead of assuming one delivery method.
- Screenshot capture is resolver-driven. Runtime capture paths rank candidate OS windows against automation targets and fail closed on ambiguity instead of silently picking the wrong window.
- `SCRIPT_KIT_AI_LOG=1` is the compact log format used throughout the agentic tooling and test harness.
- ScriptList `getState` and `getElements` project the grouped visible rows, so root file results, fallback rows, and other grouped-only entries are observable through state-first proof.

## Operational Rules

These rules describe the behavior constraints that automation changes should preserve.

- Use stdin JSON protocol to start or drive runtime flows.
- Reuse healthy sessions before cold-starting a new app process when the proof does not require a clean launch.
- Prefer `waitFor` or `batch` receipts over guessed sleeps when the surface supports them.
- Prefer state-first proofs (`getElements`, `getState`, targeted `batch`) for non-visual checks, and escalate to screenshots or native focus only when state cannot prove the behavior.
- Keep automation target identity stable across a proof run.
- Treat focus-stealing native input as a last resort, especially for attached popups that expose state snapshots but may not expose an independent GPUI key handle.
- For screenshot-library work, use `surface-navigate --capture --strict-window` before ad hoc screenshots.
- Treat screenshot ambiguity as a hard failure, not a best-effort capture.
- Treat blank or black screenshots as infrastructure failures, not visual proof.
- Do not use SDK `keyboard.*` or `mouse.*` helpers as automation proof. They reject with `UnsupportedSdkFeatureError`; use `batch.setInput` / `batch.forceSubmit` for direct prompt mutation or submit behavior, and use `simulateKey` only when testing key routing with a follow-up `getState`, `getElements`, or `waitFor` receipt.

## Filterable Surface Matrix

The matrix makes migrated filterable launcher surfaces repeatable state-first checks.

`scripts/agentic/filterable-surface-matrix.ts` declares each migrated surface's entry command, `promptType`, `SurfaceKind`, `getElements` list semantic id, and narrowing filter text. The runner starts a session, enters the real surface, samples empty and filtered `getState`/`getElements` receipts, and asserts `stateResult.surfaceContract.surfaceKind` plus `automationSemanticSurface` match the matrix, `visibleChoiceCount` equals the list item count, and `totalCount` contains at least the visible rows plus required input/list chrome. Attached actions popups can be proven through `stateResult.activePopupContract`. AURP-06 starts with `current-app-commands-visible-rows`, the surface migrated in AURP-05.

AURP-11 expands the matrix with `clipboard-history-visible-rows` and `emoji-picker-visible-rows`, then resets each case's filter after entry so multi-case runs cannot inherit prior filter state. AURP-16 adds `app-launcher-visible-rows`, `design-gallery-visible-rows`, and `process-manager-visible-rows` because their entry commands and state/element receipts are stable without browser or window-manager setup. [[tests/filterable_surface_agentic_matrix_contract.rs#matrix_declares_expanded_filterable_surface_cases]] and [[tests/filterable_surface_agentic_matrix_contract.rs#matrix_declares_stable_sibling_filterable_surface_cases]] pin these cases.

AURP-18 starts normalizing the stable sibling surfaces behind named row-projection helpers. App Launcher now routes filtered entries, visible row labels, `getState` counts, `getElements` rows, wheel reanchor counts, and Tab AI target capture through `app_launcher_*` helpers pinned by [[tests/app_launcher_visible_rows_contract.rs#app_launcher_declares_visible_row_helper_family]] and [[tests/app_launcher_visible_rows_contract.rs#app_launcher_tab_ai_targets_use_visible_row_helpers]].

Process Manager follows the same AURP-18 owner shape through `process_manager_*` helpers, including PID-aware filtering shared by render, keyboard, `getState`, `getElements`, and Tab AI target capture. [[tests/process_manager_visible_rows_contract.rs#process_manager_declares_visible_row_helper_family]] and [[tests/process_manager_visible_rows_contract.rs#process_manager_tab_ai_targets_use_visible_row_helpers]] pin the contract.

## Surface Navigator

`surface-navigate` is the warm-session, state-first entrypoint for known app surfaces and screenshot-library capture.

The navigator defaults to `--group filterable-main`, reuses `FILTERABLE_SURFACE_MATRIX`, enters each surface through its real runtime command, reveals and settles the app window, waits for the expected `promptType`, promotes the window to an exact target, samples `getState` and `getElements`, and optionally performs safe non-submitting `batch` interactions before capture. Its default output root is `.notes/image-library/`.

Screenshot captures route through `verify-shot --strict-window` with an exact automation target, so ambiguous, wrong-window, blank, or black PNGs are infrastructure failures rather than visual proof. The first supported group is the stable main-hosted filterable surfaces: Current App Commands, Clipboard History, Emoji Picker, App Launcher, Design Gallery, and Process Manager. [[tests/agentic_surface_navigator_contract.rs#surface_navigator_captures_strict_image_library_outputs]] and [[tests/verify_shot_strict_window_contract.rs#strict_window_requires_exact_target_or_window_id]] pin the contract.

Image-library capture writes one sidecar receipt per PNG plus `.notes/image-library/manifest.json`. Each manifest entry records the promoted exact automation target, native `osWindowId`, final pre-capture state/elements observation, strict capture identity, and PNG content audit. Navigator screenshots must be gated by a final `getState`/`getElements` snapshot after any safe interaction and immediately before capture. Current matrix surfaces declare filter-only safe interaction because semantic row selection is not yet supported for these main filterable prompt variants; any future selection attempt must fail closed on a failed batch receipt. [[tests/agentic_image_library_manifest_contract.rs#navigator_writes_sidecar_receipts_and_manifest]] pins the durable proof format.

The `--group attached-popup` path starts with `actions-dialog-attached-popup` from `ATTACHED_POPUP_SURFACE_MATRIX`. It opens the host window, uses the stdin Cmd+K `simulateKey` path to open Actions Dialog, promotes `{kind:"actionsDialog",index:0}` to the exact `actions-dialog` id, samples `getElements`, and captures the parent native window through `verify-shot --strict-window`. `verify-shot` skips the pre-capture `show` command for exact attached-popup captures so it does not dismiss or obscure the popup, and missing `targetBoundsInScreenshot` marks the receipt as an infrastructure error. Manifest entries for attached popups carry `surfaceClass`, `windowKind`, `preCaptureInspection`, `preCaptureElements`, `popupCapture.strategy`, and `popupCapture.targetBounds`. The active matrix now includes the ACP Chat slash Prompt Popup after repeated fresh runtime proof with `parent_capture_with_crop`, and cases may declare an expected automation id so kind/index promotion cannot accept the wrong popup. [[tests/agentic_attached_popup_surface_navigator_contract.rs#attached_popup_matrix_declares_actions_dialog_active_cases]], [[tests/agentic_attached_popup_surface_navigator_contract.rs#attached_popup_matrix_declares_prompt_popup_active_case]], [[tests/agentic_attached_popup_surface_navigator_contract.rs#attached_popup_promotion_can_require_exact_automation_id]], and [[tests/verify_shot_strict_window_contract.rs#strict_window_skips_show_for_attached_popup_exact_capture]] pin this attached-popup slice.

The `--group attached-popup` matrix supports host fixtures for Actions Dialog and Prompt Popup screenshots. Hosted Actions Dialog cases enter a stable `filterable-main` matrix surface through its real `triggerBuiltin` command, wait for the host `promptType`, sample host `getState`/`getElements`, then open Actions Dialog through stdin Cmd+K and capture the popup with the same exact-target `parent_capture_with_crop` proof. Hosted Actions Dialog cases currently cover Clipboard History, Emoji Picker, and App Launcher. Current App Commands, Design Gallery, and Process Manager are stable `filterable-main` surfaces, but they are not attached-popup hosted Actions Dialog cases while `live_actions_host_for_view` excludes generic built-in list views. Promoting any of them requires a product behavior change: the view must provide live selection-specific actions, then prove stdin Cmd+K opens the shared Actions Dialog before the matrix case can become active. The active Prompt Popup case opens Agent Chat with `triggerBuiltin tab-ai`, uses protocol `setAcpInput "/"`, requires `expectedAutomationWindowId: "acp-mention-popup"` while promoting `{kind:"promptPopup",index:0}`, and captures through `parent_capture_with_crop` under `--case all`. [[tests/agentic_prompt_popup_fixture_contract.rs#prompt_popup_fixture_is_active_attached_popup_case]] and [[tests/agentic_prompt_popup_fixture_contract.rs#prompt_popup_fixture_uses_protocol_acp_input_not_native_input]] pin the promoted boundary.

Multi-surface screenshot-library sweeps use `--fresh-per-case` to isolate each captured view in its own short-lived session. Warm state receipts are still the default for non-visual proof, but visual sweeps must avoid stale rendered frames when switching between main-hosted prompt variants. The manifest records whether fresh per-case isolation was used. [[tests/agentic_surface_navigator_contract.rs#surface_navigator_can_isolate_screenshot_cases]] covers this guard.

The `--group all-active` path runs every active `filterable-main` and `attached-popup` matrix case into one combined image-library manifest. It does not include candidates, each entry records its `sourceGroup`, and dispatch routes by that per-case group so attached-popup crop proof and Prompt Popup exact-id proof stay intact. Fresh visual sweeps include the source group in each temporary session name to avoid collisions as more families become active. [[tests/agentic_surface_navigator_all_active_contract.rs#all_active_group_combines_active_matrices]] and [[tests/agentic_surface_navigator_all_active_contract.rs#all_active_dispatch_routes_by_source_group]] pin the combined sweep.

## Batch target capabilities

Targeted automation batches declare command support in one capability owner before runner extraction.

`prompt_handler` maps each resolved batch target to `AutomationBatchTargetKind`, exposes allow-lists through `BatchTargetCapabilities`, and builds unsupported-command errors with `unsupported_batch_command_error`. `is_acp_wait_condition` is the shared classifier for ACP-specific waits so targeted `waitFor` does not fork the setup/live ACP condition list.

## Agentic session directory canonicalization

Agentic dev sessions canonicalize `SCRIPT_KIT_SESSION_DIR` so `/tmp` and `/private/tmp` resolve to the same session tree.

`scripts/agentic/session.sh` preserves the caller-provided or default session directory as `SESSION_DIR_RAW`, creates it, then stores `pwd -P` in `SESSION_DIR` before computing per-session paths. `dev.sh` exports the same physical path before launching `cargo watch`, so `dev-relaunch.sh` and later `session.sh status dev-watch` calls read and write one canonical tree. Pinned by `tests/agentic_session_dir_contract.rs`.

## Codex Stop Hook Continuation

The repo-local Codex Stop hook can keep one scoped continuation run alive while an active marker exists.

`.codex/hooks/stop-continue-agentic-testing.ts` handles Codex `Stop` events from `.codex/hooks.json`. It selects the newest enabled `.codex/hooks/stop-runs/*.active.json`, locks continuation to the first `session_id` that trips the hook, and writes only ignored state under `.codex/hooks/stop-runs/`.

Agentic-testing runs generate one bounded improvement pass with `$agentic-testing` proof when UI/protocol/behavior surfaces are touched. Marketing-infographic runs generate one `$imagegen` pass from `.codex/hooks/marketing-infographics.md`, using Script Kit brand assets, theme docs, and a rotating style direction. Both modes require `lat check` before completion. Codex can keep the same `turn_id` across Stop continuations, so the hook records `lastTurnId` for diagnostics but must not treat a repeated `turn_id` as a duplicate event. Each selected run appends ignored evidence to `.codex/hooks/stop-runs/*.events.jsonl`. A matching `.pause` file or an assistant final message containing `[stop-hook:pause]` alone on a line disables further continuation for the selected run.

## Surface-proof CLI

`surface-proof` is the default seconds-first proof entrypoint for already-open product surfaces. It promotes one exact automation target and returns a machine-readable proof bundle.

`surface-proof` records state-first evidence in `proofBundle.usage`, `proofBundle.capabilities`, `proofBundle.state`, `proofBundle.elements`, and `proofBundle.targetIdentity`, so agents can assert that ordinary surface proof used `getState`/`getElements` without screenshot capture, native input, `show`, or fixed sleeps.

### Surface classes

`main` proves the launcher without forcing `show`. Attached popups use popup semantics plus visibility waits, and `acpDetached` uses an exact-id replayable scenario.

`actionsDialog` and `promptPopup` follow the attached-popup path. `main` stays state-first through `getState`, `inspectAutomationWindow`, and `getElements`; attached popups, detached ACP, and Notes should use `getElements(target)` plus `inspectAutomationWindow(target)`, with `getAcpState(target)` for ACP state, instead of relying on targeted `getState` for secondary-window state.

Menu-syntax trigger popups are `promptPopup` windows with a specialized element cache. [[src/windows/automation_surface_collector.rs#upsert_menu_syntax_prompt_popup_snapshot]] records the live `TriggerPickerSnapshot` rows for targeted `getElements`, so agentic proofs can assert popup tokens such as `has:shortcut` and `;note` from semantic receipts rather than pixels.

ScriptList `getState` includes `filterInputDecorations`, a receipt read from the live GPUI input state rather than recomputing tokenizer spans. It reports the rendered input text plus chip ranges and roles, so state-first proofs can catch stale decoration bugs such as `f:` surviving after the input becomes `~/` or `/tmp`.

### DropPrompt Native Drop Proof

DropPrompt native drop proof should exercise the OS/GPUI file-drop path and inspect redacted receipts before submitting.

The empty-state receipt is `getState.promptType:"drop"` with `stateResult.drop.fileCount:0`, no dropped-file rows from `getElements`, and `activeFooter` showing Submit disabled with `actionDisabled:"no_files"`. After a native file drop onto `window:drop`, `getState.drop.files[]` and `getElements` file rows should expose only `index`, basename `name`, and byte `size`; paths are reserved for the final SDK submit payload. Enter or footer Submit should then resolve the script with the full `FileInfo[]`, while Escape should still cancel through the null path. [[tests/drop_prompt_native_drop_contract.rs#drop_prompt_wires_gpui_external_paths_to_prompt_state]] pins the source-side native hook.

### Non-obvious constraints

Attached popups must preserve parent identity and `parent_capture_with_crop`. Detached surfaces must preserve exact-id stability. Main should stay state-first unless a visual assertion explicitly requires capture.

## Root Search Frame Stability Proof

Root unified search has a dedicated state-first proof for target stability after late provider work.

`scripts/agentic/root-search-frame-stability.ts` drives the main launcher through `session.sh` with a deterministic delayed root-file provider fixture, sets a root query, waits for the ScriptList input state, and samples intermediate `getState.mainWindowPreflight` plus `getElements` receipts until the matching global provider settles. It requires observed provider loading and a warmed cache result count, and fails on any same-query change to selected key, selected role, Enter action, visible row fingerprint, visible results, visible count, or element row fingerprint.

`scripts/agentic/root-search-visual-stability.ts` is the visual escalation path for the same contract. It spawns the debug binary directly, shows the real launcher, captures timestamped window screenshots while `macos-input.ts` sends native keystrokes, records app logs and state receipts, and writes a contact sheet plus receipt under `.test-output/`. It fails if any same-input selected key, role, visible-result fingerprint, or visible-row fingerprint changes, and it parses `FILTER_PERF` logs to reject slow grouping or handler samples. Its default fixture mode proves that late provider completion cannot replace a cold visible frame; `--no-fixture --warm-provider --expect-visible-file-results` proves the real file provider can warm, then show actual root-file rows immediately and stably on the replayed query.

`scripts/agentic/root-passive-frame-stability.ts` uses the same state-first path for cache-refreshable passive browser sources. It samples `getState.mainWindowPreflight.rootPassiveFrame`, waits until Browser Tabs and Browser History snapshot status is not refreshing, and fails if the selected key, Enter action, visible-row fingerprint, or passive-frame receipt changes for the same query.

## Screenshot pixel audit

Screenshot proof must include content-level validation before a PNG can count as reviewable evidence.

The Rust capture path in [[src/platform/screenshots_window_open.rs]] runs a macOS Screen Recording preflight before xcap capture, then audits the RGBA buffer before PNG encoding. Captures with no sampled pixels, no non-transparent pixels, solid-like single-bucket frames, or dark-empty low-variety frames fail with `automation.capture_screenshot.permission_failed` or `automation.capture_screenshot.blank_image_rejected` instead of writing misleading proof. The audit records `maxLuma` and `nonBlackRatio` so valid dark UI with sparse visible chrome is distinct from an all-black permission failure.

The same non-prompting Screen Recording preflight is surfaced through `computer/list_permissions`, so MCP clients can diagnose missing capture permission before attempting screenshot proof.

The Bun proof helper in [[scripts/agentic/verify-shot.ts]] independently audits generated PNGs and records `screenshotReceipt.contentAudit`. Its default output root is `.test-screenshots/`, and a blank audit flips the screenshot receipt to an infrastructure failure even when the capture command wrote a file. [[scripts/agentic/verify-shot-blank-rejection-matrix.ts]] generates transparent, solid black, solid white, solid gray, and valid-dark-UI PNG fixtures so the blank/solid/dark-positive split stays deterministic.

## Window metadata

`AutomationWindowInfo.semanticSurface` re-keys as the main window's active subview changes — defaults to `scriptList`, flips to the subview tag on `triggerBuiltin`.

Subview tags currently wired through `triggerBuiltin`: `fileSearch`, `clipboardHistory`, `appLauncher`, `browserTabs`, `emojiPicker`, `windowSwitcher`, `processManager`, `currentAppCommands`, `designGallery`, `settings`, `kitStoreBrowse`, `kitStoreInstalled`, and `acpChat`. Each arm in the stdin dispatchers must have a matching `AppView` variant in `AppView::surface_contract()` — without it the surface stays at the prior value. `acpChat` is the Main-hosted ACP chat surface (flipped via `triggerBuiltin tab-ai` / `tabai`); detached ACP popups use the popup-level surface class documented in [[lat.md/automation#Automation#Surface-proof CLI#Surface classes]] and are not keyed through this map.

The re-key goes through [[src/app_impl/trigger_builtin_dispatch.rs#ScriptListApp#rekey_main_automation_surface_after_trigger_builtin_dispatch]], which delegates to [[src/app_impl/automation_surface.rs#ScriptListApp#rekey_main_automation_surface_from_current_view]] after dispatch mutates `current_view`. The stdin dispatchers call the triggerBuiltin helper after `dispatch_trigger_builtin`, so they do not recompute bounds, focus, title, or raw surface tags they do not own. The `AppView → semanticSurface` mapping lives inside `SurfaceKind::surface_contract()` next to the enum in `src/main_sections/app_view_state.rs`; `semantic_surface_for_main_view()` delegates to that registry.

Current-view-derived main-window re-keying is owned by `rekey_main_automation_surface_from_current_view`. About, parent confirm, embedded ACP entry, embedded ACP return-to-origin, and triggerBuiltin post-dispatch paths call that helper instead of duplicating `semantic_surface_for_main_view(&self.current_view)` plus `update_automation_semantic_surface("main", …)`. Hard resets to ScriptList still use explicit `scriptList` writes where existing hide/detach contracts pin reset ordering.

`sync_main_automation_window` preserves the existing `semanticSurface` on window-bounds / focus-change events — if it silently hardcoded `scriptList`, a resize following a subview transition would clobber the re-key and the automation consumer would see the subview on screen yet `scriptList` in the registry.

The same invariant applies to the actions-popup synthesis path. [[src/actions/window.rs#resolve_actions_popup_parent_automation_id]] upserts a main-window automation record when the actions popup opens without an explicit `parent_automation_id`; because `upsert_automation_window` is a full-replace (see [[src/windows/automation_registry.rs#upsert_automation_window]], which does `state.windows.insert(id, info)`), the upsert must read the registry's existing `semanticSurface` and pass it through, falling back to `scriptList` only when no prior record exists. Previously this site hardcoded `scriptList`, which wiped `clipboardHistory` / `fileSearch` / `acpChat` every time the user pressed Cmd+K on a non-main-menu host. Fix landed Run 7 Pass #21 (commit 69d5846f0); pinned at source level by `tests/actions_popup_parent_preserves_semantic_surface_contract.rs` (Run 9 Pass #27) with 4 assertions: the function `resolve_actions_popup_parent_automation_id` exists at source level, its body calls `list_automation_windows()` followed by `.and_then(|w| w.semantic_surface)` and threads the result through `semantic_surface: Some(preserved_semantic_surface)` in the upsert payload (read-before-upsert ordering), the body forbids any hardcoded `semantic_surface: Some("scriptList".to_string() | .into() | String::from())` literal in the upsert call, and the four-phrase Run 7 Pass #21 anchor-comment block (citing the `actions-cmdk-clipboard-main-surface-flip` anomaly) appears verbatim before the upsert — defending against a refactor that consolidates the `upsert_automation_window` call sites in `src/actions/window.rs` into a shared helper and drops the `preserved_semantic_surface` read, or a "cleanup" that deletes the 8-line comment and invites the next contributor to revert to a hardcoded literal.

Hide has two entry points and BOTH must reset-then-rekey to stay consistent. The non-RPC hide path ([[src/main_sections/window_visibility.rs#hide_main_window_helper]], used by Escape / menu dismissal) calls `view.reset_to_script_list(ctx)` followed by `update_automation_semantic_surface("main", Some("scriptList"))`. The stdin RPC path (`ExternalCommand::Hide`) duplicates that sequence inside each of the three triple-embedded dispatchers — `src/main_entry/runtime_stdin_match_core.rs`, `src/main_entry/runtime_stdin.rs`, and `src/main_entry/app_run_setup.rs` — because the dispatcher body is inlined at each site rather than delegating to the helper. Order matters: `reset_to_script_list` first (flips `AppView::current_view` → `ScriptList`), then the re-key pins the automation surface to the now-true value; the reverse opens a race where a window-bounds observer could clobber the re-key before the view mutated. The RPC/helper parity is CI-gated by `tests/hide_rpc_surface_reset_contract.rs`.

## Actions dialog escape is filter-text-agnostic

Pressing Escape inside the shared actions dialog always pops a child route or closes the dialog — never consumes the keystroke to clear filter text first.

[[src/actions/dialog.rs#ActionsDialog#handle_escape]] decides between `PoppedRoute` (back-navigate to parent route via `self.pop_route(cx)`) and `CloseDialog` (root-level dismissal) without reading `self.search_text`. This is the Raycast / Spotlight UX contract: one Escape dismisses, regardless of whether the user has typed a filter. The opposite convention — macOS native `NSSearchField` inside an alert, where the first Escape clears the field — would force users on every Cmd+K host (main, notes, ACP, clipboard, emoji picker, file search, app launcher) to press Escape twice to close after filtering. Structural anchors pinning this invariant live in `tests/actions_dialog_escape_filter_agnostic_contract.rs`: the body of `handle_escape` MUST contain `self.pop_route(cx)`, MUST NOT contain `self.search_text.clear()` or `self.search_text.is_empty()`, and `ActionsDialogEscapeOutcome` MUST carry exactly the two documented variants — a new `FilterCleared` / `SearchTextCleared` variant is the enum-level shape this regression would take. Falsifier: a commit that adds a third outcome variant, a `search_text.clear()` call inside `handle_escape`, or any `search_text.is_empty()` gate in the escape path flips one of the 5 tests red.

When the native actions popup itself receives Escape or Cmd+K, the launcher interceptor routes those close keys back through `route_key_to_actions_dialog` before applying the secondary-window skip. Embedded ACP can also open actions while the launcher visibility flag is false, so actions close keys are routed before the `is_main_window_visible()` guard. `close_actions_popup` detects actions-window origin, activates the main window, and leaves pending focus for the main render path instead of focusing the popup window being removed.

## Drill-down route stack preserves parent state across push/pop

The shared actions dialog's drill-down route stack ([[src/actions/dialog.rs#ActionsDialog#push_route]] / [[src/actions/dialog.rs#ActionsDialog#pop_route]]) round-trips parent UI state — search text, selection, scroll — so Escape from a child sub-menu lands back at the parent with its filter intact.

The canonical live consumer is the ACP agent-switch sub-menu at [[src/ai/acp/view.rs]] — users open Cmd+K on the ACP chat, pick "Switch agent", drill into a child route listing available agents, and Escape returns them to the parent ACP actions menu with whatever search text they had typed still present. That round-trip is powered by four carefully-ordered calls: (a) `push_route` calls [[src/actions/dialog.rs#ActionsDialog#snapshot_current_route_state]] BEFORE `route_stack.push(...)` so `last_mut()` writes to the parent's state (not the freshly-allocated child's); (b) `push_route` then calls [[src/actions/dialog.rs#ActionsDialog#apply_route_state_from_route]] to clear search_text and load the child's actions; (c) `pop_route` guards on `self.route_stack.len() <= 1` so the root is never popped (without the guard the stack goes empty and the next `push_route` silently corrupts via the no-op `last_mut()` branch in `snapshot_current_route_state`); (d) `pop_route` calls [[src/actions/dialog.rs#ActionsDialog#apply_route_state]] — the STATEFUL variant that assigns `self.search_text = state.search_text.clone()` — NOT the push-variant `apply_route_state_from_route` which clears. Pass #11 (`tests/actions_dialog_escape_filter_agnostic_contract.rs`) pinned the `handle_escape` delegation chain but NOT the `pop_route` body; a "simplify `pop_route` to `self.route_stack.pop(); true`" refactor would flip zero Pass #11 tests red while destroying the parent-state-restore on every drill-down return. Structural anchors pinning this invariant live in `tests/actions_dialog_route_stack_contract.rs`: `push_route` body MUST call `snapshot_current_route_state(` BEFORE `route_stack.push(`, AND `apply_route_state_from_route(`; `pop_route` body MUST contain a depth guard on `self.route_stack.len()` AND call `apply_route_state(` (NOT `apply_route_state_from_route(` — the symmetric negative pin); `apply_route_state` body MUST assign `self.search_text = state.search_text` AND MUST NOT call `self.search_text.clear()` (the symmetric negative pin that catches a "unify both apply_* fns" refactor). Falsifier: any reorder of snapshot vs push inside `push_route`, any removal of the depth guard inside `pop_route`, any swap of `apply_route_state` ↔ `apply_route_state_from_route` in either direction, or any substitution of `self.search_text.clear()` for the restore assignment in `apply_route_state` flips at least one of the 5 tests red.

## Session send parse receipts

`session.sh send SESSION_NAME CMD` is fire-and-forget by default. Pass `--await-parse [--timeout MS]` to upgrade to a receipt-bearing call that returns one of three `parseOutcome` values.

Default return: `{status:"ok",sent:true}` with no wait. Await return: `parsed` (with `commandType`), `parseError` (with truncated `error` text), or `timeout`.

The receipt-correlation protocol records the app.log byte offset BEFORE writing to the input FIFO, then polls the log tail for the next `stdin_command_parsed` or `stdin_parse_failed` event after that offset. stdin is line-serialized inside the app (single listener thread in [[src/stdin_commands/mod.rs#start_stdin_listener]]), so exactly one parse event corresponds to one send — the first event past the recorded offset is the receipt for this send. Ordering is pinned by `tests/session_send_await_parse_contract.rs::cmd_send_records_log_offset_before_fifo_write`.

This closes the silent-drop gap Pass #8 of Run 4's attacker probe exposed: `triggerBuiltin` with an unknown field (e.g. `builtinId` instead of `name`) was silently dropped by the parser, yet `session.sh send` still returned `sent:true`. Without `--await-parse` that gap still exists for fire-and-forget callers; with the flag it surfaces as `parseOutcome:"parseError"` + `error:"unknown variant …"` — the running binary's receipt is now structurally visible to automation without grepping app.log. The three outcome names are stable contract; add new outcomes alongside rather than renaming existing ones.

`--timeout` must be a non-negative integer (regex `^[0-9]+$`). Non-numeric input (e.g. `--timeout abc`) is rejected at flag-parse time with a structured `{"status":"error","error":{"code":"invalid_timeout",...}}` envelope and exit code 1. Validation is required because `set -euo pipefail` combined with `local deadline_ms=$(( now_ms + timeout_ms ))` would otherwise treat the non-numeric value as an unbound variable and kill the function before any JSON envelope is emitted — violating the invariant that `session.sh send` ALWAYS returns a JSON envelope. The `invalid_timeout` error code is stable contract; pinned by `tests/session_send_await_parse_contract.rs::cmd_send_rejects_non_numeric_timeout`.

Arg-order typos are rejected at flag-parse time with a `flag_as_command` error code. The correct invocation is `session.sh send SESSION_NAME JSON [--await-parse [--timeout MS]]`; the mirror-image typo `session.sh send SESSION_NAME --await-parse JSON` (flag BEFORE the JSON payload) pre-fix bound `cmd="--await-parse"` via `local cmd="${2:-}"`, wrote the literal 13-byte string `--await-parse\n` to the input FIFO, and returned `{sent:true}` — the stdin parser logged `stdin_parse_failed line_len=13 error="invalid number at line 1 column 2"` but the caller saw no signal. The post-fix guard rejects any `$cmd` beginning with `--` with `{"status":"error","error":{"code":"flag_as_command",...}}` BEFORE touching the FIFO — the stdin parser never sees the mangled payload. The guard must appear after `missing_command` (so empty CMD still errors with `missing_command`, not `flag_as_command`) and before the FIFO write. The `flag_as_command` code is stable contract; pinned by `tests/session_send_await_parse_contract.rs::cmd_send_rejects_flag_as_command` (4 asserts: guard literal, `json_error "flag_as_command"` literal, ordering after `missing_command`, ordering before FIFO write).

Concurrent `--await-parse` sends with distinct `requestId` fields DO correlate correctly on BOTH the happy and sad paths. When a requestId is present in the sent payload and matches the conservative charset `^[A-Za-z0-9_.:/-]+$`, the happy-path grep is scoped to `cid=stdin:req:<request_id> ` — the correlation_id format emitted by [[src/stdin_commands/mod.rs#start_stdin_listener]] via `tracing::info!(correlation_id = %correlation_id)`. Sends without a requestId (or with one containing shell/grep metachars) fall back to legacy offset-first grep, which preserves single-caller serial-usage correctness. Cross-correlation race pinned against by `tests/session_send_await_parse_contract.rs::cmd_send_scopes_happy_path_grep_on_request_id`.

The sad-path (`parseError`) correlation is symmetric. The listener's `Err(_)` arm calls [[src/stdin_commands/mod.rs#extract_request_id_lenient]] on the raw un-deserialized line to pull the requestId BEFORE full serde parsing fails, then builds `correlation_id = "stdin:req:<id>"` when extractable. The helper mirrors the shell-side sed pattern and charset regex exactly so a requestId accepted by one side is accepted by the other. The shell-side sad-path grep is scoped on `cid=stdin:req:<req_id> ` symmetrically with the happy-path grep, and falls back to legacy offset-first grep when the lenient extract yields nothing (for payloads where the `"requestId":"..."` structure is itself broken). Cross-correlation symmetry is pinned by `tests/session_send_await_parse_contract.rs::cmd_send_scopes_sad_path_grep_on_request_id` + `tests/stdin_parse_error_recovery_contract.rs::parse_error_arm_scopes_correlation_id_on_request_id` + `tests/stdin_parse_error_recovery_contract.rs::extract_request_id_lenient_module_helper_exists` (the last guards the Rust/shell charset mirror — modifying one side without the other desyncs the two).

`session.sh rpc` surfaces the same parse failures on its typed-response path. When a caller passes `--expect TYPE` and the app rejects the payload at parse time (e.g. malformed `target:{kind:"main"}` that omits the required `type` discriminator for `AutomationWindowTarget`), the underlying `await-response.ts` helper never sees a matching response and exits with a generic `timeout` code. `cmd_rpc` then post-hoc scans the app.log tail since its recorded `start_offset` for `stdin_parse_failed` events scoped to `cid=stdin:req:<request_id> ` and, on match, replaces the timeout envelope with `{status:"error", error:{code:"parse_error", message:"<extracted error>"}}`. The scoped grep mirrors the `cmd_send --await-parse` sad-path protocol and uses the same Pass #13 lenient-extract correlation_id. The `parse_error` code is stable contract (distinct from the `timeout` fallback it replaces and from the `parseError` *value* used on `cmd_send --await-parse`'s `parseOutcome` field). Pinned by `tests/session_rpc_parse_error_surface_contract.rs` (6 tests: request-id scoping, non-zero-exit gating, code name, truncation+JSON-escape, start_offset window, charset-boundary fallback). The scoped grep is itself gated on `request_id` matching `^[A-Za-z0-9_.:/-]+$` (mirroring `cmd_send`'s charset gate at line ~390 of session.sh) and falls back to an unscoped `grep -m1 'event_type=stdin_parse_failed'` on the log tail when the charset check fails — without this fallback, requestIds with chars outside the conservative charset (e.g. `a+b`, `a\b`) would land on a Rust-side `stdin:parse:<uuid>` cid that the scoped grep never matches, silently degrading the post-hoc scan to a generic `timeout` envelope (closed as Run 8 Pass #12 — `cmd-rpc-scan-misses-requestid-charset-boundary`). The post-hoc `cmd_rpc` scan is the fallback; preemptive parse-failure detection lives in [[scripts/agentic/await-response.ts]] and runs inside the typed-response poll loop, short-circuiting with a `parse_error` envelope + exit code 3 within one poll cycle (~50ms) of the parse failure landing in app.log — down from the full `--timeout` budget (3000-5000ms) in the post-hoc-only regime. The preemptive scan mirrors the shell charset gate (`^[A-Za-z0-9_.:/-]+$`), the scoped cid marker with trailing space (`cid=stdin:req:${requestId} `), the `event_type=stdin_parse_failed` literal, the 200-char error truncation, and the ` at line N column M` suffix strip — so both detection paths produce structurally symmetric `parse_error` envelopes. Charset-unsafe requestIds skip the preemptive scan (Rust routes them to `cid=stdin:parse:<uuid>` which the scoped grep cannot match) and fall through to the shell's post-hoc unscoped-grep fallback. Exit code 3 is stable contract: programmatic callers MAY key on it to distinguish "parse error surfaced preemptively" from "generic timeout" (exit 1) or "infrastructure error" (exit 2). Pinned by `tests/agentic_await_response_preempt_parse_failure_contract.rs` (7 tests: exit code constant, charset gate, scoped cid marker with trailing space, event_type literal match, error-length cap + suffix strip, preempt-before-typed-scan ordering inside the poll loop, `parse_error` code emission).

`parse_stdin_command` distinguishes "known automation verb, wrong payload" from "unknown verb" before falling back to the SDK `Message` parser. The dual-dispatch entry at [[src/stdin_commands/mod.rs#parse_stdin_command]] tries `ExternalCommand` first; on failure it captures the deserialize error under `ext_err`, then checks whether `raw["type"].as_str()` is a member of [[src/stdin_commands/mod.rs#EXTERNAL_COMMAND_VERBS]]. If so, it short-circuits with `anyhow::anyhow!("automation_payload_mismatch: \"<verb>\" is a known automation verb but the payload did not validate: <ext_err>")` — surfacing the field-level serde diagnostic (e.g. `unknown field \`value\`, expected \`text\` or \`requestId\``) instead of the SDK fallback's less-helpful `unknown variant \`<verb>\`, expected one of \`hello\`, \`arg\`, \`submit\`, …`. Only genuinely unknown verbs reach the Message fallback now. Pinned by `src/stdin_commands/mod.rs::tests::parse_stdin_command_surfaces_external_command_error_for_known_verb_with_wrong_field` + `..._for_missing_required_field` + `parse_stdin_command_unknown_type_still_uses_sdk_message_fallback`. The `automation_payload_mismatch` prefix is stable contract — callers MAY grep for it to distinguish field-typo errors from unknown-verb errors.

The same parser validates the optional `protocolVersion` envelope before typed dispatch. Missing envelopes remain v1, explicit v1 and v2 both dispatch, and future unsupported versions are rejected before UI side effects while incrementing `stdin_unsupported_protocol_version_total`. Invalid non-integer envelopes reject as parse failures without consuming the unsupported-version counter, matching the ingress observer.

### Session forwarder ownership

`session.sh start` prunes orphan forwarders for the named session before resume or fresh launch.

The session wrapper uses a background bash forwarder plus a `cat` reader to relay writes from `<session>/input` into the app's primary stdin FIFO. If stale forwarders survive for the same input path, a later `session.sh send` can write successfully while an orphan consumes the line, producing `parseOutcome:"timeout"` despite a healthy app PID and live recorded forwarder. `cleanup_orphan_session_forwarders` matches both the primary pipe and input FIFO, preserves the recorded live forwarder process tree on resume, and cleans all path-matched forwarders on stale replacement and stop. Fresh starts send an immediate `getState` keepalive before waiting for readiness so the no-stdin watchdog cannot exit the app before the caller sends its first command. The startup readiness marker matches compact logs via `STARTUP_READY ` so warm sessions no longer report `ready:false` after a successful startup.
