# Menu Syntax

Menu syntax is the ScriptList-owned parser layer for deterministic power-user commands. It keeps legacy handoffs separate while `;`, `:`, `#`, and `!` power syntax stay collision-safe.

## Power Syntax Grammar

Power syntax treats first-character symbols as verbs while inline tokens refine the sentence.

The canonical grammar is: `/` selects an agent skill in chat, `@` selects current context, `;` captures local structured data, `:` refines launcher search, `#` labels inside `;` and `:`, and `!` runs a registered Script Kit command. `+` remains a legacy capture alias while `~`, `>`, and `?` keep their existing launcher meanings.

Syntax only owns input when the first token clearly opts in. Plain text, URLs, `localhost:3000`, `C#`, `hello!`, unknown `;target` / legacy `+target` heads, and top-level `#tag` stay normal launcher search.

## Power Syntax Config Surface

`powerSyntax.*` is an authoring-only config surface for future parser wiring.

[[scripts/config-schema.ts]] and [[scripts/config-cli.ts]] expose schema-only knobs for enabling Power Syntax, choosing capture sigils (`;`, `+`, or `both`), disabling `>` command invocation, and configuring Cmd+Enter AI fallbacks. [[kit-init/config-template.ts]] documents the built-in defaults in a commented block. The Rust parser in [[src/menu_syntax/parse.rs#parse]] deliberately continues using its current defaults until a follow-up wires these options through runtime config.

## Parser Boundary

[[src/menu_syntax/parse.rs#parse]] is the only public classifier for power syntax. It must not claim legacy handoff triggers, unknown keyword heads, unknown capture heads, or top-level tags.

Built-in capture targets live in [[src/menu_syntax/payload.rs#KNOWN_CAPTURE_TARGETS]], and script metadata can register additional capture targets at runtime. Unregistered heads — `;github`, `+github`, `+1`, `+react component`, `localhost:3000`, `not-a-target: stuff`, `#work` — fall through to normal fuzzy search where legacy compatibility allows it.

Prefix ownership is target-gated: `;target` is the canonical capture prefix, `+target` is its legacy alias, and `target:` is the keyword alias, but all three only become parser-owned after `target` is built in or registered from metadata. Payload v1 still serializes capture-prefix invocations with the `plus` grammar surface for compatibility, even when the user typed `;target`.

## Advanced Query

[[src/menu_syntax/filter.rs#apply_advanced_query]] applies typed filters after normal fuzzy search. Filters and tags narrow results; search words still go through the existing launcher ranker.

Supported filters: `type:/kind:`, `shortcut:`, `source:`, `plugin:`, `name:`, `desc:`, `alias:`, `tag:`, `has:`, `meta.<path>:`, and leading `-` for negation. `shortcut:any` means any keyboard shortcut, `shortcut:none` means no shortcut, and literal shortcuts still match exactly. `:#tag` is sugar for `tag:<tag>`. `#tag` by itself remains plain launcher search; inside `;` capture it labels saved data, and after `:` it filters tagged launcher rows. `source:` is broad (includes `kit_name`); `plugin:` is precise (plugin pair only).

Refine mode is search, not capture. A `:` filter popup can stay open for hints, but it does not blank the main launcher results; complete inputs like `:type:script deploy` show the filtered catalog. Partial filter text narrows rows, and applying a filter replaces only the active token so existing filters stay intact.

Declarative launcher filters are order-independent trailing-colon heads. Source heads are valueless tokens such as `files:`/`f:`, `notes:`/`n:`, `clipboard:`/`c:`, `tabs:`/`t:`, `history:`/`h:`, `apps:`/`a:`, `scripts:`/`s:`, `commands:`/`cmd:`, `conversations:`/`ai:`, `dictation:`/`d:`, and `windows:`/`w:`; `processes:`/`p:` is intentionally not committed until root process rows exist. Property heads such as `type:` still require values and narrow by row kind rather than source. [[src/menu_syntax/query.rs#parse_filter_query]] strips known source heads into free text and records include/exclude gates, while a leading `-` excludes a source and exclusion wins conflicts. A bare leading `:` is now only the discoverability trigger: it opens the filter picker and inserts committed heads such as `files:` or `type:`, but `:f` is not committed source syntax. Unknown filter-looking heads, quoted tokens, URLs, and capture keyword aliases remain literal or capture-owned. Empty filtered launcher states render filter-specific recovery copy instead of generic no-results guidance, so users can predict that removing a filter widens the same search.

## Command Invocation

Command invocation makes `!` an explicit run surface for registered Script Kit commands, never a shell escape.

[[src/menu_syntax/command.rs#command_slug]] normalizes script and scriptlet names into command heads shared by the picker and executor. [[src/menu_syntax/execute.rs#command_env]] exposes `command.v1` metadata (`head`, `fields`, and `tags`) while argv after ` -- ` is passed to the resolved script/scriptlet. Unknown heads such as `!important` show no command match and do not spawn the typed text. Duplicate heads are disabled in the picker and treated as ambiguous at execution time rather than silently running the first match.

## Capture Metadata

[[src/menu_syntax/metadata.rs#handler_specs_from_extra_map]] reads `menuSyntax` handler specs from typed script metadata. Capture handlers opt into `capture.v1` and one or more targets.

Scriptlet authors use [[src/menu_syntax/metadata.rs#handler_specs_from_yaml_like_string]] against their fenced `metadata` block text. The shape is validated at lookup time, not at load time, so malformed entries are silently skipped.

[[src/menu_syntax/filter.rs#registered_capture_targets_from_scripts]] derives dynamic `;target` heads from script `capture.v1` metadata. The ScriptList input parser and trigger popup pass those targets through context-aware parse helpers so `;github` remains search until a script registers `github`; `+github` is retained as a legacy alias.

Capture target picker labels use built-in names for built-in targets and the first non-empty `menuSyntax.label` from registered `capture.v1` script metadata for dynamic targets, falling back to a humanized slug. Both filtered target rows and exact focused target rows read from this catalog.

## Menu Syntax Doctor

`metadata.menuSyntax` validation library that emits actionable diagnostics with JSON-path pointers. The `menu-syntax-doctor` CLI subcommand (Pass 37, [[src/bin/menu_syntax_doctor.rs#main]]) calls this engine and uses its `has_errors()` flag to set its exit code.

[[src/menu_syntax/doctor.rs#validate]] takes a `serde_json::Value` (Array or single-spec Object form) and returns a [[src/menu_syntax/doctor.rs#DoctorReport]] containing `Vec<DoctorIssue>`. Each [[src/menu_syntax/doctor.rs#DoctorIssue]] carries a JSONPath-like `path` (e.g. `$[2].targets[0]`, `$[0].args[1].name`), a [[src/menu_syntax/doctor.rs#DoctorSeverity]] (Error / Warning), and a human-readable `message`. Errors fail the CLI exit code; warnings are advisory. The validator covers: unknown family (suggesting the three known ones), missing required fields per family (capture: `targets`; command: `head`; skill: `slug`), malformed slugs (whitespace, leading `!`/`/`, empty), unknown `accepts` tokens against the canonical list (`tags`/`date`/`duration`/`url`/`priority`/`kv`), malformed capture requirement list shapes, unknown `required`/`optional`/`forbidden` tokens, duplicate command heads across the array (case-insensitive, paths joined: `$[0], $[1]`), and missing `name` fields on args/flags. Custom (non-built-in) capture target slugs emit a Warning rather than Error so authors can declare new targets without tripping the CLI exit code. [[src/menu_syntax/doctor.rs#validate_at_path]] keeps the same pure checks while rooting diagnostics at wrapper paths such as `$.menuSyntax` or `$.metadata.menuSyntax`, so author-pasted metadata reports `$.menuSyntax[0].targets` instead of `$[0].targets`. The receipt fixture is `tests/menu_syntax_doctor.rs` (29 tests across good / bad-slug / bad-field / duplicate-command / mixed / path-root categories) — `cargo test --test menu_syntax_doctor`. CLI surface ships in `src/bin/menu_syntax_doctor.rs`: reads JSON or YAML from `--path <file>` or stdin (JSON-first, YAML fallback, then YAML-frontmatter extraction), selects raw `menuSyntax`, nested `metadata.menuSyntax`, or a direct spec value, runs `doctor_validate_at_path`, prints either human-readable summary (`menu-syntax-doctor: 1 error(s), 0 warning(s)` followed by `[error] $.path: msg` lines) or `--json` payload (`{"issues":[…]}`), exits `0` on no-error reports, `1` when `has_errors()`, `2` on argv/IO failure (distinct from spec failure). Pinned by `tests/menu_syntax_doctor_cli.rs` (11 integration tests built via `CARGO_BIN_EXE_menu-syntax-doctor`): good-stdin-human → 0, good-stdin-json → 0 with valid JSON payload, bad-stdin-human → 1 with "error" in stdout, bad-stdin-json → 1 with `severity:error` in payload, good-via-`--path` flag → 0, unknown-flag → 2 with stderr message, YAML-via-stdin → 0 (dual-parse fallback), metadata-wrapper JSON → `$.menuSyntax[...]`, nested metadata-wrapper JSON → `$.metadata.menuSyntax[...]`, and YAML frontmatter → `$.menuSyntax[...]`.

## Tolerant Diagnostics

Best-effort menu-syntax readers keep valid data flowing while skipped malformed data stays visible.

Missing local history/artifact files are normal and do not warn. Corrupt rows that are skipped must be observable through bounded diagnostics: artifact reports expose skipped counts and capped warnings, history report readers do the same for tag/key/argv pools, and `menu-syntax-doctor` reports handler metadata mistakes with precise JSON paths. Tolerant runtime readers keep returning valid rows/specs where possible instead of failing capture or command flow.

## Command Schema Surface

Command authors declare argv schema (label, description, positional args, named flags, usage) on a `command.v1` `menuSyntax` entry; the launcher's main hint card surfaces those declarations as schema-derived rows when the user types `>head`.

## Capture Target Taxonomy Audit

Capture targets are pinned into core, special-case, and metadata-driven groups before any registry extraction.

The core built-ins are `todo`, `cal`, `note`, `social`, and `link`. [[src/menu_syntax/payload.rs#KNOWN_CAPTURE_TARGETS]] is the parser-owned list, and the bare `;` trigger picker exposes exactly those five targets in that order. [[src/menu_syntax/capture_schema.rs#builtin_target_slugs]] is the schema-owned builtin list and must stay schema-backed without implying picker visibility.

`mcal` is intentionally special: [[src/menu_syntax/payload.rs#is_known_capture_target]] treats it as parser-known and [[src/menu_syntax/capture_schema.rs#builtin_schema]] gives it the calendar schema, but the default picker does not list `;mcal` until script metadata registers that target. Shipped examples such as `gcal`, `github`, `expense`, `snippet`, `fixture`, `reminder`, `snooze`, and `defer` remain metadata-driven custom targets; they can register labels and schemas through `menuSyntax`, but parser-known status and builtin schemas must not drift to include them.

## Capture Field Schemas

Capture targets declare which payload fields are required, optional, or forbidden so the UI can refuse to ship nonsense (e.g. `;cal` without a date).

[[src/menu_syntax/capture_schema.rs#builtin_schema]] returns the `CaptureFieldSchema` for built-in targets (`todo`, `note`, `link`, `cal`, `social`). Each schema lists required fields by [[src/menu_syntax/capture_schema.rs#FieldRequirement]] (Body, Url, Priority, Duration, Tag, AnyDate, DateRole, or Kv("key")). [[src/menu_syntax/capture_schema.rs#CaptureFieldSchema#missing_required]] returns the unsatisfied required fields for a parsed [[src/menu_syntax/payload.rs#CaptureInvocation]] — empty Vec means the payload is ready to ship. The module is pure; UI surfacing and Enter-gating live in subsequent layers.

`;cal` requires Body and AnyDate, so `;cal Buy milk` reports `[AnyDate]` missing and `;cal Design review start:"friday 2pm"` reports an empty missing list. `;link` requires Url (not Body), so a bare `;link Some title` reports `[Url]` missing. Unknown targets like `;github` return `None` from `builtin_schema` — dynamic schemas are sourced from script metadata in a later story.

## Dynamic Capture Schemas

Scripts declare per-target field requirements (`required` / `optional` / `forbidden`) on a `capture.v1` `menuSyntax` entry; the launcher builds a [[src/menu_syntax/capture_schema.rs#CaptureFieldSchema]] for any target a `builtin_schema` does not know.

[[src/menu_syntax/payload.rs#MenuSyntaxHandlerSpec]] gained `required`/`optional`/`forbidden` `Vec<String>` fields (all `#[serde(default)]`). Tokens use the lowercase enum-name vocabulary plus colon-prefixed namespaces: `"body"`, `"url"`, `"priority"`, `"duration"`, `"tag"`, `"date"` (any role), `"date:start|end|due|at|any"` (specific role), `"kv:KEY"` (named kv pair). [[src/menu_syntax/metadata.rs#parse_field_requirement_token]] resolves a single token; [[src/menu_syntax/metadata.rs#dynamic_capture_schema_from_spec]] returns `Option<CaptureFieldSchema>` for the spec's first non-empty target (lowercased). Unknown tokens drop silently — the doctor surface flags them as Warnings during authoring. Returns `None` when family is not `capture.v1` or no usable target exists. Pass 33 added two-phase dedup: within each list duplicates collapse to the first occurrence, and cross-list precedence is `required > optional > forbidden` so a token declared in multiple lists ends up only in the highest-precedence one — closes the `dynamic-schema-no-dedup-across-required-optional` filing from Pass 24. Demo: `scripts/examples/menu-syntax/capture-expense-ledger.ts` declares `;expense` with `required: ["body","kv:amount"]`. Receipt: `cargo test --lib menu_syntax::metadata` (20 tests) covers token-vocabulary, kv namespace, date-role namespace, unknown-token drop, target lowercasing, non-capture filter, missing-target None, all-empty-list yields empty schema; `cargo test --test menu_syntax_run11_attacker_pass24_dynamic_schema` (27 tests) covers within-list dedup, cross-list precedence (required>optional>forbidden), optional-wins-over-forbidden, distinct-fields-survive-falsifier.

## Capture Payload Validation

`validate(payload, schema)` collapses required-field checks and well-formedness checks into one decision the UI can branch on without re-running ad-hoc rules.

[[src/menu_syntax/capture_schema.rs#validate]] returns [[src/menu_syntax/capture_schema.rs#ValidationResult]] = `Ready` | `Incomplete { missing: Vec<FieldRequirement> }` | `Malformed { field: FieldRequirement, reason: String }`. Malformed wins over Incomplete: a bad `url`, non-numeric `amount=`, or a satisfied `schema.forbidden` field short-circuits before missing-field accounting so the author sees the actual error. URL well-formedness requires `http://` or `https://` scheme; `amount` kv values must parse as `f64` AND be `is_finite()` after stripping a leading `$`/`+`/`-` — `NaN`/`inf`/`infinity` parse as numeric to f64 but are rejected because currency cannot be NaN or infinite (Pass 26 fix). Empty kv values fall through to Incomplete (treated as "not provided"), not Malformed. The forbidden sweep runs after well-formedness so `;cal` with `priority=p1` returns `Malformed { field: Priority, reason: "priority is not allowed for ;cal" }` even when other required fields are also missing — wrong-shape beats incomplete-shape (Pass 26 fix). The function is pure; callers thread the result into `MenuSyntaxMainHintSnapshot.capture_validation` and the [[src/menu_syntax/capture_gate.rs#decide_capture_gate]] surface. Receipt: `cargo test --lib menu_syntax::capture_schema` covers ready / missing-body / missing-date / malformed-amount / malformed-url / amount-format-acceptance / empty-amount-falls-to-incomplete / nan-and-infinity-rejected / forbidden-priority-on-cal-malformed / forbidden-url-on-cal-beats-incomplete / ready-unaffected-when-no-forbidden-set (25 tests).

## Skill Spec Metadata

Scripts and scriptlets register `/slug` skills via `menuSyntax` entries with `family: "skill.v1"` so the launcher can list them under `:type:skill` filters without auto-registering a `!command`.

[[src/menu_syntax/skill.rs#skill_specs_from_value]] extracts [[src/menu_syntax/skill.rs#SkillSpec]] (`slug`, `label`, `description`, `context_requirements: Vec<String>`, `accepts_capture_target`) from a raw `menuSyntax` JSON value (single-object or array shape). Only `family == "skill.v1"` entries are returned (case-insensitive — `SKILL.V1`/`Skill.V1`/`sKiLl.v1` all match per Pass 30); `slug` is required and trimmed (entries with empty/whitespace slugs are silently skipped — authoring mistakes belong to [[src/menu_syntax/doctor.rs]], not this extractor). Duplicate slugs (after trim) collapse to one entry with declaration-order winning — the FIRST occurrence is kept so the runtime UI doesn't render two visually-identical rows for the same `/review` skill (Pass 30). `contextRequirements` filters non-string array entries so a malformed mixed array still yields the valid declarations. The TypeScript `SkillHandlerSpec` mirror lives in `kit-init/types/menu-syntax.d.ts`. Demo: `scripts/examples/menu-syntax/skill-review-current-file.ts` declares a `/review` skill with `contextRequirements: ["currentFile"]`. Receipt: `cargo test --lib menu_syntax::skill` covers single-object/array forms, missing-slug skip, slug-trim, mixed-type contextRequirements filtering, default-on-missing-fields, family-filter, non-object inputs, case-insensitive family match across 5 spellings + falsifier, dedup-keeps-first, dedup-preserves-distinct-slugs, dedup-treats-trimmed-slug-as-same (13 tests).

## Capture Validation Snapshot

`MenuSyntaxMainHintSnapshot` carries a structured `capture_validation` and a multi-chip `status_chips` so the hint card can render `; capture` / `needs body` / `needs date` rows without re-running validation rules.

[[src/menu_syntax/main_hint.rs#MenuSyntaxMainHintSnapshot]] gained `status_chips: Vec<MenuSyntaxMainHintChip>` (multi-chip alternative to the original `status_chip` — capture surfaces use the multi-chip form, other surfaces keep the single-chip form for backward compat) and `capture_validation: Option<MenuSyntaxCaptureValidationSnapshot>`. [[src/menu_syntax/main_hint.rs#MenuSyntaxCaptureValidationSnapshot]] mirrors the [[src/menu_syntax/capture_schema.rs#ValidationResult]] shape with serializable fields: `target`, `status` ([[src/menu_syntax/main_hint.rs#MenuSyntaxCaptureValidationStatus]] = `ready` | `incomplete` | `malformed`), `can_submit`, `missing_field_labels: Vec<String>`, `malformed_field_label`, `malformed_reason`. The `capture_composer_hint` constructor resolves accepted NL dates before validation, then calls a private helper that synthesizes an empty payload when only `;target` is typed, runs `validate()` against the post-NL snapshot, and emits chips per missing field. `;cal` (no payload) → chips `["; capture", "needs body", "needs date"]` and `captureValidation.status="incomplete"`. `;mcal Lunch tomorrow at 12pm til 1pm` → chips `["; capture", "ready"]` because the resolved DateRange satisfies AnyDate. `;cal Buy milk start:friday` → chips `["; capture", "ready"]` and `captureValidation.status="ready", canSubmit=true`. `;link ftp://nope` → chips `["; capture", "malformed"]` and `captureValidation.malformedFieldLabel="url"`. `;github` (no schema) → chips `["; capture"]` only and `captureValidation=None`. Receipt: `cargo test --lib menu_syntax::main_hint::tests::capture_validation` covers the four cases plus a serde camelCase serialization pin, and `capture_validation_uses_resolved_nl_state_for_mcal` pins the resolved snapshot path. The hint-card renderer in `src/render_script_list/mod.rs::render_menu_syntax_main_hint` (Pass 23) consumes `status_chips`: when non-empty it skips the leading mode chip (already rendered) and emits per-missing-field chips via the existing `render_menu_syntax_hint_chip`; the legacy single `status_chip` is suppressed in that case so capture surfaces don't double-render.

Run 14 Pass 30 adds a schema-derived `Priority choices` row when a priority-capable target has no priority yet and the raw last token is `p`; `;todo p` shows `p1 p2 p3 p4`, while priority-forbidden targets like `;cal p` omit it.

## Unresolved Date Surfacing

`MenuSyntaxMainHintSnapshot.unresolved_dates` carries the date-slot phrases the parser recognized but could not interpret (e.g. `due:asdf`), so the hint card and the state-receipt RPC can warn the user about silently-ignored input.

[[src/menu_syntax/main_hint.rs#MenuSyntaxMainHintSnapshot]] gained `unresolved_dates: Vec<UnresolvedDate>` (`#[serde(default, skip_serializing_if = "Vec::is_empty")]`, camelCase serialization). The `capture_composer_hint` constructor calls [[src/menu_syntax/date.rs#resolve_capture_dates]] (`MenuSyntaxClock::local_now()` for the clock) and threads the `unresolved_dates` through; when non-empty, it also pushes a row `{label:"Unresolved", value:"due: asdf, …"}` into the hint card's row list so the warning is visible in both the rendered card and the `getState` snapshot. `UnresolvedDate` derives `Eq` (in addition to `PartialEq`) so it can live inside the snapshot's `Eq`-deriving struct. Receipt: `setFilter ";todo Thing due:asdf"` → `getState.menuSyntaxMainHint.unresolvedDates:[{role:"due",source:"asdf",sourceSpan:[16,20]}]` AND a `rows:[{…},{label:"Unresolved",value:"due: asdf"}]` entry.

## Calendar ICS Artifact

Calendar capture writes a deterministic `.ics` artifact and opens it with Calendar.app instead of mutating Calendar through AppleScript.

## Capture Handler Filtering

Capture mode treats the typed body as payload, not fuzzy text. Handler rows come from metadata-matching scripts and scriptlets via [[src/menu_syntax/filter.rs#scripts_handling_capture]], not from searching the captured task body.

The trigger picker's `;<text>` target filtering applies only before a target is locked; after an exact target and body boundary, typed text is payload and is not fuzzy-matched against handlers or targets.

Wildcard `targets: ["*"]` opts a handler in for every capture target within `capture.v1`. Handlers in other families never match capture mode, even if their `targets` list includes a known capture target.

[[src/menu_syntax/handler_index.rs#rank_handlers_for_target]] orders matched handlers by a lexicographic score: exact-target > `defaultHandler:true` > user-authored (plugin != "main") > accepts-match boost. Shipped `main/*` examples sort below user handlers in the same priority bucket. Accepts boost is capped at 3 so it never crosses priority buckets. [[src/menu_syntax/handler_index.rs#rank_scripts_handling_capture]] is the dedup-by-path convenience used by [[src/scripts/grouping.rs#build_capture_mode_results]].

[[src/menu_syntax/handler_index.rs#explain_capture_handler_ranking]] derives a state-first explanation from that same sorted and path-deduped order. Capture composer hints add `Handler`, `Why selected`, `Other matches`, and `Handler conflict` rows to `menuSyntaxMainHint`; multiple matching `defaultHandler:true` handlers become a warning while Enter still runs the deterministic winner.

## Date Resolution

Date inference is a post-parse step in [[src/menu_syntax/date.rs#resolve_capture_dates]]. Explicit date keys win over suffix inference, and resolved dates serialize as payload data with ISO time, original phrase, timezone, and granularity.

[[src/menu_syntax/date.rs#MenuSyntaxClock]] injects timezone and base time so DST and fixed-clock tests run without `Local::now()`. `local_now` prefers parseable `TZ`, then OS local IANA detection from `/etc/localtime` or macOS `systemsetup -gettimezone`, with UTC only as a last resort.

[[src/menu_syntax/date.rs#resolve_date_phrase]] normalizes smart quotes first, then runs deterministic local pre-parsers before `chrono-english`: shorthand (`noon`, `midnight`, `eod`, `eom`), `at `-prefixed time forms, exact time-of-day words, numeric time forms, absolute dates, relative offsets, timezone suffixes, and time ranges.

[[src/menu_syntax/date.rs#ResolvedDate]] carries an optional `end_iso` field for range expressions like `9-10am` (serialized as `endIso`, omitted from JSON when None so existing payload files stay byte-identical). [[src/menu_syntax/date.rs#UnresolvedDate]] reports recognized-but-unparseable date slots (e.g. `due:asdf`) so the snapshot/UI can surface them. Pass 33+ extension: [[src/menu_syntax/date.rs#ResolvedCaptureInvocation]] now carries a `unresolved_dates: Vec<UnresolvedDate>` field (serializes as `unresolvedDates`, `#[serde(default)]` so older payload JSON still deserializes). `resolve_capture_dates` previously dropped phrases that `resolve_date_phrase` returned None for; it now collects them into `unresolved_dates` so a `;todo Thing due:asdf` payload produces `dates: []`, `unresolvedDates: [{role:"due",source:"asdf",sourceSpan:...}]` instead of silently swallowing the bad phrase. Mixed payloads (one resolved + one unresolved) split into the two lists by phrase. ## Composable Natural Language Capture

The natural-language tail is parsed by four composable sub-parsers so capture handlers beyond `;mcal` — `;todo`, `;reminder`, `;snooze`, `;defer` — opt into the parts of the grammar they need.

[[src/menu_syntax/nl_phrase.rs#resolve_capture_nl_phrase]] is now a thin orchestrator that runs the four parsers in order and joins remaining tokens as the subject. Each parser returns a resolved value, a `MenuSyntaxFragment` (role + source span + status), and the unconsumed token tail. The orchestrator preserves the existing `CaptureNlResolution` shape additively — older callers see only the previously-resolved fields.

[[src/menu_syntax/nl_anchor.rs#parse_anchor]] owns date anchors: `today`, `tomorrow`, `tom`, weekdays (`mon..sun` plus full names), `next <weekday>`, and `in N <unit>` relative-now durations. [[src/menu_syntax/nl_time.rs#parse_time_or_range]] owns time, ranges, bare-hour pm/am inference, and `from <s> til <e>`; range spans merge the absorbed anchor token. [[src/menu_syntax/nl_duration.rs#parse_duration]] owns `for N <unit>` plus compact forms (`30m`, `30mins`, `1h`, `2hrs`); the `in 30 minutes` shape is reusable for relative-now anchors via [[src/menu_syntax/nl_duration.rs#parse_relative_duration_after_in]]. [[src/menu_syntax/nl_recurrence.rs#parse_recurrence]] owns recurrence frequency: `every day`/`daily`, `weekly`/`every week`, `biweekly`/`fortnightly`/`every other week`, `bimonthly`/`every other month`, `every weekday`, `every weekend`, `every <weekday>` and comma/`and` weekday lists, numeric intervals (`every N days|weeks|months|years`, 1..=30), lead-ins (`repeat|repeats|repeating|recurring|recurs <phrase>`), and terminators (`for N days|weeks|months|years|times|occurrences`, `until <date-phrase>`) which extend the emitted RRULE with `INTERVAL`, `COUNT`, or `UNTIL`. RRULE output uses canonical key order `FREQ`, `INTERVAL`, `BYDAY`, `COUNT`, `UNTIL`.

`;mcal ... at 12` uses the same `MenuSyntaxClock` timezone as date resolution, so local timezone detection prevents the natural-language body parser from anchoring the event in UTC.

Standalone time-of-day anchors are accepted in the NL body even when no day word is present. Exact bare tokens `noon`, `midnight`, `morning`, `afternoon`, `evening`, and `tonight` resolve in `MenuSyntaxClock.timezone` to local today at 12:00, 00:00, 09:00, 14:00, 19:00, and 20:00 respectively. `at `-prefixed forms share the same vocabulary plus numeric clock tokens: `at 12`, `at 12pm`, `at 9`, `at 9am`, `at 14:00`, `at noon`, and `at midnight`. Bare unambiguous clock tokens such as `3pm` and `14:00` also resolve as standalone body anchors; bare digits-only body tokens still require `at` to avoid stealing subject numbers. The standalone-bare fallback only fires when no day anchor is present, so `Morning coffee tomorrow` keeps `Morning coffee` in the subject. Standalone body times stay on local today even if that time has already passed; users must say `tomorrow ...` when they want tomorrow. `;mcal Lunch w/ Mindy at noon for 30 min` therefore yields subject `Lunch w/ Mindy`, start today 12:00, and end today 12:30.

[[src/menu_syntax/date.rs#resolve_capture_dates_with_accepts]] runs the orchestrator for `cal`/`mcal` or handlers that accept `dateRange`, `duration`, `recurrence`, `relativeDate`, `daily`, `multiWeekday`, `monthly`, or `yearly`. [[src/menu_syntax/capture_gate.rs#decide_capture_gate_with_accepts]] probes the NL parser for `AnyDate` so `;mcal Lunch tomorrow 12pm`, `;todo Renew passport tomorrow`, `;snooze in 30 minutes`, and `;defer until next week` all pass the gate without explicit `start:` syntax. [[src/menu_syntax/date.rs#RecurrenceFrequency]] gained `Daily`, `Monthly`, `Yearly` variants serialized lowercase; `ResolvedRecurrence.weekdays` is omitted when empty so daily/monthly/yearly payloads stay compact.

The shipped local-first handlers now exercise those parser paths end-to-end: [[scripts/examples/menu-syntax/snooze-task.ts]] accepts `relativeDate` phrases like `;snooze in 30 minutes Review PR #432` and writes `$SK_PATH/menu-syntax/snoozed.jsonl`; [[scripts/examples/menu-syntax/defer-task.ts]] accepts date/relativeDate phrases like `;defer until next week Refactor settings panel` and writes `$SK_PATH/menu-syntax/deferred.jsonl`.

## Execution Payload

Capture execution writes a versioned JSON payload to a temp file and passes its path through `KIT_MENU_SYNTAX_PAYLOAD_PATH`. Scripts read the payload instead of parsing argv.

## Payload Retention

Payload tempfiles (`capture_v1-*.json`) accumulate over time. A retention policy prunes them without touching user-authored artifacts.

[[src/menu_syntax/retention.rs#plan_retention]] is pure: it takes a listing of `PayloadListing { path, created_at_unix }` plus a `now_unix` clock and a `RetentionConfig`, and returns a `RetentionPlan { keep, delete }` sorted newest-first. Policy defaults match Oracle iter 004: always keep newest `KEEP_NEWEST_DEFAULT = 250`, hard cap `HARD_CAP_DEFAULT = 1000`, age cutoff `AGE_CUTOFF_DAYS_DEFAULT = 30 days` applied only OUTSIDE the newest-250 floor. The planner sorts by `created_at_unix` desc with path-ascending tie-break so identical inputs always produce identical plans.

[[src/menu_syntax/retention.rs#apply_retention_plan]] is the thin filesystem helper. Missing paths are treated as successful no-ops so concurrent passes don't spuriously fail. The module never enumerates files — callers filter for `capture_v1-*.json` and pass the listing in, so user JSONL files, markdown notes, `.ics` calendars, and social drafts are untouchable by construction.

## Tag History Pool

Per-target `#tag` history feeds a future autocomplete popup so users don't re-type `#errands` from scratch on every `;todo` capture.

[[src/menu_syntax/history.rs#HistoryStore]] owns an injectable `base: PathBuf` (resolved menu-syntax root) so tests construct a `tempfile::TempDir`-backed store rather than mutating `$SK_PATH`. Free functions [[src/menu_syntax/history.rs#record_tags]] and [[src/menu_syntax/history.rs#read_tag_pool]] wrap a `from_env()` default. Storage is append-only JSONL at `<base>/<target>/tags.history.jsonl` with rows `{ts:u64, tag:String}` — minimal shape so retention and privacy stay decoupled from autocomplete ranking. The pure ranking transform [[src/menu_syntax/history.rs#build_tag_pool]] groups by tag, sorts by count desc → last_seen_ts desc → tag asc, and returns the full pool uncapped (callers truncate at the popup boundary).

`record_payload_tags(&GrammarPayload)` is the canonical entry point from the capture pipeline: negated tags (advanced-query `-#tag`) are filtered out at the boundary so an anti-tag never poisons a positive-suggestion autocomplete pool. Unsafe target slugs (anything outside `[A-Za-z0-9_-]`) are hex-encoded into `~<hex>` directory names so a slug like `../todo` cannot escape the history root.

`execute_menu_syntax_capture_script` in `src/app_execute/menu_syntax_execution.rs` calls `record_payload_tags` and `record_payload_fields` after the validation gate passes and before the script spawns. Failures are warned-and-continued — history is best-effort and the capture itself is the contract. The conversion goes `&CaptureInvocation -> GrammarPayload::from(&inv)` (Pass-5 lossless round-trip), so a refactor of the From impl that drops `kv` into a non-Free `FieldKind` would silently stop tag/field history. Pinned by `capture_invocation_round_trips_into_history_pools` test in `src/menu_syntax/history.rs` which asserts `priority` (schema-bound) does NOT land in history while `owner` (free-form `kv`) does.

## Key Value History Pool

Per-target free-form `key:value` history feeds the same future autocomplete popup as tags but on a per-key basis (`;cal start:` should drop down past `start:` values).

[[src/menu_syntax/history.rs#HistoryStore#record_field]] appends rows `{ts:u64, value:String}` to `<base>/<target>/keys/<key>.history.jsonl`. [[src/menu_syntax/history.rs#HistoryStore#record_payload_fields]] is the canonical entry point from the capture pipeline: it iterates `payload.fields` and writes only `FieldKind::Free` non-negated fields. Schema-bound fields are excluded because the script's declared enum is the source of truth for those keys (story F precedence); query/meta predicates are excluded because they belong to refine semantics, not capture history.

[[src/menu_syntax/history.rs#build_value_pool]] is the pure ranking transform. Story C's spec says values surface most-recent-first, so the sort is `last_seen_ts desc → count desc → value asc`. That tie-break order means a value typed twice five minutes ago beats a value typed once five minutes ago at the same timestamp, while a single value typed today beats a value typed twice last week. Unsafe key names (anything outside `[A-Za-z0-9_-]`) are hex-encoded the same way unsafe target slugs are, so a key like `../start` cannot escape the per-target keys dir.

## Command Argv History Pool

Per-command-head argv history feeds the future `>head --` autocomplete popup so users don't re-type frequent argv combos.

[[src/menu_syntax/history.rs#CommandHistoryStore]] is constructed independently of `HistoryStore` because commands are not menu-syntax artifacts — storage lives at `<sk_path>/commands/<head>/argv.history.jsonl`, not `<sk_path>/menu-syntax/...`. Same boundary discipline applies: append-only JSONL rows `{ts:u64, argv:Vec<String>}`, `target_component` hex-encoding for unsafe heads (so `../deploy` cannot escape), pure ranking transform [[src/menu_syntax/history.rs#build_argv_pool]] groups by joined-argv key (NUL separator, the only POSIX-forbidden argv char) and sorts `last_seen_ts desc → count desc → argv asc`. Empty argv vectors are silently skipped at write time; the file does not exist after a no-op record call. Free `record_argv(head, argv)` and `read_argv_pool(head)` wrappers use `from_env()` for default `~/.scriptkit/commands` resolution.

Wired into the live executor (Run 14 Pass 15): [[src/app_execute/menu_syntax_execution.rs#ScriptListApp#execute_menu_syntax_command_invocation]] calls `CommandHistoryStore::from_env().record_argv(&head, &invocation.argv)` once a command head matches exactly one script-or-scriptlet, BEFORE spawning the handler. Best-effort — record failure logs `menu_syntax_command_argv_history_record_failed` and proceeds. Ambiguous and not-found cases skip the record so the pool only reflects executed commands.

## Schema Overrides History

Pure ranking transform that merges schema-declared enum values with the user's free-form history pool so the popup shows blessed options first and dimly de-prioritizes anything the user typed off-spec.

[[src/menu_syntax/schema_overrides.rs#merge_enum_with_history]] takes `enum_values: &[String]` (from `CommandArgSpec.values` / `CommandFlagSpec.values`) and a `history: &[ValueFrequency]` slice (already recency-desc → count-desc per [[src/menu_syntax/history.rs#build_value_pool]]). It emits `Vec<RankedSlotValue { value, source, count, last_seen_ts }>` where `source` is one of `SchemaEnum`, `SchemaEnumWithHistory`, or `HistoryOnly`. Render order is schema enums in declared order first, then history-only values in input order (recency-first). A value present in both the enum and history collapses to a single `SchemaEnumWithHistory` row carrying the history count/recency for tie-breaking — never double-rendered. Empty enum falls through to a pure-history list (every row `HistoryOnly`); empty history yields a pure schema list (every row `SchemaEnum`, count/ts both 0). UI hint: render `HistoryOnly` rows dimmed with a "previously used" badge so the user can tell which values are blessed by the schema.

Wired into the popup snapshot (Run 14 Pass 18): [[src/menu_syntax/capture_history_picker.rs#build_history_picker_snapshot_with_overrides]] is the schema-aware companion to `build_history_picker_snapshot`. For KeySlot snapshots with non-empty `key_enum_values`, it reads the per-key history pool, runs `merge_enum_with_history`, and emits each `HistoryPickerRow` carrying `source: Some(SlotValueSource)` so the UI receives the dim/bless hint inline. Empty `key_enum_values` and TagSlot kinds delegate to the legacy builder so `source` stays `None` (skip-if-none) and the JSON shape is unchanged for callers that haven't adopted the new function. `HistoryPickerRow.source` is a new optional field with `#[serde(default, skip_serializing_if = "Option::is_none")]` for backward compatibility.

Live `getState` wire-through (Run 14 Pass 19): [[src/menu_syntax/capture_history_picker.rs#snapshot_from_filter_text_with_overrides]] is the schema-aware companion to `snapshot_from_filter_text`. It accepts a `lookup_enum_values: FnOnce(&str, &str) -> Vec<String>` closure, resolves `(target, kind)` via the existing parser, calls the lookup ONLY for KeySlot kinds (TagSlot skips it — pinned by a panicking-lookup test), and routes through `build_history_picker_snapshot_with_overrides`. The prompt_handler `Message::GetState` arm calls this variant.

Real data source (Run 14 Pass 21): [[src/menu_syntax/payload.rs#MenuSyntaxHandlerSpec]] gained a NEW `kv_enums: BTreeMap<String, Vec<String>>` field (`#[serde(default, skip_serializing_if = "BTreeMap::is_empty")]`) that capture.v1 specs can populate to declare per-key enum overrides. [[src/menu_syntax/schema_overrides.rs#capture_kv_enum_values_for_specs]] is the lookup walker — given `(target, key, &[&MenuSyntaxHandlerSpec])` it returns the first non-empty `kv_enums[key]` whose `targets` list contains `target` (case-insensitive trimmed slug) or the wildcard `*`. The prompt_handler closure now collects every loaded script's capture.v1 specs via `script_menu_syntax_specs(script)` and calls this helper, so any script declaring `kv_enums: { env: ["prod","staging"] }` for a target lights up schema-first popup ranking end-to-end without further wiring.

## Capture History Picker

Cursor-context detection plus pure snapshot building for the future autocomplete popup that drops down when the user parks the cursor on a bare `#` or fresh `key:` inside an active capture composer.

[[src/menu_syntax/capture_history_picker.rs#detect_history_picker_context]] inspects `input[..cursor]` and returns `Some(HistoryPickerKind::TagSlot)` when the prefix ends in a bare `#` after whitespace or start-of-input, OR `Some(HistoryPickerKind::KeySlot { key })` when the prefix ends in `key:` where `key` is one ASCII word since the last whitespace (`a-z0-9_-.` only). Mid-word positions, double-colons, and slots whose value has already been typed return `None` so the popup never fights with regular typing.

[[src/menu_syntax/capture_history_picker.rs#build_history_picker_snapshot]] is the pure builder. Given a target, a kind, and a `HistoryStore` it reads the matching pool ([[src/menu_syntax/history.rs#HistoryStore#try_read_tag_pool]] for tag slots, [[src/menu_syntax/history.rs#HistoryStore#try_read_key_pool]] for key slots) and shapes the rows into a `HistoryPickerSnapshot { target, kind, rows }`. Returns `None` when the pool is empty so the UI never renders an empty popup. Sort order is inherited from the underlying pool: tags come back count-desc → recency-desc; values come back recency-desc → count-desc.

[[src/menu_syntax/capture_history_picker.rs#snapshot_from_filter_text]] is the convenience entry point used by `getState`. It treats the cursor as at end-of-input, parses the filter text with [[src/menu_syntax/parse.rs#parse]] to resolve the active capture target (only `Capture` and `Incomplete::MissingCaptureBody` arms qualify — bare prefixes and advanced-query inputs return None), then delegates to `build_history_picker_snapshot`.

The `getState` response carries the snapshot in a `captureHistoryPicker` field on `Message::StateResult` — populated when the cursor sits on a slot trigger AND the matching pool is non-empty, otherwise omitted (skip_serializing_if). Automation can use presence as a "popup is being offered" signal without grepping app.log. Keyboard navigation and the rendered overlay land in the next slice; this slice ships the contract.

## Trigger Picker

The trigger picker is a popup-menu model that surfaces `;` targets, `:` qualifiers, and `!` commands so users never have to guess trigger syntax.

[[src/menu_syntax/trigger_picker.rs#build_trigger_picker_snapshot]] returns an `Option<TriggerPickerSnapshot>` for any input that routes to menu syntax. Legacy triggers (`~ / @ > ?`), plain fuzzy text, and unknown `;foo` or legacy `+foo` heads return `None`, preserving the parser boundary.

Advanced-query mode (`:`) emits static filter rows (`type:script`, `shortcut:cmd+k`, `source:`, `plugin:`, `name:`, `desc:`, `alias:`, `#`, `tag:`, `has:menuSyntax`, `meta.category:`, `-type:app`, etc.), typo fixes for unknown filter heads (`:typ:script` → `type:script`), recent queries whose raw parse is advanced-query, and an "Open Menu Syntax help" footer action. Concrete rows close the popup on selection; open-value rows like `source:`, `#`, and `tag:` keep it open so the user can type a value.

Capture mode emits one row per known target for bare `;`, a single focused row for exact targets like `;todo`, and slug+label ranked target rows for filtered `;<text>`. If the filter has zero matches, the picker shows an empty target list and only the "Create capture handler for ;<text>…" footer — no fallback to the full target list, so a typed slug like `;gcal` cannot auto-focus an unrelated row. The footer title echoes the canonical `;` capture sigil. Focused exact-target rows and filtered rows both resolve through the same [[src/menu_syntax/trigger_picker.rs#capture_target_catalog]], so dynamic `capture.v1` targets registered by scripts use the same `menuSyntax.label` in `;mca` filtered results and `;mcal` exact-target focus. Handler rows and capture-artifact rows are reserved row kinds for explicit future pickers; they do not render while the user is composing capture text.

Pressing Enter on the create-handler footer writes a local scaffold via [[src/menu_syntax/trigger_picker.rs#create_capture_handler_scaffold]], opens it in the user's editor, shows a HUD, closes the popup, and clears the launcher filter. The scaffold path is `~/.scriptkit/plugins/main/scripts/capture-<slug>-handler.ts`; if that file exists, the dispatcher writes `capture-<slug>-handler-2.ts`, then `-3.ts`, and so on, so existing user code is never overwritten. Pressing Cmd+Enter on the same footer emits `AiScaffoldHandler` and seeds the existing AI prompt surface with the slug, typed `;<slug>` token, and nearest target list instead of writing a file.

When the create-handler footer is the only enabled row (typed slug had zero matches), the main hint surface routes through [[src/menu_syntax/main_hint.rs#capture_create_handler_hint]] instead of the generic capture companion copy. The setup-focused snapshot replaces the misleading generic state with a calmer panel: title `No capture target named ;<slug>`, mode chip `; capture`, status chip `new target`, three rows (`Action` → "Create blank capture handler", `File` → scaffold path, `Registers` → `capture.v1 target "<slug>"`), primary hint `Press Enter to create the handler scaffold.`, secondary hint `Press Cmd+Enter to ask AI to draft the handler first.`, and an empty `examples` list (so the user does not see unrelated `;todo`/`;note`/`;link` examples for an unknown slug). When the typed slug is within one edit of an existing catalog entry, [[src/menu_syntax/trigger_picker.rs#nearest_capture_target_for_slug]] adds a muted `Similar → ;cal — Calendar event` row; the near-miss is non-interactive and never repopulates the popup with rejected fuzzy candidates. `MenuSyntaxMainHintKind::CapturePickerCompanion` stays unchanged so automation consumers do not need a new variant. Pinned by `unknown_slug_no_match_hint_is_setup_focused`, `unknown_slug_no_match_hint_drops_choose_target_copy`, and `known_slug_picker_companion_keeps_examples` in `src/menu_syntax/main_hint.rs::tests`.

Command mode (`!`) emits registered script and scriptlet command rows. Accepting a row inserts `!<slug> `, then continued typing becomes argv/field composition owned by command invocation rather than fuzzy search. Rows with duplicate normalized heads render disabled with a duplicate badge.

## Menu Syntax Trigger Popup

Menu syntax owns a detached popup window anchored below the main filter input, consistent with the ACP `/` slash picker and `@` mention picker. It supersedes the inline takeover and the mode chip.

The popup follows a pure state machine + GPUI entity split. [[src/app_impl/menu_syntax_trigger_popup.rs#plan_trigger_popup_transition]] is the owner-neutral decision function: given the current [[src/app_impl/menu_syntax_trigger_popup.rs#MenuSyntaxTriggerPopupState]], the raw filter text, and a `TriggerPickerContext`, it returns a `TriggerPopupTransition` enum the owner maps onto window lifecycle. Legacy triggers (`~ / @ > ?`) always produce `Close`; advanced-query and capture target prefixes produce `Open` or `Update`; capture body composition (`;todo ` / `;todo Take out trash`) produces `Close` so the target picker does not linger over the composer. Legacy-unclaimed text with no snapshot produces `NoChange`. Selection persistence is by row id so the cursor does not snap back to the top on every filter rebuild, and [[src/app_impl/menu_syntax_trigger_popup.rs#trigger_popup_row_is_default_selectable]] prevents footer actions from becoming the default keyboard selection.

Partial triggers stay open with a filtered snapshot. Capture target input uses the trigger picker's central slug+label scoring before a target is locked, so `;daily` can rank the `;note` Daily note row first even though `daily` is not a slug. [[src/app_impl/menu_syntax_trigger_popup.rs#partial_trigger_snapshot]] still synthesizes popup state when the parser returns None but the raw filter looks like a partial trigger. Capture zero-match filters do not close the popup; they show an empty target list and only a typed-target "Create capture handler for ;<text>…" footer. Advanced-query `:` partials keep their existing row filtering behavior.

The detached GPUI window pages normal rows separately from footer rows. [[src/app_impl/menu_syntax_trigger_popup_window.rs#trigger_popup_visible_row_count]] and [[src/app_impl/menu_syntax_trigger_popup_window.rs#trigger_popup_normal_row_capacity]] keep the footer pinned under the visible normal rows, so long qualifier or command lists still show their explicit action footer without rendering a second footer surface. When an action footer exists, the popup omits the shared synopsis strip below the list so selected-row details cannot appear as a second footer.

When composer-style menu syntax owns the input, the main launcher list is suppressed. Without suppression, fuzzy search for `;t`, capture-handler rows for `;todo Take out trash`, command rows for `!dep`, or the fallback section "Use ';t' with…" would render behind the text-driven composer. `filtered_results`, `get_filtered_results_cached`, and `get_grouped_results_cached` inside `src/app_impl/filtering_cache.rs` short-circuit to empty when [[src/app_impl/menu_syntax_trigger_popup.rs#MenuSyntaxTriggerPopupState#owns_main_list]] says a `;` or `!` popup owns input, or when [[src/menu_syntax/mode.rs#MenuSyntaxMode#capture_composer_owns_input_for]] / `command_owns_input_for` says composition owns input. Advanced-query `:` popups are excluded from this ownership gate so structured search results remain visible. Because grouped results are normally keyed by debounced `computed_filter_text`, the grouped cache also has a live-filter ownership bypass so stale Suggested rows cannot repaint during the coalescing tick. `render_script_list` paints a read-only grammar hint before consulting `item_count`, instead of the normal "No results" empty state, while composer syntax owns input. The grouped cache is explicitly invalidated inside both `handle_filter_input_change` and `set_filter_text_immediate` after popup ownership flips or capture composition starts so real typing and automation `setFilter` receipts share the same visual gate.

## Main Menu Hint Surface

The main menu hint surface makes grammar-owned input discoverable without reintroducing selectable rows.

[[src/menu_syntax/main_hint.rs#build_menu_syntax_main_hint]] builds a pure [[src/menu_syntax/main_hint.rs#MenuSyntaxMainHintSnapshot]] from the live filter text, parser mode, popup snapshot, selected popup row, scripts, and scriptlets. Capture picker states explain the selected `;target`; capture composers preview body, tags, priority, dates, URLs, and fields; command states show registered command resolution, argv, fields, tags, ambiguity, and unknown-head safety. Advanced-query `:` remains search, but bare or partial refine input now renders an `AdvancedQueryGuide`, `:#` teaches tag-filter boundaries, and zero-result structured queries explain the active filters and search words.

[[src/app_impl/menu_syntax_main_hint.rs#ScriptListApp#menu_syntax_main_hint_snapshot]] is the ScriptList adapter. [[src/render_script_list/mod.rs#render_menu_syntax_main_hint]] renders the snapshot as a full-height read-only panel in the list area, not as `GroupedListItem` data. The panel keeps persistent examples open via the snapshot `examples` list, while the normal empty state separately nudges top-level `#tag` search toward `:#tag` filtering or `;... #tag` capture labels. The filtering cache continues to return empty choices for `;` and `!` owned states, so keyboard selection and automation counts cannot target hint content as runnable rows.

The read-only panel owns a dedicated free-scroll handle, so tall capture or command composer hints can scroll without reusing the launcher selection list. While menu syntax owns the main list, ScriptList Up/Down keys scroll this panel before input history or list selection logic runs.

Automation receives the same snapshot through `stateResult.menuSyntaxMainHint`. This lets the agentic-testing harness verify `;todo` capture composer, `!command` invocation, duplicate/unknown command warnings, and advanced-query empty states without scraping pixels or relying on stale launcher rows.

[[src/app_impl/menu_syntax_trigger_popup.rs#adapt_trigger_picker_row]] maps each `TriggerPickerRow` into the neutral [[src/components/inline_picker.rs#InlinePickerRow]] shape shared with the ACP pickers. `FooterAction` rows map to `InlinePickerRowKind::Action`; everything else becomes `InlinePickerRowKind::TextTrigger`. Badges convert to `InlinePickerBadgeTone::Neutral` chips. [[src/app_impl/menu_syntax_trigger_popup.rs#trigger_popup_row_highlight_indices]] computes ACP-style ordered-character highlights from the live `;` / `:` filter, offsetting trigger-prefixed tokens so the typed trigger stays neutral. The popup snapshot carries the just-typed raw filter text so row highlights do not lag behind `InputEvent::Change`. The main launcher input separately accents the active power-syntax prefix via [[src/menu_syntax/mode.rs#prefix_span_for_input_with_targets]], matching `/text` and `@text` input highlighting. The shared `render_soft_compact_picker_row` helper inside `src/components/inline_dropdown` renders every menu-syntax row using the same cell as the ACP popups.

The main launcher input also accents the power-syntax prefix. [[src/menu_syntax/mode.rs#prefix_span_for_input]] supplies the prefix byte span, `src/render_script_list/mod.rs` installs it on the vendored input state, and `vendor/gpui-component/crates/ui/src/input/element.rs` turns that range into plain-text runs colored with the theme accent. This makes `;todo`, `:type:script`, and `>deploy` visually match ACP `/skill` highlighting without changing the stored input text.

Window mechanics live in the binary-only [[src/app_impl/menu_syntax_trigger_popup_window.rs#MenuSyntaxTriggerPopupWindow]] GPUI entity plus a singleton slot. [[src/app_impl/menu_syntax_trigger_popup_window.rs#sync_menu_syntax_trigger_popup_window]] mirrors ACP's `sync_mention_popup_window` line-for-line: open a child NSWindow via `inline_popup_window_options` + `configure_inline_popup_window`, or `set_inline_popup_window_bounds` + `set_snapshot` when the window is already open against the same parent. Closing is idempotent via [[src/app_impl/menu_syntax_trigger_popup_window.rs#close_menu_syntax_trigger_popup_window]]. Open/close transitions fire from `ScriptListApp::handle_filter_input_change` in [[src/app_impl/filter_input_change.rs]] after the state-machine tick — not from `filtering_cache.rs`, which only owns result grouping.

Keyboard dispatch routes `InlinePickerKeyIntent` / `apply_intent` through [[src/menu_syntax/trigger_picker_keys.rs#InlinePickerKeyIntent]] and [[src/menu_syntax/trigger_picker_keys.rs#apply_intent]] at four intercept sites, all gated on `is_menu_syntax_trigger_popup_window_open`: Arrow Up/Down in the `arrow_interceptor` inside `src/app_impl/startup.rs` ahead of the main-list nav; Tab / Shift+Tab and plain Enter in the `tab_interceptor` also inside `src/app_impl/startup.rs` ahead of the ACP routing branches; Escape in `src/render_script_list/mod.rs` ahead of the existing filter-clear → hide-window chain. Capture target Apply/Accept commits `;target ` and closes the picker so continued typing edits the capture body. Once the body exists, [[src/app_impl/selection_fallback.rs#ScriptListApp#execute_selected]] routes Enter through the ranked capture handler instead of main-list selection.

Escape ordering is popup → filter → window: first Escape dismisses the popup and leaves the filter text untouched; second Escape clears the filter; third Escape hides the main window. This order is unique to menu syntax — ACP pickers have their own two-step escape that stays inside the ACP composer.

Parser-boundary guardrails from earlier iters still apply: the popup never opens for `~ / @ > ?`, for plain fuzzy search, or for `;unknown-target` / legacy `+unknown-target`. The state machine's `starts_with_legacy_trigger` check fires ahead of `build_trigger_picker_snapshot`, and `None` snapshots collapse to `Close`.
