# Scripting

Script Kit GPUI runs TypeScript through Bun while keeping the local SDK as a real product surface.

The scripting contract is split between `scripts/kit-sdk.ts`, the generated `kit://sdk-reference` MCP resource, and Bun-driven repo tooling.

## Key Facts

These facts describe the current SDK, resource, and tooling contract.

- `scripts/kit-sdk.ts` still documents itself as a message-passing SDK layered over the Rust app.
- The generated `kit://sdk-reference` resource is the current concise source of truth for exposed script APIs, script directories, and harness workflow.
- The live SDK reference still includes Script Kit helpers such as `exec`, clipboard helpers, filesystem helpers, `getState`, `getElements`, `waitFor`, and `batch`.
- Bun is still used directly in repo-side tooling for process execution and config inspection, especially in `scripts/config-cli.ts` and the `scripts/agentic/` helpers.
- `scripts/lat-llm-key-helper.sh` wraps `LAT_LLM_KEY_HELPER` fetches with a shared on-disk cache so repeated `lat search` calls can reuse one fetched key across terminals.
- Script discovery is plugin-based. `~/.scriptkit/plugins/main/scripts/` is the default personal plugin, and scriptlets are discovered from `~/.scriptkit/plugins/*/scriptlets/*.md`.
- Loaded scripts can be validated at startup via [[src/scripts/validation.rs#validate_script_catalog]], which returns a [[src/scripts/validation.rs#ScriptCatalogReport]] pairing the kept catalog with a [[src/scripts/validation.rs#ValidationReport]]. Duplicate `shortcut` / `alias` / `keyword` / `trigger` declarations across scripts are detected exhaustively by [[src/scripts/validation.rs#detect_binding_collisions]] — every colliding script is emitted as a [[src/scripts/validation.rs#FailedScript]] with `related` pointers at its peers, and the kept catalog excludes them so dispatch never races on ambiguous bindings. [[src/scripts/loader.rs#read_scripts_report]] wraps `read_scripts()` + validation in one accessor.
- The `kit://failed-scripts` MCP resource ships the validation report as a schema-versioned JSON envelope so MCP clients, dev tools, and the in-launcher diagnostics view can enumerate excluded scripts, the duplicate bindings, and each failure's related peers. Served by [[src/mcp_resources/mod.rs#read_kit_failed_scripts_resource]] — it calls `read_scripts_report()` at read time so the response always reflects the current disk state. The pure envelope builder [[src/mcp_resources/mod.rs#build_failed_scripts_document]] is split from the handler so tests can assert envelope shape against hand-built [[src/scripts/validation.rs#ValidationReport]]s without touching the filesystem.
- The launcher surfaces the validation report inline as a pinned "Script Issues" row at the top of the grouped view whenever scripts were excluded. [[src/scripts/types.rs#ScriptIssueMatch]] is the synthetic `SearchResult` variant, [[src/scripts/grouping.rs#prepend_script_issues_row]] inserts it at `flat_results[0]` and shifts every existing [[src/list_item/mod.rs#GroupedListItem]] index by +1 so the grouped items, flat results, and selection index stay aligned. [[src/scripts/grouping.rs#get_grouped_results_with_validation]] wires this through the filter cache so the row also appears when the query hints at `issue / failed / validation / hidden`. Enter on the row calls [[src/app_impl/selection_fallback.rs#ScriptListApp#open_script_issues_view]], which switches the launcher into a read-only [[src/main_sections/app_view_state.rs#AppView]]`::ScriptIssuesView` rendered by [[src/render_prompts/other.rs#ScriptListApp#render_script_issues_view]] against the in-memory [[src/scripts/validation.rs#ValidationReport]] that startup + scriptlet-refresh both populate on `ScriptListApp.script_validation_report`. Inside the view, Escape returns to the script list and Cmd+C copies a markdown diagnostics dump produced by [[src/render_prompts/other.rs#ScriptListApp#format_script_issues_diagnostics]], so colliders are both browsable and shareable without leaving the launcher.
- The launcher also ships an in-product **SDK Reference** browse surface reachable as the `Open SDK Reference` builtin (keywords `sdk`, `api`, `docs`, `scripting`). The entry is a [[src/builtins/mod.rs#BuiltInFeature]]`::SdkReference` variant that routes into [[src/main_sections/app_view_state.rs#AppView]]`::SdkReferenceView`, a split list+preview view rendered by [[src/render_builtins/sdk_reference.rs#ScriptListApp#render_sdk_reference_view]] against the exact same [[src/mcp_resources/mod.rs#SdkFunctionRef]] objects that power `kit://sdk-reference` — so the in-app reference and the agent-facing MCP resource cannot drift. The UI pulls its data via [[src/mcp_resources/mod.rs#sdk_reference_entries_for_ui]], filters rows with the pure case-insensitive matcher [[src/mcp_resources/mod.rs#filter_sdk_reference_entries]] (matches name / signature / description / category), and Cmd+C or Enter copies a markdown card built by [[src/mcp_resources/mod.rs#format_sdk_reference_entry_markdown]] so users can paste a ready-to-use snippet into their editor while writing scripts.
- Every [[src/mcp_resources/mod.rs#SdkFunctionRef]] carries a GPUI support status so the launcher can warn before authors paste APIs the shell does not implement. The [[src/mcp_resources/mod.rs#SdkSupport]] enum serializes as `supported | unsupported | experimental` (lowercase, always emitted) plus an optional `unsupportedNote` pointing at a working alternative. The launcher treats feedback APIs as a deliberate pair: `hud(message)` is fire-and-forget in-launcher overlay, and `notify(message)` is an OS-level Notification Center delivery — both `Supported`, each correct for its own intent. `notify()` is implemented via the [`notify-rust`](https://crates.io/crates/notify-rust) crate from [[src/execute_script/mod.rs#execute_script_dispatch_notify_command]] (which normalizes missing title/body through [[src/execute_script/mod.rs#execute_script_normalize_notify_fields]] and delivers on a dedicated thread so a stuck Notification Center never wedges script-message dispatch). The current `Unsupported` bearer is `menu(...)` — still stubbed on the Rust side — which keeps [[src/mcp_resources/mod.rs#format_sdk_reference_entry_markdown]]'s `> ⚠ Unsupported in GPUI` blockquote and the right-aligned `Unsupported in GPUI` pill in [[src/render_builtins/sdk_reference.rs#ScriptListApp#render_sdk_reference_view]] exercised end-to-end so the warning surface never bit-rots between flag flips. Filter ranking is unchanged — unsupported entries stay discoverable (pinned by `filter_sdk_reference_entries_includes_unsupported_results`). The wire shape is pinned at schema version 5 ([[src/mcp_resources/mod.rs#SDK_REFERENCE_SCHEMA_VERSION]]) with backward deserialization for pre-v5 payloads (`sdk_function_ref_deserializes_old_shape_as_supported`). Starter templates are lint-checked at test time against the live Unsupported list by `script_templates_do_not_reference_unsupported_sdk_apis`, so a future starter cannot silently depend on a stub API. The harness workflow's own embedded example scriptlet and test-script (shipped inside the `kit://sdk-reference` document via [[src/mcp_resources/mod.rs#build_harness_workflow]]) are linted against the same derived needle list by `harness_workflow_examples_do_not_reference_unsupported_sdk_apis`, and the copy-date scriptlet in particular is pinned to `hud(...)` by `harness_workflow_example_scriptlet_uses_hud_for_feedback` — because that example's intent is launcher-local feedback, not Notification Center delivery. The same discipline reaches shipped `kit-init/` guidance: the integration test [[tests/kit_init_unsupported_sdk_audit.rs]] walks every `.md` / `.ts` / `.tsx` / `.js` / `.jsx` file under `kit-init/` and fails the build if a runnable line calls `name(` for any SDK entry that the live `kit://sdk-reference` still marks `Unsupported`. The guard auto-retargets whenever an entry flips Supported (as `notify` did in this iteration) without any test-file edits — it pulls the Unsupported name list from [[src/mcp_resources/mod.rs#sdk_reference_entries_for_ui]] at runtime. Coverage is pinned by `kit_init_shipped_examples_do_not_call_unsupported_sdk_apis`. Marking coverage is pinned by `sdk_reference_marks_notify_as_supported_system_notification_api` (the SDK Reference advertises `notify()` as a real OS-notification API and contrasts it with `hud`), `sdk_reference_marks_every_documented_unsupported_api`, `sdk_support_serde_roundtrips_lowercase`, `format_sdk_reference_entry_markdown_warns_for_unsupported`, `format_sdk_reference_entry_markdown_does_not_warn_for_supported`, and `sdk_reference_supported_count_exceeds_unsupported_count`. `notify()`'s dispatch path itself is pinned by `test_execute_script_normalize_notify_fields_passes_through_title_and_body` (preserves round-trip strings including AppleScript-hostile `"` and `\`) and `test_execute_script_normalize_notify_fields_defaults_missing_fields` (title-only → reuses title as body, body-only → defaults to `Script Kit` title, both-empty → short-circuits).
- Starter-script creation leans on a Rust-owned template catalog served by the `kit://script-templates` MCP resource. Harness agents and the launcher's forthcoming catalog view both consume the same [[src/mcp_resources/mod.rs#ScriptTemplateRef]] objects — so a newly-created script from MCP and a newly-created script from the launcher UI produce byte-identical bodies. The catalog envelope ([[src/mcp_resources/mod.rs#ScriptTemplatesResourceDocument]]) is built by [[src/mcp_resources/mod.rs#build_script_templates_document]], the UI slice by [[src/mcp_resources/mod.rs#script_template_entries_for_ui]], filtering by the pure matcher [[src/mcp_resources/mod.rs#filter_script_template_entries]] (title / description / category), file instantiation by [[src/mcp_resources/mod.rs#render_script_template_file]] (substitutes `{{NAME}}` / `{{DESCRIPTION}}`), and the preview-pane markdown by [[src/mcp_resources/mod.rs#format_script_template_markdown]]. v1 templates ship with `blank-starter` (the current arg→div shape, preserved as row #1 so the fastest "new script" path feels identical to the pre-catalog experience) and `choice-list` (`arg("Pick one", ["A","B","C"])`). Every v1 template intentionally omits `alias:` / `shortcut:` / `keyword:` / `trigger:` — the fields [[src/scripts/validation.rs#detect_binding_collisions]] treats as fatal — so newly-created scripts cannot land on disk in a hidden-by-validation state. The `starter_templates_do_not_emit_collision_binding_fields` test in [[src/mcp_resources/mod.rs]] enforces that invariant at compile+test time.
- Script creation is split across two launcher commands with disjoint discovery terms. `builtin/new-script` is the direct fast path: the entry is named `New Script`, carries only direct creation keywords (`new`, `script`, `create`, `blank`, `typescript`, `code`), routes through `BuiltInFeature::ScriptCommand(ScriptCommandType::NewScript)` into the existing naming-prompt flow, and creates a blank script. The separate `builtin/new-script-from-template` command owns every template-discovery keyword (`template`, `starter`, `boilerplate`, `scaffold`, `choice`, `arg`, plus shared `new`/`script`) and is the ONLY command that can open the template catalog. The split is pinned at test time by `new_script_fast_path_label_does_not_claim_template` (fast-path name/description cannot mention template/starter), `new_script_fast_path_does_not_own_template_keywords` (fast-path keywords cannot include `template`/`starter`/`boilerplate`/`scaffold`), `new_script_from_template_owns_template_discovery_terms` (template path keywords MUST include all discovery terms), and `builtins_include_new_script_fast_path_and_template_path` (both commands exist and route to their correct features). This guarantees a user searching `template` cannot accidentally land on the blank fast path, and a user searching `new script` gets the fast path they expect. The template command ([[src/builtins/mod.rs#BuiltInFeature]]`::NewScriptFromTemplate`) routes into [[src/main_sections/app_view_state.rs#AppView]]`::ScriptTemplateCatalogView`, a filterable split list+preview rendered by [[src/render_builtins/script_templates.rs#ScriptListApp#render_script_template_catalog_view]] against the same [[src/mcp_resources/mod.rs#ScriptTemplateRef]] objects the MCP resource ships. Enter on a row hands the chosen template to [[src/app_impl/naming_dialog.rs#ScriptListApp#show_naming_dialog_for_script_template]], which reuses the standard [[src/prompts/naming/prompt.rs#NamingPromptConfig]] flow — the chosen template surfaces in the naming dialog's new `Template` section ([[src/prompts/naming/prompt.rs#TemplateSelection]]) and rides the submit payload through as `template_id` ([[src/prompts/naming/validation.rs#build_submit_payload]]). On submit, [[src/app_impl/naming_dialog.rs#ScriptListApp#handle_naming_dialog_completion]] resolves the id back to a template via [[src/mcp_resources/mod.rs#find_script_template]] and overwrites the freshly-created script file with [[src/mcp_resources/mod.rs#render_script_template_file]]'s output before opening the editor — create-then-overwrite sequencing guarantees the editor never reads the file mid-write and the `{{NAME}}`/`{{DESCRIPTION}}` substitutions always produce byte-identical output to the MCP path. Cmd+C on the catalog copies the markdown card, and Esc returns to the launcher; see [[src/render_builtins/script_templates.rs]] for the footer hints and key context. Because every template omits the collision-prone binding fields enforced by the `starter_templates_do_not_emit_collision_binding_fields` test in [[src/mcp_resources/mod.rs]], scripts generated from the UI path cannot land hidden by [[src/scripts/validation.rs#detect_binding_collisions]] either.

## Runtime Contract

These rules describe the stable runtime boundary between scripts and the app.

- Scripts talk to the Rust shell through JSON messages rather than embedding UI logic locally.
- `kit://sdk-reference` is generated from Rust code, which makes it a better current reference than older narrative migration docs.
- Bun-native process helpers and the local SDK coexist. Repo tooling uses Bun directly where that is simpler, while scripts still rely on the Script Kit APIs for prompts, automation, clipboard, and app integration.

## LAT LLM Key Helper Cache

The repo ships a shell helper that can stand in for `LAT_LLM_KEY_HELPER` and reuse one fetched key across terminal sessions.

Set `LAT_LLM_KEY_FETCH_CMD` to the command that prints the key, such as the current `op item get ... --reveal` form, then point `LAT_LLM_KEY_HELPER` at `scripts/lat-llm-key-helper.sh`.

On macOS the helper prefers the login Keychain for the cached secret and only falls back to the shared cache file on platforms where Keychain is unavailable or when `LAT_LLM_KEY_CACHE_BACKEND=file` is set.

Use `LAT_LLM_KEY_CACHE_TTL_SECONDS` to control how long the shared cache lives, `LAT_LLM_KEY_CACHE_FILE` to move the file backend, and `scripts/lat-llm-key-helper.sh --clear` to force the next lookup back through 1Password.
