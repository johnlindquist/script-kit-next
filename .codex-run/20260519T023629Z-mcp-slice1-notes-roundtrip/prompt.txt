Runtime safety for shell calls:

Do not use blind 30000ms foreground waits for slow build/test/install commands.

For cargo build/test in Rust/GPUI repos:
- Prefer a background shell wrapper that writes .codex-run/<run>/pid, status, and output.log.
- Launch wrapper with exec_command yield_time_ms=1000 and max_output_tokens=12000.
- Poll through fresh exec_command status probes every 60s, yield_time_ms=1000, max_output_tokens=80000.
- Total cap: 15m incremental; 25m cold/full.

For npm install or pnpm install:
- Use the same wrapper.
- Poll every 45s.
- Total cap: 12m normally; 20m cold/no node_modules.

For quick checks:
- exec_command yield_time_ms=30000, max_output_tokens=30000.
- If a session remains open, at most two write_stdin polls with yield_time_ms=5000.

For unknown commands:
- exec_command yield_time_ms=60000, max_output_tokens=60000.
- If still running after three write_stdin polls of 10000ms and no known live pid exists, stop with STUCK_SESSION.

Before every write_stdin poll, prove liveness through a known pid. If the tool does not expose a pid, create one with a shell wrapper.
If three consecutive polls return no output and no known live pid exists, stop polling and print STUCK_SESSION reason=blind_empty_poll_budget.
If a known pid is dead and no status file exists, print STUCK_SESSION reason=dead_pid_without_status.
