
012 Root Unified Source Filters / Source Chips / Lazy Paging Executive Summary Root Unified Source Filters are declarative launcher filters that turn ordinary root launcher input into a scoped source query. The committed source-head model is simple: a standalone trailing-colon token such as files:, f:, clipboard:, c:, tabs:, t:, commands:, cmd:, conversations:, ai:, vault:, v:, dictation:, d:, or windows: is parsed as a source selector, stripped out of the user’s search text, and recorded as an include or exclude gate. Attached query text is also supported, so c:skip, clipboard:skip, f:s, files:s, h:https://example.com, and spaced forms like c: skip all search the selected source using the stripped suffix. Leading : is not committed source syntax; it is the discovery/refine picker path. Quoted source-looking tokens and unknown heads stay literal. root-source-filters-atlas The core contract is not “show a different menu.” It is “run the root launcher through a source-filter frame.” In source-filter mode, the launcher keeps the ScriptList surface, but grouping suppresses disallowed sources, fallback rows, and unrelated primary intent. Positive source heads explicitly enable their source for the current stripped query even when that source is disabled for ordinary passive root search. Passive/local source queries use direct lookup for explicit filters rather than warming a future cache-only frame. File and passive frames are keyed by both stripped query and source-filter set, preventing late async results from bleeding between source modes. root-source-filters-atlas Files have special paging behavior. Explicit Files filters start with a source-chip page budget, allow one-character ASCII alphanumeric queries like f:s, and expand when keyboard selection nears the last few Files rows. This must happen without pressing Enter, without leaving the selected file below the footer, and without snapping selection back to the top when provider rows arrive. root-source-filters-atlas Status/source-chip rows are metadata, not launcher choices. They can say “Loading more…”, “No results”, “Showing N of M”, “No more results”, or disabled/provider-unavailable states, but they are not SearchResults, cannot be selected, cannot be action subjects, and must not inflate ScriptList row counts or scroll height. They are still exposed through automation as role:"status" and kind:"sourceStatus" so tests can prove source paging and exhaustion without screenshots. root-source-filters-atlas root-source-filters-atlas Processes are the only major mismatch in this snapshot. src/menu_syntax/source_heads.rs has a descriptor for processes: / p:, but the parser contract and source-filter tests still treat processes: and p: as uncommitted until root process rows exist. A negative-only source filter such as -files: png is different: it allows every source except Files, but still drops fallback rows because they have no root source identity. The ScriptList input is plain text. The launcher performs primary fuzzy search, optional root Files/passive append, and fallback rows. Discovery/refine state. Input begins with :. This opens filter discovery or advanced-query refinement. It does not commit :f or :files as source syntax. Complete predicates such as :type:script deploy become advanced queries; bare or partial states remain picker/hint states. root-source-filters-atlas Source-filter parse state. Input contains at least one recognized source head outside quotes. parse_filter_query tokenizes input, classifies source heads and advanced predicates, strips attached query suffixes into free text, and returns an AdvancedQuery. Unknown filter-looking tokens do not claim the input. root-source-filters-atlas Source-filter set state. The app stores include/exclude gates in RootUnifiedSourceFilterSet. Positive includes create a selected-source set. Excludes remove sources before include matching. Negative-only filters remain active filters but do not satisfy source-only browse because browse requires positive includes. root-source-filters-atlas Config override state. Explicit positive source heads override source options for that query. Files become enabled with global search, directory browse, recent files, explicit query intent, and source-chip visible limit. Passive/local sources set enabled = true, min_query_chars = 0, and raise max results to the explicit target. root-source-filters-atlas Frame acquisition state. Passive and file results are read through query frames. The passive key includes query, advanced flag, source filters, and per-source options. The file key includes query, advanced flag, source filters, file search mode, and file options. root-source-filters-atlas root-source-filters-atlas Grouping state. The grouping engine builds normal results, filters them by root sources if any source filter is active, appends Windows, Files, AI Vault status, and passive sections only when allowed, then appends source-chip status rows. Fallback handoff rows are suppressed in source-filter mode. root-source-filters-atlas root-source-filters-atlas Browse state. If stripped search text is empty, no advanced predicate is active, and there is at least one positive include, SourceFilterBrowseMode may raise browse targets and publish a hint snapshot such as warming, typeToLoadMore, or showingRecent. root-source-filters-atlas root-source-filters-atlas Render/receipt state. 012 Root Unified Source Filters / Source Chips / Lazy Paging Executive Summary Root Unified Source Filters are declarative launcher filters that turn ordinary root launcher input into a scoped source query. The committed source-head model is simple: a standalone trailing-colon token such as files:, f:, clipboard:, c:, tabs:, t:, commands:, cmd:, conversations:, ai:, vault:, v:, dictation:, d:, or windows: is parsed as a source selector, stripped out of the user’s search text, and recorded as an include or exclude gate. Attached query text is also supported, so c:skip, clipboard:skip, f:s, files:s, h:https://example.com, and spaced forms like c: skip all search the selected source using the stripped suffix. Leading : is not committed source syntax; it is the discovery/refine picker path. Quoted source-looking tokens and unknown heads stay literal. root-source-filters-atlas The core contract is not “show a different menu.” It is “run the root launcher through a source-filter frame.” In source-filter mode, the launcher keeps the ScriptList surface, but grouping suppresses disallowed sources, fallback rows, and unrelated primary intent. Positive source heads explicitly enable their source for the current stripped query even when that source is disabled for ordinary passive root search. Passive/local source queries use direct lookup for explicit filters rather than warming a future cache-only frame. File and passive frames are keyed by both stripped query and source-filter set, preventing late async results from bleeding between source modes. root-source-filters-atlas Files have special paging behavior. Explicit Files filters start with a source-chip page budget, allow one-character ASCII alphanumeric queries like f:s, and expand when keyboard selection nears the last few Files rows. This must happen without pressing Enter, without leaving the selected file below the footer, and without snapping selection back to the top when provider rows arrive. root-source-filters-atlas Status/source-chip rows are metadata, not launcher choices.
