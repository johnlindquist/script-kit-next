Feature Atlas 031 — Legacy chat() Prompt 1. Capability summary and ACP boundary Feature 031 is the legacy Script Kit SDK chat() prompt surface. It opens a GPUI conversational prompt from TypeScript scripts and supports two major operating modes: Script callback mode — the script owns conversation behavior through onInit, onMessage, and the chat.* controller helpers. Built-in AI mode — the Rust/GPUI runtime owns provider selection, model selection, streaming, setup prompts, slash/context transforms, Mini AI reuse, and handoff into the full AI window. This feature is not the ACP ai* API family. The ACP APIs are a separate agent/AI surface. This chapter should treat legacy chat() as its own prompt type with its own protocol messages, UI state, routing rules, persistence behavior, tests, and gaps. The sharpest boundary: chat() is a prompt UI plus conversational message controller. ACP ai* APIs are newer AI/agent APIs and should not be documented as equivalent. Built-in AI mode inside chat() may call configured AI providers, but that does not make it ACP. Callback mode can be fully script-defined and may not use any built-in provider at all. A practical way to reason about the feature: TypeScript script -> SDK chat() entry -> protocol Message::Chat -> Rust prompt handler -> ChatPrompt GPUI view -> either script callback routing or built-in provider streaming The important implementation detail is that useBuiltinAi is true only when neither onInit nor onMessage is supplied. That means an onInit-only chat is not automatically a built-in AI chat. It is a script-managed chat surface unless the script itself drives responses through the controller helpers. 2. It emits a protocol payload shaped like: { type: "chat", id, placeholder, messages, hint, footer, actions, model, models, saveHistory, useBuiltinAi} The SDK decides useBuiltinAi as follows: useBuiltinAi = true only when onInit is absent and onMessage is absent Activation modes: Script optionsRuntime modeExpected ownershipNo onInit, no onMessageBuilt-in AI modeRust/GPUI provider registry and ChatPrompt own AI responseonMessage presentCallback modeScript receives submits and sends replies/streamsonInit presentCallback-style/script-managed modeScript can seed messages or start work after prompt opensonInit + onMessage presentFull callback modeScript initializes and handles each user submit Practical warning: onInit disables built-in AI because useBuiltinAi is false when onInit exists. User stories 3.1 Script author opens a basic built-in chat A script calls chat() without onInit or onMessage. The SDK emits Message::Chat with useBuiltinAi: true. Rust creates ChatPrompt, loads provider/model state, and the user can converse with the configured provider. Expected behavior: User types text. Empty input without image is ignored. Submit expands context mentions and slash-command transforms. ChatPrompt adds the user message. ChatPrompt creates an empty streaming assistant message. ChatPrompt selects provider/model. Streaming reveal begins. Missing provider/model creates an assistant error message instead of silently failing. 3.2 Script author owns the whole conversation A script calls chat({ onMessage }). Expected behavior: useBuiltinAi is false. User submit is routed to SDK as ChatSubmit. SDK appends the user message locally. SDK awaits onMessage. Script can call: chat.addMessage chat.startStream chat.appendChunk chat.completeStream chat.setError chat.clearError chat.clear The listener remains active for more submits. 3.3 Script initializes prompt, then controls responses A script calls chat({ onInit, onMessage }). Expected behavior: Prompt opens first. SDK awaits onInit after opening. Script can seed messages, show an initial assistant message, start a stream, or prepare external state. Later submits route to onMessage. 3.4 User opens Mini AI before configuring providers The runtime opens inline-ai-setup. Expected behavior: User sees setup choices, including Configure and Claude Code. Setup keyboard controls work across setup buttons. Configure opens the API key prompt. Claude Code enables config path. Escape exits setup/inline mode according to Mini AI close behavior. 3.5 User continues inline Mini AI in full AI window Expected behavior: Inline conversation is not saved as a duplicate when transferring. Runtime opens the AI window with pending messages. Main window is hidden or removed according to handoff path. The transferred conversation should preserve context as pending messages. 3.6 User manages chat through actions Expected behavior: Cmd+K opens actions for ChatPrompt. User can select model. User can copy the last response. User can clear the conversation after destructive confirmation. User can capture screen area. User can expand/continue into full chat when supported by the current mode. 4.
