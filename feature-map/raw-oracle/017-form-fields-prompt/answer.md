017 Form and Fields Prompt / form() / fields() Executive Summary form() is an implemented SDK prompt surface. The TypeScript SDK sends a FormMessage with { type: "form", id, html, actions }, Rust routes it to PromptMessage::ShowForm, builds a FormPromptState, installs AppView::FormPrompt, focuses the form prompt, sizes the window based on discovered fields, renders native GPUI form controls, and submits a JSON object keyed by field names. fields() is SDK-visible but not currently a full GPUI prompt surface in the provided source. The SDK sends { type: "fields", id, fields, actions } and resolves a JSON array response, but the Rust app state has no AppView::FieldsPrompt. Instead, the visible Rust-side support in this bundle is PromptMessage::FieldsComingSoon { id, field_count }, whose handler logs an unsupported/stubbed warning and shows a “fields() prompt coming soon” toast. Older SDK tests and FORM_FIELDS_PARITY_REPORT.md describe fields() as blocked or unimplemented. The exact current conversion from wire Message::Fields to FieldsComingSoon is not included in the bundle, so the safe product truth is: fields() exists in the SDK, but GPUI backend rendering/submission is incomplete or intentionally stubbed. form() and fields() share the developer-facing concept of structured input collection, but their current product status differs sharply. form() owns a real prompt entity, footer surface, focus target, actions dialog host, automation identity, validation path, and element collection. fields() currently has SDK message/runtime support and tests documenting intended behavior, but no complete Rust prompt entity or rendered surface. What Users Can Do Users can call await form(html) from SDK scripts to show a native GPUI form generated from HTML. Supported or mentioned controls include text-like <input> fields, password, email, number, date, time, datetime-local, month, week, url, search, tel, color, <textarea>, checkbox, and select. The parity report says hidden, submit, and button inputs are intentionally skipped, while radio, range, and file are not supported. Users can type into form fields, tab between fields, click fields to focus them, enter multiline text in textareas, submit with Enter from normal fields, and submit with Command+Enter from textarea fields. A bare Enter inside a textarea is forwarded to the field as a newline instead of submitting. Users can provide SDK actions to form(html, actions). Those actions are stored as SDK actions, shown through the shared actions dialog route, and owned by ActionsDialogHost::FormPrompt. Users can call await fields(fieldDefs) from SDK scripts. The SDK accepts either strings or FieldDef objects, normalizes string fields into { name, label }, sends a fields message, and expects an array of strings back. Current GPUI behavior should not be treated as implemented; source shows a coming-soon handler rather than a real fields prompt. Core Concepts FormPrompt is the implemented Rust prompt entity for form(). AppView::FormPrompt { id, entity } stores the SDK prompt id and an entity containing FormPromptState. It is categorized as SurfaceKind::PromptEntity, receives FocusTarget::FormPrompt, and has a native footer surface id of "form_prompt". FormPromptState is created from SDK HTML plus theme-derived FormFieldColors. The bundle shows FormPromptState::new(id.clone(), html, colors, cx), followed by form_state.fields.len() to determine window sizing. The parser implementation itself is not included in full, so exact parsing details beyond tests/report snippets should be treated as bundle-backed but not exhaustively proven. fields() is an SDK API and wire message shape, not a complete Rust prompt entity in the provided source. There is no AppView::FieldsPrompt in app_view_state.rs, and the Rust prompt message visible in this bundle is FieldsComingSoon. Entry Points globalThis.form = async function form(html: string, actionsInput?: Action[]): Promise<Record<string, string>> lives in scripts/kit-sdk.ts. It generates an id, serializes actions if provided, sends a FormMessage, waits for a prompt response, parses msg.value as JSON, and resolves an object or {} on parse/type failure. globalThis.fields = async function fields(fieldDefs: (string | FieldDef)[], actionsInput?: Action[]): Promise<string[]> also lives in scripts/kit-sdk.ts. It generates an id, maps string field definitions to { name: f, label: f }, serializes actions, sends a FieldsMessage, waits for a response, parses msg.value as JSON, and resolves an array or [] on parse/type failure. Rust form entry is PromptMessage::ShowForm { id, html, actions } in src/prompt_handler/mod.rs. It prepares the window, stores SDK actions, derives form field colors from theme, creates FormPromptState, creates the GPUI entity, sets current_view = AppView::FormPrompt, clears launcher input focus, sets pending focus to FormPrompt, resizes, and notifies. Rust fields entry in the visible bundle is PromptMessage::FieldsComingSoon { id, field_count }, which logs state = "stubbed" and calls show_prompt_coming_soon_toast("fields()", cx). Rendering entry is render_impl.rs, where AppView::FormPrompt { entity, .. } calls self.render_form_prompt(entity, cx). Element collection entry is collect_elements.rs, where AppView::FormPrompt delegates to collect_form_prompt_elements. User Workflows A normal form() workflow starts when the script calls form(html). The SDK sends a form message with a prompt id. Rust switches to AppView::FormPrompt, focuses the prompt entity, and sizes the window as ArgPromptWithChoices when fields exist or DivPrompt when no fields are found. The user fills fields, moves with Tab or Shift+Tab, and submits. The SDK receives a JSON object and resolves it to the script. A textarea workflow differs at Enter. If the focused field type is textarea, plain Enter is forwarded to the field, while Command+Enter submits. This is encoded in form_enter_behavior: non-enter keys are ignored, textarea+no-command forwards to field, otherwise Enter submits. A validation workflow runs on submit. Before submit, collect_form_submit_validation_errors(form, cx) checks field values. Email fields use email validation, number fields use number validation, and other field types are accepted. If validation errors exist, the handler shows a HUD message for a longer duration and does not submit. An actions workflow starts when a form is created with actions. SDK actions are serialized and stored. The form render path uses shared prompt key preamble handling with ActionsDialogHost::FormPrompt. When the actions dialog is open, a shared backdrop with id "form-actions-backdrop" can dismiss it, preserving FormPrompt host routing. A forced submit workflow can be triggered by protocol automation. BatchCommand::ForceSubmit and PromptMessage::ForceSubmit include AppView::FormPrompt in the list of prompt views that can submit by current prompt id. The forced value is converted to a string from string/null/JSON and submitted through the generic prompt response path. This bypasses normal user editing semantics; whether it bypasses form validation is likely yes from the visible code path, but the exact submit function behavior is outside the snippets. A fields() workflow currently starts in SDK the same way: the script calls fields([...]), the SDK sends a fields message, and it waits for a JSON array. In GPUI, the source shown does not install a fields view. The expected current visible result is a coming-soon toast if the message is routed to FieldsComingSoon; older tests report an unhandled-message failure. Treat both as evidence of incomplete support, with the precise current wire mapping as a proof gap. When no fields are parsed, ShowForm resizes as ViewType::DivPrompt; when fields exist, it uses ViewType::ArgPromptWithChoices. That does not mean it becomes a div or arg prompt; it remains AppView::FormPrompt, but uses those sizing profiles. Keystrokes And Commands Enter submits from non-textarea form fields. Enter in a textarea inserts/forwards input and does not submit. Command+Enter submits from a textarea. Tab focuses the next field. Shift+Tab focuses the previous field. Global shortcuts are processed first through handle_prompt_key_preamble_default with ActionsDialogHost::FormPrompt. The renderer passes stop_propagation_on_global_shortcut: false and stop_propagation_when_handled: false, so propagation behavior is intentionally permissive compared with stricter prompt types. Actions And Menus form() accepts optional SDK actions. The SDK stores handlers in __kitActionsMap, clears the map for new action input, de-duplicates with a seen set, serializes actions, and attaches them to the form message. Rust stores form actions in self.sdk_actions = actions during ShowForm. The form renderer advertises action availability through self.has_nonempty_sdk_actions() in prompt chrome audit. Actions dialog ownership is explicit: form key handling uses ActionsDialogHost::FormPrompt, and the backdrop config also uses ActionsDialogHost::FormPrompt. ActionsDialogHost includes a FormPrompt variant in app_view_state.rs. The form actions backdrop uses shared helper render_actions_backdrop, backdrop id "form-actions-backdrop", log message "Form actions backdrop clicked - dismissing dialog", and show_pointer_cursor: true. Tests assert these invariants. Automation And Protocol Surface current_prompt_type() returns "form" for AppView::FormPrompt. This is the automation identity for active form prompts. getState prompt identity uses AppView::FormPrompt as "form" with id and default prompt metadata in the visible tuple. The full state snapshot shape is outside the snippets, but form is included in prompt classification. getElements supports form prompt. collect_elements.rs maps AppView::FormPrompt into collect_form_prompt_elements, finalizes it under "form-prompt", and starts the element list with "form-fields". ForceSubmit supports form prompt in both batch protocol and prompt message handling. AppView::FormPrompt is included beside arg, div, term, and editor prompt in prompt id lookup. Values are stringified from string/null/other JSON before submission. Protocol submit smoke tests include a submit-fields-form case that types John, tabs, types Doe, then taps cmd+enter. Data, Storage, And Privacy Boundaries form() receives raw HTML from the SDK script and parses it into native field definitions/entities. That HTML is prompt-local input data. The bundle does not show persistent storage for form HTML. Submitted form() values are collected into a JSON object with field names as keys. Password fields are masked visually per parity report, but submitted password values are still plain field values in the prompt response payload. They should be treated as sensitive. fields() submitted values, when implemented, are expected by the SDK as a JSON array of strings. Since backend rendering is incomplete, callers should not rely on this flow in GPUI. Validation errors expose field identity/type enough to show a HUD message. They should not include sensitive field values unless helper code outside this bundle does so; visible validation code collects invalid field labels/types, not values. SDK actions may carry action metadata and are stored in the process action map. Action handlers are not serialized directly; serializable action descriptors are sent to Rust. The action map is cleared when new actions are supplied for form() or fields(). Checkbox values are represented as form values; exact serialization as "true"/"false" or another string is not proven in snippets, though the parity report says checkbox is a boolean-value form element. Treat exact wire value as a proof gap. Error, Empty, Loading, And Disabled States Invalid email submit value blocks submit and shows HUD. Invalid number submit value blocks submit and shows HUD. Unknown or unsupported text-like input types mostly pass through as text-field-like controls for form() according to the parity report. Exact behavior for malformed HTML or malformed field attributes is not visible. No-field form() prompts still render as AppView::FormPrompt, but resize using ViewType::DivPrompt. This is the closest visible empty state. fields() has an explicit unsupported/stubbed state in current Rust prompt handling: FieldsComingSoon logs a warning with prompt "fields()", id, field count, and state "stubbed", then shows a coming-soon toast. Tests assert the warning text is "fields() prompt coming soon." SDK response parsing is defensive. form() resolves {} if response JSON is missing, invalid, or not an object. fields() resolves [] if response JSON is missing, invalid, or not an array. Disabled states for individual form submit buttons are not a major concept because submit/button inputs are skipped and the prompt owns submission. There is no evidence of a disabled form footer state in the snippets. Loading state is not explicitly represented for forms in the provided source. FormPrompt construction appears synchronous after receiving the message. Invariants And Regression Risks form() must remain AppView::FormPrompt, not fall back to launcher, div, or arg prompt identity. current_prompt_type() must return "form" for active forms. native_footer_surface() must return "form_prompt" for active forms. FocusTarget::FormPrompt must be set when showing a form, or typed input can route to the wrong surface. Clicking form fields must synchronize parent focus state, or key routing can follow stale focus. Textarea Enter behavior must not regress: plain Enter adds a newline; Command+Enter submits. Validation must block invalid email and number fields before prompt submission. Password fields must remain visually masked, but agents must remember visual masking is not data secrecy after submit. SDK action routing must preserve ActionsDialogHost::FormPrompt; otherwise actions may run against the wrong host or backdrop. fields() must not be documented as implemented until a real Rust view, prompt message, renderer, focus target, element collection, and submit path exist. The parity report is partly historical. It says fields fell through as unhandled; current snippets show FieldsComingSoon. Agents should check current routing before making claims or changing behavior. Color/date/time fields currently render as text-field-like controls, not native pickers. Adding native pickers would change UX, validation, automation element shape, and tests. Agent Notes Before changing this feature, search/expand the architecture graph around form prompt, fields prompt, prompt runtime, protocol automation, and verification. lat.md/design.md already records that native form() field colors and focus synchronization are part of the design contract. Do not implement fields() by pretending it is form() unless the product decision is to translate field definitions into form HTML or shared form state. A real implementation needs a clear owner: either reuse FormPromptState from structured field definitions or create a separate FieldsPrompt with equivalent focus, footer, actions, validation, automation, and submit behavior. Do not trust older reports blindly. FORM_FIELDS_PARITY_REPORT.md is useful, but current source shows FieldsComingSoon, which may be newer than the “falls through to unhandled” statement. When adding new field types, update all layers together: SDK FieldDef typing, wire message shape if needed, parser or field conversion, renderer component, validation, element collection, tests, and lat.md. When working with password fields, avoid logs or screenshots that reveal submitted values. The visual field may be masked, but automation and SDK responses can still carry secrets.
