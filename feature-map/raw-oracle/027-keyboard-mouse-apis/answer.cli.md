027 Keyboard and Mouse APIs Executive Summary The current Keyboard and Mouse APIs are SDK-visible stubs, not reliable native input controls. The SDK exposes: keyboard.type(text: string): Promise<void> keyboard.tap(...keys: string[]): Promise<void> mouse.move(positions: Position[]): Promise<void> mouse.leftClick(): Promise<void> mouse.rightClick(): Promise<void> mouse.setPosition(position: Position): Promise<void> But every one of those helpers currently logs a warning saying it is not yet implemented in the GPUI app, sends a fire-and-forget message, and resolves without waiting for any backend action or receipt. The relevant SDK implementation is in scripts/kit-sdk.ts:2731-2792. The Rust protocol has typed Message::Keyboard and Message::Mouse variants, plus KeyboardAction, MouseAction, and constructors in src/protocol/message/variants/system_control.rs, src/protocol/types/primitives.rs, and src/protocol/message/constructors/general.rs. However, the SDK payload shapes do not cleanly match the Rust protocol shapes. For example, SDK keyboard.tap() sends keys: string[], while the Rust protocol variant expects keys: Option<String>. SDK mouse helpers send positions, button, or position, while the Rust protocol expects a single optional data: MouseData. Operator truth: do not use SDK keyboard.* or mouse.* as proof of UI behavior, prompt mutation, native macOS input, terminal typing, mouse movement, or click delivery. The current reliable automation alternatives are protocol simulateKey, batch.setInput, waitFor, getState, and getElements, depending on the workflow. The main risk is false confidence. The repo contains many smoke tests and benchmarks that call keyboard.tap() or keyboard.type(), but several either skip real system input, tolerate any string result, document manual fallback, or explicitly expect the API to fail. These tests are not evidence that the SDK keyboard/mouse APIs generate real input. What Users Can Do Users can call the SDK helpers from a Script Kit GPUI script. They can write script logic that invokes keyboard.type("hello"), keyboard.tap("enter"), mouse.leftClick(), and similar helpers. The script will not block waiting for a backend confirmation. The Promise normally resolves after the SDK sends the message. What users cannot currently rely on from the provided source context: Text being typed into the active prompt. Arrow keys moving selection. Enter submitting a prompt. Escape dismissing a prompt. Cmd+K opening the Actions dialog. Mouse cursor movement. Left or right clicks. Mouse position updates. Any receipt proving that a keyboard or mouse action landed. Any native macOS event delivery. Any focus guarantee. Core Concepts 1. SDK exposure is not backend implementation The SDK defines global objects and functions. That only proves scripts can call them. It does not prove the app processes the messages, generates native events, routes input to the focused prompt, or returns receipts. For this feature, the SDK is ahead of the runtime. 2. Protocol-visible messages are not native input Rust defines Message::Keyboard and Message::Mouse. That means keyboard and mouse messages exist as protocol shapes. It does not prove the app has a native-event handler, macOS Accessibility integration, focus targeting, or a result envelope. The provided backend excerpts show message variants and constructors, not an execution path that dispatches to native input. 3.
