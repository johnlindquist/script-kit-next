030 ACP Chat SDK APIs Executive Summary The ACP Chat SDK APIs are script-facing TypeScript globals in scripts/kit-sdk.ts that send JSON protocol messages to the Rust/GPUI app over stdout/stdin. The Rust protocol surface is defined in src/protocol/message/variants/ai.rs, with shared result data types in src/protocol/types/ai.rs. The currently proven end-to-end APIs are: aiIsOpen() aiGetActiveChat() aiListChats(limit?, includeDeleted?) aiGetConversation(chatId?, limit?) aiStartChat(message, options?) aiFocus() aiGetStreamingStatus(chatId?) aiDeleteChat(chatId, permanent?) The storage-backed APIs that do not require the AI window to be open are: aiGetActiveChat() with fallback to most recently updated chat when no active window state exists aiListChats() aiGetConversation() aiDeleteChat() aiGetStreamingStatus() reads a global SDK-visible streaming snapshot, not storage, and does not require opening the window The UI-thread APIs are: aiStartChat() — opens/focuses the AI window, creates a script-sourced chat, saves optional system/user messages, attaches optional image/context parts, and optionally starts streaming. aiFocus() — opens/focuses the AI window and returns whether it was already open. Declared but not proven handled by the app: aiAppendMessage() aiSendMessage() aiSetSystemPrompt() aiOn() aiSubscribe aiUnsubscribe pushed events: aiStreamChunk, aiStreamComplete, aiNewMessage, aiError These have SDK/protocol shapes and SDK-local bookkeeping, but the provided Rust dispatch only routes AiStartChat and AiFocus to UI prompt messages, and direct handlers explicitly return None for append/send/system-prompt/subscribe/unsubscribe. What Users Can Do A Script Kit script can inspect ACP Chat state, read stored chat metadata, read stored messages, start a new ACP Chat conversation, focus the ACP window, query streaming status, and delete chats. A script can start a new chat with: user-authored text optional system prompt optional model ID optional image path, read by the SDK and sent as base64 optional typed context parts: { kind: "resourceUri", uri, label } { kind: "filePath", path, label } optional noResponse flag to create the chat without triggering AI response streaming A script can list or read chat history from storage without opening ACP Chat. That matters for automation workflows that audit, export, or prune history headlessly. A script can focus ACP Chat without creating a new chat. This is a UI affordance, not a storage operation. A script can call aiAppendMessage, aiSendMessage, aiSetSystemPrompt, and aiOn, but based on the attached runtime dispatch these should be treated as declared or partially wired surfaces, not proven runtime features. Core Concepts SDK globals scripts/kit-sdk.ts declares the TypeScript globals and result types. The ACP section distinguishes built-in ACP Chat from the older inline chat() prompt. Inline chat() is UI-only and does not call AI providers; the ai* functions target Script Kit’s built-in ACP/BYOK AI chat surface. Important SDK types: AiChatInfo id title modelId provider createdAt updatedAt isDeleted optional preview messageCount AiMessageInfo id role content createdAt optional tokensUsed AiChatOptions systemPrompt imagePath modelId noResponse parts AiContextPartInput resourceUri filePath Protocol messages Rust protocol variants live in src/protocol/message/variants/ai.rs. Every request uses requestId for correlation, except pushed subscription events, which use subscriptionId. src/protocol/message/constructors/prompts.rs includes all AI request/response variants in request_id(), so protocol correlation recognizes the AI SDK messages. Direct handlers vs UI-thread handlers src/ai/sdk_handlers.rs directly handles: AiIsOpen AiGetActiveChat AiListChats AiGetConversation AiDeleteChat AiGetStreamingStatus These return a response message without needing the UI prompt handler. src/execute_script/mod.rs maps only: AiStartChat → PromptMessage::AiStartChat AiFocus → PromptMessage::AiFocus src/main_sections/prompt_messages.rs only defines prompt-message variants for those two AI SDK UI-thread operations. src/prompt_handler/mod.rs handles those two prompt messages. ACP window meaning aiIsOpen() checks super::is_ai_window_open() and, only when open, includes get_active_chat_id(). Based on the attached code, this means the Rust AI window handle/global open state, not merely “there is a stored active chat” and not a detached browser/remote ACP session. It is about the Script Kit GPUI AI window state. ACP vs older inline chat The SDK comment is explicit: chat() is an inline main-window prompt where the script owns generation. The ai* APIs control built-in ACP Chat with BYOK providers and SQLite history. User Workflows Workflow: check whether ACP Chat is open A script calls await aiIsOpen(). The SDK sends: JSON{ "type": "aiIsOpen", "requestId": "..." } Rust handles this in handle_ai_is_open. It calls is_ai_window_open(). If the window is open, it includes the globally published active chat ID from get_active_chat_id(). If the window is closed, activeChatId is omitted even if storage has chats. Result: TypeScript{ isOpen: boolean, activeChatId?: string } Operator note: this is a window-open check, not a storage-exists check. Workflow: inspect current/most recent chat A script calls await aiGetActiveChat(). The direct handler first tries the active chat ID from SDK-visible window state. If none exists, it falls back to storage::get_all_chats().into_iter().next(). That means it may return the most recent chat even when the window is not open. Result is AiChatInfo | null. Operator note: “active” is not purely UI-active. With no active window state, it becomes “most recent stored chat” by fallback. Workflow: list chat history A script calls await aiListChats(limit, includeDeleted). The direct handler reads storage with storage::get_all_chats(). If includeDeleted is true, it extends the result with storage::get_deleted_chats(). It computes totalCount before truncation, applies limit, then maps each chat to AiChatInfo. The SDK returns only the chats array and discards totalCount. Operator note: scripts cannot currently observe totalCount through the TypeScript return value, even though the protocol sends it. Workflow: read messages from a chat A script calls await aiGetConversation(chatId?, limit?). If chatId is provided, the handler parses it as a ChatId. Invalid format returns an empty message list with chatId set to the invalid string and hasMore: false. If no chatId is provided, the handler tries active chat ID, then falls back to most recent stored chat. If no chat exists, it returns empty chatId and empty messages. If limit is provided, it calls storage::get_recent_messages; otherwise it calls storage::get_chat_messages. The SDK returns only messages, discarding protocol fields chatId and hasMore. Operator note: hasMore is protocol-visible but not SDK-return-visible. Workflow: start a new ACP Chat A script calls: TypeScriptawait aiStartChat("Review this", { systemPrompt: "You are a reviewer.", imagePath: "/tmp/screenshot.png", modelId: "claude-3-5-sonnet-20241022", noResponse: false, parts: [ { kind: "resourceUri", uri: "kit://context?profile=minimal", label: "Current Context" }, { kind: "filePath", path: "/tmp/example.rs", label: "example.rs" } ]}); SDK behavior: Generates requestId. If imagePath is present, reads the file synchronously through Node fs.readFileSync. Base64-encodes the image. Sends AiStartChat with message, optional systemPrompt, optional image, optional modelId, noResponse, and optional parts. Rust behavior: execute_script maps AiStartChat to PromptMessage::AiStartChat. prompt_handler opens/focuses the AI window via crate::ai::open_ai_window(cx). It pre-generates a real ChatId. It sets should_submit = !no_response. It resolves provider from requested modelId when possible. It converts protocol AiContextPartInput to runtime AiContextPart. It queues crate::ai::start_ai_chat(...). It sends AiChatCreated immediately with chat ID/title/model/provider/streamingStarted. The AI window handles the queued StartChat command. AiApp::handle_start_chat resolves model/provider, prepares outbound message/context, saves optional system prompt, creates chat in storage, saves user message, attaches image if present, updates in-memory state, publishes active chat ID, updates preview/count caches, clears composer, and optionally starts streaming. noResponse: true means the chat/user message are created but start_streaming_response is not called. Protocol result streamingStarted is false. noResponse: false or omitted means should_submit is true and streaming is started after persistence. Workflow: focus ACP Chat A script calls await aiFocus(). Rust opens/focuses the AI window. The response includes only { wasOpen } in the SDK, though the protocol includes success and wasOpen. Operator note: the SDK drops success; failures resolve as { wasOpen: false } in fallback cases. Workflow: query streaming status A script calls await aiGetStreamingStatus(chatId?). The direct handler reads get_streaming_snapshot(), optionally filters by requested chatId, and returns isStreaming, optional chatId, optional partialContent. Important boundary: this is a global SDK-visible snapshot published by the AI window, not a subscription and not storage. Workflow: delete a chat A script calls await aiDeleteChat(chatId, permanent?). The direct handler parses the chatId. Invalid format returns success: false with error. If permanent is true, it calls storage::delete_chat_permanently; otherwise it calls storage::delete_chat for soft deletion. The SDK resolves void and currently ignores the success/error payload in its promise handler, so app-side deletion failures are not surfaced by the shown SDK implementation. Workflow: subscribe to events A script calls: TypeScriptconst unsubscribe = await aiOn("streamChunk", handler, chatId); SDK behavior: Sends aiSubscribe with events: [eventType] and optional chatId. On aiSubscribed, stores handler in local aiSubscriptions. Returns an unsubscribe function that deletes local handler and sends aiUnsubscribe. Declared event handling exists in SDK-local _handleAiEvent, which dispatches pushed aiStreamChunk, aiStreamComplete, aiNewMessage, and aiError. Runtime gap: the attached Rust direct handler returns None for AiSubscribe and AiUnsubscribe, and execute_script does not map them to PromptMessage. No app-side subscription manager is proven in the bundle. Treat live subscriptions as unproven. It creates a visible ACP Chat state: new selected chat input cleared placeholder updated scroll forced to bottom optional streaming begins active chat ID published for SDK-visible state The exact window mode, bounds, and detached/full ACP distinctions are not proven from this bundle. Rust Message::request_id() recognizes all AI request/response variants and optional AiError.request_id. Data, Storage, And Privacy Boundaries Chat metadata boundary AiChatInfo exposes: chat ID title model ID provider timestamps deletion state preview, currently None in sdk_handlers::chat_to_info message count handle_ai_list_chats computes message count by reading each chat’s messages. That can be expensive if many chats exist; the bundle does not show batching. Conversation content boundary aiGetConversation returns full message content for stored messages. This includes user, assistant, and system roles. AiMessageInfo does not expose image attachments or context part receipts; it exposes text content and optional token count. System prompt boundary During aiStartChat, if systemPrompt is provided, AiApp::handle_start_chat saves it as the first stored system message before saving the user message. aiSetSystemPrompt is declared but not proven handled, so post-creation system prompt mutation should not be considered reliable from the attached evidence. Image boundary The SDK reads imagePath from local disk and base64-encodes it before sending over protocol. Runtime then attaches it as ImageAttachment::png(img_base64.clone()) to the user message. Important implications: The SDK assumes PNG attachment semantics at runtime. The protocol field is base64 image data, not a file path. If SDK file read fails, it logs to stderr and still sends the message without image. The attached AiMessageInfo result type does not expose image attachments back to scripts. Context parts boundary For aiStartChat, protocol parts are converted to runtime AiContextPart and passed into start_ai_chat. AiApp::handle_start_chat calls prepare_outbound_user_message(&message, &parts), producing a receipt and final user content. Persistence behavior: If context resolution is Ready, saved user content uses the final assembled content. If Partial, saved user content also uses final assembled content and records a user-visible streaming error. If Blocked, saved user content falls back to cleaned authored content rather than unresolved injected context. Tests in src/ai/window/chat.rs verify these saved-content decisions. Deletion boundary aiDeleteChat(chatId, false) soft-deletes. aiDeleteChat(chatId, true) permanently deletes. Invalid IDs return protocol failure. SDK currently does not surface failure details in its return type. Error, Empty, Loading, And Disabled States SDK mock/autosubmit states The SDK defines mock results for auto-submit/test mode. Many SDK tests capture stdout and rely on message shapes, not actual app behavior. Examples: aiStartChat mock result returns mock-chat-id. aiListChats mock result returns empty chats. aiGetConversation mock result returns empty messages. aiGetStreamingStatus mock result returns isStreaming: false. These are useful for SDK-shape tests but should not be mistaken for runtime integration proof. Storage errors aiListChats returns an empty list and total count 0 if storage read fails. aiGetConversation returns an empty message list if message retrieval fails. aiGetActiveChat returns null if storage lookup fails or active chat cannot be found. Invalid chat ID aiGetConversation(invalidId) returns empty messages with hasMore: false. aiDeleteChat(invalidId) returns success: false with an error string. AI window open failure aiStartChat attempts to send AiChatCreated with empty chat_id, empty title, defaulted model ID, empty provider, and streaming_started: false so the SDK does not hang. aiFocus returns success: false in protocol if open_ai_window(cx) fails, but the SDK return type only exposes wasOpen. Provider/model gaps AiApp::handle_start_chat resolves model/provider from requested model/provider, selected model, or defaults. Tests cover provider disambiguation and fallback. If a selected chat’s provider is unavailable, src/ai/window/chat.rs records a provider error message. Unproven API hangs/fallback risk For aiAppendMessage, aiSendMessage, aiSetSystemPrompt, aiSubscribe, and aiUnsubscribe, the SDK sends messages and waits for response shapes. Because app-side handling is not proven in the attached dispatch, real scripts may hang or fall back only in test/autosubmit conditions.
