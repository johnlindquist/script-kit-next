025 System Feedback and Prompt Control APIs Executive Summary Feature 025 covers the script-facing utility APIs that send feedback to the user or mutate an already-running prompt: beep(), say(), notify(), hud(), setStatus(), menu(), setActions(), and setInput(). The important truth is split: hud(), setActions(), and setInput() have real app-side behavior proven in the supplied source. hud() creates standalone HUD overlay windows. setActions() updates SDK action state, shortcuts, and an open actions dialog. setInput() mutates prompt/composer/search/editor input depending on the active surface or batch target. beep(), say(), notify(), setStatus(), and menu() are SDK/protocol-visible but should not be treated as implemented native features. The SDK explicitly warns that these are not implemented in the GPUI app. setStatus() reaches a Rust prompt-handler arm, but that arm only logs receipt; the bundle does not prove a status UI, tray update, native notification, speech, or sound side effect for the others. Operationally: use hud() for user-visible feedback today. Use setActions() for prompt-level actions and shortcuts. Use setInput() or batch setInput for automation-safe prompt mutation. Treat the rest as typed future/stub surfaces until backend behavior is added and verified. What Users Can Do Users and scripts can currently do the following: Show standalone feedback with hud(message, { duration }). The HUD does not steal focus, does not reveal the launcher, and auto-dismisses after its duration. It is managed independently from main-window visibility. Replace or update prompt actions with setActions(actions). Actions can have names, descriptions, shortcuts, optional submit values, optional JS handlers, visibility flags, and close flags. The SDK stores handlers in process memory and sends a serializable action list to Rust. Programmatically set prompt input with setInput(text). Direct SDK setInput() is fire-and-forget. Batch setInput is automation-grade because it can target a specific surface and return a batchResult. Use action shortcuts or the actions dialog to trigger actions. Visible actions with shortcuts are normalized and registered by Rust. The SDK receives actionTriggered messages and dispatches to the matching JS action handler or submits the action value. Attempt beep(), say(), notify(), setStatus(), and menu(). These functions serialize and send messages, but the SDK warns that they are not implemented. setStatus() logs on the Rust side. Do not promise users audible beeps, speech, native notifications, visible status chrome, or tray/menu changes from these calls based on the supplied source. Core Concepts Fire-and-forget control calls The SDK section labels these APIs as fire-and-forget messages. They send JSON protocol messages and resolve immediately or return immediately without waiting for a response envelope. Important examples: hud() returns void. setInput() returns void. setActions() is declared Promise<void> but only serializes, stores handlers, sends the message, and does not await an app response. beep(), say(), notify(), setStatus(), and menu() are async functions but only warn and send messages. This means automation should not infer success from the SDK call returning. For state-changing operations, verify with getState, getElements, waitFor, or a batch result.
