024 Confirm Prompt and Dialogs / confirm() Executive Summary Confirm Prompt and Dialogs is the explicit yes/no decision layer for Script Kit GPUI. It covers the TypeScript SDK confirm() API, the Rust protocol type: "confirm" message, the in-window AppView::ConfirmPrompt state, and the native attached confirm-popup fallback used when the main window cannot host the state. There are two UI routes: In-window confirm state: preferred when the main window is visible and the startup-registered router can reach the main ScriptListApp. It becomes AppView::ConfirmPrompt, renders title/body in the main content area, and uses the native footer for Confirm/Cancel. Parent/native popup fallback: used when the main window is hidden or the active GPUI context is not the main root. It opens an attached popup window, registered as a PromptPopup-class automation surface, commonly id’d as confirm-popup. The contract is fail-closed: user cancellation, Escape, null submit values, dialog-open failure, and SDK auto-submit fallback all resolve as false. Primary source anchors: scripts/kit-sdk.ts, src/protocol/message/variants/prompts_media.rs, src/prompt_handler/mod.rs, src/confirm/parent_dialog.rs, src/app_impl/about_route.rs, src/main_sections/app_view_state.rs, src/app_impl/ui_window.rs, src/main_sections/render_impl.rs, src/windows/automation_surface_collector.rs. What Users Can Do Users can: Call await confirm() from a Script Kit script and receive a boolean. Provide a custom message: await confirm("Delete this?"). Provide positional button labels: await confirm("Delete this?", "Delete", "Keep"). Provide config labels: await confirm({ message, confirmText, cancelText }). Confirm or cancel by keyboard. Confirm or cancel through footer buttons in the in-window state. Confirm or cancel destructive built-ins such as Quit Script Kit, Empty Trash, Restart, Shut Down, Log Out, Stop All Processes, Clear Suggested, and similar confirmation-gated commands. Confirm destructive clipboard, file, script, sharing, and adjacent action flows. Inspect and drive confirm dialogs through automation using getState, getElements, listAutomationWindows, and targeted batch selection. Core Concepts SDK confirm() Source: scripts/kit-sdk.ts. Supported shapes: TypeScriptconfirm(): Promise<boolean>confirm(message: string): Promise<boolean>confirm(message: string, confirmText?: string, cancelText?: string): Promise<boolean>confirm(config: ConfirmConfig): Promise<boolean> ConfirmConfig: TypeScriptinterface ConfirmConfig { message: string confirmText?: string cancelText?: string} SDK defaults: No argument → message defaults to "Are you sure?". Positional labels are passed through as confirmText and cancelText. Config labels are passed through as config.confirmText and config.cancelText. SDK-side auto-submit fallback is { value: "false" }. A SubmitMessage with value === null resolves false. A SubmitMessage with value === "true" resolves true. Any other non-null value resolves false. Protocol confirm message Source: src/protocol/message/variants/prompts_media.rs. Rust protocol shape: Rust#[serde(rename = "confirm")]Confirm { id: String, message: String, #[serde(rename = "confirmText", skip_serializing_if = "Option::is_none")] confirm_text: Option<String>, #[serde(rename = "cancelText", skip_serializing_if = "Option::is_none")] cancel_text: Option<String>,} The SDK sends: JSON{ "type": "confirm", "id": "<prompt-id>", "message": "<message>", "confirmText": "<optional>", "cancelText": "<optional>"} Rust converts this into PromptMessage::ShowConfirm. ParentConfirmOptions ParentConfirmOptions is the shared Rust UI configuration for both confirm routes. The bundle shows these fields in callers: title body confirm_text cancel_text confirm_variant width destructive helper constructor: ParentConfirmOptions::destructive(...) SDK confirm maps to: title: "Confirm" body: SDK message confirm text: SDK confirmText or "OK" cancel text: SDK cancelText or "Cancel" Destructive built-ins and actions use more specific labels such as "Quit", "Empty Trash", "Move to Trash", "Delete", and "Delete All". A script calls confirm(). The SDK creates a prompt id, registers a pending handler with fallback { value: "false" }, sends type: "confirm", and waits for a submit response. Rust prompt handler entry Source: src/prompt_handler/mod.rs. PromptMessage::ShowConfirm is classified as PromptMessageRoute::ConfirmDialog. The handler builds ParentConfirmOptions and calls: Rustcrate::confirm::confirm_with_parent_dialog(...) Result mapping: Ok(true) → sends Message::Submit { value: Some("true") } Ok(false) → sends Message::Submit { value: Some("false") } Err(error) → logs error and sends false This is the core fail-closed behavior. SDK sends type: "confirm" with message "Are you sure?". Rust prompt handler opens confirm UI. User confirms or cancels. Rust sends submit value "true" or "false". SDK resolves Promise<boolean>. SDK custom message and labels Script calls await confirm("Delete?", "Delete", "Keep"). SDK sends message, confirmText, and cancelText. Rust uses title "Confirm", body "Delete?", confirm label "Delete", cancel label "Keep". In-window route or popup fallback is chosen. Result resolves to true only if the confirm path is chosen. In-window confirm Main window is visible and router can reach ScriptListApp. AppView::ConfirmPrompt replaces the current view. Previous view is stored in previous. Body renders in the main content area. Native footer shows: Apply / ↵ / confirm label Close / Esc / cancel label Tab toggles focused button. Enter resolves according to focused button. Escape resolves false. Resolution sends bool to awaiting caller and restores previous. Popup fallback confirm Main window is hidden or context is not the main root. open_parent_confirm_dialog creates a native popup. Popup is attached to parent when possible. Automation sees it as a prompt popup, with confirm-specific semantic elements. Confirm/cancel callbacks run according to user choice. Popup close/teardown removes the popup surface.
