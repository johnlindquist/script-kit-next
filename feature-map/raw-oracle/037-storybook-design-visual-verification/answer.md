037 Storybook, Design Explorer, and Visual Verification Executive Summary Storybook is Script Kit GPUI’s visual state lab. It is a developer-facing browser and CLI-backed catalog for reviewing launcher states, popup states, secondary-window states, shared component primitives, and adoptable visual variations. Its durable purpose is not to preserve every historical mockup; it is to expose supported visual states through real or production-adjacent render paths. The Storybook catalog separates three roles: canonicalState: supported app states that should remain in primary catalog coverage. adoptableVariation: selectable visual variants that can be persisted and resolved into live product configuration. designExperiment: archived comparison studies that should not appear in the primary registered catalog. The primary catalog is supposed to contain only canonical states and adoptable variations. Archived design experiments can remain in helper modules, but they are not valid as primary proof of supported product state. Storybook evidence quality is explicit. A variant should declare whether it is liveSurface, presenterFixture, or design-only. liveSurface is the strongest Storybook representation because it renders through the live surface path or a read-only live override. presenterFixture is acceptable for deterministic presenter-backed state review. Old PNG-backed runtimeFixture coverage is treated as migration debt and must not appear in the registered catalog. Visual verification has a proof ladder. State-first receipts such as getState, getElements, waitFor, and batch are preferred for behavior and routing. Screenshots are only valid proof when the acceptance criterion is visual, and even then they must be strict-window, target-aware, and content-audited. Blank, black, wrong-window, ambiguous-window, or missing-crop screenshots are infrastructure failures, not product evidence. Design Gallery is a product built-in surface, separate from Storybook. Storybook is a developer lab. Design Gallery is reachable through triggerBuiltin design-gallery, has its own product surface contract, and uses a select-only native footer. The design-picker visual matrix is another verification tool that captures and checks shipped design catalog variants; it is not the Storybook browser itself. The standalone feature_explorer XState mockup should be understood as a feature/state contract exploration surface. Storybook proves visual states. feature_explorer explores feature/state model behavior. Treat any overlap as complementary, not interchangeable. What Users Can Do Developers and agents can use Storybook to inspect registered visual coverage without launching ordinary product flows. The standalone binary supports opening the browser, selecting a story, preselecting a variant, entering compare mode, adopting a selected variant, printing catalog JSON, printing adopted surface-resolution JSON, and capturing screenshots. Supported Storybook CLI entry points shown in src/bin/storybook.rs include: cargo run --bin storybook cargo run --bin storybook -- --story "main-menu" cargo run --bin storybook -- --story "main-menu" --screenshot cargo run --bin storybook -- --catalog-json cargo run --bin storybook -- --surface-resolution-json cargo run --bin storybook -- --story <ID> --variant <ID> --adopt cargo run --bin storybook -- --story <ID> --compare Inside the Storybook browser, users can: Browse stories grouped by category. Filter/search the story list. Preview the selected story. Switch between single-preview and compare-preview modes. Move through stories and variants by keyboard. Persist/adopt selected variants. Capture Storybook screenshots. Review story role, surface, selected variant, comparable status, and per-variant metadata through catalog diagnostics. Agents can use --catalog-json as the machine-readable entry point. The JSON snapshot includes story count, comparable story count, surfaces, story IDs, names, categories, roles, surfaces, selected variant IDs, comparable status, adopted surface coverage, and variant props. Developers can use Design Gallery as a product surface, not a Storybook feature. It is a built-in filterable surface reachable through triggerBuiltin design-gallery, triggerBuiltin designgallery, or triggerBuiltin design gallery. It renders gallery items, supports filtering, exposes visible rows, and owns a select-only footer contract. Agents can run visual verification sweeps through scripts/agentic/surface-navigator.ts. The navigator enters known surfaces through real runtime commands, collects state/elements receipts, promotes exact automation targets, and optionally captures strict-window screenshots into .notes/image-library/ with sidecar receipts and a manifest. Core Concepts Storybook browser StoryBrowser in src/storybook/browser.rs is the main interactive UI. It owns the story list, selected story index, selected variant index, filter string, theme name, design variant, preview mode, selection store, status line, focus handle, screenshot directory, compare scroll handle, and cached live preview. The browser supports two preview modes: Single: one selected story/variant preview. Compare: side-by-side variant comparison for stories with multiple variants. Single-preview mode can use a cached live view through render_live_variant. Compare mode remains stateless and uses compare rendering. Story trait The Story trait in src/storybook/story.rs defines the shape of a registered preview: id() name() category() catalog_role() surface() render() render_live_variant() render_compare_variant() variants() The default catalog role is DesignExperiment, which is intentionally unsafe as a primary catalog default. Registered primary stories must override this to CanonicalState or AdoptableVariation. Catalog role StoryCatalogRole has three values: CanonicalState AdoptableVariation DesignExperiment The durable rule in lat.md/storybook.md is that the primary registered catalog should contain only canonical states and adoptable variations. src/storybook/diagnostics.rs has tests asserting catalog stories are not designExperiment. Story surface Story surfaces classify visual coverage. Examples visible in the provided context include Main Menu, Dictation Overlay, Built-In Browser, About Surface, Component, Footer, Header, Action Dialog, and other registered lab surfaces exported through src/storybook/mod.rs. Representation quality StorybookRepresentation in src/storybook/adoption.rs describes evidence quality: LiveSurface serializes as liveSurface. PresenterFixture serializes as presenterFixture. DesignExperiment serializes as designExperiment. lat.md/storybook.md also names runtimeFixture as an old PNG-backed path that must not appear in the registered catalog. src/storybook/diagnostics.rs includes a test that rejects registered variants whose representation prop is runtimeFixture. Data source quality StorybookDataSource in src/storybook/adoption.rs describes the source behind a variant: ProductionState DeterministicProductionFixture MockDesignOnly Primary-catalog comparison panels must not imply production parity when either side is mock design-only data. Footer hint source FooterHintSource in src/storybook/adoption.rs describes where footer hints come from: ActiveFooterState MainWindowFooterConfig MockDesignOnly None Main-menu Storybook states must source footer hints through the same production footer contract as runtime launcher states. Mock-only footer hints are rejected for primary main-menu coverage. Adoptable surface AdoptableSurface in src/storybook/adoption.rs is the shared adoption contract for a live-rendered surface backed by Storybook variants. It uses stable VariationId values, a story ID, specs, and live values. adopted_surface_live resolves on-disk Storybook selection into a typed live value. Catalog diagnostics src/storybook/diagnostics.rs builds the machine-readable catalog snapshot. It reads the selection store, walks all_stories(), groups stories by surface, marks comparable stories, includes selected variant IDs, and serializes variant metadata. Design Gallery Design Gallery is a built-in product surface. lat.md/design.md states that Design Gallery owns a select-only native footer surface. Its fallback GPUI footer is a single ↵ Select hint routed through main_window_footer_slot, and native Run is guarded so it cannot fall through to launcher execution. src/render_builtins/design_gallery.rs renders the surface, filters gallery items, exposes visible rows and labels, and owns its key handler. Surface Navigator surface-navigate is the warm-session, state-first entry point for known app surfaces and screenshot-library capture. Its default output root is .notes/image-library/. It can run groups such as filterable-main, attached-popup, and all-active, and it can isolate visual sweeps with --fresh-per-case. Verify Shot verify-shot.ts is the screenshot proof helper. It collects state/probe receipts where applicable, inspects target windows, captures screenshots through exact target identity where required, audits PNG content, records popup crop strategy, emits optional vision review prompts, and exits with distinct failure modes. Screenshot proof versus state proof State receipts prove behavior, routing, and semantics. Screenshots prove rendered appearance only when they are targeted, content-audited, and tied to state/elements receipts. It is pinned by tests/design_picker_visual_matrix_script_contract.rs. User Workflows Workflow: inspect Storybook interactively Run cargo run --bin storybook. The app opens a normal window titled Script Kit Storybook. The left sidebar lists stories grouped by category. The user filters or navigates the story list. The preview panel renders the selected story. In single-preview mode, the browser may use a cached live preview if the story supports render_live_variant. The status bar shows available commands. Result: the user can visually inspect registered canonical states and adoptable variations. Proof path: for source-only changes, use catalog/source tests. For UI rendering changes, launch Storybook and add visual proof only when the acceptance criterion is visual. Workflow: open a specific Storybook story Run cargo run --bin storybook -- --story "main-menu". The binary constructs StoryBrowser. It calls select_story("main-menu"). If the story exists, the browser selects it and renders its preview. If the story does not exist, interactive mode emits an unknown-story error and exits with the invalid-story path. Result: the requested story is selected on open. Proof path: --catalog-json can list available story IDs before interactive launch. Workflow: open a specific Storybook variant Run cargo run --bin storybook -- --story "main-menu" --variant "populated-results". The binary selects the story. It then calls the variant-selection path. If the variant exists for that story, the selected variant index changes. If the variant is missing, the binary reports available variants. Result: the Storybook browser starts on the requested variant. Proof path: src/storybook/diagnostics.rs catalog snapshot includes variant IDs; src/bin/storybook.rs validates the requested variant. Workflow: compare variants Open a story with multiple variants. Press Tab in single-preview mode, or pass --compare at launch. The browser switches PreviewMode::Compare. The preview renders side-by-side variant panels. Use left/right arrows to move between variants. Use number keys 1 through 9 to focus variants, based on the browser status text. Press Enter to adopt the selected variant when adoption is supported. Press Esc to exit compare mode. Press Tab to return to single-preview mode. Result: the user can compare variant treatments and adopt a selection. Proof path: compare-mode source contracts require fixture/data-source labels through ComparePanelContract and reject primary-catalog mock-only production claims. Workflow: persist/adopt a Storybook variant from CLI Run cargo run --bin storybook -- --story <story-id> --variant <variant-id> --adopt. The binary validates that both story and variant are supplied. It validates that the story exists. It validates that the variant exists within that story. It calls save_selected_story_variant. It prints a JSON success envelope and exits without opening a GPUI window. Result: the selection store records the selected variant. Failure modes: Missing --story or missing --variant returns structured adopt_failed-style validation. Unknown story returns a structured unknown_story JSON error. Unknown variant returns a structured unknown_variant JSON error. Store write failure returns adopt_failed. Proof path: --surface-resolution-json can confirm how persisted selection resolves to live typed surface values. Workflow: capture a Storybook screenshot from the browser Open Storybook. Navigate to a story. Press Cmd+Shift+S. StoryBrowser::capture_screenshot enumerates OS windows with xcap. It finds a window whose title contains Storybook or whose app name contains storybook. It captures the image, encodes PNG, and writes into test-screenshots. It logs storybook_screenshot_saved. Result: a PNG screenshot is written for visual review. Boundary: this Storybook screenshot path is less strict than the agentic verify-shot path. For operator-grade visual proof, prefer strict-window capture through the agentic tooling when available. Workflow: capture Storybook screenshot from CLI Run cargo run --bin storybook -- --story "main-menu" --screenshot. The binary opens the Storybook window. A background thread waits briefly for rendering. It calls capture_storybook_screenshot. It enumerates xcap windows and captures the first title containing Storybook. It writes a PNG under test-screenshots/storybook-<timestamp>.png. It prints a JSON success envelope with the path. Result: a Storybook screenshot is captured and the process exits. Failure modes: No Storybook window found. Window capture fails. PNG encoding fails. File-system write fails. Proof boundary: this path checks that a screenshot was written, but it does not perform the same strict target and blank-image audit as verify-shot.ts. It walks registered stories. It groups surface counts and comparable counts. It prints JSON. Result: agents can inspect story role, surface, comparable status, selected variant, and per-variant props without opening a GPUI window. Proof path: cargo test --test storybook_adoption_contract, cargo test --test storybook_main_menu_render_path_contract, and diagnostics tests verify the catalog contract. Workflow: run image-library capture for active surfaces Run surface-navigate with --capture. Choose a group: filterable-main attached-popup all-active The navigator starts or reuses a session. For each case, it enters the surface through real runtime commands. It waits for the expected prompt type or popup contract. It records state/elements observations. It writes .notes/image-library/manifest.json unless overridden. Result: a durable screenshot library with machine-readable receipts. Workflow: isolate visual sweep cases Run surface navigator with --fresh-per-case. Each case gets its own short-lived session. The manifest records whether fresh-per-case isolation was used. Warm-session reuse remains the default for non-visual proof. Result: visual captures avoid stale rendered frames when switching between different main-hosted prompt variants. Workflow: verify attached popup screenshot proof Open a host surface. Open the attached popup, such as Actions Dialog or ACP slash Prompt Popup. Promote the exact popup target, optionally requiring an expected automation ID. verify-shot.ts classifies attached popup window kinds such as ActionsDialog and PromptPopup. The screenshot captures the parent window. The receipt records popupCapture.strategy: "parent_capture_with_crop". The receipt must include popupCapture.targetBounds. Missing target bounds turns the proof into an infrastructure error. Result: attached popup visual proof captures the parent plus exact crop bounds rather than pretending the popup is a standalone window. Workflow: run design-picker visual matrix Start a running Script Kit launcher with the required design-picker code. Run scripts/agentic/design-picker-visual-matrix.ts. Choose --designs all or a subset. Choose --sizes mini,full or one size. Include --capture-screenshots to write screenshots. Include --verify-state-receipts to assert the design receipt state. Include --cleanup to hide the launcher after the matrix. Result: every shipped design catalog ID can be checked across mini/full sizes with optional screenshots and state receipts. Boundary: this script is for Design Picker persistence/visual matrix verification, not Storybook catalog browsing. Theme synchronization lat.md/storybook.md states that Storybook theme reveal must apply selected theme state before preview windows become visible. Theme-sensitive proofs should compare browser and preview token snapshots before reveal, then close child previews without terminating the primary browser. Operational meaning: Do not claim theme parity from a visible preview unless selected theme tokens were applied before reveal. Child preview close must not kill the primary browser. Dark/light preview state, chooser selection, and secondary-window chrome must stay synchronized. Child preview and popup windows Storybook has a counted lifecycle registry for primary and child windows. The context names child preview and popup windows as Storybook-owned surfaces. The registry differentiates primary browser windows from children so the lab remains alive while any Storybook-owned surface is open. The specific child-window rendering APIs are not fully included in the provided bundle, so exact open/close UI controls are an inference. The invariant is explicit: register child windows and only quit once primary and child counts are both zero.
