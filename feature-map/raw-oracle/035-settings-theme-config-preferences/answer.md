035 Settings, Theme, Config, and Preferences Executive Summary Feature 035-settings-theme-config-preferences covers the user-facing and automation-facing surfaces that let Script Kit GPUI users inspect and change operational settings, choose and customize themes, and persist runtime preferences. There are two major UI surfaces in scope: Settings Hub: a mini built-in list rendered by src/render_builtins/settings.rs. It exposes operational rows such as Theme Designer, Dictation Setup, microphone selection, permission helpers, suggested-item reset, snap-mode controls, and window-position reset. It supports filtering, keyboard selection, row activation, actions popup routing, footer ownership, and automation receipts through the same visible-row projection described in lat.md/builtins.md. Theme Chooser: a dedicated theme-design surface rendered by src/render_builtins/theme_chooser.rs. It lists stock presets and user themes, live-previews selection, exposes customization controls for accent, opacity, vibrancy/material, and UI font size, owns a native footer, owns handled key/click propagation, and uses an explicit dismissal policy so temporary native focus churn during theme updates does not eject the user back to the launcher. The durable configuration boundary is split deliberately: ~/.scriptkit/config.ts owns operational preferences: hotkeys, command overrides, runtime preferences, layout, theme preset selection, dictation preference, AI preference, window-management preference, built-in enablement, and related config schema fields. ~/.scriptkit/theme.json owns the active theme color/font/vibrancy override payload. ~/.scriptkit/themes/<slug>.json owns user-authored saved themes that can be listed alongside bundled presets. Script SDK and config CLI TypeScript files expose config shapes for scripts and tooling, but not every schema field is necessarily wired into Rust runtime behavior. The highest-risk regression areas are footer double-rendering, theme chooser handled inputs leaking into the launcher filter, config schema drift between Rust and TypeScript, theme opacity/contrast regressions, stale config-cache behavior, and confusing the config-backed preset selection with the color payload stored in theme.json. What Users Can Do Users can open the Settings Hub and operate configuration-related actions from a compact built-in list. They can: Open Theme Designer and move into the Theme Chooser. Run Dictation Setup. Open Select Microphone for dictation device choice. Clear launcher suggested/recent history through Clear Suggested Items. Check macOS permissions through Check Permissions. Open permission-assistant flows for Accessibility and Screen Recording. Prompt macOS for Accessibility permission. Open the Accessibility pane in macOS System Settings. Change window snap behavior when available: Disable Window Snapping. Snap Mode: Simple. Snap Mode: Expanded. Snap Mode: Precision. Reset saved window positions when custom positions exist. Users can open the Theme Chooser and: Search/filter available theme presets. Navigate preset rows by keyboard. Click a preset row to select and preview it. Apply or commit the current theme. Undo/exit with Escape or Cmd+W. Use Cmd+K to open theme-specific actions. Remix/surprise the theme with Cmd+J. Reset customizations to the selected preset defaults with Cmd+R. Customize accent, opacity, vibrancy, material, and UI font size. See live preview changes applied to GPUI component colors and native vibrancy/material. Use a native footer row rather than a duplicated GPUI footer strip. Users and scripts can persist preferences through config surfaces: config.ts.theme.presetId selects a theme preset before theme.json overrides. config.ts.dictation.selectedDeviceId stores the preferred microphone device ID. config.ts.ai.selectedModelId and config.ts.ai.selectedAcpAgentId store AI runtime choices. config.ts.windowManagement.snapMode stores window snap mode. config.ts.commands[commandId].shortcut stores durable command shortcut overrides. config.ts.layout stores layout sizing preferences. Font and UI sizing are split: config.ts.editorFontSize and config.ts.terminalFontSize are config-level fields with Rust getters. Theme UI/mono font sizes live in theme.json / theme payloads and are pushed into gpui-component theme state. config.ts.uiScale exists as a config field/default test surface, but the provided context only shows it as test-gated getter behavior, so treat runtime UI-scale wiring as not established by this bundle. Core Concepts Settings Hub The Settings Hub is a mini built-in list, not the Theme Chooser itself. Its row definitions live in src/render_builtins/settings.rs: SettingsItem contains name, description, icon, and action. SettingsAction enumerates row actions such as ChooseTheme, DictationSetup, SelectMicrophone, ClearSuggested, permission actions, snap-mode actions, and ResetWindowPositions. get_settings_items() builds the base list and then appends conditional rows based on current snap mode and custom window-position state. settings_visible_row_names, settings_selected_visible_row, and settings_selected_visible_row_name form the visible-row projection used by render and automation. The important design rule from lat.md/builtins.md is that Settings renderer, automation, and state receipts share the same filtered-row projection. This means getElements, getState, and the visible UI should agree on which settings rows exist and which row is selected. Theme Chooser Theme Chooser is a dedicated AppView::ThemeChooserView { filter, selected_index }, not a generic built-in list. Its implementation is centered in src/render_builtins/theme_chooser.rs. Supporting split files exist for customize controls, preview panel, list/header, and key setup: src/render_builtins/theme_chooser.rs src/render_builtins/theme_chooser_customize_controls.rs src/render_builtins/theme_chooser_preview_panel.rs src/render_builtins/theme_chooser_list_header.rs src/render_builtins/theme_chooser_setup_key.rs The maintained source appears to have canonical behavior consolidated in theme_chooser.rs, with older/split files still visible in the bundle. Regression tests reference theme_chooser.rs markers such as "theme_chooser_font_size_click" and stop-propagation guards. Theme Chooser owns: Preset filtering. Keyboard navigation. Click selection. Live preview. Persistent theme application. Theme-specific actions dialog catalog. Native footer surface theme_chooser. Explicit dismiss policy instead of blur-dismiss policy. Theme Payload Boundaries There are three different theme-related storage concepts: Preset selection in config config.ts.theme.presetId is represented by Rust ThemeSelectionPreferences { preset_id: Option<String> } and TypeScript ThemeSelectionPreferences { presetId?: string }. This selects a preset. Active theme override payload ~/.scriptkit/theme.json stores the actual theme color/font/vibrancy override payload. src/theme/mod.rs describes loading the theme from this file. src/theme/presets.rs::write_theme_to_disk writes a Theme to that file. User-authored saved theme files ~/.scriptkit/themes/<slug>.json stores named user themes. src/theme/user_themes.rs owns listing, saving, directory creation, slug/file behavior, atomic writes, and validation. Do not collapse these into one concept. A preset ID in config.ts is not the same as the full active color payload in theme.json.
