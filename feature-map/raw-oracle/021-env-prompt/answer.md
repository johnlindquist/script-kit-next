021 Env Prompt / env() Executive Summary Env Prompt is the Script Kit GPUI feature behind env(): it gives scripts a way to resolve an environment variable, optionally ask the user for a value, and persist secret values in the app’s encrypted local secrets store. The current product truth is: env(key, promptFn?) in scripts/kit-sdk.ts first returns process.env[key] when already set. If a custom prompt function is provided, the SDK calls that function, stores the returned value in process.env[key], and does not show the GPUI EnvPrompt. Only when the value is missing and no prompt function is supplied does the SDK send an env message to the Rust app. The SDK auto-detects secrets by key name. Keys whose lowercase name contains secret, password, token, or key are sent as secret prompts. Secret submissions are persisted by Rust through src/secrets.rs into ~/.scriptkit/secrets.age. Non-secret submissions are provided back to the running script only and are not persisted by EnvPrompt. The code repeatedly uses “keyring” terminology, but the current storage implementation is not the macOS Keychain. It is an age-encrypted local file, cached in memory, with secure Unix file permissions when written. Older smoke-test text still describes “system keychain” behavior and env(key, { secret: true }) options; that is not supported by the visible SDK implementation in the bundle. EnvPrompt is a prompt entity routed through AppView::EnvPrompt. It owns focus, form-like DivPrompt sizing, a native footer surface named env_prompt, and a Submit-only footer. Native footer Run dispatches directly to EnvPrompt::submit before any launcher execute_selected fallback. Launcher AI is deliberately omitted. When a stored secret exists and the EnvPrompt has no custom title or prompt, Rust auto-submits the stored secret without showing UI. When contextual title or prompt text is present, such as API key configuration flows, EnvPrompt shows an update UI instead of auto-submitting, allowing the user to replace or delete the stored value. Automation support exposes EnvPrompt through prompt type env, AppView::EnvPrompt, layout prompt type env, getElements entries for key/value/status, active footer metadata through the native footer contract, and redacted secret input values. Secret values must not leak through logs, screenshots, automation state, or Tab AI snapshots. What Users Can Do Users can resolve an environment variable without seeing UI when the value already exists in the script process environment. Users can run a script that calls env("SOME_KEY") and, if no process value exists, see an EnvPrompt asking for the value. Users can type a value into the prompt and submit it with Enter or the Submit footer button. Users can press Escape to cancel. In the SDK resolver, a null submitted value exits the script process. Users can use secret-style prompts for keys that look sensitive by name. The SDK marks keys as secret when the key name contains secret, password, token, or key. Users can submit secret values that are stored persistently in ~/.scriptkit/secrets.age. Users can submit non-secret values that are returned to the current script run and placed into process.env, but are not stored by EnvPrompt. Users can update an existing stored secret when EnvPrompt is shown with contextual title or prompt text. Users can delete an existing stored value from the EnvPrompt UI via the “Delete stored value” click target when exists_in_keyring is true. Users can receive API-key configuration prompts triggered by app flows such as inline chat configuration. Those prompts are direct Rust-created EnvPrompt instances, not SDK env() calls. Users cannot currently use an options object such as env("TEST_SECRET_KEY", { secret: true }) with the visible SDK implementation. That older smoke-test usage conflicts with the current signature and implementation. User Workflows Workflow: existing process environment value A script calls env with a key. The SDK checks process.env[key]. If it is defined and non-empty, the SDK returns that string immediately. No Rust message is sent, no EnvPrompt is shown, and no secret store lookup occurs. This is the fastest and least visible path. It is also the path that preserves ordinary environment-variable semantics for scripts launched with preconfigured environment. Proof: scripts/kit-sdk.ts around lines 2477-2481. Workflow: custom prompt function A script calls env with a key and a custom prompt function. If no process env value exists, the SDK calls the supplied function, stores the result into process.env[key], and returns it. This path bypasses EnvPrompt completely. The custom prompt function may call another prompt, but that is outside EnvPrompt itself. Proof: scripts/kit-sdk.ts around lines 2482-2487. Workflow: missing key, no custom prompt, no stored secret The SDK sends an env message to Rust. Rust routes it to ShowEnv. The prompt handler checks stored secret info. If no existing stored value is found, it creates an EnvPrompt entity, sets AppView::EnvPrompt, clears focused input to None, sets pending focus to EnvPrompt, sizes as ViewType::DivPrompt, and notifies the UI. The user sees a form-like prompt with title, description, input field, storage hint, and “Script is running and waiting for {key}” status copy. Proof: src/prompt_handler/mod.rs around lines 6132-6182; src/prompts/env/render.rs around lines 23-91. Workflow: missing key, secret detected The SDK marks secret true when the lowercase key contains secret, password, token, or key. The EnvPrompt renders a secret-labeled field, masks input when secret reveal is off, and persists submitted non-empty values in ~/.scriptkit/secrets.age. Proof: scripts/kit-sdk.ts around lines 2502-2510; src/prompts/env/helpers.rs around lines 22-37; src/prompts/env/prompt.rs around lines 114-130. Workflow: missing key, non-secret The SDK sends secret false or absent. EnvPrompt still validates non-empty input and returns the value to the SDK. It does not call secrets::set_secret; the value is only available to the current script run through the SDK resolver and process.env[key]. Proof: src/prompts/env/prompt.rs around lines 98-132; scripts/kit-sdk.ts around lines 2498-2500. Workflow: existing stored secret, no contextual prompt/title Rust checks secrets::get_secret_info and then calls EnvPrompt::check_keyring_and_auto_submit only if the prompt has no prompt text or title. If secrets::get_secret returns a value, the prompt callback is invoked immediately and the UI is not shown. This is the “stored secret silently satisfies env()” behavior. Proof: src/prompt_handler/mod.rs around lines 6148-6176; src/prompts/env/prompt.rs around lines 79-97. Workflow: existing stored secret with contextual prompt/title When the EnvPrompt has title or prompt copy, auto-submit is skipped. The prompt is shown in update mode. It uses replacement placeholder copy, update description, “Configured {relative time}” status, and a Delete stored value click target. This is the API-key configuration behavior: if a key already exists, configuration still lets the user update it rather than silently returning the stored value. Proof: src/prompts/env/prompt.rs has_prompt_or_title around lines 66-78; src/prompt_handler/mod.rs around lines 6168-6177; src/prompts/env/render.rs around lines 95-123. Workflow: submit value The user presses Enter or clicks Submit. EnvPrompt::submit checks for empty or whitespace-only input. If valid and secret, it writes the secret. Then it clears validation error and invokes the submit callback with Some(value). The SDK receives the submitted value, writes it into process.env[key], and resolves the promise. Proof: src/prompts/env/helpers.rs around lines 39-45; src/prompts/env/prompt.rs around lines 98-132; scripts/kit-sdk.ts around lines 2493-2500. Workflow: empty submit If the input is empty or whitespace, EnvPrompt::submit sets validation error to “Value cannot be empty”, notifies the UI, logs a blocked submit, and stays open. The SDK resolver treats a null submitted value as cancellation and exits the script process.
