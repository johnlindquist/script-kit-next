039 Logging, Diagnostics, and Transaction Observability Executive Summary Logging, Diagnostics, and Transaction Observability is an operator/developer-facing feature set, not a user-facing product surface. It gives humans and AI agents enough runtime evidence to debug Script Kit GPUI without guessing from screenshots alone. The system spans four related but distinct observability layers: Runtime structured logging: tracing output from the app, including compact AI logs and full JSONL/session logs. Safe value logging: byte-capped previews and rate-limited emission for untrusted user-provided strings. Protocol transaction traces: persisted flight-recorder records for waitFor and batch automation transactions. Diagnostics resources and audit logs: MCP-readable transaction/protocol stats resources plus persistent AI preflight audit JSONL. The most important boundary is that logs are evidence, not UI. A runtime receipt, protocol trace, or MCP resource can prove state and command outcomes even when no screenshot is useful. Screenshots remain valuable for visual layout proof, but automation correctness should prefer state-first receipts, structured logs, protocol traces, and MCP resources. This chapter covers: ./dev.sh and compact log-driven iteration. Full JSONL logs versus compact AI logs. Safe user-value previews, byte caps, and rate-limited untrusted-value logging. Stable debug trace markers including DO_IN_TRACE, SCROLL_TRACE, and script_kit::input_history. Transaction traces, trace replay, bounded retention, malformed JSONL recovery, and MCP transaction resources. AI preflight audit persistence, correlation-level dedupe, compaction, and schema drift handling. Protocol stats/reporting and source-audit logging contracts. Verification recipes for logging safety, trace propagation, transaction persistence, and runtime receipt proof. What Users Can Do Operators and developers can: Start the dev loop with ./dev.sh and get compact AI-friendly logs by default. Override compact logs with standard verbose logs or debug-level tracing. Tail the latest session JSONL log for full structured evidence. Filter compact output for stable trace markers during live reproduction. Use protocol waitFor and batch automation receipts to prove UI state changes. Persist transaction traces on demand or on failure. Replay/idempotently resolve repeated transaction requests by request_id plus command fingerprint. Read the latest transaction trace through MCP instead of manually tailing files. Read protocol health counters through kit://diagnostics/protocol-stats. Inspect AI preflight audit history without relying on ephemeral stderr. Verify that logging sites do not dump raw user content. Verify that malformed JSONL entries do not poison diagnostic readers. Use source-audit tests to prevent trace/logging contract drift. End users do not directly operate most of this feature. They benefit indirectly because developer and agentic workflows can diagnose failures faster, reproduce protocol bugs more deterministically, and avoid unsafe raw-content logging. Core Concepts Operator/dev-loop observability versus user-facing UI This feature is primarily for developers, automation agents, and operators. It is not a visible launcher feature, preference pane, or end-user workflow. The user-facing app may emit logs while users interact with it, but logs are not themselves UI state. The UI state should be proven through getState, getElements, transaction receipts, MCP resources, and screenshots only when visual evidence is necessary. Compact AI logs versus full JSONL logs SCRIPT_KIT_AI_LOG=1 enables compact stderr logs in the shape: SS.mmm|L|C|message Where SS.mmm is elapsed time since startup, L is a compact level code, and C is a compact component/category. This format is optimized for AI agents and dev-loop iteration because it removes repeated metadata and saves token budget. Full JSONL/session logs are richer structured logs written under the Script Kit logs directory, including the latest-session JSONL file. JSONL logs are better for durable analysis, structured field inspection, historical replay, and tooling. Compact logs are better for live console reading. Structured logs versus transaction traces Structured logs are continuous event streams. They describe runtime events, warnings, trace markers, counters, and diagnostic messages as the app runs. Transaction traces are bounded, schema-versioned records of protocol automation transactions. A transaction trace captures the request id, command fingerprint, status, elapsed time, failed command index, command traces, before/after snapshots, poll observations, and typed errors. A transaction trace is closer to a flight recorder than to a general log. Runtime receipts versus screenshots Runtime receipts are protocol-level evidence. A batch or waitFor receipt can prove that the app saw a command, observed state, matched or failed a condition, and returned a typed result. Screenshots only prove pixels at a moment in time. For automation and regression proof, receipts are usually stronger than screenshots. Screenshots should be used when the claim is visual: layout, focus ring placement, clipped text, wrong window selection, or styling. Privacy-safe value previews versus raw user content dumps Untrusted strings must not be logged raw. This includes stdin text, chat titles, dictation queries, triggerBuiltin names, ACP command displays, pasted text, and similar user-controlled values. The safe logging contract uses byte-capped previews and metadata fields instead of dumping the full string. The shared preview helper returns a log-safe display value with: Raw byte count. Safe byte count. Truncation flag. UTF-8 boundary preservation. Trailing whitespace trimming. Ellipsis inside the byte budget. The shared rate limiter buckets by category, key length, and hash of the key, so the raw untrusted string is not retained in limiter state. Stable trace markers Some verbose trace flows intentionally use stable marker strings so an operator can filter a live ./dev.sh session while reproducing a bug. Known markers and targets include: DO_IN_TRACE: follows shared filter input changes, current-app command normalization, intent resolution, and built-in execution routing for “Do in Current App”. SCROLL_TRACE: follows current-app command list rendering, wheel-step accumulation, scroll metrics, wheel-owned selection notes, and reanchor decisions. script_kit::input_history: follows main-menu Up/Down routing, render-paced key-repeat coalescing, render acknowledgments, obsolete-pending cancellation, programmatic filter echo suppression, history indices, and ScriptList key-up receipts. Trace fields that include user-entered text must use safe previews, not raw values. Protocol stats Protocol stats are atomic counters and derived health flags for Rust/Bun protocol-boundary issues. The MCP resource kit://diagnostics/protocol-stats exposes a JSON report containing snapshot, health, and thresholds. Thresholds are strict for malformed protocol input and more tolerant for expected noisy traffic such as unknown triggerBuiltin typos. AI preflight audits AI preflight audits are persistent JSONL records for message-preparation and context-resolution decisions. They are schema-versioned, bounded, compacted, deduped by correlation id during reads, and tolerant of malformed or unsupported-schema lines. User Workflows Workflow: Live dev-loop diagnosis with compact logs Start the app with ./dev.sh. Confirm the runner reports compact AI log mode. Reproduce the behavior. Filter compact logs for stable markers such as DO_IN_TRACE, SCROLL_TRACE, or script_kit::input_history. Use the compact stream for fast iteration. Use full JSONL logs if structured fields or durable evidence are needed. Expected result: the operator can correlate the user action with runtime events without reading a verbose full log stream. Workflow: Deep structured log inspection Reproduce the issue in a dev or test session. Tail or inspect ~/.scriptkit/logs/latest-session.jsonl. Look for structured fields such as request id, correlation id, trace target, stage, status, error, byte counts, truncation flags, and suppression counters. Avoid treating a screenshot as proof when JSONL state evidence is available. Expected result: the operator gets durable structured evidence suitable for regression notes and agentic debugging. Workflow: Verify safe logging for untrusted values Find the log site that includes user-controlled values. Confirm it uses log_user_value or log_user_value_with_limit. Confirm the emitted fields include: *_preview *_bytes *_safe_bytes *_truncated suppressed Confirm repeated noisy values pass through log_rate_limit. Run the structured logging source audit. Expected result: logs contain bounded previews and accounting metadata, not raw user content dumps. Workflow: Trace a failing protocol transaction Send a waitFor or batch stdin JSON command with transaction tracing enabled. Use trace: "onFailure" when success traces would be too noisy. Read the command receipt first. If the command failed, inspect the included trace or persisted trace. Use MCP kit://transactions/latest when available instead of manual file tailing. Use command-level before, after, polls, elapsed times, typed errors, and failure suggestions to identify the failing state transition. Expected result: the operator sees why the transaction failed without relying on screenshots. Workflow: Prove idempotent replay Execute a transaction with a stable request_id. Execute the same payload again with the same request_id. Confirm the second execution resolves from the existing trace and does not repeat side effects. Execute a different payload with the same request_id. Confirm the executor rejects it as a different transaction payload. Expected result: repeated automation calls are safe when they are truly identical and rejected when request id reuse would hide a different command. Workflow: Inspect protocol health Read kit://diagnostics/protocol-stats. Check snapshot for raw counters. Check health.ok. Check health.flags for threshold-crossed counters. Check thresholds to understand expected tolerance. Expected result: protocol-boundary health is machine-readable, stable, and does not require grepping app logs. Workflow: Inspect AI preflight decisions Reproduce an Agent Chat or ACP preflight path. Inspect structured ai_preflight_audit logs for the active stage and correlation id. Read ~/.scriptkit/logs/ai-preflight-audits.jsonl for durable records. Confirm malformed lines or unsupported schemas do not prevent valid audits from loading. Confirm duplicate correlation ids collapse to the latest/kept record during read. Expected result: context-resolution and message-preparation decisions are auditable across process lifetime. Automation And Protocol Surface Stdin JSON automation contract Automation uses stdin JSON messages to drive the app. Command-line script arguments are explicitly not the durable automation contract. Agentic tools should choose among protocol-level batch input, GPUI event simulation, and native input based on the surface being tested. Receipts waitFor and batch are backed by a transaction executor that records: Request id. Success/failure. Per-command result entries. Before/after state snapshots. Poll observations. Elapsed timings. Failed command index. Typed error codes. Actionable failure suggestions. Optional transaction trace. Receipts are the preferred proof for automation behavior because they show what the runtime observed and returned. Trace modes Transaction trace mode controls inclusion/persistence: Off: no trace included/persisted. On: trace included/persisted regardless of success. OnFailure: trace included/persisted only on failure. should_include_trace(mode, success) is the central inclusion policy. Trace schema and persistence Transaction traces are schema-versioned. The executor uses TRANSACTION_TRACE_SCHEMA_VERSION when constructing traces. Persisted traces are JSONL records appended through append_transaction_trace. Trace records include: schema_version request_id command_fingerprint status started_at_ms total_elapsed_ms failed_at commands Command traces include command names, before/after snapshots, poll observations, and timing/error details. Idempotency and replay The executor computes a stable command fingerprint for the payload. For a repeated request_id: If an existing trace has the same fingerprint, the executor returns the prior trace/result instead of repeating side effects. If an existing trace has no fingerprint, it is treated as legacy and ignored with a warning. If an existing trace has a different fingerprint, the executor rejects the request as a different transaction payload. The replay script scripts/agentic/tx_trace_replay_idempotency.ts and tests/tx_trace_replay_idempotency_contract.rs cover this behavior. MCP transaction resources src/mcp_resources/transaction_resources.rs exposes: kit://transactions/latest kit://transactions/schema The latest resource reads persisted traces through read_latest_transaction_trace. If no trace exists, it returns a JSON payload with: JSON{ "kind": "transactionTrace", "status": "empty", "message": "No transaction traces found"} Malformed resource URIs should produce actionable errors. Request-id filtering is part of the resource contract. Protocol stats resource kit://diagnostics/protocol-stats returns a ProtocolStatsReport with camelCase fields: snapshot health thresholds The report is generated from atomic counters in src/protocol_stats.rs. Tracked counters include: stdin_parse_failed_total stdin_command_too_large_total stdin_unsupported_protocol_version_total trigger_builtin_unknown_total trigger_builtin_deprecated_name_total Health flags are stable ordered and threshold based. Protocol ingress observation The protocol ingress observer parses raw JSONL lines and composes version reading with deprecation validation. Unsupported protocol versions are counted before normal parse dispatch. The unsupported-version warning is rate-limited per (stdin_unsupported_protocol_version, found) window, and the protocol-stats health resource flips unhealthy on the first unsupported version. Compact automation logs Agentic tooling and test harnesses use SCRIPT_KIT_AI_LOG=1. The startup readiness marker matches compact logs via STARTUP_READY so warm sessions can correctly report readiness after successful startup. Data, Storage, And Privacy Boundaries Runtime logs Compact logs are emitted to stderr/console. Full session logs are available as JSONL, including ~/.scriptkit/logs/latest-session.jsonl. Logs may include structured fields, trace targets, component categories, elapsed time, status, and byte-count metadata. Logs must not include raw user-controlled content unless a specific owning feature has explicitly approved and tested that disclosure. Safe user-value previews Default safe preview budget: SAFE_USER_VALUE_MAX_BYTES: 200 bytes. Important details: The cap is bytes, not characters. The helper walks back to a UTF-8 char boundary. The helper trims trailing whitespace. The helper appends … inside the byte budget. The helper reports raw and safe byte counts. The helper reports whether truncation occurred. Required structured fields: *_preview*_bytes*_safe_bytes*_truncatedsuppressed The preview is allowed diagnostic content. The raw full value is not. Rate-limited untrusted-value logging The rate limiter: Buckets by (category, key.len(), hash(key)). Does not retain the raw untrusted key. Uses a default 30-second window. Garbage-collects stale buckets after 120 seconds. Caps the map at 2,048 keys with pruning. Prevents hostile clients from growing the limiter unbounded by randomizing keys. This limiter is complementary to occurrence-count gates such as protocol_stats::should_log_occurrence(total). Occurrence counters are useful for aggregate thresholds; rate limiting is needed for same-key bursts. Transaction trace storage Transaction trace persistence: JSONL append-only under the default transaction log path or an explicit test path. Maximum size: 10 * 1024 * 1024 bytes. Compaction keep count: 2,000 valid traces. Malformed entries are skipped with warnings during read/compaction. Readers stream line-by-line and do not load the whole log into memory just to find the latest trace. A process-local mutex serializes append/read/compact operations. Transaction traces may contain UI state snapshots and command parameters. They should be treated as diagnostic records, not public telemetry. AI preflight audit storage AI preflight audit persistence: Default path: ~/.scriptkit/logs/ai-preflight-audits.jsonl. Schema version: AI_PREFLIGHT_AUDIT_SCHEMA_VERSION = 2. Maximum size: 5 * 1024 * 1024 bytes. Compaction keep count: 2,000 audits. Malformed JSONL lines are skipped with warnings. Unsupported schema versions are skipped with warnings. Read path dedupes by correlation id. The structured ai_preflight_audit log event includes counts and identifiers, including final user content length, but should avoid dumping raw final user content into normal logs. Protocol stats storage Protocol stats are in-memory atomic counters exposed through MCP. They are not the same as persisted transaction traces or JSONL session logs. The MCP resource makes current health visible without requiring file access. Privacy boundary summary Allowed: Safe previews within byte budget. Byte counts. Truncation flags. Suppression counts. Correlation ids. Request ids. Schema versions. Status and error codes. Aggregate counters. State snapshots where explicitly part of protocol trace contract. Risky or forbidden without explicit contract: Full raw prompt text. Full pasted text. Full dictation transcript. Raw stdin payloads. Raw unknown trigger names. Raw chat titles. Raw ACP command display strings. Unbounded JSON dumps of user-controlled protocol payloads. Error, Empty, Loading, And Disabled States Compact logging disabled State: SCRIPT_KIT_AI_LOG=0 Result: Standard verbose logs are emitted. Compact log filtering assumptions do not apply. Operators should not expect SS.mmm|L|C|message formatting. Debug logging not enabled State: RUST_LOG not set to debug/trace Result: Some trace markers may not appear. Operator should rerun with RUST_LOG=debug or a targeted module filter. Missing latest session log Possible causes: App has not started. Log path not created. Different session/log path active. Process crashed before log initialization. Expected handling: Do not fabricate log proof. Use direct runtime stderr or restart with SCRIPT_KIT_AI_LOG=1. Transaction trace file missing read_latest_transaction_trace returns None, not an error. MCP latest transaction resource returns an empty JSON payload with status empty and message No transaction traces found. Malformed transaction trace JSONL Malformed lines are skipped with warnings. Valid later entries remain readable. Compaction rewrites only retained valid entries. Regression risk: a malformed line must not poison the whole trace log. Oversized transaction trace log When the trace log exceeds 10 MiB: Compaction runs before append. Valid traces are read. Malformed traces are skipped. Latest 2,000 valid traces are retained. Log is rewritten. Legacy transaction trace without fingerprint When idempotency lookup finds a trace with an empty command fingerprint: The executor warns. The legacy trace is ignored. The new request executes normally. Same request id with different transaction payload The executor rejects the command. This prevents accidental request-id reuse from hiding different side effects. Trace mode disabled When TransactionTraceMode::Off: Runtime receipts still exist. Trace inclusion/persistence does not happen. Operators should not expect MCP latest trace to update from that command. OnFailure trace mode with successful command When TransactionTraceMode::OnFailure and success is true: No trace is included/persisted. This is expected, not a missing-trace bug. Protocol stats unhealthy When thresholds are crossed: health.ok becomes false. health.flags contains stable ordered flag names. Operators should inspect the corresponding counters. Strict threshold examples: Any stdin parse failure. Any too-large stdin command. Any unsupported protocol version. Expected-noise threshold examples: Unknown triggerBuiltin names tolerate several typos before flagging. Deprecated triggerBuiltin names tolerate a high volume of legacy SDK calls before flagging. AI preflight audit file missing read_preflight_audits returns an empty list. Malformed AI preflight audit JSONL Malformed lines are skipped with warnings. Valid audits continue loading. Unsupported AI preflight audit schema Unsupported schema versions are skipped with warnings. Readers do not try to coerce unknown schemas. AI preflight audit oversized When the audit log exceeds 5 MiB: Compaction runs. Valid/deduped audits are read. Latest 2,000 are retained. Log is rewritten. Rate-limited log suppression When repeated untrusted values hit the limiter: The log site should emit or account for suppressed. Missing repeated log lines may be expected. 039 Logging, Diagnostics, and Transaction Observability Executive Summary Logging, Diagnostics, and Transaction Observability is an operator/developer-facing feature set, not a user-facing product surface. It gives humans and AI agents enough runtime evidence to debug Script Kit GPUI without guessing from screenshots alone. The system spans four related but distinct observability layers: Runtime structured logging: tracing output from the app, including compact AI logs and full JSONL/session logs. Safe value logging: byte-capped previews and rate-limited emission for untrusted user-provided strings. Protocol transaction traces: persisted flight-recorder records for waitFor and batch automation transactions. Diagnostics resources and audit logs: MCP-readable transaction/protocol stats resources plus persistent AI preflight audit JSONL. The most important boundary is that logs are evidence, not UI. A runtime receipt, protocol trace, or MCP resource can prove state and command outcomes even when no screenshot is useful. Screenshots remain valuable for visual layout proof, but automation correctness should prefer state-first receipts, structured logs, protocol traces, and MCP resources. This chapter covers: ./dev.sh and compact log-driven iteration. Full JSONL logs versus compact AI logs. Safe user-value previews, byte caps, and rate-limited untrusted-value logging. Stable debug trace markers including DO_IN_TRACE, SCROLL_TRACE, and script_kit::input_history. Transaction traces, trace replay, bounded retention, malformed JSONL recovery, and MCP transaction resources. AI preflight audit persistence, correlation-level dedupe, compaction, and schema drift handling. Protocol stats/reporting and source-audit logging contracts. Verification recipes for logging safety, trace propagation, transaction persistence, and runtime receipt proof. What Users Can Do Operators and developers can: Start the dev loop with ./dev.sh and get compact AI-friendly logs by default. Override compact logs with standard verbose logs or debug-level tracing. Tail the latest session JSONL log for full structured evidence. Filter compact output for stable trace markers during live reproduction. Use protocol waitFor and batch automation receipts to prove UI state changes. Persist transaction traces on demand or on failure. Replay/idempotently resolve repeated transaction requests by request_id plus command fingerprint. Read the latest transaction trace through MCP instead of manually tailing files. Read protocol health counters through kit://diagnostics/protocol-stats. Inspect AI preflight audit history without relying on ephemeral stderr. Verify that logging sites do not dump raw user content. Verify that malformed JSONL entries do not poison diagnostic readers. Use source-audit tests to prevent trace/logging contract drift. End users do not directly operate most of this feature. They benefit indirectly because developer and agentic workflows can diagnose failures faster, reproduce protocol bugs more deterministically, and avoid unsafe raw-content logging. Core Concepts Operator/dev-loop observability versus user-facing UI This feature is primarily for developers, automation agents, and operators. It is not a visible launcher feature, preference pane, or end-user workflow. The user-facing app may emit logs while users interact with it, but logs are not themselves UI state. The UI state should be proven through getState, getElements, transaction receipts, MCP resources, and screenshots only when visual evidence is necessary. Compact AI logs versus full JSONL logs SCRIPT_KIT_AI_LOG=1 enables compact stderr logs in the shape: SS.mmm|L|C|message Where SS.mmm is elapsed time since startup, L is a compact level code, and C is a compact component/category. This format is optimized for AI agents and dev-loop iteration because it removes repeated metadata and saves token budget. Full JSONL/session logs are richer structured logs written under the Script Kit logs directory, including the latest-session JSONL file. JSONL logs are better for durable analysis, structured field inspection, historical replay, and tooling. Compact logs are better for live console reading. Structured logs versus transaction traces Structured logs are continuous event streams. They describe runtime events, warnings, trace markers, counters, and diagnostic messages as the app runs. Transaction traces are bounded, schema-versioned records of protocol automation transactions. A transaction trace captures the request id, command fingerprint, status, elapsed time, failed command index, command traces, before/after snapshots, poll observations, and typed errors. A transaction trace is closer to a flight recorder than to a general log. Runtime receipts versus screenshots Runtime receipts are protocol-level evidence. A batch or waitFor receipt can prove that the app saw a command, observed state, matched or failed a condition, and returned a typed result. Screenshots only prove pixels at a moment in time. For automation and regression proof, receipts are usually stronger than screenshots. Screenshots should be used when the claim is visual: layout, focus ring placement, clipped text, wrong window selection, or styling. Privacy-safe value previews versus raw user content dumps Untrusted strings must not be logged raw. This includes stdin text, chat titles, dictation queries, triggerBuiltin names, ACP command displays, pasted text, and similar user-controlled values. The safe logging contract uses byte-capped previews and metadata fields instead of dumping the full string. The shared preview helper returns a log-safe display value with: Raw byte count. Safe byte count. Truncation flag. UTF-8 boundary preservation. Trailing whitespace trimming. Ellipsis inside the byte budget. The shared rate limiter buckets by category, key length, and hash of the key, so the raw untrusted string is not retained in limiter state. Stable trace markers Some verbose trace flows intentionally use stable marker strings so an operator can filter a live ./dev.sh session while reproducing a bug. Known markers and targets include: DO_IN_TRACE: follows shared filter input changes, current-app command normalization, intent resolution, and built-in execution routing for “Do in Current App”. SCROLL_TRACE: follows current-app command list rendering, wheel-step accumulation, scroll metrics, wheel-owned selection notes, and reanchor decisions. script_kit::input_history: follows main-menu Up/Down routing, render-paced key-repeat coalescing, render acknowledgments, obsolete-pending cancellation, programmatic filter echo suppression, history indices, and ScriptList key-up receipts. Trace fields that include user-entered text must use safe previews, not raw values. Protocol stats Protocol stats are atomic counters and derived health flags for Rust/Bun protocol-boundary issues. The MCP resource kit://diagnostics/protocol-stats exposes a JSON report containing snapshot, health, and thresholds. Thresholds are strict for malformed protocol input and more tolerant for expected noisy traffic such as unknown triggerBuiltin typos. AI preflight audits AI preflight audits are persistent JSONL records for message-preparation and context-resolution decisions. They are schema-versioned, bounded, compacted, deduped by correlation id during reads, and tolerant of malformed or unsupported-schema lines. Automation And Protocol Surface Stdin JSON automation contract Automation uses stdin JSON messages to drive the app. Command-line script arguments are explicitly not the durable automation contract. Agentic tools should choose among protocol-level batch input, GPUI event simulation, and native input based on the surface being tested. Receipts waitFor and batch are backed by a transaction executor that records: Request id. Success/failure. Per-command result entries. Before/after state snapshots. Poll observations. Elapsed timings. Failed command index. Typed error codes. Actionable failure suggestions. Optional transaction trace. Receipts are the preferred proof for automation behavior because they show what the runtime observed and returned. Trace modes Transaction trace mode controls inclusion/persistence: Off: no trace included/persisted. On: trace included/persisted regardless of success. OnFailure: trace included/persisted only on failure. should_include_trace(mode, success) is the central inclusion policy. Trace schema and persistence Transaction traces are schema-versioned. The executor uses TRANSACTION_TRACE_SCHEMA_VERSION when constructing traces. Persisted traces are JSONL records appended through append_transaction_trace. Trace records include: schema_version request_id command_fingerprint status started_at_ms total_elapsed_ms failed_at commands Command traces include command names, before/after snapshots, poll observations, and timing/error details. Idempotency and replay The executor computes a stable command fingerprint for the payload. For a repeated request_id: If an existing trace has the same fingerprint, the executor returns the prior trace/result instead of repeating side effects. If an existing trace has no fingerprint, it is treated as legacy and ignored with a warning. If an existing trace has a different fingerprint, the executor rejects the request as a different transaction payload. The replay script scripts/agentic/tx_trace_replay_idempotency.ts and tests/tx_trace_replay_idempotency_contract.rs cover this behavior. MCP transaction resources src/mcp_resources/transaction_resources.rs exposes: kit://transactions/latest kit://transactions/schema The latest resource reads persisted traces through read_latest_transaction_trace. If no trace exists, it returns a JSON payload with: JSON{ "kind": "transactionTrace", "status": "empty", "message": "No transaction traces found"} Malformed resource URIs should produce actionable errors. Request-id filtering is part of the resource contract. Protocol stats resource kit://diagnostics/protocol-stats returns a ProtocolStatsReport with camelCase fields: snapshot health thresholds The report is generated from atomic counters in src/protocol_stats.rs. Tracked counters include: stdin_parse_failed_total stdin_command_too_large_total stdin_unsupported_protocol_version_total trigger_builtin_unknown_total trigger_builtin_deprecated_name_total Health flags are stable ordered and threshold based. Protocol ingress observation The protocol ingress observer parses raw JSONL lines and composes version reading with deprecation validation. Unsupported protocol versions are counted before normal parse dispatch. The unsupported-version warning is rate-limited per (stdin_unsupported_protocol_version, found) window, and the protocol-stats health resource flips unhealthy on the first unsupported version. Compact automation logs Agentic tooling and test harnesses use SCRIPT_KIT_AI_LOG=1. The startup readiness marker matches compact logs via STARTUP_READY so warm sessions can correctly report readiness after successful startup. Data, Storage, And Privacy Boundaries Runtime logs Compact logs are emitted to stderr/console. Full session logs are available as JSONL, including ~/.scriptkit/logs/latest-session.jsonl. Logs may include structured fields, trace targets, component categories, elapsed time, status, and byte-count metadata. Logs must not include raw user-controlled content unless a specific owning feature has explicitly approved and tested that disclosure. Safe user-value previews Default safe preview budget: SAFE_USER_VALUE_MAX_BYTES: 200 bytes. Important details: The cap is bytes, not characters. The helper walks back to a UTF-8 char boundary. The helper trims trailing whitespace. The helper appends … inside the byte budget. The helper reports raw and safe byte counts. The helper reports whether truncation occurred. Required structured fields: *_preview*_bytes*_safe_bytes*_truncatedsuppressed The preview is allowed diagnostic content. The raw full value is not. Rate-limited untrusted-value logging The rate limiter: Buckets by (category, key.len(), hash(key)). Does not retain the raw untrusted key. Uses a default 30-second window. Garbage-collects stale buckets after 120 seconds. Caps the map at 2,048 keys with pruning. Prevents hostile clients from growing the limiter unbounded by randomizing keys. This limiter is complementary to occurrence-count gates such as protocol_stats::should_log_occurrence(total). Occurrence counters are useful for aggregate thresholds; rate limiting is needed for same-key bursts. Transaction trace storage Transaction trace persistence: JSONL append-only under the default transaction log path or an explicit test path. Maximum size: 10 * 1024 * 1024 bytes. Compaction keep count: 2,000 valid traces. Malformed entries are skipped with warnings during read/compaction. Readers stream line-by-line and do not load the whole log into memory just to find the latest trace. A process-local mutex serializes append/read/compact operations. Transaction traces may contain UI state snapshots and command parameters. They should be treated as diagnostic records, not public telemetry. AI preflight audit storage AI preflight audit persistence: Default path: ~/.scriptkit/logs/ai-preflight-audits.jsonl. Schema version: AI_PREFLIGHT_AUDIT_SCHEMA_VERSION = 2. Maximum size: 5 * 1024 * 1024 bytes. Compaction keep count: 2,000 audits. Malformed JSONL lines are skipped with warnings. Unsupported schema versions are skipped with warnings. Read path dedupes by correlation id. The structured ai_preflight_audit log event includes counts and identifiers, including final user content length, but should avoid dumping raw final user content into normal logs. Protocol stats storage Protocol stats are in-memory atomic counters exposed through MCP. They are not the same as persisted transaction traces or JSONL session logs. The MCP resource makes current health visible without requiring file access. Privacy boundary summary Allowed: Safe previews within byte budget. Byte counts. Truncation flags. Suppression counts. Correlation ids. Request ids. Schema versions. Status and error codes. Aggregate counters. State snapshots where explicitly part of protocol trace contract. Risky or forbidden without explicit contract: Full raw prompt text. Full pasted text. Full dictation transcript. Raw stdin payloads. Raw unknown trigger names. Raw chat titles. Raw ACP command display strings. Unbounded JSON dumps of user-controlled protocol payloads. Error, Empty, Loading, And Disabled States Compact logging disabled State: SCRIPT_KIT_AI_LOG=0 Result: Standard verbose logs are emitted. Compact log filtering assumptions do not apply. Operators should not expect SS.mmm|L|C|message formatting. Debug logging not enabled State: RUST_LOG not set to debug/trace Result: Some trace markers may not appear. Operator should rerun with RUST_LOG=debug or a targeted module filter. Missing latest session log Possible causes: App has not started. Log path not created. Different session/log path active. Process crashed before log initialization. Expected handling: Do not fabricate log proof. Use direct runtime stderr or restart with SCRIPT_KIT_AI_LOG=1. Transaction trace file missing read_latest_transaction_trace returns None, not an error. MCP latest transaction resource returns an empty JSON payload with status empty and message No transaction traces found. Malformed transaction trace JSONL Malformed lines are skipped with warnings. Valid later entries remain readable. Compaction rewrites only retained valid entries. Regression risk: a malformed line must not poison the whole trace log. Oversized transaction trace log When the trace log exceeds 10 MiB: Compaction runs before append. Valid traces are read. Malformed traces are skipped. Latest 2,000 valid traces are retained. Log is rewritten. Legacy transaction trace without fingerprint When idempotency lookup finds a trace with an empty command fingerprint: The executor warns. The legacy trace is ignored. The new request executes normally. Same request id with different transaction payload The executor rejects the command. This prevents accidental request-id reuse from hiding different side effects. Trace mode disabled When TransactionTraceMode::Off: Runtime receipts still exist. Trace inclusion/persistence does not happen. Operators should not expect MCP latest trace to update from that command. OnFailure trace mode with successful command When TransactionTraceMode::OnFailure and success is true: No trace is included/persisted. This is expected, not a missing-trace bug. Protocol stats unhealthy When thresholds are crossed: health.ok becomes false. health.flags contains stable ordered flag names. Operators should inspect the corresponding counters. Strict threshold examples: Any stdin parse failure. Any too-large stdin command. Any unsupported protocol version. Expected-noise threshold examples: Unknown triggerBuiltin names tolerate several typos before flagging. Deprecated triggerBuiltin names tolerate a high volume of legacy SDK calls before flagging. AI preflight audit file missing read_preflight_audits returns an empty list. Malformed AI preflight audit JSONL Malformed lines are skipped with warnings. Valid audits continue loading. Unsupported AI preflight audit schema Unsupported schema versions are skipped with warnings. Readers do not try to coerce unknown schemas. AI preflight audit oversized When the audit log exceeds 5 MiB: Compaction runs. Valid/deduped audits are read. Latest 2,000 are retained. Log is rewritten. Rate-limited log suppression When repeated untrusted values hit the limiter: The log site should emit or account for suppressed. Missing repeated log lines may be expected. The absence of spam is a feature, not evidence that the event stopped happening. Regression risks Replacing byte caps with character caps can allow oversized emoji/combining-mark strings into logs. Logging raw stdin/chat/dictation/paste values can leak private content. Using occurrence counters alone can allow same-key spam inside a short window. Changing marker strings can break dev-loop filters and agentic debugging scripts. Returning screenshots as proof when receipts are available can hide protocol/state failures. Letting malformed JSONL abort a reader can permanently break diagnostics after one bad line. Removing compaction can allow unbounded growth in transaction or preflight audit logs. Treating request id alone as idempotency proof can hide changed payloads. Ignoring schema versions can misread future audit/trace formats. Changing protocol stats key casing can break MCP consumers. Changing protocol health flag ordering can break stable contract tests. Persisting traces for every success by default can create unnecessary diagnostic volume. Omitting trace on failure can make automation failures non-actionable. Emitting final user content in AI preflight logs would violate the intended privacy boundary. Letting transaction resource URI parsing accept malformed requests silently can lead agents to inspect the wrong trace. Depending only on cargo check can miss source-audit contract drift.
