Executive summary Feature 038 is the intersection of two live systems: Agent Skills Catalog — discovery, routing, search, and staging of repo-local or plugin-owned SKILL.md files. AI Context Catalog — typed, schema-versioned context attachments that flow into ACP Agent Chat through picker, portal, slash, mention, main-menu, SDK, and focused-target paths. The core contract is simple: the UI should never inline large skill/context bodies into the composer. It should stage compact visible tokens/chips, preserve typed AiContextPart records, and resolve those records at submit time into deterministic prompt blocks and receipts. The canonical ownership map says mcp-context-resources owns MCP resources and resource-backed catalogs, acp-context-composer owns ACP composer context parts and attachment tokens, acp-chat-core owns Agent Chat lifecycle, sdk-script-execution owns SDK/script execution behavior, and protocol-automation owns stdin JSON receipts and state proof. agent-skills-ai-context-atlas 2. Product definition Feature name: Agent Skills and AI Context Catalog Feature ID: 038 Primary user value: let users and agents attach the right operational context — desktop state, files, selected/focused targets, pasted text, MCP resources, and reusable skills — to ACP Agent Chat without losing source identity, ordering, previewability, or failure diagnostics. Non-negotiable behavior: Skills are discoverable from plugin-owned skills/<skill_id>/SKILL.md files. Skill selection produces a visible slash token or chip, not a pasted skill body. Context attachments are represented as typed AiContextPart variants. Context resolution happens at submit time. Partial context failure is surfaced through receipts; successful context is not discarded because another part failed. Preview metadata is synchronous and cheap. Portal return must restore the originating ACP/composer state. Verification should use the narrowest proof that can fail on the changed contract. 3. Live data model 4.1 AiContextPart is the central attachment contract The feature’s single most important type is AiContextPart. It is serde tagged with kind using camelCase, so this is both Rust model and protocol shape. The live variants are: resourceUri — MCP resource, such as kit://context?profile=minimal filePath — local file attachment skillFile — local skill attachment selected from slash mode or main menu focusedTarget — explicit surface-native target, such as selected script/file/clipboard entry ambientContext — display-only promoted Ask Anything chip textBlock — raw text, URLs, logs, pasted snippets, or similar inline material The enum definition and label/source accessors pin this as the central model. agent-skills-ai-context-atlas agent-skills-ai-context-atlas 4.2 Resolution output At submit time: ResourceUri resolves through mcp_resources::read_resource. FilePath reads file contents and can fall back to metadata-only if unreadable. SkillFile resolves through build_staged_skill_prompt. FocusedTarget resolves into a deterministic <context source="focusedTarget"> block. AmbientContext is display-only and produces no prompt block. TextBlock resolves as explicit text content. The resolver returns ContextResolutionReceipt with attempted, resolved, failures, and prompt_prefix, which is the machine-verifiable proof of what reached the prompt. agent-skills-ai-context-atlas 4.3 Prepared message receipt prepare_user_message_with_receipt takes raw user text and pending parts, resolves every part, records per-part outcomes, aggregates a ContextResolutionReceipt, and returns a schema-versioned PreparedMessageReceipt with a final send decision. agent-skills-ai-context-atlas This is the correct submit-path boundary. New feature code should not invent separate “context assembly” paths unless it can prove parity with this receipt model. 5. Agent Skills Catalog 5.1 Durable workspace location The live workspace contract centers plugin content under ~/.scriptkit/plugins/; main/ is the default personal plugin, scriptkit/ owns bundled skills, and plugin skills live under ~/.scriptkit/plugins/*/skills/. The setup layer also seeds bundled verb-named skills, example skills, config/theme templates, ACP agent catalog files, and migration from legacy ~/.kenv. agent-skills-ai-context-atlas agent-skills-ai-context-atlas 5.2 Discovery contract discover_plugin_skills scans each plugin’s skills directory. Each skill must live at: <plugin_root>/skills/<skill_id>/SKILL.md Results are sorted deterministically by (plugin_id, skill_id). Title resolution order is frontmatter title → first # H1 → skill_id; description comes only from frontmatter description. agent-skills-ai-context-atlas The implementation skips plugins without a skills directory, skips entries without SKILL.md, reads frontmatter, derives plugin title fallback, logs plugin_skill_cataloged, and pushes PluginSkill records. agent-skills-ai-context-atlas 5.3 Search contract fuzzy_search_skills matches against title, skill_id, plugin_title, and description, returning SkillMatch results sorted by highest relevance score. Empty query returns every skill with score 0, preserving catalog availability for browse mode. agent-skills-ai-context-atlas Scoring favors exact title match heavily, then title substring, title word-boundary bonus, Nucleo fuzzy match, skill_id, plugin_title, and description. Ties sort by title, not file path or plugin load order. agent-skills-ai-context-atlas 5.4 Skill attach contract When the attachment portal sees a selected SearchResult::Skill, it constructs an AiContextPart::SkillFile with path, label/title, skill_name, owner_label, and slash_name. agent-skills-ai-context-atlas Slash command acceptance follows the same “compact visible token, staged hidden content” rule: accepting default, plugin, or Claude Code skills inserts a literal /{slash-name} token into the composer. Plugin/Claude skill bodies attach as AiContextPart::SkillFile and reach the agent at submit time via the shared staged-skill resolver. agent-skills-ai-context-atlas 6. Edge cases and failure modes Do not paste skill bodies into the composer. The visible composer gets /{slash-name} while SkillFile carries the body to submit-time resolution. Pasting bodies breaks compact composer UX, duplicates context, and makes dedup harder. Do not treat ambient chips as resolved prompt content. AmbientContext is display-only; actual desktop capture is staged separately. The preparation outcome marks it as DisplayOnly. agent-skills-ai-context-atlas Do not collapse all failure cases into raw-message-only submit. Missing context should produce blocked or partial receipts, not silently discard attachments. Do not assume detached ACP supports every portal. Detached ACP currently supports local AcpHistory; broader main-window portals depend on the embedded launcher return contract. agent-skills-ai-context-atlas Do not rely on older AI-context docs. Current drift: tab_ai_submit is screenshot-free, tab_ai is the screenshot profile, kit://context supports diagnostics and per-field flags, FocusedTarget and attachment portal are live code, and ContextPreviewInfo is the preview contract. agent-skills-ai-context-atlas Do not use legacy skill names as canonical routing. Legacy .claude/skills/* material may be mined, but the 26-skill taxonomy is canonical. agent-skills-ai-context-atlas 12.
