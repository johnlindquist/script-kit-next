016 Prompt Runtime Core / arg() / select() / div() / md() Executive Summary Feature 016 is the core prompt runtime that lets SDK scripts ask the GPUI app for user input through four primary SDK surfaces: arg() collects a single string from either typed text or a selected choice. It supports placeholder/config overloads, choices, SDK actions, filtering, focused text input, Enter submission, empty-choice behavior, and an actions menu. select() presents a selectable list and returns an array of selected values. The current SDK message always sends multiple: true, so this chapter should treat the SDK select() surface as multi-select unless future code proves a single-select overload. Internally, the Rust select prompt supports both multi-select and single-select behavior, because row activation toggles when multiple is true and submits immediately when false. div() renders HTML content in a prompt panel and resolves when the user submits through supported div interactions or a forced/protocol submit. It can carry SDK actions and container styling options. It does not own a text input focus. md() is an SDK helper that converts Markdown into HTML strings for div() or other HTML-consuming prompt surfaces. It is not a prompt by itself and does not send a runtime message. The Rust prompt handler routes SDK messages into AppView::ArgPrompt, AppView::SelectPrompt, and AppView::DivPrompt, assigns focused-input state, creates submit callbacks, resizes the window to the prompt type, exposes state through test/protocol helpers, and supports automation paths such as getState, getElements, simulateKey, selectByValue, selectFirst, and ForceSubmit. The core invariant: SDK prompt calls create a unique prompt id, send a typed message to Rust, Rust installs an app view with the same id, user/protocol interaction submits that id with a value, and the SDK pending promise resolves with the expected return shape. What Users Can Do Users can call await arg(...) from a script to ask for one value. They can type free text, filter choices, select a choice, press Enter, use an actions menu, or submit an empty/no-choice prompt depending on the state. Users can call await select(placeholder, choices) to present a list of choices and gather multiple selections. They can search/filter, move focus, toggle the focused row, toggle all filtered rows, click rows, and submit the selected set. Users can call await div(...) to show rendered HTML. The div prompt can include links or submit-capable HTML, can show SDK actions, and can be submitted through UI/protocol mechanisms. It is a panel surface, not a text-entry surface. Users can call div(md(markdown)) to render Markdown content as HTML. The SDK helper performs conversion before the prompt message is sent. Based on the visible SDK snippet, fenced code blocks are converted to <pre><code> with an optional class from the fence language. Users and agents can inspect the active prompt type, current prompt id, input value, selected value, and visible elements through protocol/test probes. Agents can simulate keys and use safe submit helpers without reaching into private UI internals. Core Concepts Prompt id. Each SDK prompt call uses nextId() in scripts/kit-sdk.ts, stores a pending resolver keyed by that id, and sends a message containing the id. Rust stores the same id in AppView::*Prompt. Submit paths look up the current view id and submit a response for that id. Prompt message. The SDK sends discriminated messages: ArgMessage: { type: "arg", id, placeholder, choices, actions? } DivMessage: { type: "div", id, html, containerClasses?, actions? } SelectMessage: { type: "select", id, placeholder, choices, multiple } These are shown in scripts/kit-sdk.ts. App view. Rust routing installs prompt-specific app state: AppView::ArgPrompt { id, placeholder, choices, actions } AppView::SelectPrompt { id, entity } AppView::DivPrompt { id, entity } Rendering dispatches from src/main_sections/render_impl.rs to render_arg_prompt, render_select_prompt, and render_div_prompt. Submit callback. PromptHandler::make_submit_callback creates a callback that sends a prompt response back to the SDK. Div, select, and adjacent prompt surfaces use this path. Arg uses app-level submit helpers that eventually submit the current prompt id and value. Focused input. Arg owns the main filter/text input and sets focused_input = FocusedInput::ArgPrompt. Div sets focused_input = FocusedInput::None. Select sets focused_input = FocusedInput::None and focuses FocusTarget::SelectPrompt. Actions host. Actions menus are scoped by ActionsDialogHost so shortcuts and dismiss behavior route to the active surface. Arg uses ActionsDialogHost::ArgPrompt; div uses ActionsDialogHost::DivPrompt. Select is listed as a native footer surface and has prompt routing, but the provided snippets do not prove a select-specific actions host implementation. Receipts. The app reports prompt identity through helpers such as current_prompt_type(), current_input_value(), current_selected_value(), and element collection. These make automation deterministic. Entry Points SDK arg() globalThis.arg = async function arg(placeholderOrConfig?: string | ArgConfig, choicesInput?: ChoicesInput, actionsInput?: Action[]): Promise<string>. It accepts either a placeholder string or a config object. It normalizes choices and actions, registers a pending resolver, sends an ArgMessage, and resolves with the submitted string. SDK select() globalThis.select = async function select(placeholder: string, choices: (string | Choice)[]): Promise<string[]>. It normalizes choices, sends a SelectMessage with multiple: true, and resolves with an array. The visible pending handler resolves [] for one fallback/empty path. SDK div() globalThis.div = async function div(htmlOrConfig?: string | DivConfig, actionsInput?: Action[]): Promise<string | void>. It accepts an HTML string or config, serializes actions, registers a pending resolver, sends a DivMessage, and resolves with msg?.value. SDK md() globalThis.md = function md(markdown: string): string. It returns HTML. It is synchronous and does not register a pending prompt. It is an SDK rendering helper, commonly paired with div(). User Workflows Arg: free-text submit. A script calls await arg("Name"). SDK sends an arg message with a prompt id. Rust sets AppView::ArgPrompt, clears arg_input and filter_text, sets selected index to zero, focuses FocusedInput::ArgPrompt, requests focus for FocusTarget::MainFilter, and resizes to a mini prompt or no-choice prompt depending on choices. The user types text. Enter resolves through submit_arg_prompt_from_current_state, which chooses either the selected choice value or typed text via resolve_current_arg_submit_outcome. The SDK promise resolves to a string. Arg: choice submit. A script calls await arg("Pick one", choices). The prompt opens with choices and text focus. Typing filters the choice list. If a filtered choice is selected, Enter submits that choice’s value. Automation can call selectByValue, selectBySemanticId, or selectFirst on choice-backed prompts; those helpers only support ArgPrompt, MiniPrompt, and MicroPrompt in the visible code. Arg: no choices. When there are zero choices, Rust may use ViewType::ArgPromptNoChoices and still focuses arg input. The visible handler has a no-choice branch that clears input, sets FocusedInput::ArgPrompt, focuses AppRoot, and resizes to ArgPromptNoChoices. Exact conditions differ across arg/mini/micro prompt routes; treat no-choice layout as explicit and tested separately. Arg: actions. Arg can include serialized SDK actions. render_arg_prompt routes keyboard preamble through ActionsDialogHost::ArgPrompt, draws an actions backdrop with id "arg-actions-backdrop", and preserves pointer dismissal. Simulated Cmd+K toggles arg actions. In the actions dialog, Enter can execute an action; Escape closes the dialog and restores arg focus. Select: multi-select list. A script calls await select("Pick", choices). SDK sends type: "select" with multiple: true. Rust constructs SelectPrompt::new(id, placeholder, choices, multiple, submit_callback, theme), stores it in AppView::SelectPrompt, sets no global text input focus, focuses FocusTarget::SelectPrompt, and resizes based on choice count. The prompt owns list focus and selection state. Select: toggle and submit. When multiple is true, clicking a row toggles selection instead of submitting. Keyboard intent ToggleFocusedSelection calls toggle_selection; ToggleAllFiltered calls toggle_select_all_filtered. Submit resolves selected values as an array. The visible code proves row activation branches between this.toggle_selection(cx) and this.submit() depending on this.multiple. Select: single-select internal behavior. Although SDK select() sends multiple: true, Rust select internals support multiple: false: mouse activation submits immediately instead of toggling. This should be treated as internal/reusable behavior unless another SDK entry point exposes it. Div: HTML prompt. A script calls await div("<h1>Hello</h1>") or passes a config. SDK sends HTML and optional container classes/actions. Rust prepares the UI for a div prompt, stores SDK actions, creates a submit callback, builds ContainerOptions, creates a DivPrompt, sets AppView::DivPrompt, sets focused_input = FocusedInput::None, focuses app root, resizes to ViewType::DivPrompt, and notifies GPUI. Div: links and submit. The prompt renders HTML and can return a submitted value. The SDK resolver uses msg?.value, so a div submission with no value resolves undefined/void, while value-bearing submit paths resolve that string. Tests listed in the bundle include tests/smoke/test-div-submit-links.ts, which indicates link/submit behavior is covered, but the snippet does not show the implementation details. Markdown to div. A script calls await div(md(markdown)). md() converts Markdown to HTML before the div prompt is created. The visible conversion includes fenced code blocks. The bundled tests include tests/sdk/test-md.ts and tests/smoke/test-md-div-integration.ts, which are recommended proof targets for Markdown rendering and div integration. Visual And Focus States Arg is rendered by render_arg_prompt. It has an input cursor/selection renderer, a visible-window calculation for long input, choice rendering, and an actions backdrop. The arg input visible range is clamped using minimum and maximum visible characters, with truncation flags for left/right overflow. Arg sets FocusedInput::ArgPrompt, requests FocusTarget::MainFilter, clears arg_input, clears filter_text, and resets arg_selected_index = 0 on entry. No stale input should carry across arg prompts. Select is rendered by render_select_prompt through render_prompts/other.rs. The wrapper uses a simple prompt shell and a select-owned key handler. Select state includes filtered choices, focused index, hovered index, selected set, and a list scroll handle. Resetting the query scrolls to item 0. Div is rendered by render_div_prompt. It uses ActionsDialogHost::DivPrompt, an actions backdrop id of "div-actions-backdrop", and GPUI entity rendering. Div sets FocusedInput::None and FocusTarget::AppRoot; it should not accidentally receive or report text input. Surface metadata maps arg, select, and div to prompt surfaces. native_footer_surface() returns "arg_prompt", "select_prompt", and "div_prompt" for footer ownership. Keystrokes And Commands Arg uses the main prompt input. Enter submits current state through submit_arg_prompt_from_current_state. Escape can cancel by submitting None through the prompt response path in simulateKey. Cmd+K toggles arg actions in the visible simulateKey route. When the arg actions dialog is open, Enter executes the selected action, Escape closes the dialog, restores FocusedInput::ArgPrompt, and focuses the app handle. Select uses select-owned keyboard intent handling. The visible key intent dispatch supports: move up/down, submit, backspace, append character, toggle focused selection, toggle all filtered, let global handle, ignore. Mouse activation on rows is conditional: multi-select toggles, single-select submits. Div key handling goes through render_div_prompt with a prompt key preamble configured as dismissable and host-owned by ActionsDialogHost::DivPrompt. The preamble stops propagation for global shortcuts and handled keys. Exact div-specific key commands are not shown in the bundle; treat Enter/link behavior as proven by smoke-test names, not by implementation snippets. Automation simulateKey explicitly shows arg/mini prompt routing. The provided snippets do not prove equivalent select/div simulateKey internals, so verification should use the smoke tests and protocol receipts before relying on unsupported keys. Data, Storage, And Privacy Boundaries Typed arg text is transient prompt input stored in self.arg_input and exposed to automation through current_input_value(). It is submitted back to the SDK when the user chooses text-submit. It should not be stored as persistent app data by this feature. Choice data crosses the SDK/App boundary. Choice labels, values, and metadata sufficient for rendering and semantic selection are sent in prompt messages and exposed through element collection/test probes. Treat choice metadata as script-provided data visible to automation. Select selection state is held inside SelectPrompt and returned as string[] to the SDK. In multi-select mode, toggling changes UI state without resolving until submit. Div HTML is script-provided rendered content. It is sent from SDK to Rust and displayed in the app. It may include links or submit-bearing elements. Since it is rendered content, do not treat it as private from UI automation; getElements exposes the div panel and tests may inspect behavior. Markdown passed to md() is converted locally in the SDK into HTML. The Markdown source itself is not a prompt message unless the caller also sends it. The resulting HTML is sent through div(). Action payloads are serialized from SDK actions and stored as current SDK actions while the prompt is active. Action execution should be scoped to the active prompt host. Return values go back to the SDK pending resolver only for the prompt id that submitted. Arg resolves a string, select resolves a string array, and div resolves a string or void. Error, Empty, Loading, And Disabled States Empty arg choices are explicit. The handler distinguishes zero-choice layouts and can resize to ArgPromptNoChoices. Arg still supports typed input unless disabled elsewhere; no disabled arg path is proven in the snippet. Empty select choices are explicit. The handler chooses ViewType::ArgPromptNoChoices when select choice count is zero, despite the prompt being select. This is probably a shared no-list visual layout. Verify empty select with SDK and smoke tests because the naming is easy to misread. Empty div content is allowed by SDK shape because div() accepts optional htmlOrConfig and resolves through msg?.value; exact default HTML is not shown. Treat blank div as a valid panel unless implementation proves rejection. Loading states are not proven in the provided snippets for arg/select/div. If choice loading or disabled metadata exists elsewhere, this bundle does not show it. Mark any loading/disabled behavior as unverified until Choice type and render code are expanded. Disabled choices are not proven in the snippets. The atlas should not claim disabled choice keyboard behavior without additional source.
