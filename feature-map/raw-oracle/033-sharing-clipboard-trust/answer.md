033 Sharing and Clipboard Trust Install Executive Summary Script Kit GPUI feature 033 turns selected launcher items into portable clipboard share bundles, then lets another Script Kit GPUI instance detect that clipboard payload and install it only after an explicit trust prompt. The feature has two halves: Export/share path: the existing copy_deeplink action becomes Share for portable shareable launcher results. It serializes a ScriptShareBundle, base64-url-encodes it, prefixes it with scriptkit-share://v1/, copies that URI to the clipboard, and marks it as recently exported so the same local app does not immediately prompt the sender to import their own share. Import/trust path: startup spawns a lightweight clipboard watcher outside tests. The watcher polls clipboard change count every 350 ms, only inspects clipboard text after a change or detector error, extracts a scriptkit-share://v1/... URI even when wrapped in surrounding text, suppresses recently exported and recently prompted duplicates, then opens a parent confirmation dialog. Only Install writes files. Ignore, close, or dialog failure prevents install. The critical security model is simple and blunt: this is not a signed bundle, not remote transport, not automatic trust, and not a sandbox. The clipboard payload contains file contents. Install is gated by a trust prompt, path validation, known top-level directories, plugin-id normalization, and writing only under ~/.scriptkit/plugins/<plugin-id>/. Primary ownership is in src/script_sharing.rs, src/app_actions/handle_action/files.rs, src/actions/builders/script_context.rs, src/app_impl/startup.rs, and the shared confirmation helper in src/confirm/parent_dialog.rs. Recent-export suppression When a local user shares something, the app writes the share URI to the clipboard. Without suppression, the startup watcher would see the new clipboard text and immediately ask the sender to install their own share. Clipboard watcher spawn_clipboard_share_watcher returns an async_channel::Receiver<ClipboardShareImport> and spawns a thread. Visible behavior from src/script_sharing.rs: Creates a bounded channel with capacity 8. Keeps an optional arboard::Clipboard. Uses ClipboardChangeDetector. Sleeps for 350 ms each loop. Calls detector.has_changed().unwrap_or(true). If the detector says unchanged, it continues without inspecting clipboard text. If the detector errors, it treats that as changed and may inspect clipboard text. After valid detection, it sends ClipboardShareImport { uri, bundle }. If the receiver is closed, send_blocking fails and the watcher breaks. The important privacy boundary is that normal unchanged clipboard polls do not read clipboard payload text. Data, Storage, And Privacy Boundaries Clipboard payload exposure A share URI contains encoded JSON with file contents. Base64 URL-safe encoding is transport encoding, not encryption. Practical consequence: Anyone or anything that can read the clipboard can read the shared script/skill/agent contents after decoding. This includes clipboard managers, other apps with clipboard access, and logs if the URI is logged. The files.rs snippet logs share_uri = %share_uri while copying. That is operationally sensitive: the logged URI can contain the full payload. If logging behavior is changed, treat share URIs as secret-like content. Clipboard inspection boundary The watcher avoids reading clipboard text unless: the pasteboard change detector says the clipboard changed, or change detection errors, because unwrap_or(true) treats that as changed. This limits passive clipboard reads during stable clipboard state, but it does not avoid reading changed text. Trust boundary The trust prompt is the main user-facing security gate. Install does not proceed until the user explicitly clicks Install. Ignore, close, cancel, helper failure, or confirmation error prevents installation. There is no evidence in the supplied bundle of: cryptographic signatures, publisher verification, remote attestation, notarized bundles, hash pinning, malware scanning, sandboxed execution, automatic trust based on plugin metadata. Do not imply any of those exist. SearchResult::Script, SearchResult::Scriptlet, SearchResult::Skill, and SearchResult::Agent. Which fall back to plain deeplinks? Anything else. If it has a launcher command id, it copies scriptkit://commands/{commandId}. Otherwise it copies scriptkit://run/<deeplink-name>. What exact files are included? Scripts include scripts/<filename>. Scriptlets include scriptlets/<filename>. Skills include the collected skills/<skill_id>/ directory with SKILL.md as entry. Agents include agents/<filename>. How are plugin manifest fields resolved? Bundle construction uses resolve_plugin_manifest. Visible script code passes script plugin id, plugin title, kit name, and script name. Visible skill code includes skill plugin id and skill title. Agent and scriptlet resolver details are partly outside the packed windows. Install then normalizes the plugin id, chooses a unique final id/root, writes plugin.json, and trims at least visible manifest fields such as repo_url. Full fallback precedence should be verified in the complete resolve_plugin_manifest implementation. What prevents copied local share links from prompting immediate import? mark_recently_exported_share records the just-copied URI hash and time. should_ignore_recently_exported_share suppresses it for RECENT_EXPORT_TTL, which is 5 seconds. What does the clipboard watcher read, and when does it avoid reading payload data? It polls clipboard change count every 350 ms. If ClipboardChangeDetector::has_changed() returns false, it continues without reading clipboard text. If changed, or if change detection errors, it may read clipboard text and try to extract/decode a share URI. What trust prompt does the user see? Title: Install Shared <Kind>? Body: Script Kit found a shared <kind> on your clipboard. followed by title, plugin label, file count, and Only install it if you trust the sender. Buttons: Install and Ignore. What happens on Install vs Ignore? Install runs install_share_bundle. On success it refreshes scripts and skills, returns to ScriptList, and shows an installed HUD. Ignore, close, cancel, dialog-open failure, or confirmation error prevents install. Where does installation write files? Under ~/.scriptkit/plugins/<unique-plugin-id>/. How are plugin id collisions handled? unique_plugin_root chooses a sibling directory. Visible code tries suffixed forms like <plugin-id>-shared-2 through <plugin-id>-shared-999, then a hash suffix. The first collision fallback before -2 is likely <plugin-id>-shared, but that exact omitted line should be verified. What file paths are rejected? Empty paths, absolute paths, paths containing .., and paths outside known top-level roots such as assets/file.txt. Which top-level directories are allowed? scripts/, scriptlets/, skills/, and agents/. Which source paths refresh after successful install? Visible startup code calls app.refresh_scripts(cx) and app.refresh_skills(cx). No explicit refresh_agents call is visible; agent discovery is likely covered through script/plugin refresh, but that is an inference. What can agents verify without touching real clipboard or filesystem? Encoding/decoding, wrapped URI extraction, prompt copy, action labels, shareable-kind boundary, plain deeplink source audits, and invalid path validation. Install can be verified safely with a temp HOME, not the real filesystem. What tests should run before changing share bundle format, action availability, startup watcher, or install behavior? Run the script sharing unit tests, action builder tests, the command namespace source audit, parent-dialog/helper tests, cargo check --lib, cargo fmt --check, git diff --check, and lat check. What should not be assumed? Do not assume binary asset support, remote transport, OS-level protocol handling, signed bundles, trusted plugin metadata, automatic install, publisher verification, atomic rollback, or sandboxing. Do not write verification notes, plans, or artifacts unless the local workflow explicitly does that outside this response. Do not mutate the real clipboard in automated tests unless the harness is explicitly designed for it. Prefer temp HOME for install tests.
