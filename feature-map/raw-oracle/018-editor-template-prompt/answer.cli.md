018 Editor and Template Prompt / editor() / template() Executive Summary Feature 018 has two related but distinct prompt surfaces: editor() opens the full-height AppView::EditorPrompt surface. It is a GPUI editor-oriented child-content prompt with content, language, optional SDK actions, full-height window sizing, native footer ownership, submit-on-Cmd+Enter / Cmd+S, and snippet/tabstop behavior when the content or explicit editor template contains tabstops. The prompt handler routes it through PromptMessage::ShowEditor, builds an EditorPrompt, stores SDK actions, installs a submit callback for "editor", sets pending_focus = FocusTarget::EditorPrompt, and defers a guarded resize to ViewType::EditorPrompt. template() opens the dedicated AppView::TemplatePrompt surface. It is not the same thing as editor snippet mode. It parses placeholders from a template string into editable fields, renders a field list plus preview, validates values, and submits a rendered template string. The prompt handler routes it through PromptMessage::ShowTemplate, builds prompts::TemplatePrompt, installs a submit callback for "template", sets pending_focus = FocusTarget::TemplatePrompt, and resizes to ViewType::DivPrompt. The overlap is easy to get wrong: EditorPrompt::with_template(...) is editor snippet mode inside the full-height editor, while TemplatePrompt::new(...) is a form-like template placeholder prompt. The SDK exposes both editor() and template(), and the Rust app owns separate AppView variants, focus targets, element collectors, footer behavior, and sizing contracts for them. The largest operational risks in the bundled source are parity gaps: simulateKey has a rich EditorPrompt arm but no dedicated TemplatePrompt arm; ForceSubmit supports EditorPrompt but not TemplatePrompt; physical editor Escape appears to exit snippet mode but otherwise propagate, while protocol simulateKey Escape cancels the editor prompt; and TemplatePrompt footer UI advertises an Actions button even though the visible actions host map does not include TemplatePrompt. What Users Can Do Users can open a full-height editor prompt from SDK code: TypeScriptconst result = await editor("// code here", "typescript", [ { name: "Format Code", shortcut: "cmd+shift+f" },]) That creates AppView::EditorPrompt, not TemplatePrompt. The editor can display code or text, track language, accept text edits through the embedded editor entity, submit the current content, and optionally expose SDK actions through the shared actions dialog. If the editor content contains explicit tabstops, or if the protocol route supplies an explicit editor template, the same EditorPrompt enters snippet mode and Tab / Shift+Tab navigation moves across snippet tabstops. Users can open a dedicated template prompt: TypeScriptconst result = await template("Hello {{name}}, your email is {{email}}") That creates AppView::TemplatePrompt. The prompt parses placeholders, shows editable fields, validates required and slug-like values, renders a preview, and submits the rendered template. The SDK also preprocesses $SELECTION and $CLIPBOARD before sending the prompt, so selected text or clipboard text can become part of the template string. Users can submit editor content with Cmd+Enter or Cmd+S. They can move through editor snippet tabstops with Tab and Shift+Tab when snippet mode is active. They can open editor actions with Cmd+K when actions exist. In the template prompt, the footer exposes Submit, Next Field, and Actions; Next Field is wired through the footer’s Ai action branch specifically for TemplatePrompt, not through Agent Chat. Core Concepts EditorPrompt is full-height editing. src/prompt_handler/mod.rs builds an EditorPrompt from PromptMessage::ShowEditor. It sets editor_height = px(700.0 - window_resize::layout::FOOTER_HEIGHT), then creates either EditorPrompt::with_template(...) or EditorPrompt::with_height(...). src/window_resize/mod.rs maps ViewType::EditorPrompt to max_height, and src/app_impl/ui_window.rs maps AppView::EditorPrompt to ViewType::EditorPrompt. This is the full-height prompt path. TemplatePrompt is a prompt entity, not the full editor. src/prompt_handler/mod.rs handles PromptMessage::ShowTemplate by calling prompts::TemplatePrompt::new(...), setting AppView::TemplatePrompt, and resizing to ViewType::DivPrompt. In src/main_sections/app_view_state.rs, TemplatePrompt belongs to SurfaceKind::PromptEntity, while EditorPrompt belongs to SurfaceKind::PromptChildContent. Editor snippet mode is not template(). ShowEditor can route into EditorPrompt::with_template(...) when the protocol message has a template field, or when snippet::analysis::contains_explicit_tabstops(&content_str) detects tabstops in the editor content. This uses ParsedSnippet::parse(&template) inside src/editor/mod.rs. Dedicated template() uses TemplatePrompt::parse_template_inputs(...) and placeholder substitution logic in src/prompts/template/prompt.rs. Prompt ids are preserved. Both app views carry id: String. getState maps AppView::EditorPrompt { id, .. } to promptType: "editor" with that id, and AppView::TemplatePrompt { id, .. } to promptType: "template" with that id. Submissions use prompt-specific submit callbacks created by make_submit_callback("editor") or make_submit_callback("template"). Focus is explicit. AppView::EditorPrompt stores a FocusHandle; TemplatePrompt stores one inside the entity. FocusTarget::EditorPrompt and FocusTarget::TemplatePrompt exist in both app state and focus_coordinator. src/app_impl/theme_focus.rs applies pending editor focus to either EditorPrompt or ScratchPadView, while template focus only applies to AppView::TemplatePrompt. Native footer ownership is registered per view. AppView::native_footer_surface() maps EditorPrompt to "editor_prompt" and TemplatePrompt to "template_prompt". Editor rendering routes footer hints through main_window_footer_slot(self.clickable_universal_hint_strip(cx)); TemplatePrompt footer buttons are resolved in src/app_impl/ui_window.rs. The SDK creates a prompt id and sends an editor prompt. The Rust app creates AppView::EditorPrompt, focuses the editor, and resizes the window to the full editor height. The user edits text. Cmd+Enter or Cmd+S calls EditorPrompt::submit, which reads current content and invokes the submit callback with Some(content). Automation can also call simulateKey with key: "enter", modifiers: ["Cmd"]; the dispatcher reads editor.content(...) and calls submit_prompt_response(id, Some(content), ctx). Open editor snippet mode A user or script can reach editor snippet mode in two ways proven by source: One path is an explicit template value in the ShowEditor message. The other path is passing content that snippet::analysis::contains_explicit_tabstops(...) detects as tabstop-bearing. In either case, PromptMessage::ShowEditor builds EditorPrompt::with_template(...), which parses the template through ParsedSnippet::parse(&template). In snippet mode, Tab advances tabstops, Shift+Tab goes backward for physical editor key handling, Escape exits snippet mode, and choices can display a choice popup. When the user leaves snippet mode, the prompt is still an editor prompt; it does not become TemplatePrompt. Use editor actions If editor() receives SDK actions, the prompt handler stores them in self.sdk_actions. AppView::EditorPrompt maps to ActionsDialogHost::EditorPrompt in src/app_impl/actions_dialog.rs. The editor wrapper passes ActionsDialogHost::EditorPrompt to the shared key preamble and renders an actions backdrop with id "editor-actions-backdrop". Cmd+K is the editor actions entry. The shared actions dialog owns popup keyboard routing. Editor action shortcuts are filtered through editor_reserved_shortcut_reason(...); reserved editor shortcuts such as submit, undo/redo, find, and clipboard/selection operations are preserved for editor behavior instead of being consumed by SDK actions. Open and complete a template prompt A user or script calls template("Hello {{name}}"). The SDK preprocesses $SELECTION and $CLIPBOARD if present, then sends a template prompt. Rust creates AppView::TemplatePrompt, parses placeholders, initializes one value slot per unique placeholder, and renders fields plus a preview. The user fills the current field, moves to the next field, and submits. On submit, the prompt validates values and renders the template in a single substitution pass. The single-pass behavior is pinned by tests: if one replacement value is {{second}}, it is not recursively expanded after insertion. Use template footer For AppView::TemplatePrompt, src/app_impl/ui_window.rs builds footer buttons: Run with key label ↵ and visible label Submit. Ai with key label ⇥ and visible label Next Field. Actions with key label ⌘K and visible label Actions. The FooterAction::Ai dispatch branch special-cases TemplatePrompt and calls prompt.next_input(cx). For all other views, FooterAction::Ai opens Agent Chat. That distinction is load-bearing. The source bundle does not show an ActionsDialogHost::TemplatePrompt variant or a TemplatePrompt arm in actions_host_for_view(...). So TemplatePrompt’s visible Actions footer is not proven to have a working actions dialog host in this snapshot. Visual And Focus States EditorPrompt visual state The editor prompt is a full-height child-content prompt. src/window_resize/mod.rs maps ViewType::EditorPrompt to max_height, and src/app_layout/build_layout_info.rs has a special branch for AppView::EditorPrompt that emits an EditorContent prompt component. The layout explanation says the editor fills the content area and footer ownership is routed through the shared main-window footer slot. src/render_prompts/editor.rs wraps the editor entity in a full-width, full-height, overflow-hidden container with key context "editor_prompt". It uses main_window_footer_slot(self.clickable_universal_hint_strip(cx)), not a prompt-local footer. tests/minimal_chrome_audit.rs explicitly pins that EditorPrompt should not reference PromptFooter and that the wrapper should not instantiate PromptFooter::new(...). Editor focus is handled twice: ShowEditor stores a focus handle in the app view and sets pending_focus = FocusTarget::EditorPrompt. EditorPrompt::render calls ensure_initialized(...), then if needs_focus is true, focuses the embedded editor state and clears needs_focus. TemplatePrompt visual state TemplatePrompt is rendered by src/prompts/template/render.rs through FocusablePrompt::new(content), key context "template_prompt", and its own focus handle. It uses shared prompt styling helpers such as prompt_field_style(...) and prompt_text_palette(...), and tests require a scrollable placeholder field region with "template-fields-scroll" and .overflow_y_scrollbar(). The description text changes when no placeholders exist: This template has no editable placeholders. Review the preview and press Enter to submit. The template prompt has native footer surface id "template_prompt" and footer buttons resolved in src/app_impl/ui_window.rs. It is sized as a DivPrompt, not as a full-height editor. Template focus is applied by src/app_impl/theme_focus.rs: if the current view is AppView::TemplatePrompt, it reads the entity focus handle, focuses it, and sets focused_input = FocusedInput::None. Keystrokes And Commands EditorPrompt physical key handling Inside src/editor/mod.rs, editor key handling includes: Cmd+Enter / Cmd+Return: submit current content. Cmd+S: submit current content. Tab in snippet mode: next_tabstop(...). Shift+Tab in snippet mode: prev_tabstop(...). Escape in snippet mode: exit_snippet_mode(...) and notify. Escape outside snippet mode: propagates. Choice popup visible: Up: move popup selection up. Down: move popup selection down. Enter / Return without Cmd: confirm choice. Escape: cancel choice popup. Tab without Shift: confirm choice and advance to next tabstop. Other keys: cancel popup and continue normal handling. The editor wrapper in src/render_prompts/editor.rs adds shared actions and shortcut behavior: Cmd+K toggles actions when editor actions are available. Shared action shortcut matching is allowed only after editor_reserved_shortcut_reason(...) declines the shortcut. Reserved editor shortcuts are preserved for editor/native behavior. The visible reserved set includes submit, save-submit, undo/redo, find, and clipboard/selection shortcuts. EditorPrompt protocol simulateKey The simulateKey dispatcher first routes keys to any open/current actions dialog path. Then the AppView::EditorPrompt arm handles: Choice popup Up / Down / Enter / Escape / Tab. Non-Cmd Tab: if snippet mode, next_tabstop_public(...); otherwise log no-op. Cmd+Enter: read editor content and submit it. Escape without Cmd: submit None and cancel script execution. Everything else: log unhandled. Known parity gaps from visible source: simulateKey Shift+Tab does not call prev_tabstop_public(...); it follows the same non-Cmd Tab branch and advances. simulateKey Cmd+S is not handled by the editor arm, even though physical Cmd+S submits. simulateKey Escape cancels the editor prompt, while physical no-snippet Escape appears to propagate. TemplatePrompt physical key handling The visible template renderer imports helpers for Backspace, Enter, Tab, and printable characters, and wraps the content with FocusablePrompt. The shown intercepted-key branch proves: Escape calls submit_cancel() and returns handled. The full Tab / Enter / printable handling implementation is not fully included in the bundle. Based on the visible imports, methods used by footer routing, and tests, it is reasonable to infer TemplatePrompt supports typed input, field navigation, and submit from keyboard. The exact physical key map should be verified locally from the full src/prompts/template/render.rs. TemplatePrompt protocol simulateKey The visible runtime_stdin_match_simulate_key.rs has no dedicated AppView::TemplatePrompt arm. Unless the generic actions toggle helper handles a specific Cmd+K, TemplatePrompt falls into the unhandled-view warning path. This conflicts with tests/smoke/test-template-tab-nav.ts, whose instructions say to use simulateKey {"type":"simulateKey","key":"tab"} to test template Tab navigation. In the bundled source, that runtime path is not implemented. For automation, prefer batch.setInput / try_set_prompt_input(...) plus footer selection or direct protocol submit routes where available, rather than assuming simulateKey Tab works for TemplatePrompt. Actions And Menus EditorPrompt actions EditorPrompt has first-class shared-actions support. Proven source: actions_host_for_view(...) maps AppView::EditorPrompt to ActionsDialogHost::EditorPrompt. ActionsDialogHost includes EditorPrompt. handle_cmd_k_actions_toggle(...) routes prompt hosts including EditorPrompt. request_focus_restore_for_actions_host(...) maps ActionsDialogHost::EditorPrompt to FocusRequest::editor_prompt(). render_editor_prompt(...) passes ActionsDialogHost::EditorPrompt to handle_prompt_key_preamble(...). The editor render path adds render_actions_backdrop(...) with backdrop id "editor-actions-backdrop" and close host ActionsDialogHost::EditorPrompt. Editor action execution has two paths: Select an action in the actions dialog. Trigger a non-reserved SDK shortcut. Reserved shortcuts are intentionally not action shortcuts. For example, Cmd+S and Cmd+Enter submit editor content rather than triggering SDK actions with those shortcuts. TemplatePrompt actions TemplatePrompt action support is not proven. The footer for TemplatePrompt includes an Actions button, and the button reflects actions_open = self.show_actions_popup || crate::actions::is_actions_window_open(). However, the visible ActionsDialogHost enum does not include TemplatePrompt, actions_host_for_view(...) does not map AppView::TemplatePrompt, and simulateKey has no TemplatePrompt-specific action branch. So the safe operator statement is: TemplatePrompt has footer chrome that advertises Actions, but the bundled source does not prove TemplatePrompt has a functioning actions host or SDK action catalog. Treat TemplatePrompt actions as an open gap until runtime proves Cmd+K or footer Actions opens a valid activePopupContract. Data, Storage, And Privacy Boundaries Editor and template prompts can expose sensitive text through several paths. Editor content Editor content can include arbitrary user text, code, secrets, clipboard material pasted by the user, or generated content. It flows through: SDK editor(content, language, actions?). PromptMessage::ShowEditor. EditorPrompt pending initialization and editor state. EditorPrompt::content(...). Submit callback response payload. Protocol ForceSubmit values. Automation getElements if the collector includes content previews or line summaries in the full local source. Tab AI context snapshots; tests/tab_ai_input_coverage.rs expects EditorPrompt can have captured input_text. Logging risk: ShowEditor logs full content when it auto-detects a template in content: Rusttracing::info!(category = "UI", content = %content_str, "Auto-detected template in content"); That can leak editor content to logs whenever tabstop detection fires. Template source and values TemplatePrompt data can include template source, selected text, clipboard text, and all field values. It flows through: SDK template(templateStr, options). SDK preprocessing of $SELECTION and $CLIPBOARD. PromptMessage::ShowTemplate. TemplatePrompt::template. TemplatePrompt::values. Preview rendering. Submit rendered output. getElements, which exposes a preview of template-source and each field value. Logging risk is stronger for TemplatePrompt: PromptMessage::ShowTemplate logs template = %template. TemplatePrompt::new logs "TemplatePrompt::new template: {}" with the full template.
