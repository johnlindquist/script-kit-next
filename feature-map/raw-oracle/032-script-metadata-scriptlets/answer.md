032 Script Metadata, Scriptlets, and Execution Catalog — Feature Atlas 1. Capability summary This feature is the runtime catalog layer for Script Kit GPUI. It answers four operational questions: What executable things exist? Scripts are discovered from plugin-scoped scripts/ directories. Scriptlets are discovered from plugin-scoped markdown bundles under scriptlets/. What metadata describes them? Scripts support typed metadata = { ... } declarations, legacy // Key: comments, schema declarations, schedule comments, and binding fields such as shortcut, alias, keyword, and trigger. Scriptlets support markdown headings, HTML comment metadata, fenced metadata / schema blocks, tool fences, groups, actions, shared companion actions, and named inputs. What is safe to expose or execute? Scripts with duplicate shortcut, alias, keyword, or trigger bindings are excluded from the kept script catalog by src/scripts/validation.rs#validate_script_catalog. The diagnostics are exposed in the launcher and through kit://failed-scripts. What can humans and agents inspect? Agents can read schema-versioned MCP resources: kit://scripts, kit://scriptlets, kit://failed-scripts, kit://sdk-reference, and kit://script-templates. The launcher uses the same Rust-owned SDK reference and script template catalog surfaces, which prevents UI/resource drift. User capabilities Users can: Browse scripts and scriptlets in the launcher. Run scripts and scriptlets through search, shortcuts, aliases, keywords, command syntax, and actions. See validation problems through the pinned “Script Issues” row when scripts are excluded. Open an in-app SDK Reference surface. Create a blank script or create from a curated template catalog. Script author capabilities Authors can: Put personal scripts under ~/.scriptkit/plugins/main/scripts/. Put plugin scripts under ~/.scriptkit/plugins/<plugin-id>/scripts/. Put scriptlet bundles under ~/.scriptkit/plugins/<plugin-id>/scriptlets/*.md. Use typed metadata, legacy comments, schema declarations, and scriptlet metadata fences. Define scriptlet actions with H3 headings and shared companion .actions.md files. Use tool-specific scriptlet execution modes: shell, TypeScript, interpreter, template, open, edit, paste, type, submit, and transform. Agent capabilities Agents can: Read catalog metadata through kit://scripts and kit://scriptlets. Read excluded-script diagnostics through kit://failed-scripts. Read supported SDK APIs and harness workflow guidance through kit://sdk-reference. Read script starter templates through kit://script-templates. Use the same catalog data that the launcher uses for SDK Reference and script template UI. 2. Workspace and discovery model Plugin root model Discovery is plugin-scoped, not legacy-root-scoped. The default personal plugin is: ~/.scriptkit/plugins/main/ Scripts live under: ~/.scriptkit/plugins/<plugin-id>/scripts/ Scriptlets live under: ~/.scriptkit/plugins/<plugin-id>/scriptlets/*.md The architecture page lat.md/scripting.md explicitly states that ~/.scriptkit/plugins/main/scripts/ is the default personal plugin and scriptlets are discovered from ~/.scriptkit/plugins/*/scriptlets/*.md. Scriptlet discovery There are two scriptlet loading surfaces in the bundle: src/scripts/scriptlet_loader/loading.rs#read_scriptlets src/scripts/scriptlet_loader/loading.rs#load_scriptlets read_scriptlets defaults to the main plugin scriptlets directory and manually splits by ## headings. Script validation contract Validation entry point src/scripts/validation.rs#validate_script_catalog is the validation entry point. It accepts an already-loaded catalog: RustVec<Arc<Script>> It returns: RustScriptCatalogReport { scripts: Arc<[Arc<Script>]>, validation: Arc<ValidationReport>,} The scripts field is the kept catalog. Scripts with fatal issues are excluded. src/scripts/loader.rs#read_scripts_report wraps: Rustlet scripts = read_scripts();Arc::new(validate_script_catalog(scripts)) Current fatal validation: duplicate bindings The current implemented fatal validation is duplicate binding detection. src/scripts/validation.rs#detect_binding_collisions checks collisions across: shortcut alias keyword trigger The binding extraction path is script_bindings: shortcut comes from script.shortcut. alias comes from script.alias. keyword comes from script.typed_metadata.keyword. trigger comes from script.typed_metadata.extra["trigger"] when it is a string. Bindings are bucketed by: Rust(BindingKind, normalized_value) That means alias: "cmd k" does not collide with shortcut: "cmd k" because binding kind is part of the key. This invariant is pinned by the unique_bindings_across_kinds_do_not_collide unit test. Normalization behavior The tests prove: Duplicate shortcuts normalize case, such as "cmd shift k" vs "Cmd Shift K". Duplicate aliases normalize case, such as "GC" vs "gc". Empty or whitespace-only binding values are skipped. trigger collisions are detected through the typed metadata extra map and normalize case. Three-way collisions produce related pointers to every peer. Failure model For a duplicate binding, every offending script gets its own fatal issue. Each ScriptValidationIssue includes: severity path script_name field message kind related The duplicate-binding kind is: RustScriptValidationKind::DuplicateBinding { binding: BindingKind, value: String,} Each failed script is represented by FailedScript and contains: path name fatal Kept/excluded catalog behavior validate_script_catalog partitions issues into fatal and warning issues. Scripts with no fatal issues are kept. Scripts with fatal issues are moved into failed_scripts. Warning issues remain in the report but do not exclude the script. Current implementation evidence shows duplicate bindings are fatal. The enum also defines metadata parse, schema parse, and invalid value issue kinds, but the visible validation function only calls detect_binding_collisions. Failed-script diagnostics resource src/mcp_resources/mod.rs#FAILED_SCRIPTS_RESOURCE_URI is: kit://failed-scripts src/mcp_resources/mod.rs#build_failed_scripts_document builds a schema-versioned envelope from ValidationReport. src/mcp_resources/mod.rs#read_kit_failed_scripts_resource calls crate::scripts::read_scripts_report() at read time, then serializes the document. Envelope fields: RustFailedScriptsResourceDocument { schema_version, validation_schema_version, total_candidates, valid_count, fatal_count, warning_count, failed_scripts, warnings,} The resource intentionally reflects current disk state at read time rather than only the startup in-memory state. User stories and agent stories User stories Run a personal TypeScript script A user saves demo.ts under ~/.scriptkit/plugins/main/scripts/. On startup or refresh, the loader discovers it, extracts metadata, attaches plugin_id = "main", indexes the body, and shows it in the launcher. Repair a duplicate shortcut Two scripts declare cmd shift k. Validation excludes both from the kept catalog. The launcher shows a pinned “Script Issues” row. The user opens diagnostics, sees both paths and related peers, edits one shortcut, refreshes, and both scripts become visible again. Run a markdown scriptlet A user saves a markdown bundle under ~/.scriptkit/plugins/main/scriptlets/tools.md. Each H2 section with a valid tool fence becomes a runnable scriptlet. H1 headings group the rows. H3 sections become actions. Use a shared scriptlet action A user defines shared actions in tools.actions.md. The loader skips that file as standalone, but the parser can attach actions to parent scriptlets without leaking template placeholders or global shortcuts. Create a starter script from template A user opens New Script From Template, chooses “Choice List,” enters a name, and the launcher renders the Rust-owned body template. The same template is available to agents through kit://script-templates. Use command syntax A user types >copy-date -- arg1 arg2. Agent stories Inventory available automations An agent reads kit://scripts and kit://scriptlets to list available automations with metadata and schema hints. Avoid broken scripts An agent reads kit://failed-scripts before suggesting a script invocation. If the desired script is excluded, it reports the duplicate binding and related peers instead of pretending the script does not exist. Generate a supported script An agent reads kit://sdk-reference, checks support status, avoids unsupported SDK APIs, and uses the documented metadata format and script directory. Create from canonical template An agent reads kit://script-templates, chooses blank-starter or choice-list, renders the same body the launcher would render, and avoids collision-bearing binding fields. Diagnose scriptlet behavior An agent inspects the scriptlet bundle format: H1 group, H2 scriptlet, H3 actions, fenced metadata/schema, and tool fence. For each plugin, scan <plugin-root>/scripts. Load .ts and .js entries. Extract typed metadata, legacy comments, and schema through extract_metadata_full. Read body text for content search. Startup scriptlet lifecycle Discover plugin roots. For each plugin, scan <plugin-root>/scriptlets/*.md. Skip .actions.md companion files as standalone entries. Parse markdown with parse_markdown_as_scriptlets. Convert parser-level scriptlets into runtime scriptlets. Attach plugin identity and file path/anchor. Sort by group, then name. Launcher renders scriptlet rows and actions. Validation lifecycle Validation currently applies to scripts through read_scripts_report. Fatal duplicate binding issues: remove scripts from the kept catalog populate ValidationReport.failed_scripts surface through kit://failed-scripts surface through launcher issue row Warnings are represented in the model but not strongly evidenced by current implemented checks. Refresh lifecycle The loader test read_scripts_from_dir_reloads_updated_body_content proves reload reads current disk body text. lat.md/scripting.md says startup and scriptlet-refresh both populate ScriptListApp.script_validation_report. The exact refresh trigger path is outside the visible snippets, but the intended state model is that validation diagnostics stay in sync after refresh. Hidden vs excluded Hidden scripts are metadata-controlled discoverability behavior. Excluded scripts are validation failures. Do not conflate them: Hidden scripts may still be valid and potentially invokable through specific flows. Excluded scripts are removed from the kept catalog to prevent ambiguous dispatch. The snippets prove excluded behavior. Hidden behavior should be verified in search/grouping code. Execution lifecycle For a scriptlet: User selects scriptlet or action. Runtime calls run_scriptlet. Content is processed for conditionals. Tool is normalized. Tool-specific execution path runs. ScriptletResult reports exit code, stdout, stderr, and success. For full scripts: Discovery and metadata are proven. TypeScript-through-Bun SDK execution is documented by lat.md/scripting.md. Exact runner/session lifecycle should be documented from src/executor/runner.rs, src/prompt_handler/mod.rs, and SDK skill files. Known risks, gaps, and ambiguities 1. Typed metadata parse errors are not propagated into validation Evidence: src/metadata_parser/mod.rs#extract_typed_metadata returns errors. src/scripts/metadata.rs#extract_full_metadata reads typed_result.metadata but the visible code drops typed_result.errors. src/scripts/validation.rs#ScriptValidationKind includes MetadataParse. src/scripts/validation.rs#validate_script_catalog only calls detect_binding_collisions. Impact: malformed typed metadata may silently fall back or disappear rather than producing a diagnostics row. 2. Schema parse errors are not proven to reach validation Evidence: src/scripts/metadata.rs#extract_full_metadata calls extract_schema. src/scripts/validation.rs#ScriptValidationKind includes SchemaParse. The visible validate_script_catalog implementation does not inspect schema parse errors. Impact: invalid schema declarations may not be surfaced in kit://failed-scripts. 3. Scheduling may bypass validation Evidence: src/scripts/scheduling.rs#register_scheduled_scripts separately scans plugin script directories. It uses schedule metadata comments and registers with Scheduler. The visible scheduling code does not call read_scripts_report or validate_script_catalog. Inferred risk: a script excluded from the launcher due to duplicate bindings might still be scheduled if it has // Cron: or // Schedule: metadata. 4. Typed cron/schedule support is ambiguous Evidence: TypedMetadata includes cron and schedule fields in codefence simple metadata mapping and likely in typed metadata. extract_schedule_metadata explicitly parses // Cron: and // Schedule: comments. register_scheduled_scripts references comment schedule extraction. Impact: authors may expect typed metadata.cron or metadata.schedule to register schedules, but the provided snippets only prove legacy comment scheduling. 5. Scriptlet validation-aware parser appears underused Evidence: src/scriptlets/mod.rs#parse_scriptlets_with_validation returns errors and frontmatter. It is marked #[allow(dead_code)]. src/scripts/scriptlet_loader/loading.rs#load_scriptlets calls parse_markdown_as_scriptlets, not parse_scriptlets_with_validation. Impact: scriptlet parse errors may be logged or dropped instead of surfaced in a diagnostics resource. 6. Scriptlet binding collisions are not proven to be excluded Evidence: Script validation works on Script. Scriptlets have shortcut, keyword, and metadata trigger. No visible validate_scriptlet_catalog equivalent is shown. .actions.md skipping prevents one class of accidental global shortcut leakage, but not all duplicate scriptlet bindings. Impact: duplicate scriptlet shortcuts/keywords/triggers may still race or rank unpredictably unless handled elsewhere. 7. Legacy read_scriptlets can conflict with plugin-scoped model Evidence: read_scriptlets defaults to plugin_scriptlets_dir("main"). load_scriptlets is plugin-scoped across discovered plugins. Impact: production callers should be audited to ensure they use load_scriptlets, not the main-only legacy loader. 8. Scriptlet simple metadata is intentionally flat Evidence: src/scriptlet_metadata/mod.rs#parse_simple_metadata only supports key: value. lat.md/menu-syntax.md says scriptlet metadata blocks are line-by-line only. Complex menuSyntax is not load-time validated for scriptlets and malformed entries are silently skipped at lookup time. Impact: authors cannot reliably declare nested menu syntax directly inside scriptlet metadata today. 9. kit://failed-scripts reads current disk state, not necessarily in-memory launcher state Evidence: src/mcp_resources/mod.rs#read_kit_failed_scripts_resource calls read_scripts_report() at read time. Launcher diagnostics use in-memory ScriptListApp.script_validation_report. Impact: resource reads can reflect disk changes before or after the launcher’s cached report refreshes. This is useful for agents but can create momentary mismatch with UI. 10. Mixed JSON casing in resource entries Evidence: ScriptsResourceDocument uses camelCase envelope fields. tests/script_resources.rs#kit_scripts_json_uses_camel_case_envelope asserts entry field has_schema. Impact: clients should not assume all nested fields are camelCase until the entry structs are audited and versioned. 11. Frontmatter/icon resolution may be future-facing Evidence: resolve_scriptlet_icon and parse_scriptlets_with_validation are marked dead-code allowed. Current plugin loader uses parse_markdown_as_scriptlets. Impact: documented frontmatter defaults may not apply in the active loader path unless confirmed in the full parser. 12. Environment allowlist details are not visible in snippets Evidence: Prompt says executor enforces environment allowlist. src/executor/scriptlet.rs#run_scriptlet snippet does not show the allowlist. Impact: the maintained chapter should document exact environment variables and restrictions from the full executor implementation. 13. read_scripts() still returns unvalidated scripts Evidence: read_scripts() remains unchanged. read_scripts_report() is additive.
