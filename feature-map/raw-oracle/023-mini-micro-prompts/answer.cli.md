023 Mini and Micro Prompts / mini() / micro() Executive Summary Mini and Micro Prompts are compact, choice-backed prompt surfaces exposed through the TypeScript/Bun SDK as mini(placeholder, choices) and micro(placeholder, choices). Both APIs return Promise<string>, normalize string choices into { name, value } choice objects, send real prompt messages, and resolve with the submitted value or ''. The SDK comments and runtime warnings currently say mini() and micro() are “not yet implemented” and suggest using arg() instead, but the Rust app snapshot contradicts that warning. The Rust side contains real Message::Mini / Message::Micro routing, PromptMessage::ShowMini / PromptMessage::ShowMicro, AppView::MiniPrompt / AppView::MicroPrompt, render dispatch, state reporting, element collection, batch selection support, Tab AI context coverage, and tests. Treat the SDK warning as stale product copy, not as the actual capability boundary. MiniPrompt is the compact arg-like list prompt. It uses shared arg choice state, compact minimal-list chrome, a native-footer-backed footer slot, choice filtering, selected values, Enter submit, Escape cancel, and ViewType::MiniPrompt sizing in the window-size calculation path. It must not inherit the full ArgPrompt width. MicroPrompt is also choice-backed and arg-like internally, but visually it is an ultra-compact inline prompt. It is explicitly footerless, stays off native-footer routing, uses compact no-footer chrome, and must remain distinct from mic() / microphone media behavior. micro() is not a microphone API. The visible PromptMessage::ShowMini arm still performs an immediate resize_to_view_sync through ViewType::ArgPromptNoChoices / ViewType::ArgPromptWithChoices, even though the later window-size calculation path correctly uses mini_prompt_view_type() / ViewType::MiniPrompt. simulateKey explicitly handles ArgPrompt | MiniPrompt but not MicroPrompt. The lat.md/protocol.md prompt-family list mentions micro but not mini, while source code proves Mini support elsewhere. These are not fatal to the feature, but they are operator-visible proof gaps and regression traps. What Users Can Do Script authors can call: TypeScriptconst value = await mini("Pick one", ["Alpha", "Beta", "Gamma"]); or: TypeScriptconst value = await micro("Pick one", ["Alpha", "Beta", "Gamma"]); Both calls return a string. A string choice becomes { name: "Alpha", value: "Alpha" }. A Choice object is passed through as-is. The submitted value is choice.value, not necessarily choice.name. Users can type into the prompt to filter choices. The shared arg-like filter path matches against choice.name.toLowerCase().contains(filter), based on src/app_impl/prompt_ai.rs. The state model tracks both total choices and visible filtered choices. Users can submit the selected choice. For MiniPrompt, both renderer tests and protocol simulation prove Enter submission. src/render_prompts/mini.rs contains the is_key_enter(key) path and calls submit_arg_prompt_from_current_state(&prompt_id, cx). src/main_entry/runtime_stdin_match_simulate_key.rs also routes simulateKey Enter for AppView::MiniPrompt. Users can cancel MiniPrompt with Escape. The simulateKey path sends None for the prompt response and calls cancel_script_execution. The SDK callback collapses missing msg.value to '', so cancellation is not distinguishable from an intentional empty string by the plain Promise<string> return value. Users can inspect MiniPrompt and MicroPrompt through automation. getState reports promptType, promptId, placeholder, inputValue, choiceCount, visibleChoiceCount, selectedIndex, selectedValue, focus, visibility, surface contract, and footer state. getElements uses the shared choice collector for ArgPrompt, MiniPrompt, and MicroPrompt. Automation can select values by value or semantic ID through batch commands. select_choice_by_value, select_choice_by_semantic_id, select_first_choice, and submit_current_value all include AppView::MiniPrompt and AppView::MicroPrompt. Dictation-style prompt input is allowed for both MiniPrompt and MicroPrompt. src/app_impl/ui_window.rs includes both in can_accept_dictation_into_prompt and try_set_prompt_input. Core Concepts MiniPrompt is SDK mini(), not Mini main-window mode. mini() creates AppView::MiniPrompt. Mini main-window mode uses MainWindowMode::Mini and ViewType::MiniMainWindow. Mini AI / Mini ACP sizing uses ViewType::MiniAiChat. These are adjacent windowing concepts, not the same feature. MicroPrompt is SDK micro(), not mic(). micro() creates AppView::MicroPrompt, a compact text/choice prompt. mic() is a media/microphone stub area. tests/minimal_chrome_audit.rs explicitly guards that mic() remains a stub until a real media surface exists and that micro() remains distinct as "ultra_compact_no_footer" prompt chrome. Both Mini and Micro reuse arg-like choice state. They share: arg_input arg_selected_index filtered_arg_choices get_filtered_arg_choices selected-value calculation set_input_text try_set_prompt_input submit_current_value select_choice_by_value select_choice_by_semantic_id collect_choice_view_elements Mini owns footer space; Micro does not. AppView::native_footer_surface() returns Some("mini_prompt") for MiniPrompt, but MicroPrompt is grouped with footerless surfaces and returns None. Mini uses main_window_footer_slot(...); Micro must not. MiniPrompt sizing should use compact prompt width. ViewType::MiniPrompt exists in src/window_resize/mod.rs. width_for_view(ViewType::MiniPrompt) returns Some(MINI_MAIN_WINDOW_WIDTH), and tests pin MINI_MAIN_WINDOW_WIDTH == 480.0 while FULL_MAIN_WINDOW_WIDTH == 750.0. src/app_impl/ui_window.rs exposes mini_prompt_view_type() -> ViewType::MiniPrompt and uses it in the AppView::MiniPrompt sizing arm. MicroPrompt sizing is ultra compact and footerless. PromptMessage::ShowMicro resizes to ViewType::ArgPromptNoChoices with item count 0. AppView::MicroPrompt has no native footer surface. The minimal chrome audit checks that Micro stays footerless and off native footer routing. State contracts matter more than screenshots. The repo’s automation model expects state-first proof through getState, getElements, getLayoutInfo, batch, and waitFor. Screenshots are secondary and mostly for visual regressions.
