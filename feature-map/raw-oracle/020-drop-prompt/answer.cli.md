020 Drop Prompt / drop() Executive Summary drop() is the SDK-facing file drop prompt. The public SDK API is currently a no-argument drop(): Promise<FileInfo[]>, where each returned item is shaped as path, name, and size (scripts/kit-sdk.ts lines 40–44, 1638, 2404–2423). It opens a Rust/GPUI DropPrompt surface through the protocol Message::Drop { id }, which routes to PromptMessage::ShowDrop and then to AppView::DropPrompt (src/prompt_handler/mod.rs lines 990–994, 6184–6212). The product contract is: DropPrompt owns file-submit semantics, uses a focused drop-target UI, routes Enter / native footer Run to DropPrompt::submit, omits launcher AI, exposes Submit plus ⌘K Actions, and keeps empty submit disabled with actionDisabled:"no_files" through the active footer snapshot (lat.md/design.md lines 33–34; src/app_impl/ui_window.rs lines 547–570). The most important proof gap in the provided bundle: DropPrompt has a handle_drop method that sets dropped_files, but it is marked #[allow(dead_code)], and the shown render path does not prove a GPUI .on_drop(...) hook or drag-over state transition for SDK drop() (src/prompts/drop.rs lines 69–74, 81–172). So the atlas can describe the intended and internal state/submission behavior, but the actual OS-file-drop event wiring is not proven by this snapshot. Another important mismatch: embedded docs show drop("Drop files here"), but the SDK declaration and implementation shown here accept no arguments. Internal ShowDrop supports optional placeholder and hint, but public SDK drop() does not pass them (src/setup/embedded_agents_part_000.md lines 131–135 versus scripts/kit-sdk.ts lines 1638 and 2404). What Users Can Do Users can call SDK drop() from a script and receive an array of file metadata objects. The resolved metadata shape is: FileInfo[], where each file has path: string, name: string, and size: number. Users see a DropPrompt surface with default copy: Drop files here and hint text: Drag and drop files to upload Those defaults come from DropPrompt::render when no placeholder or hint is supplied (src/prompts/drop.rs lines 90–97). The UI body shows a folder icon, placeholder, hint, and, once dropped_files is non-empty, a count like N file(s) dropped (src/prompts/drop.rs lines 129–150). The provided source does not show individual file names rendered in the visible UI body. File names and paths are exposed through automation elements once files are present (src/app_layout/collect_elements.rs lines 1192–1218). Users can submit only after files exist. Empty submit is intentionally blocked: the native footer keeps Submit visible but disabled with actionDisabled:"no_files", and DropPrompt::submit itself no-ops when dropped_files is empty (src/app_impl/ui_window.rs lines 551–564; src/prompts/drop.rs lines 49–64). Users can cancel with Escape. Cancellation calls the submit callback with None; the SDK treats a null submit value as Escape and exits the script process (src/prompts/drop.rs lines 66–68; scripts/kit-sdk.ts lines 2409–2413). Users are shown a ⌘K Actions footer affordance. The footer button is wired to the shared footer Actions dispatcher, but the provided bundle does not prove DropPrompt has a supported ActionsDialog host. Treat actual DropPrompt action-menu behavior as a gap unless verified at runtime (src/app_impl/ui_window.rs lines 222–233, 565–570; src/main_sections/app_view_state.rs lines 1080–1095). Core Concepts SDK API. drop() is a TypeScript/Bun global. It sends a protocol message of type drop with a generated prompt id, waits for a submit message, parses the submitted JSON, and resolves to FileInfo[] (scripts/kit-sdk.ts lines 2404–2428). Prompt id. The SDK-generated id is carried through Message::Drop { id }, PromptMessage::ShowDrop { id, placeholder, hint }, AppView::DropPrompt { id, entity }, and eventually back through the submit callback (src/prompt_handler/mod.rs lines 990–994, 6184–6208). DropPrompt entity state. DropPrompt owns id, optional placeholder, optional hint, dropped_files, is_drag_over, focus_handle, on_submit, theme, and design_variant (src/prompts/drop.rs lines 10–20). Dropped file metadata. Rust-side dropped files are DroppedFile { path, name, size }, where size is u64 (src/prompts/drop.rs lines 21–26). Submitted JSON contains only path, name, and size (src/prompts/drop.rs lines 51–59). No MIME type, file kind, modification time, directory flag, or file contents are part of SDK drop() in the shown source. Prompt-owned surface. DropPrompt is classified as SurfaceKind::PromptEntity, not launcher list content (src/main_sections/app_view_state.rs lines 463–472). Layout info gives it a prompt-owned DropContent component instead of launcher list/preview (src/app_layout/build_layout_info.rs lines 148–179). Native footer ownership. AppView::native_footer_surface() maps DropPrompt to drop_prompt (src/main_sections/app_view_state.rs lines 527–536). The design contract says DropPrompt owns file-submit footer semantics and omits launcher AI (lat.md/design.md lines 33–34). Footer-disabled state. The empty state is represented in footer state by disabling the Submit button with disabled_reason("no_files"), which surfaces through active footer snapshots as actionDisabled:"no_files" (src/app_impl/ui_window.rs lines 557–564; lat.md/protocol.md lines 37–38). Automation body elements. When files are present, getElements can expose a list:dropped-files element plus one choice-like row per file. When no files are present, the DropPrompt collector returns no body elements (src/app_layout/collect_elements.rs lines 1192–1218). Entry Points TypeScript SDK: drop() is declared globally as function drop(): Promise<FileInfo[]> and implemented as globalThis.drop = async function drop(): Promise<FileInfo[]> (scripts/kit-sdk.ts lines 1638, 2404). Protocol prompt: SDK sends a DropMessage with type: 'drop' and id. The prompt handler maps Message::Drop { id } to PromptMessage::ShowDrop { id, placeholder: None, hint: None } (scripts/kit-sdk.ts lines 2404–2408 and implied send after the shown handler; src/prompt_handler/mod.rs lines 990–994). Runtime prompt creation: PromptMessage::ShowDrop creates DropPrompt::new, stores it as AppView::DropPrompt, clears legacy focused input, sets pending focus to FocusTarget::DropPrompt, sizes the window as ViewType::DivPrompt, and notifies GPUI (src/prompt_handler/mod.rs lines 6184–6212). Render dispatch: AppView::DropPrompt routes to render_drop_prompt(entity, cx) (src/main_sections/render_impl.rs lines 393–395). Footer entry: Native footer Run routes to DropPrompt::submit before launcher fallback; native footer Actions routes to the shared actions dispatcher (src/app_impl/ui_window.rs lines 196–220, 222–233). Keyboard entry: The focused DropPrompt handles Escape through submit_cancel and Enter through submit (src/prompts/drop.rs lines 152–170). Automation entry: Agents can inspect DropPrompt through getState, getElements, and getLayoutInfo; they can press keys through simulateKey, remembering that simulateKey itself is fire-and-forget and does not emit a response envelope (lat.md/protocol.md lines 28–38; src/prompt_handler/mod.rs lines 2624–2633; src/app_layout/collect_elements.rs lines 563–573).
