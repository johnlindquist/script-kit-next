029 Widget, Media, and Find APIs Executive Summary Feature 029 is mostly SDK surface, stub protocol, and negative-contract behavior, not a working GPUI feature set. Current truth from the provided bundle: widget() exists in scripts/kit-sdk.ts, warns that it is not implemented, sends a widget message, immediately returns a controller object, and supports local registration of event handlers. Rust currently routes Message::Widget to PromptMessage::WidgetComingSoon, and the prompt handler shows a coming-soon toast. The returned controller shape is real; visible floating widget behavior is not proven and is explicitly stubbed. webcam() and mic() exist as SDK globals but throw immediately before sending any protocol message. Their Rust protocol variants and coming-soon routes still exist, but the active TypeScript SDK functions do not reach them. The SDK error text says media streaming is not feasible in the current JSONL message-passing architecture. eyeDropper() exists as an SDK global and throws immediately before sending. The SDK has an EyeDropperMessage interface, but the provided Rust snippets do not prove a Rust variant, constructor, route, prompt message, or handler. The SDK comment says it could potentially be implemented via screen capture plus pixel reading, but it is not implemented. find() exists as an SDK global. It sends a find message with id, placeholder, and optional onlyin, then waits for a SubmitMessage. Escape/null exits the script. Empty/undefined submit resolves to an empty string. The call site passes an auto-submit fallback of { value: '' }. The provided Rust snippets do not prove a corresponding Message::Find, PromptMessage::Find, renderer, handler, or coming-soon route, so backend support is an explicit proof gap. The existing tests are mostly shape and availability tests. They prove that functions and controller methods exist, not that GPUI renders widgets, captures media, picks colors, or performs find/file search. What Users Can Do Users and script authors can currently do the following: Call widget(html, options?) and receive a WidgetController. Catch immediate thrown errors from webcam(), mic(), and eyeDropper(). Call find(placeholder, options?) and send a find payload from the SDK, but backend behavior is not proven by the bundle. Users cannot currently rely on: A visible floating HTML widget. Widget HTML actually being rendered by GPUI. Widget click/input/move/resize events actually being emitted by the app. Camera capture through webcam(). Microphone capture through mic(). Color picking through eyeDropper(). find() being handled by a Rust prompt route. find() being equivalent to file search, path(), or portal-backed file selection. Core Concepts SDK surface versus GPUI implementation The SDK exposes more globals than the GPUI app currently implements. This feature is a clear example: widget() has an SDK implementation that sends messages and returns a controller, but the Rust app routes it to WidgetComingSoon. webcam() and mic() have Rust variants and coming-soon prompt routes, but the current SDK throws before sending. eyeDropper() has a TypeScript message interface and SDK global, but no Rust route is proven. find() has a TypeScript message interface and SDK global, but no Rust route is proven. Message-passing boundary The SDK communicates with the app through JSONL-style message passing. That works for discrete prompts and actions, but the SDK comments explicitly reject webcam and microphone streaming as too complex for the current architecture. For this feature, there are two categories of messages: Outbound SDK-to-app messages: widget widgetAction find Historical/interface-only: webcam, mic, eyeDropper Inbound app-to-SDK messages: widgetEvent submit The current implementation only proves outbound widget, outbound widgetAction, outbound find, inbound widgetEvent dispatch handling inside the SDK, and inbound submit handling for find(). It does not prove that the app emits real widgetEvent messages or handles find. Coming-soon prompt messages Rust has explicit coming-soon prompt messages for: WidgetComingSoon WebcamComingSoon MicComingSoon These are declared in src/main_sections/prompt_messages.rs:191-207 and handled in src/prompt_handler/mod.rs:6733-6762. There is no equivalent proven coming-soon prompt message for eyeDropper() or find() in the provided snippets. Throw-before-send behavior webcam(), mic(), and eyeDropper() throw immediately in the SDK. That means the TypeScript functions do not send protocol messages during normal execution. This matters because Rust may still contain historical variants or stubs, but they are not reachable through the current SDK globals unless another caller manually emits those messages.
