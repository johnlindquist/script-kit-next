028 Window Control and Visual Inspection APIs Executive Summary Feature 028 covers the SDK and protocol surfaces that control the Script Kit GPUI main window and inspect its visual/structural state: show(), hide(), blur(), showGrid(), hideGrid(), getWindowBounds(), captureScreenshot(), and getLayoutInfo(). There are two broad classes: Fire-and-forget controls These send a command and do not give the SDK caller a meaningful completion receipt: show(), hide(), blur(), showGrid(), hideGrid(). Their SDK functions return Promise<void>, but that promise resolves after the message is sent, not after the UI has visibly changed. Operators and agents must prove postconditions with follow-up observation: getState, getElements, inspectAutomationWindow, listAutomationWindows, getWindowBounds, or screenshot proof when visual pixels matter. Receipt-backed queries These allocate a requestId, send a query, and expect a response: getWindowBounds(), captureScreenshot(), getLayoutInfo(). The response shape depends on the runtime path. The SDK script-execution path for getWindowBounds() expects a submit-style JSON value, while the stdin automation handler has a typed windowBounds response. Screenshot and layout info are typed response surfaces: screenshotResult and layoutInfoResult. The most important operator rule: use state-first proof unless the claim is visual. Bounds and layout are structural; screenshots expose user pixels and require macOS Screen Recording permission. Screenshot proof must include a content audit, not just “a PNG exists.” Primary source anchors include: SDK globals and message interfaces: scripts/kit-sdk.ts Protocol variants: src/protocol/message/variants/query_ops.rs, src/protocol/message/variants/system_control.rs Grid and layout types: src/protocol/types/grid_layout.rs Runtime dispatch: src/execute_script/mod.rs, src/prompt_handler/mod.rs Main-window stdin show/hide/grid paths: src/main_entry/runtime_stdin_match_core.rs, src/main_entry/runtime_stdin_match_tail.rs, src/main_entry/app_run_setup.rs Layout builder: src/app_layout/build_layout_info.rs Screenshot capture audit: src/platform/screenshots_window_open.rs Contracts/tests: tests/source_audits/stdin_get_window_bounds_wired.rs, tests/stdin_show_hide_simulatekey_no_response_envelope_contract.rs, tests/verify_shot_strict_window_contract.rs, tests/sdk/test-capture-screenshot.ts, tests/smoke/test-debug-grid-basic.ts, tests/smoke/test-grid-dimensions.ts, tests/smoke/test-layout-info-simple.ts, tests/smoke/test-resize-behavior.ts One proof gap matters: the architecture docs and contract test describe stdin show / hide / simulateKey as no-envelope commands, but the provided app_run_setup.rs snippet also shows a window_visibility_ack send in one Show branch. Treat that as a live dispatcher ambiguity until verified in the active build. SDK show() / hide() still have no requestId and no SDK-level response wait. What Users Can Do Users and scripts can: Show the main Script Kit launcher or prompt window. Hide the main window without necessarily killing secondary windows. Blur or defocus through the SDK blur() command, with exact runtime behavior not fully proven by the provided bundle. Overlay a debug grid on the current prompt/window. Remove the debug grid overlay. Read the main window’s current bounds. Capture a screenshot of the app window or, through protocol/runtime support, a targeted automation window. Read structural layout info: prompt type, component list, bounds, depth, parent-child metadata, and explanations. Use visual captures to verify grid overlays, prompt dimensions, HiDPI behavior, and screenshot validity. Use layout info to inspect structure without exposing private pixels. Agents can: Use show / hide with parse receipts only, then prove state separately. Use getWindowBounds to verify resize behavior. Use captureScreenshot only when a visual assertion requires pixel evidence. Use getLayoutInfo for structural inspection of the main window. Use getElements(target) and inspectAutomationWindow(target) instead of targeted getState for secondary surfaces. Core Concepts 1. SDK send is not always UI completion In scripts/kit-sdk.ts, these functions just send messages: show() hide() showGrid(options?) hideGrid() blur() They do not allocate a pending request, do not wait for a response, and do not prove that the UI finished changing. Their promise is not a visual receipt. 2. Request-backed APIs allocate requestId These SDK functions use nextId() and addPending(...): getWindowBounds() captureScreenshot(options?) getLayoutInfo() They expect a response correlated by requestId, except getWindowBounds() has two known response styles depending on path. 3. Main window is special The main launcher window is tracked as automation window id "main" and has special visibility, focus, reset, and surface-rekey behavior. lat.md/windowing.md states that main-window hide paths are main-panel-only dismissals and should use defer_hide_main_window, not broad app-level cx.hide() / ctx.hide(), because app-level hide can also conceal independent secondary hosts such as Notes if detection is stale or racing. 4. Secondary windows are not inspected by targeted getState lat.md/protocol.md says targeted getState is retained for compatibility and main-window disambiguation, but secondary-window state should be inspected with: getElements(target) inspectAutomationWindow(target) getAcpState(target) for ACP state This matters for screenshot and layout proof: do not assume a non-main target gives meaningful stateResult. 5. Screenshots are high-sensitivity evidence Screenshots can expose private user data. The capture path intentionally audits pixels before accepting the PNG so black/blank permission failures do not masquerade as valid proof. lat.md/automation.md and src/platform/screenshots_window_open.rs pin this: no sampled pixels, no non-transparent pixels, solid-like frames, and dark-empty low-variety frames are rejected as infrastructure failures. 6. Layout info is structural, not visual getLayoutInfo() returns a component tree and explanations. It does not prove color, text rendering, clipping, z-order, blur/vibrancy, or pixel-level overlay appearance. Use it before screenshot escalation, not as a replacement for visual proof when pixels matter. Entry Points SDK globals Defined in scripts/kit-sdk.ts. SDK call: getWindowBounds() SDK behavior: Allocates requestId. Sends { type: "getWindowBounds", requestId }. Registers a pending handler. Auto-submit mock uses JSON value: {"x":100,"y":100,"width":800,"height":600}. Handler parses the response value as JSON. Missing or parse-failed fields resolve to zeros. Runtime paths: Script-execution path in src/execute_script/mod.rs Handles Message::GetWindowBounds. On macOS, reads the main NSWindow frame. Flips Y relative to primary screen height. Returns a Message::Submit with id: requestId and JSON bounds in value. If the main window is missing, returns JSON {"error":"Main window not found"}. On non-macOS, returns JSON {"error":"Not supported on this platform"}. SDK consequence: because the SDK parser expects numeric x, y, width, height, error JSON resolves to zero-like bounds rather than a thrown error. Pinned constants include EXPECTED_MAIN_PANEL_LEVEL = 101, corresponding to NSPopUpMenuWindowLevel, not ordinary NSFloatingWindowLevel. Show focus handoff The stdin show path: shows main window without activation, activates the GPUI window, sends the AI window to the back, retrieves the app focus handle, focuses the window, syncs automation window state. That makes show() a visibility-plus-focus command in the shown path. Hide reset semantics Hide has a reset contract: reset current view to ScriptList, rekey automation semantic surface to scriptList, hide main panel only, preserve independent secondary hosts. lat.md/automation.md calls out that both non-RPC and stdin RPC hide paths must reset-then-rekey. Order matters: reset first, then automation surface rekey. Script-requested hide PromptMessage::HideWindow in src/prompt_handler/mod.rs sets script-requested hide state. HUD behavior intersects here: PromptMessage::ShowHud clears script_requested_hide without revealing the main window. This is the relevant Feature 025 boundary: HUD feedback must not restore/show the launcher. Blur The SDK surface exists and sends type:"blur", but the provided bundle does not show the Rust implementation branch. Operators should verify post-focus state through observation rather than assume exact semantics. Keystrokes And Commands This feature is primarily SDK/protocol driven, not keyboard driven. Relevant keyboard-adjacent behavior: Escape and Cmd+W hide/reset paths are part of the broader main-window lifecycle. lat.md/windowing.md says passive desktop click-away uses preserve-state focus-loss hide, while Escape, Cmd+W, script completion, prompt cancellation, and explicit hide/reset commands remain reset paths. ScriptList Escape routes popup-first: close menu-syntax trigger popup if open, clear non-empty filter, if opened from main menu, call go_back_or_close, otherwise hide main window. simulateKey is explicitly fire-and-forget in the protocol docs, but Feature 027 owns unsupported keyboard/mouse helper details. Do not use unsupported key/mouse helpers as proof for this feature. Command surfaces: SDK globals are the normal script API. Stdin ExternalCommand::ShowGrid / HideGrid routes grid overlay directly. Stdin show / hide routes main visibility. Protocol query messages route screenshot/layout/bounds requests. Actions And Menus These APIs are not primarily actions-menu APIs. There is no source evidence in the provided bundle that show, hide, blur, showGrid, hideGrid, getWindowBounds, captureScreenshot, or getLayoutInfo are exposed as selectable rows in the shared Cmd+K actions dialog by default. Relevant boundaries: The actions dialog has its own contracts for section headers, selection, drill-down, Enter routing, and resize behavior. Those are adjacent but not part of Feature 028. Use activePopupContract, getElements(target), or strict window screenshot targeting instead of assuming the main window is unobstructed. Attached actions/prompt popups have strict visual proof rules. tests/verify_shot_strict_window_contract.rs pins attached popup handling for ActionsDialog and PromptPopup. Data, Storage, And Privacy Boundaries Screenshot data Screenshots are sensitive. They may include: private user text, filenames, clipboard content, app names, browser/history surfaces, hidden but visually exposed UI, debug grid overlays over real content. The runtime rejects blank/black captures because otherwise missing Screen Recording permission could produce misleading “proof.” Tests write screenshots to .test-screenshots/, for example: tests/smoke/test-debug-grid-basic.ts tests/smoke/test-grid-dimensions.ts Operators should avoid saving screenshot artifacts unless the proof requires it. Layout info Layout info is structural. It reveals: window dimensions, component names, component bounds, prompt type, hierarchy, explanations. It does not expose rendered pixels, but it can reveal UI state and active prompt class. Bounds Bounds reveal screen geometry and app positioning. Hide paths may save per-display main-window position via window_state::save_main_position_for_display. Visibility and focus commands show, hide, and blur mutate user-visible focus/visibility state. They can interrupt the user by bringing Script Kit forward or moving focus away. Permissions Screenshot capture depends on macOS Screen Recording permission. lat.md/automation.md says the non-prompting Screen Recording preflight is also surfaced through computer/list_permissions, so agents can diagnose permission before screenshot proof. Do not request permissions or open System Settings from these APIs. The provided capture path checks permission and fails. Error, Empty, Loading, And Disabled States show() Known failure/edge states: No SDK response exists. A parse receipt may show the command parsed, but not that the window appeared. Active dispatcher ambiguity exists around window_visibility_ack; do not depend on it without verification. If the main window registry is stale, follow-up proof may disagree with visual state. hide() Known failure/edge states: No SDK response exists. Must not hide independent secondary windows. Hide should reset/rekey to ScriptList; if it does not, next show() may reveal stale subview state. Bounds are saved before hide in shown stdin paths. Script-requested hide can affect post-script restore behavior. HUD clears script-requested hide without showing main. blur() Known failure/edge states: No SDK response exists. Runtime implementation not visible in the provided bundle. Must be proven with focus observation. Do not treat it as Accessibility selected-text focus handoff. showGrid() Known failure/edge states: No response envelope. Overlay may not be visible if the window is hidden. Stdin external command path discards color scheme by constructing color_scheme: None. Visual proof requires screenshot. Structural proof via getLayoutInfo() does not prove grid rendering. hideGrid() Known failure/edge states: No response envelope. If grid was never active, hide should be harmless, but this is inferred; no explicit idempotence proof appears in the provided snippets. Visual absence requires screenshot proof. getWindowBounds() Script SDK path: Main window missing returns JSON error. Non-macOS returns JSON error. SDK parser resolves missing numeric fields to zeros. Parse failure resolves zeros. Therefore {x:0,y:0,width:0,height:0} may mean “real zero-like response,” “missing window,” “unsupported platform,” or “parse failure” unless corroborated with logs/typed protocol. Stdin handler path: Missing main registry bounds returns zeros and logs bounds_available:false. Response is typed windowBounds. If response_sender is missing, it logs warning and caller may time out. captureScreenshot() Failure states: screenshotResult.error present. Empty data. Non-positive width/height. Screen Recording permission denied. Blank/black image rejected. Target resolution failure. Stale native window id. Native capture/xcap failure. Infrastructure failure despite UI being correct. Loading/pending states: Request waits for correlated screenshotResult. Agents should avoid fixed sleeps; prefer visibility/state proof before capture. getLayoutInfo() Failure/edge states: Non-main target rejected and returns default LayoutInfo. Empty components can mean mock/auto-submit mode, target rejection, or a real default layout result depending runtime. Layout info does not prove pixel rendering. Agent Notes Do not wait for showResult, hideResult, showGridResult, hideGridResult, or blurResult. They are not part of the SDK contract shown here. For stdin show / hide, use parse receipts to prove the command parsed, then use a query to prove state. Treat the window_visibility_ack evidence in app_run_setup.rs as a proof gap, not a stable universal contract. Prefer getState, getElements, inspectAutomationWindow, and getWindowBounds before screenshots.
