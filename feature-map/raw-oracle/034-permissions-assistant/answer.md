034 Permissions and Permission Assistant Executive Summary Permissions and Permission Assistant covers Script Kit GPUI’s macOS privacy-permission setup path. The feature has two separate jobs: First, it gives users an in-context setup flow for Accessibility and Screen Recording. The built-ins builtin/allow-accessibility and builtin/allow-screen-recording open the correct System Settings privacy pane and present a native AppKit overlay above the live settings window. The overlay contains a drag-source row for the Script Kit host .app, so the user can drag the application into the allowlist. Second, it exposes permission state without causing side effects. Passive checks read Accessibility, Screen Recording, Microphone, and MCP-facing Event Synthesizing status without prompting, writing TCC, clicking System Settings, activating Script Kit, or changing the launcher’s accessory-app posture. The central contract is simple but strict: read status passively; help the user act manually; never silently request or mutate macOS privacy permissions. Key anchored files: src/platform/permiso/mod.rs — public assistant API, retained handle, Settings opening, teardown through Drop. src/platform/permiso/panel.rs — PermisoPanel::Accessibility and PermisoPanel::ScreenRecording, Settings pane targets, display names, receipt names. src/platform/permiso/host_app.rs — resolves the host .app bundle path for the drag payload. src/platform/permiso/drag_source.rs — native drag-source row contract and row-hide/show lifecycle. src/platform/permiso/overlay_window.rs — overlay controller and passive panel ownership. src/platform/permiso/locator.rs — System Settings window locator contract and CoreGraphics-to-AppKit coordinate conversion. src/platform/permiso_detect.rs — passive Accessibility, Screen Recording, and Microphone checks. src/mcp_computer_use_tools.rs — computer/list_permissions and computer/get_permission. src/dictation/device.rs and src/dictation/setup.rs — microphone preflight boundary for dictation setup. src/platform/screenshots_window_open.rs — Screen Recording and Event Synthesizing preflight helpers used by capture and MCP status. Source audits under tests/source_audits/*permission* and tests/dictation_setup_nux_contract.rs pin the non-prompting behavior. What Users Can Do Users can: Open the Accessibility Permission Assistant from the launcher built-in command Allow Accessibility. Open the Screen Recording Permission Assistant from the launcher built-in command Allow Screen Recording. Open the same assistant actions from the Settings surface: Allow Accessibility Allow Screen Recording See System Settings opened to the relevant Privacy & Security pane. See a native overlay above the System Settings window with a Script Kit app drag row. Drag the Script Kit .app into the macOS allowlist manually. Check permission state through read-only status surfaces. Let dictation setup detect that microphone permission is denied or not determined without causing a hidden microphone prompt. Let screenshot automation detect missing Screen Recording permission before or during capture proof. Let MCP agents observe Accessibility, Screen Recording, and Event Synthesizing status without opening Settings or requesting permissions. Users cannot, through this feature: Force-grant a permission. Edit TCC.db. Use tccutil. Have Script Kit click the allowlist for them. Have the assistant silently prompt macOS for Accessibility, Screen Recording, or Microphone access. Use the Permission Assistant to activate Script Kit or change its app activation policy. Use MCP permission tools to request, grant, click, type, focus, activate, or open System Settings. Core Concepts Permission Assistant The Permission Assistant is the user-facing setup flow for Accessibility and Screen Recording. It opens the correct macOS System Settings privacy pane, then shows a passive overlay with drag instructions and a drag-source row. The assistant is represented by PermisoAssistant in src/platform/permiso/mod.rs. Its core entry path is: PermisoAssistant::present(panel) calls present_settings_url(panel) calls overlay_window::OverlayController::present(panel) returns a PermisoHandle PermisoAssistant::present_retained(panel) calls present(panel) stores the returned handle in the global retained slot PermisoAssistant::dismiss_active() clears the retained handle, dropping the active overlay PermisoPanel src/platform/permiso/panel.rs defines the two assistant panels: PermisoPanel::Accessibility PermisoPanel::ScreenRecording Each panel owns: settings_url() display_name() receipt_name() Accessibility display name is "Accessibility". Screen Recording display name is "Screen Recording". The receipt names are: Accessibility: "accessibility" Screen Recording: "screenRecording" Passive Permission Detection src/platform/permiso_detect.rs defines: The passive checks are: Accessibility: AXIsProcessTrusted() Screen Recording: CGPreflightScreenCaptureAccess() Microphone: AVCaptureDevice.authorizationStatusForMediaType(...) These checks do not request permissions. They only read state. Accessibility and Screen Recording return Authorized or Denied on macOS in the bundled implementation. Microphone can return Authorized, Denied, NotDetermined, or Unknown. Native Overlay The overlay is not a GPUI popup. The contract in lat.md/permissions.md describes it as a separate native NSPanel subclass, PassiveOverlayPanel, owned by OverlayController. Important native behavior pinned by docs and source audits: Uses a non-activating panel style. Cannot become key. Cannot become main. Uses status-bar-level placement. Uses orderFrontRegardless. Does not mutate launcher panel invariants. Does not use WindowKind::PopUp. Does not change app activation policy. Owns its own AppKit lifetime. The filtered src/platform/permiso/overlay_window.rs shows a simplified OverlayController with present, dismiss, and Drop; lat.md/permissions.md pins additional native behavior such as positioning helpers, refresh/repositioning, spring animation, and teardown ordering. Drag Source The assistant drag row is represented by AppDragSourceView in src/platform/permiso/drag_source.rs. The drag payload is the host application bundle URL from: host_app_bundle_url() The payload must be the .app directory itself. It must not be the executable inside Contents/MacOS. Reason: macOS Privacy & Security allowlists expect applications, not arbitrary internal executable paths. Dragging the executable would target the wrong object and can produce broken or non-durable permission setup. The native contract says the view is an NSView implementing: NSDraggingSource NSPasteboardItemDataProvider It writes a .fileURL pasteboard item for the resolved host .app bundle and uses copy drag semantics. System Settings Window Locator The locator contract in lat.md/permissions.md and src/platform/permiso/locator.rs is: Resolve the running System Settings app by bundle identifier com.apple.systempreferences. Query CGWindowListCopyWindowInfo. Filter to layer 0. Require width greater than 320. Require height greater than 240. Pick the largest remaining frame by area. Re-query every refresh tick. Do not cache results across display changes, space switches, or app activation. Convert CoreGraphics top-left coordinates to AppKit bottom-left coordinates. User Workflows Workflow 1: Allow Accessibility from the launcher User intent: grant Script Kit Accessibility permission. Steps: User opens Script Kit launcher. User searches for or selects Allow Accessibility. Built-in command builtin/allow-accessibility routes to PermissionCommandType::AllowAccessibility. Execution calls PermisoAssistant::present_retained(PermisoPanel::Accessibility). present_settings_url(PermisoPanel::Accessibility) opens the macOS Accessibility privacy pane. OverlayController::present(PermisoPanel::Accessibility) presents the passive overlay. Script Kit shows HUD: Drag Script Kit into Accessibility. User drags the Script Kit .app row into the System Settings allowlist. macOS owns the actual permission grant flow. Important boundaries: Script Kit does not call AXIsProcessTrustedWithOptions. Script Kit does not click the System Settings allowlist. Script Kit does not write TCC.db. Script Kit does not use tccutil. Script Kit does not activate itself. The command arm must not call prepare_for_submit_hide. Workflow 2: Allow Screen Recording from the launcher User intent: grant Script Kit Screen Recording permission. Steps: User opens Script Kit launcher. User searches for or selects Allow Screen Recording. Built-in command builtin/allow-screen-recording routes to PermissionCommandType::AllowScreenRecording. Execution calls PermisoAssistant::present_retained(PermisoPanel::ScreenRecording). present_settings_url(PermisoPanel::ScreenRecording) opens the macOS Screen Recording privacy pane. OverlayController::present(PermisoPanel::ScreenRecording) presents the passive overlay. Script Kit shows HUD: Drag Script Kit into Screen Recording. User drags the Script Kit .app row into the System Settings allowlist. macOS owns the actual permission grant flow. Important boundaries: Script Kit does not call CGRequestScreenCaptureAccess. Script Kit does not automate System Settings. Script Kit does not take a screenshot merely to check status. Screen Recording status is checked through preflight APIs. Workflow 3: Open assistant from Settings User intent: use the Settings UI instead of launcher search. Steps: User opens Script Kit Settings. User selects Allow Accessibility or Allow Screen Recording. SettingsAction::AllowAccessibility or SettingsAction::AllowScreenRecording constructs the matching BuiltInEntry. Settings calls self.execute_builtin(&entry, cx). Execution follows the same retained assistant path as the launcher built-ins. Result: Same System Settings pane. Same passive overlay. Same drag-source row. Same HUD. Same non-prompting guarantees. Workflow 4: Drag the app row User intent: place Script Kit into the macOS privacy allowlist. Relevant code: src/platform/permiso/drag_source.rs::AppDragSourceView src/platform/permiso/host_app.rs::host_app_bundle_url() Steps: Overlay renders a native drag-source row. AppDragSourceView::new() resolves the host .app bundle URL. When dragging starts: dragging_session_will_begin_at_point() sets row_hidden = true. Native contract: a snapshot of the row is used as the drag image. Pasteboard payload is the .fileURL for the host .app directory. Drag operation is copy. When dragging ends: ended_at_point_operation() sets row_hidden = false. The exact payload boundary is critical: the dragged file URL must resolve to the real .app directory. It must not point to Contents/MacOS/.... Workflow 5: Read permission status from MCP User intent: an agent wants to know whether key permissions are granted. Use: computer/list_permissions for all permission rows. computer/get_permission for one permission id. Current MCP permission ids: accessibility screenRecording eventSynthesizing Returned row fields: id name granted status Status mapping: Some(true) → "granted" Some(false) → "notGranted" None → "unknown" Important: Microphone status is part of passive permission detection and dictation setup, but the bundled MCP permission list exposes Accessibility, Screen Recording, and Event Synthesizing — not Microphone. Workflow 6: Dictation setup detects microphone permission User intent: start or configure dictation. Relevant files: src/platform/permiso_detect.rs::microphone_authorized() src/dictation/device.rs::microphone_permission_status() src/dictation/setup.rs::build_dictation_setup_state() src/app_execute/builtin_execution.rs::open_dictation_setup_if_microphone_not_ready() tests/dictation_setup_nux_contract.rs Flow: Dictation checks microphone permission through AVFoundation authorization status. It does not call requestAccessForMediaType. If microphone permission is Denied or NotDetermined, setup state becomes DictationMicrophoneStatus::PermissionNeeded(...). Dictation readiness is false. Starting dictation when microphone is not ready shows HUD: Dictation needs microphone permission. Script Kit opens dictation setup instead of silently requesting permission. Unknown is treated as non-blocking by the start-path guard in open_dictation_setup_if_microphone_not_ready(). Hotkey note: dictation setup readiness does not require a configured hotkey. The setup model keeps hotkey configuration separate from microphone readiness. Workflow 7: Screenshot capture checks Screen Recording permission User intent: capture screenshots for automation proof. Relevant file: src/platform/screenshots_window_open.rs Flow: Capture path calls screen_capture_access_preflight(). On macOS, that calls CGPreflightScreenCaptureAccess(). If capture proceeds, the RGBA buffer is audited before PNG encoding. Captures with no sampled pixels, no non-transparent pixels, solid-like frames, or dark-empty low-variety frames are rejected. Failure uses structured errors such as: automation.capture_screenshot.permission_failed automation.capture_screenshot.blank_image_rejected This keeps missing Screen Recording permission from producing misleading black screenshots. Host app path boundary host_app_bundle_url() resolves the Script Kit host .app bundle. Resolution order: On macOS, try the main bundle path. Accept it only if it has .app extension and is a directory. Otherwise inspect ancestors of std::env::current_exe(). Return the first ancestor that is a real .app directory. If no .app ancestor exists, bail. Privacy boundary: The drag payload exposes the local path to Script Kit’s .app bundle on the pasteboard during drag. It does not expose the internal executable path. It does not expose user documents or script contents. Overlay state boundary The retained active handle is process-local state: ACTIVE_PERMISO_HANDLE Mutex<Option<PermisoHandle>> It is not durable storage. It is not a user preference. It exists to keep the native overlay alive. MCP data boundary computer/list_permissions returns only: Schema version. Permission id. Permission name. Granted boolean or null. Status string. computer/get_permission adds: Source. Scope. Found/notFound status. Warnings. It does not return: Window state. Images. App inventory. User data. Paths. Action handles. Permission detection must be passive. Use preflight/status APIs only. Do not request. Assistant setup must be manual. The assistant may open System Settings and show an overlay. It must not click, type, press, or grant. The overlay must be native and non-activating. It must not become key/main and must not change Script Kit’s activation policy. The overlay must not be a GPUI popup. It is separate from launcher WindowKind::PopUp and panel invariants. The drag payload must be the .app bundle URL. Never drag the Contents/MacOS executable. The System Settings locator must not cache stale windows. Re-query every refresh tick. Only one assistant handle should be retained. ACTIVE_PERMISO_HANDLE keeps the current overlay alive and drops the prior one on replacement. Teardown must be deterministic. Dropping the handle must dismiss the controller and release native resources. Built-in assistant commands must not hide via prepare_for_submit_hide. This is source-audited. MCP permission tools must be status-only. No runtime bridge, no System Settings opening, no action handles, no prompting. Dictation microphone preflight must not request microphone access. authorizationStatusForMediaType only. Screenshot proof must reject misleading captures. Blank/black/solid captures cannot count as valid evidence. Regression risks Replacing AXIsProcessTrusted() with AXIsProcessTrustedWithOptions would cause prompt behavior. Replacing CGPreflightScreenCaptureAccess() with CGRequestScreenCaptureAccess would prompt. Replacing authorizationStatusForMediaType with requestAccessForMediaType would prompt. Adding System Settings automation would violate the manual grant contract. Dragging the executable instead of the .app bundle would break allowlist semantics. Caching CGWindowListCopyWindowInfo results would break overlay placement after display/Space changes. Reusing the launcher popup window for the overlay would risk focus, activation, and panel invariant regressions. Removing retained handle storage would cause the overlay to disappear immediately after command execution. Adding MCP runtime bridge methods for permission status would expand the attack surface without need. Adding click, press, focus, activate, hotkey, or type behavior to permission tools would violate the observation-only contract. Treating a screenshot PNG as proof without content audit would hide Screen Recording failures. Treating dictation hotkey absence as microphone unreadiness would regress setup semantics. Adding Microphone to MCP without explicit schema/tests could confuse the current protocol contract, which lists Accessibility, Screen Recording, and Event Synthesizing only. Agent Notes Agents should treat this feature as a manual setup assistant plus read-only status layer, not as an automation grant mechanism. Practical rules: To inspect current permission state, use computer/list_permissions. To inspect one permission, use computer/get_permission. Open Questions And Gaps Filtered locator source appears stubbed. src/platform/permiso/locator.rs::settings_window_snapshot() returns None in the provided snippet, with comments describing the production locator. The docs say it resolves System Settings via CGWindowListCopyWindowInfo. Verify whether the full local implementation exists outside the focused bundle or whether this is still pending. Filtered overlay source is simplified. src/platform/permiso/overlay_window.rs in the bundle shows PassiveOverlayPanel as an empty struct and a simple OverlayController. lat.md/permissions.md documents a fuller native AppKit implementation with NSPanel, status-bar level, refresh/repositioning, anchored_origin, spring_frame_at, timers, observers, and display link teardown. Verify the actual local implementation before relying on those native details at runtime. Teardown order is documented more deeply than shown. Docs specify timer/workspace observer invalidation, then CADisplayLink, then panel order-out/release. The filtered source only shows controller.dismiss() and overlay = None. Microphone is not currently an MCP permission row. The feature scope includes passive microphone detection, and dictation setup uses it. But computer/list_permissions exposes Accessibility, Screen Recording, and Event Synthesizing only. Adding Microphone to MCP would need an explicit protocol update and audits. Accessibility and Screen Recording do not expose NotDetermined in the current passive enum mapping. The enum supports NotDetermined, but the current Accessibility and Screen Recording checks map false to Denied. Microphone uses NotDetermined. Legacy Accessibility commands remain adjacent. Settings still includes Request Accessibility Permission and Open Accessibility Settings. Those are not the Permission Assistant flow and should stay clearly separated in docs and tests. No overlay-position runtime receipt is shown in the bundle. Source audits pin structure and non-prompting behavior, but visual overlay placement still needs manual or screenshot proof. Running from an unbundled dev binary can break drag payload resolution. host_app_bundle_url() requires a real .app directory. Development workflows should account for that. SDK clipboard and selected-text boundaries are not directly auditable from this focused bundle. The safe conclusion is that Permission Assistant and MCP permission tools must not touch clipboard or selected text. The actual SDK APIs need their own atlas or source audit.
