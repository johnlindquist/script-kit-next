019 Path Prompt / path() Executive Summary The Path Prompt is the SDK-backed file and directory selection prompt exposed to scripts as path(options?). In the current source snapshot, it is a compact prompt entity, not the same surface as dedicated File Search. It lets a script show a filesystem browser rooted at options.startPath or the user’s home directory, filter visible entries by typed text, move through directory rows and file rows, navigate into directories, navigate to the parent directory, submit the selected entry, or cancel. The SDK API is small: TypeScriptinterface PathOptions { startPath?: string; hint?: string;}function path(options?: PathOptions): Promise<string>; Proof: scripts/kit-sdk.ts defines PathOptions with only startPath?: string and hint?: string, and declares path(options?: PathOptions): Promise<string> at scripts/kit-sdk.ts:27-30 and scripts/kit-sdk.ts:1636. At runtime, Message::Path { id, start_path, hint } is converted into PromptMessage::ShowPath, then into AppView::PathPrompt { id, entity, focus_handle }. The prompt is built as a PathPrompt entity with a submit callback, theme, shared path-actions state, and a focus handle. Proof: src/prompt_handler/mod.rs:968-976, src/prompt_handler/mod.rs:6060-6130, and src/main_sections/app_view_state.rs:51-56. The Path Prompt owns its own footer semantics. Native Run is labeled Select, dispatches to PathPrompt::handle_enter, and does not fall through to launcher execution. Its footer exposes only ↵ Select and ⌘K Actions; launcher AI is intentionally omitted for this prompt surface. Proof: lat.md/design.md:24, src/app_impl/ui_window.rs:209-212, and src/app_impl/ui_window.rs:527-545. Path actions are typed through PathAction, not ad hoc string matching. Supported action ids are select_file, open_directory, copy_path, copy_filename, open_in_finder, open_in_editor, open_in_quick_terminal, and move_to_trash. The parser strips an optional file: prefix and rejects unknown ids. Proof: src/app_impl/path_action.rs:1-42 and inline tests at src/app_impl/path_action.rs:44-110. The main product boundary is important: SDK path() is not File Search. Dedicated File Search has its own AppView::FileSearchView, mini/full presentation modes, thumbnail preview, drag-out support, attachment-portal behavior, root-file integration, and ~ first-character launch routing. Path Prompt may share filesystem concepts and some action helpers, but the current source proves it is a separate prompt entity and separate route. Proof: src/main_sections/app_view_state.rs:122-126, src/render_builtins/file_search.rs:417-423, src/render_builtins/file_search.rs:1389-1412, and tests/file_search_tilde_entry.rs:3-18. What Users Can Do Users can call await path() from a TypeScript/Bun script and receive a selected filesystem path as a string. They can also pass startPath to control the initial directory and hint to provide prompt copy, although the attached render snippets do not prove exactly where hint appears visually. Users can browse the current directory. Directories and files are listed as entries. The prompt sorts directories alphabetically first, then files alphabetically. Proof: src/prompts/path/prompt.rs:105-108. Users can move the selected row up and down, navigate into a selected directory, navigate to the parent directory, type to filter, delete filter characters, submit the selected path, submit typed filter text when no row is selected and the filter is non-empty, or cancel. Proof: src/prompts/path/prompt.rs:202-235, src/prompts/path/prompt.rs:236-288, and src/prompts/path/render.rs:174-193. Users can open a path-specific action menu with ⌘K. The action menu is attached to the currently selected path entry. The menu can execute actions such as copy path, open/reveal, open in editor, open in Quick Terminal, or move to trash, depending on the action id selected. Proof: src/prompts/path/prompt.rs:158-200, src/render_prompts/path.rs:120-143, src/render_prompts/path.rs:203-257, and src/app_impl/path_action.rs:1-42. Core Concepts SDK surface. path(options?) is a script-level async function. It sends a prompt message and resolves when the prompt submits. The visible options in this bundle are only startPath and hint; no proof exists in the attached source for mode flags like “files only,” “directories only,” “allow multiple,” “extensions,” or “show hidden.” Proof: scripts/kit-sdk.ts:27-30, scripts/kit-sdk.ts:1624-1644, and scripts/kit-sdk.ts:2324-2330. Prompt id. Each SDK path() call gets a generated id from nextId(). The Rust handler carries that id into PromptMessage::ShowPath and then into PathPrompt::new. The submit callback returns the same id with either Some(path) or None. Proof: scripts/kit-sdk.ts:2324-2331, src/prompt_handler/mod.rs:6060-6094, and src/prompts/path/prompt.rs:202-235. Current directory. PathPrompt::new initializes current_path from start_path when provided, otherwise from dirs::home_dir(), falling back to /. It computes a display prefix, loads entries from the current path, copies those into filtered_entries, builds render rows, and starts with selected_index = 0. Proof: src/prompts/path/prompt.rs:3-32. Entries. PathEntry values are held in entries and filtered_entries; render rows are built from the filtered entries. The attached source proves that entries have name, path, and is_dir/directory information through PathInfo and element collection. Proof: src/prompts/path/types.rs:8-20, src/prompts/path/types.rs:25-33, and src/app_layout/collect_elements.rs:1099-1113. Selection. selected_index points into filtered_entries. Submitting selects that row’s full path. If there is no selected row and filter text is non-empty, the prompt submits the filter text as a path. Proof: src/prompts/path/prompt.rs:202-221. Filtering. Typed printable characters are appended to the prompt input through handle_char; Backspace routes through handle_backspace; and set_input exists for dictation/protocol input. The attached snippets prove filter text exists and drives collected elements, but the exact matching algorithm inside update_filtered is not included in the bundle window. Treat substring/fuzzy details as a proof gap. Proof: src/prompts/path/prompt.rs:120, src/prompts/path/render.rs:188-193, src/app_impl/ui_window.rs:1311-1313, and src/app_layout/collect_elements.rs:1083-1093. Actions. ⌘K toggles the path action popup. The prompt emits PathPromptEvent::ShowActions(PathInfo) for the selected entry. The app creates an ActionsDialog::with_path, hides its search, aligns its background with the main window, marks the popup open, and synchronizes shared action state. Proof: src/prompts/path/prompt.rs:158-200 and src/render_prompts/path.rs:120-143. Footer ownership. Path Prompt is a prompt surface with its own native footer contract. Run means Select, not launcher execution. ⌘K Actions is retained. Proof: lat.md/design.md:24 and src/app_impl/ui_window.rs:209-212, src/app_impl/ui_window.rs:527-545. Window sizing. There is a small inconsistency worth calling out. PromptMessage::ShowPath immediately calls resize_to_view_sync(ViewType::ScriptList, 20) after installing the prompt, while later sizing logic maps AppView::PathPrompt to ViewType::DivPrompt. The architecture note says Path prompts own footer submission and use a compact GPUI prompt contract, but the exact final window size can depend on which sizing path runs last. Proof: src/prompt_handler/mod.rs:6119-6130 and src/app_impl/ui_window.rs:849.
