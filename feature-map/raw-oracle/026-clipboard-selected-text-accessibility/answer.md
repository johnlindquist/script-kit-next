026 Clipboard, Selected Text, and Accessibility APIs Executive Summary Feature 026 covers the user-facing and automation-facing clipboard, selected text, and accessibility permission APIs in Script Kit GPUI: copy(), paste(), clipboard.readText(), clipboard.writeText(), clipboard.readImage(), clipboard.writeImage(), getSelectedText(), setSelectedText(), hasAccessibilityPermission(), and requestAccessibilityPermission(). The implementation is split across three major layers: TypeScript/Bun SDK surface in scripts/kit-sdk.ts, where scripts call high-level globals and the SDK emits protocol messages. Rust protocol/message layer in src/protocol/message/... and src/protocol/types/primitives.rs, where clipboard, selected-text, and accessibility messages are typed and serialized. Runtime handlers in src/execute_script/mod.rs, src/executor/selected_text.rs, src/prompt_handler/mod.rs, src/selected_text.rs, and src/permissions_wizard.rs. Clipboard APIs use Message::Clipboard with ClipboardAction::{Read, Write} and ClipboardFormat::{Text, Image}. Text reads/writes use arboard::Clipboard. Image reads return raw image bytes base64-encoded, not a PNG/JPEG container. Image writes are typed in the SDK but the current executor-side ClipboardAction::Write path only calls clipboard.set_text(...), so clipboard.writeImage() is a gap: it sends base64 content, but the shown Rust write handler treats provided content as text rather than decoding and setting an image. Selected text APIs are macOS-first and depend on Accessibility permission. The SDK hides the Script Kit window, waits 20ms, then sends getSelectedText or setSelectedText. The Rust selected-text module checks Accessibility permission before reads/writes. Reads use the get_selected_text crate. Writes use a clipboard fallback: save original clipboard text, set target text into the clipboard, simulate Cmd+V via Core Graphics, wait, then best-effort restore the original clipboard. Automation over stdin uses typed response variants, not the executor’s older Submit shape. getSelectedText returns selectedText; setSelectedText returns textSet; checkAccessibility and requestAccessibility return accessibilityStatus. Source-audit tests pin these dispatcher arms, response constructors, request-id correlation, and privacy logging rules. The biggest operational risks are focus races, permission denial, clipboard restore failure, image payload ambiguity, unsupported platforms, SDK auto-submit fallbacks masking failures, and accidental logging of raw clipboard or selected text. What Users Can Do Users and scripts can: Read the current system clipboard text with paste() or clipboard.readText(). Write text to the system clipboard with copy(text) or clipboard.writeText(text). Attempt to read image data from the clipboard with clipboard.readImage(). Attempt to write image data to the clipboard with clipboard.writeImage(buffer), with the caveat that the shown executor write path appears text-only. Read selected text from the previously focused macOS app with getSelectedText(). Replace selected text in the previously focused macOS app with setSelectedText(text). Check whether Accessibility permission is granted with hasAccessibilityPermission(). Prompt for Accessibility permission with requestAccessibilityPermission(). Use $SELECTION and $CLIPBOARD inside template(...); the SDK expands $SELECTION by calling getSelectedText() and $CLIPBOARD by calling clipboard.readText(). Use clipboard-derived behavior indirectly through adjacent surfaces: Clipboard History paste. Emoji paste. Clipboard History copy/share/OCR/save/snippet actions. AI attachment from Clipboard History. Sharing trust watcher that inspects clipboard share links. Root unified Clipboard History source, covered by a different feature. Core Concepts Clipboard API The SDK defines: ClipboardAPI.readText(): Promise<string> ClipboardAPI.writeText(text: string): Promise<void> ClipboardAPI.readImage(): Promise<Buffer> ClipboardAPI.writeImage(buffer: Buffer): Promise<void> The protocol message shape in scripts/kit-sdk.ts is: TypeScriptinterface ClipboardMessage { type: 'clipboard'; id: string; action: 'read' | 'write'; format: 'text' | 'image'; content?: string;} The Rust protocol variant is: RustMessage::Clipboard { id: Option<String>, action: ClipboardAction, format: Option<ClipboardFormat>, content: Option<String>,} id is optional in Rust. When an SDK call provides an id, the executor responds with Message::Submit { id, value }. When a clipboard write arrives with no id, the executor treats it as fire-and-forget and writes text if content is present. Clipboard aliases copy(text) is a thin alias for clipboard.writeText(text). paste() is a thin alias for clipboard.readText(). These aliases live in scripts/kit-sdk.ts: TypeScriptglobalThis.copy = async function copy(text: string): Promise<void> { return globalThis.clipboard.writeText(text);};globalThis.paste = async function paste(): Promise<string> { return globalThis.clipboard.readText();}; Selected text API The SDK defines: setSelectedText(text: string): Promise<void> getSelectedText(): Promise<string> Both yield focus back to the previous/frontmost app before sending the protocol message: setSelectedText calls await hide(), waits 20ms, then sends SetSelectedTextMessage. getSelectedText calls await hide(), waits 20ms, then sends GetSelectedTextMessage. That focus handoff is pinned by tests/config_contract_alignment.rs::kit_sdk_yields_focus_before_set_selected_text. Accessibility API The SDK defines: hasAccessibilityPermission(): Promise<boolean> requestAccessibilityPermission(): Promise<boolean> The message shapes are: TypeScriptinterface CheckAccessibilityMessage { type: 'checkAccessibility'; requestId: string;}interface RequestAccessibilityMessage { type: 'requestAccessibility'; requestId: string;} checkAccessibility is read-only. It checks current trust status. requestAccessibility may trigger the macOS system permission prompt via accessibility::application_is_trusted_with_prompt() through src/permissions_wizard.rs. Request IDs Clipboard SDK calls use id. Selected text and accessibility SDK calls use requestId. Rust exposes request_id() coverage for these variants in src/protocol/message/constructors/prompts.rs, including: GetSelectedText SetSelectedText CheckAccessibility RequestAccessibility SelectedText TextSet AccessibilityStatus ClipboardHistory variants other automation variants For stdin automation, the typed response must echo the same requestId. User Workflows Workflow: copy text to clipboard Script calls copy("hello"). SDK forwards to clipboard.writeText("hello"). SDK creates a request id with nextId(). SDK registers a pending resolver. SDK sends: JSON{ "type": "clipboard", "id": "<id>", "action": "write", "format": "text", "content": "hello"} Executor receives Message::Clipboard. Because id is present, it writes with arboard::Clipboard::set_text. On success, executor sends: JSON{ "type": "submit", "id": "<id>", "value": "ok"} SDK resolves Promise<void>. Failure handling is weak: the SDK write resolver ignores the actual Submit.value and resolves on any response. Executor failures usually return empty string, not a structured clipboard error. Workflow: paste/read text from clipboard Script calls paste() or clipboard.readText(). SDK sends: JSON{ "type": "clipboard", "id": "<id>", "action": "read", "format": "text"} Executor calls arboard::Clipboard::get_text(). On success, executor returns Submit.value = <clipboard text>. On clipboard init/read failure, executor logs the error and returns Submit.value = "". SDK resolves to msg.value ?? "". Empty clipboard, unreadable clipboard, unsupported format fallback, or clipboard init failure all collapse to an empty string at the SDK surface. Workflow: read image from clipboard Script calls clipboard.readImage(). SDK sends: JSON{ "type": "clipboard", "id": "<id>", "action": "read", "format": "image"} Executor calls arboard::Clipboard::get_image(). On success, executor takes img.bytes.to_vec() and base64-encodes those bytes. SDK decodes the base64 string into a Buffer. Important limitation: the executor appears to return raw image bytes from arboard, not a PNG container with width/height metadata. The SDK receives only a Buffer, with no image dimensions or pixel-format metadata. Treat this as raw bytes unless a future implementation explicitly encodes a standard image format. On failure, executor returns an empty string; SDK decodes it into an empty Buffer. Workflow: write image to clipboard Script calls clipboard.writeImage(buffer). SDK sends: JSON{ "type": "clipboard", "id": "<id>", "action": "write", "format": "image", "content": "<base64>"} The shown executor write branch does not branch on ClipboardFormat::Image; it calls clipboard.set_text(text.clone()) whenever content exists. Result: current shown implementation likely writes the base64 string as text rather than setting an image. This is a proof gap/current implementation gap. The SDK advertises writeImage(buffer), but the provided executor context does not show a matching image decode/set-image implementation. Workflow: read selected text Script calls getSelectedText(). SDK logs benchmark markers: getSelectedText_start getSelectedText_hide_done getSelectedText_delay_done getSelectedText_request_sent getSelectedText_response SDK calls await hide() so the previous app can regain focus. SDK waits 20ms. SDK sends: JSON{ "type": "getSelectedText", "requestId": "<id>"} Runtime selected-text handler checks macOS Accessibility permission. On macOS, src/selected_text.rs::get_selected_text() calls get_selected_text_impl(). If selected text is empty, Rust returns Ok(String::new()). If the selected-text probe fails, Rust returns an error. Executor path returns: success: Submit.value = text error: Submit.value = "ERROR: <error>" SDK rejects if msg.value starts with "ERROR:"; otherwise resolves msg.value ?? "". For stdin automation, the prompt handler returns Message::selected_text_response(text, request_id), not Submit. Workflow: replace selected text Script calls setSelectedText("replacement"). SDK calls await hide(). SDK waits 20ms. SDK sends: JSON{ "type": "setSelectedText", "requestId": "<id>", "text": "replacement"} On macOS, Rust checks Accessibility permission. Rust saves original clipboard text if available. Rust writes replacement text to the clipboard. Rust waits 10ms. Rust simulates Cmd+V through Core Graphics. Rust waits 150ms. If original clipboard text existed, Rust waits another 100ms and tries to restore it. Rust returns the paste result after the restore attempt. Executor path returns: success: Submit { id, value: None } error: Submit { id, value: "ERROR: <error>" } SDK rejects on ERROR: and otherwise resolves. For stdin automation, the prompt handler returns TextSet { success, error, requestId }. Workflow: check Accessibility permission Script calls hasAccessibilityPermission(). SDK sends: JSON{ "type": "checkAccessibility", "requestId": "<id>"} Runtime checks permission. Executor path returns Submit.value = "true" or "false". SDK resolves msg.value === "true". For stdin automation, src/prompt_handler/mod.rs calls crate::permissions_wizard::check_accessibility_permission() and returns: JSON{ "type": "accessibilityStatus", "granted": true, "requestId": "<id>"} or granted: false. Workflow: request Accessibility permission Script calls requestAccessibilityPermission(). SDK sends: JSON{ "type": "requestAccessibility", "requestId": "<id>"} Runtime requests permission. On macOS, src/permissions_wizard.rs::request_accessibility_permission() calls accessibility::application_is_trusted_with_prompt(). Response is boolean granted/not granted. This may show a macOS system dialog. src/permissions_wizard.rs notes the user may need to restart Script Kit after granting permission depending on app lifecycle timing. States Clipboard reads and writes do not inherently require a UI focus handoff. They operate on the system clipboard from the script runtime. Selected text reads and writes do require focus handoff: The SDK hides the main Script Kit window before calling selected-text operations. It waits 20ms so the previous/frontmost app can regain focus. The previous app must expose selected text through macOS Accessibility for reads. The previous app must accept Cmd+V into the current focused selection for writes. A race exists if 20ms is not enough for focus restoration. setSelectedText depends on clipboard replacement plus paste simulation, not a direct AX text-set operation in the shown source. Clipboard History and Emoji paste flows use the same broad UX contract: Put target content on clipboard. Hide or keep window depending on action. Simulate Cmd+V after a delay. src/app_actions/handle_action/paste.rs::finalize_paste_after_clipboard_ready centralizes this for some action flows: HideWindow closes/resets the main window first. KeepWindowOpen queues paste without hiding. Both call spawn_clipboard_paste_simulation(). src/app_actions/handle_action/clipboard.rs::spawn_clipboard_paste_simulation() sleeps 100ms before calling selected_text::simulate_paste_with_cg(). SDK rejects only when value starts with "ERROR:". Data, Storage, And Privacy Boundaries Clipboard text Clipboard text can contain secrets. Relevant boundaries: clipboard.readText() returns raw clipboard text to the script. clipboard.writeText(text) writes raw script-provided text to the system clipboard. Executor logs action, format, request id, and write byte length, not the content in the shown clipboard branch. Clipboard History is adjacent and stores clipboard data; do not treat it as part of this SDK clipboard API, but clipboard history actions can copy/paste/save/share stored entries. Clipboard images Image reads return base64 payloads over the protocol. Boundaries: Base64 image payloads can be large. The shown image read path returns img.bytes only; no dimensions or encoding metadata are returned. The SDK returns a Buffer, but the caller may not know whether it is raw pixels, PNG bytes, or another representation. Based on the shown executor, it is raw bytes from arboard. Empty image read failure becomes an empty buffer. Selected text Selected text is high-risk local user content from the frontmost app. Privacy rules pinned in tests: Stdin getSelectedText logs text_len, not raw text. Stdin setSelectedText logs text_len, not raw text. Source-audit tests fail if raw text logging is added to those arms. Executor selected-text path logs text_len on success and error string on failure. It does not log raw selected text in the provided context. Clipboard restoration setSelectedText temporarily overwrites the system clipboard. It attempts to restore the original clipboard text after paste, but restoration is best effort: It only saves/restores original text via clipboard.get_text().ok(). If the original clipboard held non-text data, the shown restore path does not preserve it. If restore fails, Rust logs a warning and still returns the paste result. A paste failure is returned only after the restore attempt. This means setSelectedText can disturb the clipboard, especially when the original clipboard contains images, files, rich text, or other non-text data. Accessibility permission Accessibility permission grants the app broad ability to inspect and synthesize interactions with other apps. Boundaries: checkAccessibility is read-only. requestAccessibility may show a macOS system prompt. Selected text read/write requires permission. Core Graphics paste simulation is used for text insertion. Logs Safe logging principles visible in this feature: Log request IDs and lengths. Do not log raw selected text or replacement text. Clipboard write logs byte length, not content. Clipboard read failure logs error, not content. Source-audit tests enforce selected-text privacy only on stdin arms; broader logging audits for all clipboard/selected-text paths would be a useful gap to close. Error, Empty, Loading, And Disabled States Clipboard empty/failure states Clipboard text read: Empty clipboard text returns "". Clipboard init failure returns "". Clipboard read error returns "". These are indistinguishable to SDK callers. Clipboard text write: arboard::Clipboard::new() failure returns Submit.value = "". set_text failure returns Submit.value = "". Missing content returns Submit.value = "". SDK clipboard.writeText() resolves anyway when a response arrives, because the resolver ignores value content. Clipboard image read: Clipboard init failure returns empty string → empty Buffer. Image read error returns empty string → empty Buffer. No dimensions/pixel format are returned. Clipboard image write: Current shown implementation gap: likely writes base64 text instead of setting image data. No dedicated image-write error shape is shown. No-request-id write: If Message::Clipboard has id: None and action: Write with content, executor attempts text write and sends no response. If no id and read, executor continues without response.
