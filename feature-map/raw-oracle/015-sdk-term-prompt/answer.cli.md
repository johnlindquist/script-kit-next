015 SDK TermPrompt / term() / Terminal Actions / Full-height Terminal Executive Summary SDK term() is the Script Kit SDK terminal prompt surface. It is launched from TypeScript/Bun scripts through globalThis.term(command?: string, actions?: Action[]): Promise<string> in scripts/kit-sdk.ts, routed through the Rust prompt handler, and rendered as AppView::TermPrompt { id, entity }. The important product distinction is that SDK TermPrompt is not Quick Terminal. They share the TermPrompt implementation for PTY/Alacritty rendering, input, theme adaptation, and terminal command behavior, but they have different route identity, sizing, footer ownership, and close/apply-back semantics. SDK TermPrompt: Is created by the SDK prompt route as AppView::TermPrompt { id, entity }. Uses ViewType::TermPrompt sizing and remains full-height, with layout::MAX_HEIGHT. Keeps the terminal GPUI hint strip / prompt-owned footer behavior. Does not register the native quick_terminal footer surface. Returns terminal output as the SDK term() promise result. Supports command execution and interactive shell sessions. Supports SDK-provided actions when passed to term(command?, actions?). Falls back to built-in terminal commands when no SDK actions are provided. Owns the terminal actions host as ActionsDialogHost::TermPrompt. Quick Terminal: Is opened from launcher utility command flows as AppView::QuickTerminalView. Uses compact launcher height. Uses native main-window footer once the host is installed. Has Tab AI apply-back semantics that SDK TermPrompt does not inherit. Uses a warm PTY pool and Quick Terminal-specific shell/path contracts. ACP Chat is separate again. It is an AI chat surface, not a terminal surface. It may share footer concepts and agentic automation ideas, but SDK TermPrompt should not be described as ACP Chat or Quick Terminal. What Users Can Do Users can call await term() from a Script Kit SDK script to open a terminal prompt. With no command, it opens an interactive terminal shell. With a command, it starts a terminal process for that command. Users can run commands such as: TypeScriptawait term("ls")await term('echo "hello"')await term('for i in 1 2 3; do echo "Line $i"; done')await term() The SDK test file tests/sdk/test-term.ts exercises basic command execution, echo output, ANSI color output, successful exit behavior, non-zero command behavior, multi-line output, and no-command interactive terminal behavior. Users can interact with the terminal like a normal terminal surface: type printable text, press Enter, use Ctrl combinations, paste, copy, scroll, select terminal text, and use mouse interaction where terminal rendering/input supports it. Users can clear the terminal with Cmd+K. Users can open or close terminal actions with Cmd+Shift+K. Users can use terminal actions either from SDK-provided actions passed into term(command?, actions?) or from built-in terminal commands such as scroll, find, clear, reset, and action toggle. Users can close the prompt/window using the terminal close path. For Quick Terminal, Cmd+W is explicitly state-first via close_quick_terminal_main_window_state_first; SDK TermPrompt should not inherit Quick Terminal’s close/apply-back special handling. Core Concepts SDK term() API The SDK entry point is globalThis.term = async function term(command?: string, actionsInput?: Action[]): Promise<string> in scripts/kit-sdk.ts. It creates a new prompt id with nextId(), processes optional actions, stores handlers, serializes actions, and sends a terminal prompt request through the SDK prompt protocol. The exact serialized JSON fields are not fully visible in the tight bundle, so the atlas should treat the wire payload shape as “terminal prompt request with id, optional command, and optional serialized actions” unless expanded from source. Rust TermPrompt route The Rust prompt handler receives the terminal prompt request, calculates: Rustlet term_height = window_resize::layout::MAX_HEIGHT - px(window_resize::layout::FOOTER_HEIGHT); Then it creates the terminal with: Rustterm_prompt::TermPrompt::with_height( id.clone(), command, self.focus_handle.clone(), submit_callback, term_height,) On success, it wraps the prompt in a GPUI entity and installs: Rustself.current_view = AppView::TermPrompt { id, entity };self.focused_input = FocusedInput::None;self.pending_focus = Some(FocusTarget::TermPrompt); The route also uses an expected-id guard before deferred sizing/focus work, so stale async work does not resize or focus the wrong prompt after a view change. This is covered by tests/source_audits/resize_presentation_contract.rs. Shared TermPrompt implementation TermPrompt is shared by SDK TermPrompt and Quick Terminal. It owns terminal handle creation, shell command creation, terminal theme use, effective padding, terminal rendering state, and terminal output capture. For command terminals: RustTerminalHandle::with_command_and_theme(&cmd, initial_cols, initial_rows, &theme) For interactive terminals: RustTerminalHandle::new_with_theme(initial_cols, initial_rows, &theme) The initial terminal grid starts at initial_cols and initial_rows = 24, then is resized to match rendered height/width. Actions mode src/render_prompts/term.rs defines: Rustfn term_prompt_actions_mode(has_sdk_actions: bool) -> TermPromptActionsMode { if has_sdk_actions { TermPromptActionsMode::SdkActions } else { TermPromptActionsMode::TerminalCommands }} This means SDK-provided actions win when present. Without SDK actions, the terminal actions palette shows built-in terminal commands. Footer ownership SDK TermPrompt keeps the terminal GPUI hint strip. Quick Terminal registers a native footer surface named "quick_terminal" and reserves footer space via the native footer spacer. User Workflows Run a command and receive output A script calls: TypeScriptconst output = await term('echo "SDK_TERM_TEST_OUTPUT_12345"') The SDK sends a terminal prompt request. Rust creates a full-height AppView::TermPrompt. The terminal starts a PTY command process using TerminalHandle::with_command_and_theme. Output appears in the terminal grid and is captured. When the terminal is closed/submitted, the SDK promise resolves to a string. tests/sdk/test-term.ts verifies this pattern for ls, echo, ANSI output, exit 0, non-zero command plus follow-up echo, and multi-line output. Open an interactive terminal A script calls: TypeScriptconst output = await term() Rust creates TermPrompt::with_height with no command. TermPrompt creates a shell through TerminalHandle::new_with_theme. The user interacts with the shell until closing/submitting. The returned output is whatever the terminal captured for that session. Use SDK actions A script calls term(command, actions). The SDK serializes the actions and stores handlers. Renderer detects that SDK actions exist and sets TermPromptActionsMode::SdkActions. Cmd+Shift+K opens/closes the actions dialog with SDK actions rather than built-in terminal commands. Selecting an SDK action routes through the prompt/action handler for that action id. Use built-in terminal commands When no SDK actions are present, the terminal actions palette uses built-in terminal commands. terminal_actions_for_dialog() includes actions such as find/search, scroll to top, scroll to bottom, clear, reset, and toggle actions. Clear is assigned Cmd+K; toggle actions is assigned Cmd+Shift+K. Clear terminal The user presses Cmd+K. is_term_prompt_clear_shortcut(has_cmd, has_shift, key) requires command/platform modifier, no Shift, and key K. The renderer calls: Rustexecute_term_prompt_action_by_id(TERM_PROMPT_CLEAR_ACTION_ID, cx) If the actions popup is open, it closes the popup after clear. Open terminal actions The user presses Cmd+Shift+K. is_term_prompt_actions_toggle_shortcut requires command/platform modifier, Shift, and key K. The renderer calls: Rusttoggle_term_prompt_actions(actions_mode, cx, window) The host is ActionsDialogHost::TermPrompt. Automation reads terminal surface Agentic automation can call state/element collection. For SDK TermPrompt, the surface is collected with semantic prefix "term" and finalized as "term-prompt". Quick Terminal is collected separately as "quick-terminal". Visual And Focus States Full-height SDK terminal SDK TermPrompt uses the full terminal prompt height path. ViewType::TermPrompt maps to max_height in src/window_resize/mod.rs, and tests assert height_for_view_with_layout(ViewType::TermPrompt, 0, &layout) == layout::MAX_HEIGHT. The prompt route passes a content height of MAX_HEIGHT - FOOTER_HEIGHT into TermPrompt::with_height. That leaves space for the prompt-owned footer/hint area while preserving full-window terminal presentation. Footer state SDK TermPrompt keeps a GPUI terminal hint strip. It must not register: RustAppView::TermPrompt { .. } => Some("term_prompt") Quick Terminal does register: RustAppView::QuickTerminalView { .. } => Some("quick_terminal") This is pinned by tests/quick_terminal_contracts.rs and tests/main_window_footer_surface_owner_contract.rs. Focus When SDK TermPrompt is installed, focus is not an arg/input field: Rustself.focused_input = FocusedInput::None;self.pending_focus = Some(FocusTarget::TermPrompt); When the actions dialog uses ActionsDialogHost::TermPrompt, close/activation focus restoration maps to: RustFocusRequest::term_prompt() Actions popup state When the actions popup is shown, the renderer sets: Rustterm.suppress_keys = show_actions; This prevents normal terminal key handling from fighting with the actions dialog while the dialog is active. Actions And Menus The terminal actions system has two modes: SDK actions mode Active when term(command?, actions?) supplies actions. term_prompt_actions_mode(true) => TermPromptActionsMode::SdkActions. Actions are serialized in TypeScript and routed back to stored handlers. Terminal commands mode Active when no SDK actions are supplied. term_prompt_actions_mode(false) => TermPromptActionsMode::TerminalCommands. Built-in terminal commands are shown. Built-in terminal command ordering is controlled by terminal_action_sort_key in src/app_impl/actions_toggle.rs: find scroll_to_top scroll_to_bottom clear reset The clear action is in the "Session" section and uses trash icon. Reset is also a session action. Search/find uses magnifying glass. Scroll to top and scroll to bottom use navigation sections/icons. The actions list appends: RustAction::new( TERM_PROMPT_ACTIONS_TOGGLE_ACTION_ID, "Toggle Actions", Some("Open or close the terminal actions palette".to_string()), ActionCategory::Terminal,).with_shortcut("⌘⇧K").with_icon(IconName::Settings) ActionsDialogHost::TermPrompt is the owner for terminal actions. It appears in host naming, toggle handling, activation callback setup, close callback setup, and focus restoration. Data, Storage, And Privacy Boundaries Terminal output The SDK term() promise returns a string. This string is terminal output captured from the session. It may include command output, shell prompts, ANSI escape output, multi-line text, and data printed by local commands. tests/sdk/test-term.ts confirms output string handling for echo, ANSI color command, multi-line output, exit 0, and non-zero-command flows. Prompt text versus terminal output SDK TermPrompt is not a normal text input prompt. It should not be treated as arg, textarea, or chat input. Its user-visible content is terminal grid/output plus PTY input, and its returned value is terminal output. Local shell environment Commands run in a local PTY subprocess. They can read local files, environment variables, shell configuration, and current working directory depending on the terminal process setup. This is a major privacy/security boundary: term() exposes local shell behavior, not a sandboxed text prompt. Clipboard Copy/paste is a terminal interaction boundary. Pasting sends clipboard contents into the PTY. Copying can place terminal selection/output on the clipboard. Exact implementation is not visible in the tight bundle, but any changes here should treat clipboard data as sensitive. ANSI/control sequences ANSI output is expected and tested. The terminal renders escape sequences through the Alacritty/terminal adapter rather than treating output as plain text only. Returned output may contain raw or processed terminal text depending on capture implementation; the exact normalization is not visible in the tight bundle. SDK actions SDK-provided actions are serialized across the SDK/app protocol. Handlers are stored in the SDK runtime. Action payloads and labels may reveal script-specific workflow information. Built-in terminal commands do not require SDK handler storage. PTY subprocess state PTY processes are live local subprocesses. Closing the terminal should clean up the associated process/session. Exact cleanup behavior is in terminal lifecycle code referenced by the bundle but not expanded in the visible snippets. Error, Empty, Loading, And Disabled States Terminal creation failure TermPrompt::with_height returns a Result. If TerminalHandle::with_command_and_theme or TerminalHandle::new_with_theme fails, the prompt handler must handle the error rather than installing a broken AppView::TermPrompt. The exact error UI is not visible in the tight bundle. Command with no output term("true") or similar commands may produce an empty output string. Empty output is valid and should not be confused with failure. Non-zero command behavior tests/sdk/test-term.ts uses: TypeScriptawait term('false; echo "Command returned non-zero"') and treats completion as pass. This suggests term() does not simply reject on a non-zero intermediate command when the shell continues and output is captured. Whether a final non-zero exit code rejects, resolves, or is ignored is not fully proven by the visible test because the command continues with echo. ANSI output ANSI color output is tested with: TypeScriptecho -e "\x1b[31mRed\x1b[0m \x1b[32mGreen\x1b[0m Normal" The UI should render ANSI via the terminal adapter. Return string semantics for ANSI escape preservation are not fully visible. Interactive no-command session term() with no command is valid. Output length may be zero or non-zero depending on shell prompts and user activity. tests/sdk/test-term.ts labels it “Interactive terminal opened and closed.” Actions disabled/suppressed state When the actions popup is open, terminal key handling is suppressed through term.suppress_keys = show_actions. This prevents accidental PTY input while the actions dialog owns focus. Stale deferred resize/focus The prompt route guards deferred resize/focus with expected id matching: Rustmatches!(view, AppView::TermPrompt { id, .. } if id == &expected_id) This prevents old async work from mutating a newer prompt. Agent Notes Treat SDK TermPrompt as a prompt runtime surface, not a launcher utility surface. When changing terminal rendering, always ask: “Does this affect both SDK TermPrompt and Quick Terminal?” Shared TermPrompt changes probably do. Route identity, footer, sizing, apply-back, warm PTY, and Quick Terminal path behavior should remain separated. When changing footer code, run the footer owner contract tests. The most likely regression is accidentally registering AppView::TermPrompt as a native footer surface or making SDK TermPrompt render Quick Terminal’s native footer spacer. When changing window sizing, keep ViewType::TermPrompt => max_height. Quick Terminal compact sizing is intentionally implemented outside the generic ViewType::TermPrompt branch. When changing actions, preserve the host: RustActionsDialogHost::TermPrompt and preserve the shortcut split: Cmd+K: clear. Cmd+Shift+K: actions. When changing automation, keep SDK TermPrompt and Quick Terminal semantic prefixes distinct. Agentic receipts should make it obvious which terminal surface is currently active. When changing command execution or output capture, rerun tests/sdk/test-term.ts and manually verify interactive term() because automated Ctrl+C / no-command behavior is limited.
