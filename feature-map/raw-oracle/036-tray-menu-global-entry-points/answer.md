036 Tray Menu and Global App Entry Points Executive Summary Feature 036-tray-menu-global-entry-points covers Script Kit GPUI’s macOS status-bar menu and the global app entry routes that let a user reach Script Kit without starting from the main launcher UI. The tray menu is the app’s secondary entry point. It is created by TrayManager in src/tray/mod.rs, initialized after first window creation from src/main_entry/app_run_setup.rs, and built as a native macOS menu with stable action ids, section separators, native/template icons, dynamic labels, update-state mutation, and a read-only MCP observation model. The tray surface supports these user-facing actions: Open Script Kit. Open Current App Commands. Open Notes. Open AI / Agent Chat. Send Feedback. Follow Us. Open GitHub. Join Discord. Open Settings. Reload Scripts. Check for Updates. Open release page from the Version row when an update is available. Open About Script Kit. Quit Script Kit. The same feature also covers the entry-point contract around the global launcher hotkey display. The tray’s Open Script Kit row mirrors the configured launcher hotkey through tray::main_shortcut_accelerator(&config.hotkey), so the status-bar menu shows the same key equivalent the user expects from the global launcher path. The focused bundle does not expose the full hotkey registration implementation, but it does include the display conversion helper and tests that pin default meta+Semicolon plus common variants. The feature has a strict automation boundary. MCP tools can observe Script Kit’s own tray menu model through computer/list_tray_menu, computer/get_tray_menu_item, and computer/get_tray_menu_item_by_id, but those tools never open the tray menu, click status items, execute tray actions, enumerate global menu extras, refresh native menu state, or request permissions. This is explicitly pinned in lat.md/protocol.md and source audits under tests/source_audits/. The implementation distinguishes three surfaces that can otherwise be confused: Script Kit’s own tray menu model — owned by src/tray/mod.rs, exposed read-only through MCP. Frontmost app menu-bar observation — used by Current App Commands and MCP menu tools such as computer/list_menus; this reads the cached menu tree for the last tracked non-Script-Kit app. Global status items / menu extras discovery — explicitly not implemented for MCP tray tools or frontmost menu tools. The most important operator invariant is this: the tray menu is actionable for human clicks, but the MCP tray surface is observation-only. Agents can inspect rows, ids, sections, enabled state, title source, destination kind, and warning/status metadata, but they cannot receive executable handles or trigger the native tray from these tools. What Users Can Do Users can use the macOS status-bar icon as a persistent app entry point. They can open the main Script Kit launcher from the tray using Open Script Kit. The tray row displays the configured launcher shortcut when main_shortcut_accelerator can convert the configured HotkeyConfig into a muda Accelerator. The default meta+Semicolon is pinned by test_main_shortcut_accelerator_default in src/tray/mod.rs. They can open Current App Commands from the tray. The row title is dynamic. When the frontmost-app tracker has a last real app, the label becomes <localized app name> Commands; when no tracked app exists, it falls back to Current App Commands…. The tray dispatcher refreshes this row before every tray event because the tray-icon / muda stack does not expose a menu-will-open callback. The actual open route calls view.open_current_app_commands_from_tray(cx) after showing the main window. They can open the Notes window from the tray using Open Notes. The current dispatcher path uses notes::open_notes_window_without_launcher_restore(cx), which means opening Notes does not restore the launcher first. They can open the AI / Agent Chat entry using Open AI. The tray handler shows the main window, then calls view.open_tab_ai_acp_with_entry_intent(None, cx). They can open external destinations: feedback, Follow Us, GitHub, and Discord. These URLs are pinned constants: URL_FEEDBACK in src/tray/mod.rs, and URL_FOLLOW_US, URL_GITHUB, URL_DISCORD via src/branding.rs. test_tray_urls_are_https_and_pinned verifies the public destinations stay HTTPS and pinned. They can open Settings. The tray handler resolves the configured editor through config_for_tray_actions.get_editor(), expands ~/.scriptkit/config.ts, and spawns the editor process with the config path. They can reload scripts. The tray handler calls view.refresh_scripts(ctx) inside the main app entity update. They can check for updates. The tray handler starts a worker thread that calls updates::check_now(state, || {}), waits on the GPUI background executor for a fixed delay, then refreshes the Version row on the GPUI/main thread using tray_mgr.refresh_version_label(). They can use the Version row as a release-page link only when an update is available. The row id is tray.open_release_page, but it is disabled for Idle, UpToDate, Checking, and Error states. When UpdateState::Available { version, url } is present, version_label_and_enabled returns Update Available: vX.Y.Z and enabled: true; clicking then reads tray_mgr.update_state_snapshot() and opens snapshot.release_url(). They can open About Script Kit from the tray. The tray handler shows the main window, waits briefly, then calls view.open_about_surface(state, cx) with the shared update state. There is also an automation-only stdin command openAbout documented in lat.md/protocol.md, routed through ExternalCommand::OpenAbout, which opens the in-window About surface without needing tray interaction. They can quit Script Kit. The tray handler sets SHUTDOWN_REQUESTED first, kills all managed child processes through PROCESS_MANAGER.kill_all_processes(), removes the main PID file through PROCESS_MANAGER.remove_main_pid(), then calls cx.quit(). Core Concepts TrayManager TrayManager is the owner of the native tray icon and menu in src/tray/mod.rs. It owns: tray_icon: TrayIcon. current_app_commands_item: IconMenuItem. version_item: MenuItem. update_state: Arc<RwLock<UpdateState>>. The manager builds the tray icon from embedded LOGO_SVG, creates the native menu, templates menu icons on macOS, stores the observation source for MCP snapshots, exposes the global MenuEvent::receiver(), converts menu events to typed actions, and mutates dynamic row labels. The main public methods and helpers are: TrayManager::new(update_state, main_shortcut). TrayManager::create_icon_from_svg(). TrayManager::create_menu(update_state, main_shortcut). TrayManager::menu_event_receiver(). TrayManager::action_from_event(event). TrayManager::handle_action(action). TrayManager::refresh_current_app_label(). TrayManager::refresh_version_label(). TrayManager::update_state_snapshot(). TrayManager::update_state_handle(). main_shortcut_accelerator(hk). current_tray_menu_observation_snapshot(). tray_menu_observation_snapshot(update_state, launcher_shortcut_configured). handle_action() currently has no internal side effects. Actual action execution happens in the tray event dispatcher in src/main_entry/app_run_setup.rs. Native menu layout The tray menu uses Raycast-style bands separated by PredefinedMenuItem::separator(). Current sections: Open Open Script Kit Current App Commands / <App Name> Commands Open Notes Open AI Help Send Feedback… Social Follow Us GitHub Discord System Settings Reload Scripts Check for Updates… Version / Checking / Update Available row About Script Kit Exit Quit Script Kit This order is pinned twice: through the actual create_menu() append order and through the observation snapshot tests in tray_menu_observation_sections_match_create_menu_order. Icon and template rendering The tray icon and several menu-row icons are rendered from SVG. menu_icon_from_svg(svg) renders a 32px RGBA menu icon. It rewrites currentColor to white before rendering because, once AppKit marks the NSImage as a template, AppKit uses the alpha mask and ignores the source color. On macOS, template_menu_items(menu.as_ref()) walks the underlying NSMenu through muda::ContextMenu::ns_menu(), copies each NSImage, sets setTemplate:YES, and recurses into submenus. This is main-thread-only. It logs: tray.menu_item_icons_templated with a count on success. tray.menu_item_template_noop if every item came back without an image. tray.template_menu_items_not_main_thread if called off the main thread. SVG render failures are not fatal for individual rows. The row falls back to native icons where configured: Open Script Kit falls back to NativeIcon::Home. Open Notes falls back to NativeIcon::Bookmarks. Open AI falls back to NativeIcon::Bookmarks. Follow Us falls back to NativeIcon::User. GitHub falls back to NativeIcon::FollowLinkFreestanding. Discord falls back to NativeIcon::UserGroup. Other rows use native icons directly where appropriate: Current App Commands uses NativeIcon::Bookmarks. Send Feedback uses NativeIcon::Share. Reload Scripts and Check for Updates use NativeIcon::Refresh. Settings, About Script Kit, and Quit Script Kit intentionally render without icons because their natural AppKit native variants are full-color status images or add visual noise. test_create_menu_uses_native_menu_icons and test_brand_icons_render pin the allowed native icon set and confirm brand glyphs render successfully. Dynamic current-app title The current-app row is dynamic and model-backed. current_app_commands_label() reads crate::frontmost_app_tracker::get_last_real_app(): If present: format!("{} Commands", app.name). If absent: Current App Commands…. TrayManager::refresh_current_app_label() calls set_text(current_app_commands_label()). The tray dispatcher calls this before handling every event because the native API does not offer a menu-will-open hook. The observed MCP model also uses current_app_commands_label() so the read-only model reflects the same title source. Dynamic update/version title version_label_and_enabled(state) maps UpdateState to the Version row label and enabled state: Update stateLabelEnabledIdleVersion <current>falseUpToDateVersion <current>falseCheckingChecking for updates… (v<current>)falseError(_)Version <current> (update check failed)falseAvailable { version, .. }Update Available: v<version>true The Version row is created with id tray.open_release_page, but whether it behaves as an actionable release link depends on update state. This is a clean no-op/disabled row when no release URL exists. The worker thread cannot mutate the menu directly because muda requires main-thread access for NSMenuItem mutation. Workers only write shared UpdateState; the GPUI-side dispatcher later calls refresh_version_label() on the main thread. Read-only tray observation model current_tray_menu_observation_snapshot() exposes Script Kit’s own tray menu as a pure model/state snapshot. The snapshot returns: schemaVersion. source: "scriptKitTrayMenuModel". owner: kind: "scriptKitStatusItem". name: "Script Kit". scope: "ownTrayMenuOnly". sections. warnings. Each item contains: id. title. enabled. shortcut. title_source. destination_kind. The observation snapshot deliberately contains no click, press, execute, action, focus, activate, refresh, or native status-item fields. tray_menu_observation_has_no_click_or_execute_fields pins this at serialization level. Entry Points 1. macOS status-bar icon The native tray/status-bar icon is initialized after the main window is created. app_run_setup.rs defers tray initialization with a short async yield so startup-to-first-render is not blocked by tray icon rendering or menu construction. Path: Create shared Arc<RwLock<UpdateState::Idle>>. Convert configured launcher hotkey: tray::main_shortcut_accelerator(&config_for_tray_actions.hotkey). Create tray manager: TrayManager::new(update_state.clone(), main_shortcut.clone()). On success, log Tray icon initialized successfully (deferred). On failure, log Failed to initialize tray icon (deferred): .... The tray is a secondary entry point, not the primary app startup path. 2. Tray menu clicks The tray event bridge reads MenuEvent::receiver() on a blocking thread and forwards events into an async channel. The GPUI async loop then receives each event and dispatches it. Per event: Refresh current app row label on the GPUI thread. Convert the native menu event id to Option<TrayMenuAction>. Match the typed action. Execute the corresponding app behavior. Unknown ids are ignored through None. 3. Global launcher hotkey display The tray Open Script Kit row receives the same accelerator derived from config.hotkey. main_shortcut_accelerator(hk) supports known modifier strings: meta, cmd, command. ctrl, control. alt, option. shift. It supports direct Code variants for keys such as Semicolon, Space, KeyK, Comma, and Slash. The helper avoids depending on muda’s string parser for keys like Semicolon. If any modifier or key is not recognized, it returns None, and the tray row appears without a key equivalent instead of showing a misleading shortcut. The focused bundle proves display conversion, not the whole global hotkey runtime. The included tests and file names indicate additional startup/global-key contracts exist in tests/main_window_global_key_intent_contract.rs and tests/launcher_startup_entrypoint_contract.rs, but their bodies are not included in the focused bundle. 4. Open Notes hotkey and AI hotkey The focused context contains tray entry routes for Notes and AI, plus file references to hotkey modules and runtime hotkey tests, but it does not include enough source to state the exact registered Notes or AI hotkey values. Grounded behavior from this bundle: Tray Notes route calls notes::open_notes_window_without_launcher_restore(cx). Tray AI route shows the main window and calls view.open_tab_ai_acp_with_entry_intent(None, cx). Older/stale-looking src/main_entry/runtime_tray_hotkeys.rs snippets show historical tray hotkey handling names such as OpenAiChat, OpenOnGitHub, OpenManual, and LaunchAtLogin; the current app_run_setup.rs path and TrayMenuAction enum use OpenAgentChat, OpenGitHub, SendFeedback, and no LaunchAtLogin. Operator note: treat src/main_entry/app_run_setup.rs plus src/tray/mod.rs as the current tray dispatch source in this bundle. Use runtime_tray_hotkeys.rs only as a regression-risk/stale-surface clue unless the local repo confirms it is still compiled. 5. Dictation boundaries The bundle includes dictation protocol-resource docs in lat.md/protocol.md, but not tray menu rows or hotkey implementation for dictation. There is no TrayMenuAction::Dictation, no dictation row in create_menu(), and no dictation tray item in the observation snapshot. Grounded boundary: Dictation exists as protocol resources, including kit://dictation and kit://dictation-history. Dictation is not exposed as a tray menu row in this feature snapshot. Any dictation hotkey behavior must be verified outside this focused tray bundle before being included as an actionable tray/global entry claim. 6. Automation-only About entry The tray About row is for humans. Automation has a separate route: ExternalCommand::OpenAbout in app_run_setup.rs. Documented in lat.md/protocol.md as openAbout. It opens the in-window About surface without requiring tray menu use. This matters because automation proofs should not click the tray. They should use state-first protocol routes for About verification. 7. MCP read-only observation tools MCP exposes these tray observation tools: computer/list_tray_menu. computer/get_tray_menu_item. computer/get_tray_menu_item_by_id. They are part of src/mcp_computer_use_tools.rs and documented in lat.md/protocol.md. They observe Script Kit’s own tray menu model. They are not native menu automation tools. It uses a Script Kit tray icon rendered from SVG. The menu groups rows with separators. Section labels are part of the observation model, not necessarily visible native headers. Native UI shows bands separated by separators. Icon states: The headline Open Script Kit row uses the Script Kit mark. Open Notes uses assets/icons/notes.svg. Open AI uses assets/icons/agent_chat.svg. Follow Us, GitHub, and Discord use brand SVGs from assets/icons/. Native icons are used for Current App Commands, Send Feedback, Reload Scripts, and Check for Updates. Settings, About, and Quit are icon-less by design. Template states: On macOS, template_menu_items() marks copied NSImages as template images. Template images allow AppKit to tint for light mode, dark mode, highlighted rows, and disabled rows. If templating fails or returns no touched items, the menu still works; white rendered glyphs are the fallback. Main-window focus behavior: Open Script Kit, Open Current App Commands, Open AI, and Open About show the main window. Open Notes opens the Notes window without launcher restore. Settings opens an external editor, not a Script Kit surface. External URL rows open the browser. Reload Scripts refreshes runtime state without necessarily changing focus. Quit exits the app. About automation focus behavior: ExternalCommand::OpenAbout marks the main window visible, shows and activates it, syncs automation window visibility, then opens About. Current App Commands focus/identity behavior: Current App Commands are tied to a captured source app identity. Related lat notes say tray-opened filtering/execution keeps source PID and bundle identity together and fails closed on app switches/relaunch drift. MCP focus behavior: MCP tray tools do not focus, activate, open, or click anything. They only return JSON. 036 Tray Menu and Global App Entry Points Executive Summary Feature 036-tray-menu-global-entry-points covers Script Kit GPUI’s macOS status-bar menu and the global app entry routes that let a user reach Script Kit without starting from the main launcher UI. The tray menu is the app’s secondary entry point. It is created by TrayManager in src/tray/mod.rs, initialized after first window creation from src/main_entry/app_run_setup.rs, and built as a native macOS menu with stable action ids, section separators, native/template icons, dynamic labels, update-state mutation, and a read-only MCP observation model. The tray surface supports these user-facing actions: Open Script Kit. Open Current App Commands. Open Notes. Open AI / Agent Chat. Send Feedback. Follow Us. Open GitHub. Join Discord. Open Settings. Reload Scripts. Check for Updates. Open release page from the Version row when an update is available. Open About Script Kit. Quit Script Kit. The same feature also covers the entry-point contract around the global launcher hotkey display. The tray’s Open Script Kit row mirrors the configured launcher hotkey through tray::main_shortcut_accelerator(&config.hotkey), so the status-bar menu shows the same key equivalent the user expects from the global launcher path. The focused bundle does not expose the full hotkey registration implementation, but it does include the display conversion helper and tests that pin default meta+Semicolon plus common variants. The feature has a strict automation boundary. MCP tools can observe Script Kit’s own tray menu model through computer/list_tray_menu, computer/get_tray_menu_item, and computer/get_tray_menu_item_by_id, but those tools never open the tray menu, click status items, execute tray actions, enumerate global menu extras, refresh native menu state, or request permissions. This is explicitly pinned in lat.md/protocol.md and source audits under tests/source_audits/. The implementation distinguishes three surfaces that can otherwise be confused: Script Kit’s own tray menu model — owned by src/tray/mod.rs, exposed read-only through MCP. Frontmost app menu-bar observation — used by Current App Commands and MCP menu tools such as computer/list_menus; this reads the cached menu tree for the last tracked non-Script-Kit app. Global status items / menu extras discovery — explicitly not implemented for MCP tray tools or frontmost menu tools. The most important operator invariant is this: the tray menu is actionable for human clicks, but the MCP tray surface is observation-only. Agents can inspect rows, ids, sections, enabled state, title source, destination kind, and warning/status metadata, but they cannot receive executable handles or trigger the native tray from these tools. What Users Can Do Users can use the macOS status-bar icon as a persistent app entry point. They can open the main Script Kit launcher from the tray using Open Script Kit. When the frontmost-app tracker has a last real app, the label becomes <localized app name> Commands; when no tracked app exists, it falls back to Current App Commands…. The tray dispatcher refreshes this row before every tray event because the tray-icon / muda stack does not expose a menu-will-open callback. The actual open route calls view.open_current_app_commands_from_tray(cx) after showing the main window. They can open the Notes window from the tray using Open Notes. Native UI shows bands separated by separators. Icon states: The headline Open Script Kit row uses the Script Kit mark. Open Notes uses assets/icons/notes.svg. MCP focus behavior: MCP tray tools do not focus, activate, open, or click anything. They only return JSON.
