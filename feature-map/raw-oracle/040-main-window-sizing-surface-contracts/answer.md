038 Main Window Sizing and Surface Contracts Executive Summary Main Window Sizing and Surface Contracts is the cross-cutting contract that keeps the Script Kit GPUI main window understandable, automatable, and regression-proof as it moves between launcher, mini launcher, prompt, built-in, preview, utility, and assistant surfaces. This feature has two tightly coupled halves: Main-window sizing decides whether the primary launcher window is in Mini or Full mode, which ViewType is used for width and height, and which resize path owns the transition. Surface identity decides what the active AppView means behaviorally through SurfaceKind, LauncherSurfaceContract, native footer ownership, automation semantic surface tags, and runtime getState.surfaceContract receipts. The core operational rule is simple but strict: surface layout determines size, and SurfaceKind determines behavior. A single-column command surface stays Mini. A surface with a real list-plus-preview/detail layout is Full. Agents should not infer width from command importance, row count, entry source, shortcut source, or render file name. The main window has several distinct presentation families: Full launcher / ScriptList: wide main menu with optional preview/info panel, represented by MainWindowMode::Full and ViewType::ScriptList. Mini main window: compact launcher surface, represented by MainWindowMode::Mini and ViewType::MiniMainWindow. MiniPrompt: compact script prompt route, represented by AppView::MiniPrompt and ViewType::MiniPrompt; this is not the same as Mini main window. Mini AI chat / inline assistant: compact assistant route, represented by ViewType::MiniAiChat when main_window_mode is Mini. Full prompt and content surfaces: DivPrompt, editor, terminal, form, path, drop, template, and other prompt/content views, with prompt-specific ViewTypes. Split preview/detail surfaces: wide surfaces whose layout has a list pane plus preview/detail pane, such as clipboard history, full file search, ACP history, browser history, dictation history, notes browse, SDK reference, and script template catalog. The proof model is state-first. Runtime verification should prefer getState, getElements, surfaceContract, activePopupContract, activeFooter, automation window metadata, generated contract checks, and source audits. Screenshots are useful for visual regressions, but they are not the primary proof for Mini/Full classification, semantic surface identity, footer ownership, or protocol behavior. What Users Can Do Users can open and move through the main launcher without thinking about the underlying contracts: Open the standard main menu in Full or Mini mode. Open Mini main window command surfaces from menu actions, utility commands, tray/current-app commands, or automation. Open single-column built-ins such as emoji picker, app launcher, window switcher, browser tabs, design gallery, favorites, process manager, settings, current-app commands, kit browse, installed kits, search AI presets, and mini dictation in compact Mini presentation. Open split preview/detail surfaces such as clipboard history, full file search, theme chooser, ACP history, browser history, dictation history, and notes browse in Full presentation. Open MiniPrompt without inheriting Full prompt width. Use inline Mini AI / mini-hosted ACP in compact mode and full chat/ACP in full prompt mode. Filter current surfaces without accidental width changes. Use Escape, Cmd-W, hide, reset, and go-back flows without leaking stale wide bounds into hidden-window receipts. Use shared actions popups while preserving the host surface identity. Trigger built-ins over stdin and see automation metadata re-key to the actual active surface. Inspect getState.surfaceContract to understand the active route’s behavior without reading render implementation files. Inspect activePopupContract when an attached actions dialog is open. Inspect activeFooter / footer elements to prove native footer ownership and avoid screenshot-only footer validation. For operators and agents, this means every major main-window route has two questions that must be answered together: What size should the main window be? What semantic surface is active for automation and behavior policy? Those answers are owned by different but cooperating files and tests. Core Concepts Main window vs secondary windows This feature is only about the primary main window and top-level AppView routes inside it. It does not own secondary-window sizing, detached popup sizing, detached ACP popup geometry, shortcut recorder popup geometry, or platform-level AppKit activation rules. Secondary target diagnostics intentionally omit surfaceContract because they are not main AppView receipts. MainWindowMode MainWindowMode is the app-level mode state for the main window: MainWindowMode::Mini is the default. MainWindowMode::Full restores the wide launcher/full prompt behavior. set_main_window_mode is the window-backed owner. It short-circuits unchanged modes, updates active ChatPrompt mini rendering, closes shared/detached actions popups, runs deferred sizing, syncs the native footer, and emits a main_window_mode_changed trace. set_main_window_mode_state_only updates mode without directly requiring a Window. It is used by open helpers that immediately call a synchronous resize primitive themselves. ViewType ViewType is the low-level sizing input consumed by window_resize: ScriptList: Full main launcher / split preview width. MiniMainWindow: compact main launcher / Mini built-in list width. MiniPrompt: compact prompt width. MiniAiChat: compact AI/ACP width. ArgPromptWithChoices: dynamic arg prompt choice list height. ArgPromptNoChoices: compact input-only prompt height. DivPrompt: full prompt/main-window standard height. EditorPrompt: full editor height. TermPrompt: full terminal height. Width is determined by width_for_view: MiniMainWindow, MiniPrompt, and MiniAiChat return MINI_MAIN_WINDOW_WIDTH. ScriptList returns FULL_MAIN_WINDOW_WIDTH. Other prompt/content view types preserve current width by returning None. Mini main window Mini main window is a compact main-window launcher mode. It is represented by MainWindowMode::Mini plus ViewType::MiniMainWindow. It is not the same as: MiniPrompt, which is a script prompt route. MicroPrompt, which resolves to compact no-choice prompt sizing. Inline Mini AI, which resolves through ViewType::MiniAiChat. Quick Terminal, which is a utility child content surface using terminal sizing. Detached popups or secondary windows. Mini main window uses fixed max height to prevent resize jank during filter changes and view transitions. Content-aware sizing still produces structured receipts, but height_for_mini_main_window returns MINI_MAIN_WINDOW_MAX_HEIGHT. Content-aware Mini sizing receipts Mini main-window sizing uses MiniMainWindowSizing: selectable_items visible_section_headers is_empty mini_main_window_sizing_from_grouped_items builds this from grouped launcher rows and treats section headers separately from selectable rows. This avoids conflating headers with selectable items. Even though the current height is fixed, every Mini resize path should log parseable sizing data through log_mini_window_sizing under the MINI_WINDOW target. The receipt records the resize reason, sizing fields, and target height. This matters because source and runtime proofs can verify that Mini behavior stayed Mini even when visual height does not change. Synchronous vs deferred resize There are two major resize paths: resize_to_view_sync(view_type, item_count) immediately calculates height/width and calls the raw resize primitive. defer_resize_to_view(view_type, item_count, window, cx) schedules resize at the end of the current GPUI effect cycle to avoid borrow conflicts. The app-level owners are: update_window_size_deferred(window, cx) for normal GPUI/window-context updates. update_window_size() for async handlers running outside render. calculate_window_size_params() for deriving the correct (ViewType, item_count) from the current AppView. calculate_window_size_params_if_current_view(reason, predicate) for async resize tasks that must re-read current_view and skip if the route changed. The critical invariant: a follow-up deferred resize must preserve the same Mini/Full classification chosen by the open helper. Built-in filterable sizing policy Built-in filterable views open through shared helpers that take an expanded: bool flag: expanded = true sets MainWindowMode::Full and resizes to ViewType::ScriptList. expanded = false sets MainWindowMode::Mini and resizes to ViewType::MiniMainWindow. Single-column filterable built-ins stay Mini. Split preview/detail built-ins stay Full. Examples of Mini single-column surfaces include: emoji picker app launcher window switcher browser tabs design gallery, where current source marks the contract as split-preview vocabulary but sizing evidence shows ViewType::MiniMainWindow; treat the exact render shape as partly inferred because broad render files were excluded favorites process manager settings current-app commands kit store browse installed kits search AI presets mini dictation Examples of Full split preview/detail surfaces include: clipboard history full file search theme chooser ACP history browser history dictation history notes browse SDK reference script template catalog A large dataset does not justify Full width. A right-side preview/detail/content pane is what justifies Full width. File Search exception File search intentionally has two SurfaceKinds: FileSearchMini FileSearchFull Both map to AppView::FileSearchView but differ by FileSearchPresentation. They share the automation semantic surface tag fileSearch, while the surface contract distinguishes compact/no-preview behavior from split-preview behavior. Sizing evidence in calculate_window_size_params currently maps FileSearchView to ViewType::ScriptList; the Mini-vs-Full distinction depends on the route/presentation contract and omitted render/route details should be treated carefully. Surface identity AppView carries runtime payloads, focus handles, child entities, and per-route state. It is not the ideal key for agent reasoning because its variants may carry data and some grouped behaviors are shared. SurfaceKind is the payload-free identity layer. AppView::surface_kind() maps every AppView variant to a stable SurfaceKind. The match is exhaustive and intentionally has no wildcard arm. SurfaceKind::surface_contract() maps each surface kind to a LauncherSurfaceContract. This match is also exhaustive and intentionally has no wildcard arm. AppView::surface_contract() is a thin delegate through self.surface_kind().surface_contract(). Agents should key behavior expectations on SurfaceKind, not on render file names or partial enum payload guesses. LauncherSurfaceContract vocabulary Every surface contract carries behavior vocabulary and policy fields: family inputOwnership previewRole focusPolicy keyboardPolicy actionsPolicy proofPolicy visualPolicy dismissPolicy automationSemanticSurface per-active-view native footer surface in runtime/generated snapshots The vocabulary is behavior-oriented. A surface should be described as “filterable launcher list with launcher-owned input and no persistent preview” or “child-view content pane,” not as “whatever this render file seems to do.” Dismiss policy Dismiss behavior is declared in SurfaceKind::surface_contract() through DismissPolicy. It pairs triggers such as WindowBlur, BackdropClick, Escape, and CmdW with effects such as Ignore, CloseMainWindow, and LetViewHandle. Standard launcher surfaces close on normal launcher dismissal paths. Explicit surfaces are generally Cmd-W-oriented or view-handled. Runtime coexistence guards such as pinned state, focus grace, confirm popup, actions popup, shortcut recorder popup, detached ACP, dictation, and Tab AI remain separate runtime checks and should not be collapsed into per-view surface policy. Native footer identity AppView::native_footer_surface() owns the exact native footer surface id for the active view. This remains AppView-specific rather than SurfaceKind-specific because grouped surface kinds can still need distinct footer IDs. The native footer surface is exposed in: generated surface-contracts.json runtime stateResult.surfaceContract.nativeFooterSurface activeFooter getElements footer rows and buttons The footer host is not interchangeable with detached popups. NATIVE_MAIN_WINDOW_FOOTER_HEIGHT is the shared height contract for the AppKit footer host, GPUI spacer, launcher hover blocker, and footer-safe list reveal math. Generated artifacts docs/ai/contracts/surface-contracts.json is generated from the Rust registry, not hand-authored. It lists every SurfaceKind, mapped AppView variants, per-variant native footer ids, vocabulary tuple, policies, dismiss effects, and automation semantic surface. The current-view transition inventory is also generated. It exposes current_view mutation sites and helper contracts such as main automation re-keying, focus ownership, resize ownership, actions cleanup, embedded AI upsert, and state snapshot proof. It is not a public runtime receipt; runtime proof still comes from getState.surfaceContract and activePopupContract. Automation semantic surface re-keying The main automation window’s semanticSurface changes as the main view changes: default is scriptList triggerBuiltin can flip it to clipboardHistory, appLauncher, browserTabs, emojiPicker, windowSwitcher, processManager, currentAppCommands, designGallery, settings, kitStoreBrowse, kitStoreInstalled, fileSearch, acpChat, and related tags The mapping lives inside the surface contract registry via automation_semantic_surface. semantic_surface_for_main_view() delegates to that registry. rekey_main_automation_surface_from_current_view() is the owner for main-window re-keying after current_view has already changed. rekey_main_automation_surface_after_trigger_builtin_dispatch() is the triggerBuiltin post-dispatch owner and delegates to that helper. Resize/focus/bounds sync must preserve the current semantic surface. It must not silently hardcode scriptList after a subview transition. Entry Points Main menu / launcher entry The main menu can enter AppView::ScriptList in Mini or Full mode. show_script_list_with_main_filter_focus() restores AppView::ScriptList, sets main filter focus, and re-keys the main automation surface from the restored view. The caller owns filter text, list cache, sizing, and notification. open_mini_main_window clears computed filter text, schedules filter sync, sets the main placeholder, restores ScriptList with main filter focus, switches to MainWindowMode::Mini, clears hover, selects the first selectable row, invalidates grouped cache, computes grouped item count, resizes to ViewType::MiniMainWindow, and notifies. Built-in utility command entry Utility commands can open Mini main window, MiniPrompt, or other built-ins. The Mini main-window arm is source-audited to use the shared helper and to resize through ViewType::MiniMainWindow. The select microphone built-in creates AppView::MiniPrompt and resizes through ViewType::MiniPrompt, capped at five choices. Built-in filterable view entry open_builtin_filterable_view and open_builtin_filterable_view_with_filter set current_view, clear hover, mark opened_from_main_menu, set Mini/Full mode from expanded, synchronously resize to MiniMainWindow or ScriptList, set pending focus to main filter, set focused input to MainFilter, and notify. These helpers are the intended entry point for filterable built-ins. Callers should not independently call raw resize primitives unless the helper contract explicitly allows it. triggerBuiltin stdin entry triggerBuiltin dispatch routes commands into built-in surfaces. Filterable triggerBuiltin routes enter through show_filterable_view() and run_filterable_route_state_machine(), which prepares a route plan, applies one current_view = plan.next_view assignment, derives SurfaceKind, and optionally requests deferred resize. After dispatch mutates current_view, stdin dispatchers call the post-dispatch re-key helper. They must not upsert the whole window or recompute bounds/focus/title because they do not own those fields. The current-app commands triggerBuiltin path explicitly calls open_current_app_commands_from_tray(cx) and then update_window_size_deferred(window, cx). The deferred sizing arm must resolve CurrentAppCommandsView to ViewType::MiniMainWindow, not ScriptList. Tray/current-app commands Current-app commands can be opened from tray/current-app entry paths and through triggerBuiltin. They are a Mini single-column filterable surface. The regression contract specifically pins that the follow-up deferred resize path remains Mini. Special-character handoffs Special-character handoffs are entry paths where typed input or launcher state transitions can route into a prompt or filtered surface. The bundle does not include the broad render/input files that would fully enumerate these handoffs, so detailed per-character behavior should be treated as inference. The sizing rule still applies: after the handoff mutates current_view, calculate_window_size_params() must resolve the new route to the correct ViewType, and async/deferred paths must re-read current view before resizing. Prompt creation Prompt routes map to prompt-specific sizing: ArgPrompt uses ArgPromptNoChoices if no choices exist, otherwise ArgPromptWithChoices. MiniPrompt uses ViewType::MiniPrompt. MicroPrompt uses ArgPromptNoChoices. DivPrompt, FormPrompt, PathPrompt, EnvPrompt, DropPrompt, TemplatePrompt, and confirm prompt use full prompt sizing. ChatPrompt resolves through compact_ai_view_type_for_mode(main_window_mode): Mini becomes MiniAiChat, Full becomes DivPrompt. AcpChatView uses the same Mini vs Full AI sizing split. EditorPrompt and TermPrompt use child content sizing. Filter changes Filter changes can change visible row count, empty states, and selected rows, but they should not arbitrarily change Mini/Full width. For Mini main window, content-aware grouped sizing logs receipts and keeps fixed Mini height. For built-in Mini surfaces, calculate_window_size_params() must continue resolving to MiniMainWindow during follow-up deferred resize. Hide/reset/go-back Hide and reset paths snapshot or preserve Mini state so hidden-window bounds do not leak wide/full dimensions into automation receipts. The ordering matters: windowVisible:false should be established before reset or prompt cancellation can make ScriptList current, and reset/hide should re-key automation to scriptList after resetting current_view. close_and_reset_window restores MainWindowMode::Full, closes child windows and popups, and resets main state. Go-back paths clear actions popup state and then call update_window_size_deferred where appropriate. Escape and Cmd-W Escape, Cmd-W, blur, and backdrop dismissal are routed through surface dismiss policy plus runtime overlay guards. The surface contract declares what each top-level route is expected to do; runtime code keeps overlay-specific conditions separate. For state-first close paths such as Quick Terminal Cmd-W, reset ordering must avoid showing a full main-menu frame or recording wide hidden bounds. Protocol commands Protocol commands can open built-ins, simulate keys, query state, hide the main window, and list automation windows. Important protocol surfaces: triggerBuiltin opens route surfaces and then re-keys semantic surface from the now-current AppView. getState returns stateResult.surfaceContract for main-window targets. getState returns activePopupContract when an attached shared actions popup is active. listAutomationWindows exposes main-window semanticSurface, which must stay aligned with current route. ExternalCommand::Hide duplicates reset-then-rekey behavior in triple-embedded stdin dispatchers because the dispatcher body is inlined. simulateKey Cmd+K can toggle shared actions for registered actions hosts, preserving the host surface identity and exposing popup identity through activePopupContract. Sizing track A route entry chooses or restores an AppView. The entry path chooses MainWindowMode where relevant. The entry path either: calls resize_to_view_sync immediately, or marks resize as caller-owned / deferred. Follow-up changes call update_window_size_deferred or update_window_size. calculate_window_size_params() maps the current AppView plus main_window_mode to (ViewType, item_count). window_resize calculates height and optional width. Mini main window paths bypass the flat item-count path where possible to build MiniMainWindowSizing from grouped items and log structured Mini sizing receipts. The key state transitions are: ScriptList + Full → ViewType::ScriptList ScriptList + Mini → ViewType::MiniMainWindow MiniPrompt → ViewType::MiniPrompt ChatPrompt + Mini → ViewType::MiniAiChat ChatPrompt + Full → ViewType::DivPrompt AcpChatView + Mini → ViewType::MiniAiChat AcpChatView + Full → ViewType::DivPrompt CurrentAppCommandsView → ViewType::MiniMainWindow EmojiPickerView → ViewType::MiniMainWindow AppLauncherView → ViewType::MiniMainWindow WindowSwitcherView → ViewType::MiniMainWindow BrowserTabsView → ViewType::MiniMainWindow ProcessManagerView → ViewType::MiniMainWindow SettingsView → ViewType::MiniMainWindow BrowseKitsView → ViewType::MiniMainWindow InstalledKitsView → ViewType::MiniMainWindow SearchAiPresetsView → ViewType::MiniMainWindow FavoritesBrowseView → ViewType::MiniMainWindow ClipboardHistoryView → ViewType::ScriptList FileSearchView → ViewType::ScriptList in the included sizing source, while SurfaceKind splits Mini/Full by presentation ThemeChooserView → ViewType::ScriptList AcpHistoryView → ViewType::ScriptList BrowserHistoryView → ViewType::ScriptList DictationHistoryView → ViewType::ScriptList NotesBrowseView → ViewType::ScriptList SdkReferenceView → ViewType::ScriptList ScriptTemplateCatalogView → ViewType::ScriptList ScratchPadView → ViewType::EditorPrompt QuickTerminalView → ViewType::TermPrompt Surface identity track A route entry mutates current_view. AppView::surface_kind() derives the payload-free identity. SurfaceKind::surface_contract() derives vocabulary, behavior policy, dismiss policy, proof policy, visual policy, and automation semantic surface. 038 Main Window Sizing and Surface Contracts Executive Summary Main Window Sizing and Surface Contracts is the cross-cutting contract that keeps the Script Kit GPUI main window understandable, automatable, and regression-proof as it moves between launcher, mini launcher, prompt, built-in, preview, utility, and assistant surfaces. This feature has two tightly coupled halves: Main-window sizing decides whether the primary launcher window is in Mini or Full mode, which ViewType is used for width and height, and which resize path owns the transition. Surface identity decides what the active AppView means behaviorally through SurfaceKind, LauncherSurfaceContract, native footer ownership, automation semantic surface tags, and runtime getState.surfaceContract receipts. The core operational rule is simple but strict: surface layout determines size, and SurfaceKind determines behavior. A single-column command surface stays Mini. A surface with a real list-plus-preview/detail layout is Full. Agents should not infer width from command importance, row count, entry source, shortcut source, or render file name. The main window has several distinct presentation families: Full launcher / ScriptList: wide main menu with optional preview/info panel, represented by MainWindowMode::Full and ViewType::ScriptList. Mini main window: compact launcher surface, represented by MainWindowMode::Mini and ViewType::MiniMainWindow. MiniPrompt: compact script prompt route, represented by AppView::MiniPrompt and ViewType::MiniPrompt; this is not the same as Mini main window. Mini AI chat / inline assistant: compact assistant route, represented by ViewType::MiniAiChat when main_window_mode is Mini. Full prompt and content surfaces: DivPrompt, editor, terminal, form, path, drop, template, and other prompt/content views, with prompt-specific ViewTypes. Split preview/detail surfaces: wide surfaces whose layout has a list pane plus preview/detail pane, such as clipboard history, full file search, ACP history, browser history, dictation history, notes browse, SDK reference, and script template catalog. The proof model is state-first. Runtime verification should prefer getState, getElements, surfaceContract, activePopupContract, activeFooter, automation window metadata, generated contract checks, and source audits. Screenshots are useful for visual regressions, but they are not the primary proof for Mini/Full classification, semantic surface identity, footer ownership, or protocol behavior. What Users Can Do Users can open and move through the main launcher without thinking about the underlying contracts: Open the standard main menu in Full or Mini mode. Open Mini main window command surfaces from menu actions, utility commands, tray/current-app commands, or automation. Open single-column built-ins such as emoji picker, app launcher, window switcher, browser tabs, design gallery, favorites, process manager, settings, current-app commands, kit browse, installed kits, search AI presets, and mini dictation in compact Mini presentation. Open split preview/detail surfaces such as clipboard history, full file search, theme chooser, ACP history, browser history, dictation history, and notes browse in Full presentation. Open MiniPrompt without inheriting Full prompt width. Use inline Mini AI / mini-hosted ACP in compact mode and full chat/ACP in full prompt mode. Filter current surfaces without accidental width changes. Use Escape, Cmd-W, hide, reset, and go-back flows without leaking stale wide bounds into hidden-window receipts. Use shared actions popups while preserving the host surface identity. Trigger built-ins over stdin and see automation metadata re-key to the actual active surface. Inspect getState.surfaceContract to understand the active route’s behavior without reading render implementation files. Inspect activePopupContract when an attached actions dialog is open. Inspect activeFooter / footer elements to prove native footer ownership and avoid screenshot-only footer validation. For operators and agents, this means every major main-window route has two questions that must be answered together: What size should the main window be? What semantic surface is active for automation and behavior policy? Those answers are owned by different but cooperating files and tests. Core Concepts Main window vs secondary windows This feature is only about the primary main window and top-level AppView routes inside it. It does not own secondary-window sizing, detached popup sizing, detached ACP popup geometry, shortcut recorder popup geometry, or platform-level AppKit activation rules. Secondary target diagnostics intentionally omit surfaceContract because they are not main AppView receipts. MainWindowMode MainWindowMode is the app-level mode state for the main window: MainWindowMode::Mini is the default. MainWindowMode::Full restores the wide launcher/full prompt behavior. set_main_window_mode is the window-backed owner. It short-circuits unchanged modes, updates active ChatPrompt mini rendering, closes shared/detached actions popups, runs deferred sizing, syncs the native footer, and emits a main_window_mode_changed trace. set_main_window_mode_state_only updates mode without directly requiring a Window. It is used by open helpers that immediately call a synchronous resize primitive themselves. ViewType ViewType is the low-level sizing input consumed by window_resize: ScriptList: Full main launcher / split preview width. MiniMainWindow: compact main launcher / Mini built-in list width. MiniPrompt: compact prompt width. MiniAiChat: compact AI/ACP width. ArgPromptWithChoices: dynamic arg prompt choice list height. ArgPromptNoChoices: compact input-only prompt height. DivPrompt: full prompt/main-window standard height. EditorPrompt: full editor height. TermPrompt: full terminal height. Width is determined by width_for_view: MiniMainWindow, MiniPrompt, and MiniAiChat return MINI_MAIN_WINDOW_WIDTH. ScriptList returns FULL_MAIN_WINDOW_WIDTH. Other prompt/content view types preserve current width by returning None. Mini main window Mini main window is a compact main-window launcher mode. It is represented by MainWindowMode::Mini plus ViewType::MiniMainWindow. It is not the same as: MiniPrompt, which is a script prompt route. MicroPrompt, which resolves to compact no-choice prompt sizing. Inline Mini AI, which resolves through ViewType::MiniAiChat. Quick Terminal, which is a utility child content surface using terminal sizing. Detached popups or secondary windows. Mini main window uses fixed max height to prevent resize jank during filter changes and view transitions. Content-aware sizing still produces structured receipts, but height_for_mini_main_window returns MINI_MAIN_WINDOW_MAX_HEIGHT. Content-aware Mini sizing receipts Mini main-window sizing uses MiniMainWindowSizing: selectable_items visible_section_headers is_empty mini_main_window_sizing_from_grouped_items builds this from grouped launcher rows and treats section headers separately from selectable rows. This avoids conflating headers with selectable items. Even though the current height is fixed, every Mini resize path should log parseable sizing data through log_mini_window_sizing under the MINI_WINDOW target. The receipt records the resize reason, sizing fields, and target height. This matters because source and runtime proofs can verify that Mini behavior stayed Mini even when visual height does not change. Synchronous vs deferred resize There are two major resize paths: resize_to_view_sync(view_type, item_count) immediately calculates height/width and calls the raw resize primitive. defer_resize_to_view(view_type, item_count, window, cx) schedules resize at the end of the current GPUI effect cycle to avoid borrow conflicts. The app-level owners are: update_window_size_deferred(window, cx) for normal GPUI/window-context updates. update_window_size() for async handlers running outside render. calculate_window_size_params() for deriving the correct (ViewType, item_count) from the current AppView. calculate_window_size_params_if_current_view(reason, predicate) for async resize tasks that must re-read current_view and skip if the route changed. The critical invariant: a follow-up deferred resize must preserve the same Mini/Full classification chosen by the open helper. Built-in filterable sizing policy Built-in filterable views open through shared helpers that take an expanded: bool flag: expanded = true sets MainWindowMode::Full and resizes to ViewType::ScriptList. expanded = false sets MainWindowMode::Mini and resizes to ViewType::MiniMainWindow. Single-column filterable built-ins stay Mini. Split preview/detail built-ins stay Full. Examples of Mini single-column surfaces include: emoji picker app launcher window switcher browser tabs design gallery, where current source marks the contract as split-preview vocabulary but sizing evidence shows ViewType::MiniMainWindow; treat the exact render shape as partly inferred because broad render files were excluded favorites process manager settings current-app commands kit store browse installed kits search AI presets mini dictation Examples of Full split preview/detail surfaces include: clipboard history full file search theme chooser ACP history browser history dictation history notes browse SDK reference script template catalog A large dataset does not justify Full width. A right-side preview/detail/content pane is what justifies Full width. File Search exception File search intentionally has two SurfaceKinds: FileSearchMini FileSearchFull Both map to AppView::FileSearchView but differ by FileSearchPresentation. They share the automation semantic surface tag fileSearch, while the surface contract distinguishes compact/no-preview behavior from split-preview behavior. Sizing evidence in calculate_window_size_params currently maps FileSearchView to ViewType::ScriptList; the Mini-vs-Full distinction depends on the route/presentation contract and omitted render/route details should be treated carefully. Surface identity AppView carries runtime payloads, focus handles, child entities, and per-route state. It is not the ideal key for agent reasoning because its variants may carry data and some grouped behaviors are shared. SurfaceKind is the payload-free identity layer. AppView::surface_kind() maps every AppView variant to a stable SurfaceKind. The match is exhaustive and intentionally has no wildcard arm. SurfaceKind::surface_contract() maps each surface kind to a LauncherSurfaceContract. This match is also exhaustive and intentionally has no wildcard arm. AppView::surface_contract() is a thin delegate through self.surface_kind().surface_contract(). Agents should key behavior expectations on SurfaceKind, not on render file names or partial enum payload guesses. LauncherSurfaceContract vocabulary Every surface contract carries behavior vocabulary and policy fields: family inputOwnership previewRole focusPolicy keyboardPolicy actionsPolicy proofPolicy visualPolicy dismissPolicy automationSemanticSurface per-active-view native footer surface in runtime/generated snapshots The vocabulary is behavior-oriented. A surface should be described as “filterable launcher list with launcher-owned input and no persistent preview” or “child-view content pane,” not as “whatever this render file seems to do.” Dismiss policy Dismiss behavior is declared in SurfaceKind::surface_contract() through DismissPolicy. It pairs triggers such as WindowBlur, BackdropClick, Escape, and CmdW with effects such as Ignore, CloseMainWindow, and LetViewHandle. Standard launcher surfaces close on normal launcher dismissal paths. Explicit surfaces are generally Cmd-W-oriented or view-handled. Runtime coexistence guards such as pinned state, focus grace, confirm popup, actions popup, shortcut recorder popup, detached ACP, dictation, and Tab AI remain separate runtime checks and should not be collapsed into per-view surface policy. Native footer identity AppView::native_footer_surface() owns the exact native footer surface id for the active view. This remains AppView-specific rather than SurfaceKind-specific because grouped surface kinds can still need distinct footer IDs. The native footer surface is exposed in: generated surface-contracts.json runtime stateResult.surfaceContract.nativeFooterSurface activeFooter getElements footer rows and buttons The footer host is not interchangeable with detached popups. NATIVE_MAIN_WINDOW_FOOTER_HEIGHT is the shared height contract for the AppKit footer host, GPUI spacer, launcher hover blocker, and footer-safe list reveal math. Generated artifacts docs/ai/contracts/surface-contracts.json is generated from the Rust registry, not hand-authored. It lists every SurfaceKind, mapped AppView variants, per-variant native footer ids, vocabulary tuple, policies, dismiss effects, and automation semantic surface. The current-view transition inventory is also generated. It exposes current_view mutation sites and helper contracts such as main automation re-keying, focus ownership, resize ownership, actions cleanup, embedded AI upsert, and state snapshot proof. It is not a public runtime receipt; runtime proof still comes from getState.surfaceContract and activePopupContract. Automation semantic surface re-keying The main automation window’s semanticSurface changes as the main view changes: default is scriptList triggerBuiltin can flip it to clipboardHistory, appLauncher, browserTabs, emojiPicker, windowSwitcher, processManager, currentAppCommands, designGallery, settings, kitStoreBrowse, kitStoreInstalled, fileSearch, acpChat, and related tags The mapping lives inside the surface contract registry via automation_semantic_surface. semantic_surface_for_main_view() delegates to that registry. rekey_main_automation_surface_from_current_view() is the owner for main-window re-keying after current_view has already changed. rekey_main_automation_surface_after_trigger_builtin_dispatch() is the triggerBuiltin post-dispatch owner and delegates to that helper. Resize/focus/bounds sync must preserve the current semantic surface. It must not silently hardcode scriptList after a subview transition. Entry Points Main menu / launcher entry The main menu can enter AppView::ScriptList in Mini or Full mode. show_script_list_with_main_filter_focus() restores AppView::ScriptList, sets main filter focus, and re-keys the main automation surface from the restored view. The caller owns filter text, list cache, sizing, and notification. open_mini_main_window clears computed filter text, schedules filter sync, sets the main placeholder, restores ScriptList with main filter focus, switches to MainWindowMode::Mini, clears hover, selects the first selectable row, invalidates grouped cache, computes grouped item count, resizes to ViewType::MiniMainWindow, and notifies. Built-in utility command entry Utility commands can open Mini main window, MiniPrompt, or other built-ins. The Mini main-window arm is source-audited to use the shared helper and to resize through ViewType::MiniMainWindow. The select microphone built-in creates AppView::MiniPrompt and resizes through ViewType::MiniPrompt, capped at five choices. Built-in filterable view entry open_builtin_filterable_view and open_builtin_filterable_view_with_filter set current_view, clear hover, mark opened_from_main_menu, set Mini/Full mode from expanded, synchronously resize to MiniMainWindow or ScriptList, set pending focus to main filter, set focused input to MainFilter, and notify. These helpers are the intended entry point for filterable built-ins. Callers should not independently call raw resize primitives unless the helper contract explicitly allows it. triggerBuiltin stdin entry triggerBuiltin dispatch routes commands into built-in surfaces. Filterable triggerBuiltin routes enter through show_filterable_view() and run_filterable_route_state_machine(), which prepares a route plan, applies one current_view = plan.next_view assignment, derives SurfaceKind, and optionally requests deferred resize. After dispatch mutates current_view, stdin dispatchers call the post-dispatch re-key helper. They must not upsert the whole window or recompute bounds/focus/title because they do not own those fields. The current-app commands triggerBuiltin path explicitly calls open_current_app_commands_from_tray(cx) and then update_window_size_deferred(window, cx). The deferred sizing arm must resolve CurrentAppCommandsView to ViewType::MiniMainWindow, not ScriptList. Tray/current-app commands Current-app commands can be opened from tray/current-app entry paths and through triggerBuiltin. They are a Mini single-column filterable surface. The regression contract specifically pins that the follow-up deferred resize path remains Mini. Special-character handoffs Special-character handoffs are entry paths where typed input or launcher state transitions can route into a prompt or filtered surface. The bundle does not include the broad render/input files that would fully enumerate these handoffs, so detailed per-character behavior should be treated as inference. The sizing rule still applies: after the handoff mutates current_view, calculate_window_size_params() must resolve the new route to the correct ViewType, and async/deferred paths must re-read current view before resizing. Prompt creation Prompt routes map to prompt-specific sizing: ArgPrompt uses ArgPromptNoChoices if no choices exist, otherwise ArgPromptWithChoices. MiniPrompt uses ViewType::MiniPrompt. MicroPrompt uses ArgPromptNoChoices. DivPrompt, FormPrompt, PathPrompt, EnvPrompt, DropPrompt, TemplatePrompt, and confirm prompt use full prompt sizing. ChatPrompt resolves through compact_ai_view_type_for_mode(main_window_mode): Mini becomes MiniAiChat, Full becomes DivPrompt. AcpChatView uses the same Mini vs Full AI sizing split. EditorPrompt and TermPrompt use child content sizing. Filter changes Filter changes can change visible row count, empty states, and selected rows, but they should not arbitrarily change Mini/Full width. For Mini main window, content-aware grouped sizing logs receipts and keeps fixed Mini height. For built-in Mini surfaces, calculate_window_size_params() must continue resolving to MiniMainWindow during follow-up deferred resize. Hide/reset/go-back Hide and reset paths snapshot or preserve Mini state so hidden-window bounds do not leak wide/full dimensions into automation receipts. The ordering matters: windowVisible:false should be established before reset or prompt cancellation can make ScriptList current, and reset/hide should re-key automation to scriptList after resetting current_view. close_and_reset_window restores MainWindowMode::Full, closes child windows and popups, and resets main state. Go-back paths clear actions popup state and then call update_window_size_deferred where appropriate. Escape and Cmd-W Escape, Cmd-W, blur, and backdrop dismissal are routed through surface dismiss policy plus runtime overlay guards. The surface contract declares what each top-level route is expected to do; runtime code keeps overlay-specific conditions separate. For state-first close paths such as Quick Terminal Cmd-W, reset ordering must avoid showing a full main-menu frame or recording wide hidden bounds. Protocol commands Protocol commands can open built-ins, simulate keys, query state, hide the main window, and list automation windows. Important protocol surfaces: triggerBuiltin opens route surfaces and then re-keys semantic surface from the now-current AppView. getState returns stateResult.surfaceContract for main-window targets. getState returns activePopupContract when an attached shared actions popup is active. listAutomationWindows exposes main-window semanticSurface, which must stay aligned with current route. ExternalCommand::Hide duplicates reset-then-rekey behavior in triple-embedded stdin dispatchers because the dispatcher body is inlined. simulateKey Cmd+K can toggle shared actions for registered actions hosts, preserving the host surface identity and exposing popup identity through activePopupContract. Sizing track A route entry chooses or restores an AppView. The entry path chooses MainWindowMode where relevant. The entry path either: calls resize_to_view_sync immediately, or marks resize as caller-owned / deferred. Follow-up changes call update_window_size_deferred or update_window_size. calculate_window_size_params() maps the current AppView plus main_window_mode to (ViewType, item_count). window_resize calculates height and optional width. Mini main window paths bypass the flat item-count path where possible to build MiniMainWindowSizing from grouped items and log structured Mini sizing receipts. The key state transitions are: ScriptList + Full → ViewType::ScriptList ScriptList + Mini → ViewType::MiniMainWindow MiniPrompt → ViewType::MiniPrompt ChatPrompt + Mini → ViewType::MiniAiChat ChatPrompt + Full → ViewType::DivPrompt AcpChatView + Mini → ViewType::MiniAiChat AcpChatView + Full → ViewType::DivPrompt CurrentAppCommandsView → ViewType::MiniMainWindow EmojiPickerView → ViewType::MiniMainWindow AppLauncherView → ViewType::MiniMainWindow WindowSwitcherView → ViewType::MiniMainWindow BrowserTabsView → ViewType::MiniMainWindow ProcessManagerView → ViewType::MiniMainWindow SettingsView → ViewType::MiniMainWindow BrowseKitsView → ViewType::MiniMainWindow InstalledKitsView → ViewType::MiniMainWindow SearchAiPresetsView → ViewType::MiniMainWindow FavoritesBrowseView → ViewType::MiniMainWindow ClipboardHistoryView → ViewType::ScriptList FileSearchView → ViewType::ScriptList in the included sizing source, while SurfaceKind splits Mini/Full by presentation ThemeChooserView → ViewType::ScriptList AcpHistoryView → ViewType::ScriptList BrowserHistoryView → ViewType::ScriptList DictationHistoryView → ViewType::ScriptList NotesBrowseView → ViewType::ScriptList SdkReferenceView → ViewType::ScriptList ScriptTemplateCatalogView → ViewType::ScriptList ScratchPadView → ViewType::EditorPrompt QuickTerminalView → ViewType::TermPrompt Surface identity track A route entry mutates current_view. AppView::surface_kind() derives the payload-free identity. SurfaceKind::surface_contract() derives vocabulary, behavior policy, dismiss policy, proof policy, visual policy, and automation semantic surface. AppView::native_footer_surface() derives per-view native footer owner. Runtime getState exposes the contract through stateResult.surfaceContract. If attached actions are active, getState also exposes activePopupContract for ActionsDialog. Automation window metadata re-keys semanticSurface from the active AppView after route transitions.
