# wzrrd publish

Use `wzrrd` to publish sites to `https://<slug>.wzrrd.sh/`.

**Always build the site with SvelteKit + mdsvex** (see
https://svelte.dev/docs/cli/mdsvex) and publish the static `build/` output.
Never hand-write loose HTML files — author content as mdsvex pages (`.svx`)
with Svelte components for interactivity, then prerender to static files.

Public CLI and skill distribution repo: `wzrrd-sh/wzrrd-cli`.

## Install

If `wzrrd` is missing:

```bash
curl -fsSL https://wzrrd.sh/install.sh | bash
```

If `~/.local/bin` is not on PATH, add it or call `~/.local/bin/wzrrd` directly.

## Build the site (required pipeline)

Scaffold a SvelteKit project with mdsvex and the static adapter:

```bash
npx sv create site --template minimal --types ts --no-add-ons --install npm
cd site
npx sv add mdsvex sveltekit-adapter="adapter:static" --install npm
```

Then make the whole site prerender by creating `src/routes/+layout.ts`:

```ts
export const prerender = true;
export const trailingSlash = 'always';
```

`trailingSlash = 'always'` is required for multi-route sites. Without it,
adapter-static emits flat files (`about.html`), and wzrrd does NOT map
extensionless URLs to `.html` files — `/about` silently serves the root
`index.html` with a 200, so the bug looks like "routes don't work". With it,
each route becomes `about/index.html`, which wzrrd serves correctly at
`/about/` (and `/about`).

Author pages as `src/routes/<path>/+page.svx` (mdsvex lets you mix markdown
prose with imported Svelte components). Build and verify:

```bash
npm run build    # static output lands in build/
npx vite preview # optional local check
```

Preview-server gotcha: adapter-static deletes and recreates `build/` on every
build, so a running `vite preview` keeps a stale directory handle and 404s new
hashed assets (pages render unstyled). Restart the preview server after each
rebuild before judging visual output.

## Lean into Svelte for information architecture

The whole point of this pipeline is that Svelte excels at data presentation
and interactivity. Don't publish walls of static prose — design the
information architecture around exploration:

- **Structure data as data.** Put tables, lists, metrics, and timelines into
  typed objects/JSON and render them through components, not hardcoded markup.
- **Make density navigable.** Use filterable/sortable tables, client-side
  search, tabs, and expandable `<details>`-style sections so large content
  stays scannable.
- **Progressive disclosure.** Lead with a summary view; let `$state`-driven
  toggles reveal detail (per-item drill-downs, diffs, raw logs) on demand.
- **Visualize comparisons.** Prefer charts, sparklines, bar comparisons, and
  highlighted deltas (SVG + Svelte transitions are cheap) over numeric prose.
- **Mix prose and components.** mdsvex pages should interleave markdown
  narrative with interactive components — e.g. a written finding followed by
  the live, sortable evidence table that backs it.
- **Multi-page when it helps.** Use SvelteKit routes for natural sections
  (overview / details / appendix) with a persistent nav in `+layout.svelte`.
  Write internal links with trailing slashes (`/details/`, not `/details`) to
  match the `trailingSlash = 'always'` output. If different routes need
  radically different global styling (e.g. design variants), use route groups
  like `(main)/` to scope layout CSS, and add `data-sveltekit-reload` to
  cross-variant links so one route's `:global` styles can't leak into another
  via client-side navigation.

Everything must still prerender to static files — keep interactivity
client-side (no server endpoints, no `+page.server.ts` with runtime needs).

## Publish

```bash
wzrrd publish --file ./build --slug demo
```

- `--file` should point at the SvelteKit `build/` directory (it contains
  `index.html` at the root).
- Anonymous publishes work without login.
- Anonymous publishes are ephemeral and expire after 24 hours.
- Anonymous responses include `claimUrl`; share/open it if the user wants the URL to become permanent.

## Permanent publishing

```bash
wzrrd login
wzrrd publish --file ./build --slug demo
```

`wzrrd login` runs a device approval flow and stores a scoped agent token at `~/.config/wzrrd/auth.json`.

## Safety defaults

- Published sites are `noindex` by default.
- Only add `--index` when the user explicitly wants search indexing.
- Keep builds lean. Current anonymous limits are aggressive: 5 MB total, 1 MB
  per file, 50 files, 10 anonymous publishes per IP/hour. A minimal SvelteKit
  static build fits, but avoid heavy assets/fonts; if you hit the file-count
  cap, log in or trim routes.

## Diagnostics

Before debugging, run:

```bash
wzrrd doctor && wzrrd auth-status
```

## Verify multi-route publishes

Status codes are not enough: wzrrd returns 200 with root-page fallback content
for unknown paths. Verify each route by content, e.g.:

```bash
for r in "" "about/" "details/"; do
  printf "/%s -> %s\n" "$r" \
    "$(curl -s https://<slug>.wzrrd.sh/$r | grep -o '<title>[^<]*</title>')"
done
```

Each route must show its own title, not the root page's.

Second risk: over-selling Agent Chat. The better angle is trust and proof: agents receive structured desktop context, act through known surfaces, and leave verifiable transactions. Avoid autonomy language unless paired with verification.

Third risk: memory taking over the story. Brain, Day Page, and sediment are retention features. They should make the product feel useful over time, but programmability remains the headline.

Fourth risk: a heavy marketing build. For wzrrd, keep this to one SvelteKit route, mdsvex content, lightweight Svelte components, CSS/SVG-only visuals, no stock imagery, no external fonts, and no server runtime.
